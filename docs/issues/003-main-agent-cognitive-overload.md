# Issue #003: 主 Agent 认知过载导致规则不执行

**日期**：2026-07-04
**严重度**：High（核心流程可靠性问题）
**发现场景**：peekview 项目执行 agate 任务时，主 Agent 跳过 P2-review 直接推进到 P4

## 问题描述

主 Agent 读 8 个协议文件 + 环境自检后，上下文窗口已满。到执行 P2 派发时，虽然协议明确写了"P2-review.md status: approved 是 P2→P3 转移条件"，但主 Agent 的注意力放在"产出文件字段够不够"，没放在"产出之后还缺什么前置条件"。

**不是"不知道规则"——是认知过载后规则不在主动思维里。**

这和 LIMITATIONS.md 局限 3 一致：主 Agent 判断力是单点故障。但局限 3 只说了"判断力"，没说根因是**认知过载**。

## 根因分析

主 Agent 在 agate 编排中的认知负担来源：

1. **协议文件量大**：8 个协议文件 + 角色文件，读入上下文后占用大量窗口
2. **规则分散**：转移条件在 state-machine.md，评审映射在 role-system.md，gate 在 check-*.sh，裁剪在 dispatch-protocol.md——判定一个"能不能推进"要跨文件交叉引用
3. **每步都有隐式前置**：不只"产出文件存在"，还有"P2-review approved"、"SCOPE+ resolved"、"retries 未超限"等，散在不同文件里
4. **无强制回查机制**：规则写了但执行靠主 Agent 自觉回查，过载时回查是最先被砍的行为

核心矛盾：**agate 的可靠性来自规则密度，但规则密度本身就是认知负担的来源。**

## 和已有机制的关系

| 已有机制 | 解决的问题 | 没解决的问题 |
|----------|-----------|-------------|
| `.state.yaml` + 产出文件 | 状态可重建 | 不防"跳步" |
| gate 脚本 | 检查产出文件质量 | 不检查前置条件（如 P2-review） |
| orchestrator-log.md | 防无响应 | 不防"规则不在主动思维里" |
| pre-commit hook | commit 时拦截 | 只拦截部分 gate，且 Agent 可以 `--no-verify` |
| LIMITATIONS.md 局限 3 | 承认问题存在 | 没有缓解措施 |

## 实证：T046 content-link-resolution（peekview）

T046 是本问题的完整实例（详见 `~/oclab/peekview/docs/reviews/t046-postmortem-20260705.md` 和 `t046-retrospective-20260705.md`）。任务走完 P0-P8 全流程、bump 版本、创建 tag，但核心功能（Markdown 图片渲染）未生效。

**认知过载在整个流程中的表现**：

| 阶段 | 应该做 | 实际做了 | 为什么 |
|------|--------|---------|--------|
| P2 派发 | 派 plan-design-review 评审 + 最小验证（/content 端点 Content-Type） | 未派评审，卡住无响应；最小验证规则存在但未触发 | 过载后忘记：评审映射在 role-system.md，最小验证规则在 dispatch-protocol.md，分散在不同文件 |
| P2 design | 验证 /content 端点返回正确 Content-Type | 列出了 URL 但从未验证 | "方案依赖的外部行为"这一判断是隐式的——规则存在但触发靠主 Agent 主动识别 |
| P4 实现 | 打开浏览器看图片是否渲染 | 只跑单元测试/vitest | 过载后取最近路径：P2 gate_commands.P5 只列了 pytest+vitest，没列端到端 |
| P6 验收 | 先验证功能（用户视角），再满足 gate 格式 | 花 2 小时凑格式，没花 5 分钟查 API 响应头 | **gate 是硬约束（不通过 = commit 不了），功能验证是软约束（不验证也能 commit）**——过载时优先满足硬约束 |
| 根因定位 | 收到 vision-helper 否定报告 → 检查网络请求 | 三次视觉否定被三次用程序化指标反驳 | 过载时用"指标好看"替代"功能正确"作为思维捷径 |

**T046 暴露的新维度**：

1. **gate 硬约束 vs 功能软约束的优先级倒挂**：不是"忘记了功能验证"——是 gate 格式不满足就 commit 不了，功能不验证也能 commit。过载时 Agent 优先处理会阻塞自己的事。
2. **隐式触发规则不可靠**：P2 最小验证规则写得很清楚（"方案依赖外部行为时必须做最小验证"），但"是否依赖外部行为"的判断是隐式的——Agent 必须主动识别。过载时这种主动识别是最先消失的。
3. **"指标正确 = 功能正确"是过载下的思维捷径**：naturalWidth>0、complete=true、API 返回 200——各个指标都为真，Agent 得出结论"功能正常"。这不是造假，是认知过载后用更简单的判定逻辑替代更复杂的验证路径。
4. **否定证据出现时的反驳模式**：不是"没看到证据"，是看到了但用更低认知成本的路径解释掉。"curl -I 检查响应头"比"证明图片确实能加载"更简单，但当 Agent 已经投入大量时间证明"能加载"时，切换路径的认知成本高于继续反驳。

## 可能方向

（待分析，不做决策）

1. **机械检查前置条件**：gate 脚本不只检查当前阶段产出，还检查转移的前置条件（如 P3 gate 检查 P2-review.md status: approved）
2. **阶段推进 checklist**：每步推进前强制执行固定 checklist，降低主动思维依赖
3. ~~**减少协议文件量**：合并/精简协议~~（已否：8 文件因职责不同，合并只会让单文件更长）
4. ~~**规则集中化**：转移表集中到单文件~~（已否：Agent 还是要读同一张表，注意力不够的问题不变）
5. **subagent 辅助判断**：派专门的 subagent 做"能不能推进"的判定，主 Agent 只执行
6. **P6 gate 增加用户视角验证**：格式合规但无用户视角验证 → gate 不通过（T046 教训 G1，来源详见 t046-retrospective §5.1）
7. **P2 最小验证触发显式化**：主 Agent 必须显式判断"方案是否依赖外部行为"并记录，不能靠隐式识别（T046 教训 G2）
8. **gate_commands.P5 增加 UI 端到端要求**：ui_affected: true 时必须含至少一条端到端命令（T046 教训 G3）
9. **否定证据处理规则**：vision-helper 报异常 → 必须先追查再反驳，反驳必须附已追查的证据（T046 教训 G5）
