#### P1-2: P1 NEED_CONFIRM gate 检查

当前：check-gate.sh P1 分支无 NEED_CONFIRM 检查。

目标：在 P1 分支 BDD 锚点检查（~44 行）之后、终止 `echo+exit 2`（~45 行）之前插入（不是"末尾"——末尾是 `exit 2 ;;` 死代码位置）：
```bash
# P1 NEED_CONFIRM 检查（与 P6 对称）
P1_FILE="$TASK_DIR/P1-requirements.md"
NC=$(grep -cE '\[NEED_CONFIRM\]' "$P1_FILE" 2>/dev/null || echo 0)
NC=$(echo "$NC" | tail -1)
if [ "$NC" -ne 0 ]; then
    echo "GATE P1: $NC 个未解决的 NEED_CONFIRM 项" >&2
    exit 1
fi
```

注意：
- `$P1_FILE` 在 P1 分支内未定义（只在 P2 分支第 54 行赋值），必须显式定义，否则 `set -u` 崩溃
- 此检查抓"遗留未解决标记"，**不抓"主 Agent 抹掉标记后通过"**（那是 observability/audit 范畴，见 P3-2）
- 二行式是必须的——`grep -c` 无匹配时打印 "0" 并 exit 1，`|| echo 0` 再加一行 "0"，`tail -1` 归一为单行
- 语义：approved 的 P1 若仍有遗留 NEED_CONFIRM，应 exit 1 拦下，不是走到 exit 2 交主 Agent

#### P1-5: check-tdd-red.sh 加 TEST_RUNNER_FLAGS + 汇总正则可配

当前（check-tdd-red.sh ~第 51 行）：
```bash
RESULT=$($RUNNER -q 2>&1) && EXIT=0 || EXIT=$?
```

目标（标志可配）：
```bash
RUNNER_FLAGS="${TEST_RUNNER_FLAGS:--q}"
RESULT=$($RUNNER $RUNNER_FLAGS 2>&1) && EXIT=0 || EXIT=$?
```

注意：`$RUNNER_FLAGS` 不加引号——允许传多 flag（如 `--silent --no-coverage`）。默认 `-q` 保持向后兼容。

当前（汇总正则，~第 53-54 行）：
```bash
FAILED=$(echo "$RESULT" | grep -oE '[0-9]+ failed' | grep -oE '[0-9]+' || true)
ERRORS=$(echo "$RESULT" | grep -oE '[0-9]+ error' | grep -oE '[0-9]+' || true)
```

目标（汇总正则可配）：
```bash
FAIL_PATTERN="${TEST_FAIL_PATTERN:-[0-9]+ failed}"
ERROR_PATTERN="${TEST_ERROR_PATTERN:-[0-9]+ error}"
FAILED=$(echo "$RESULT" | grep -oE "$FAIL_PATTERN" | grep -oE '[0-9]+' || true)
ERRORS=$(echo "$RESULT" | grep -oE "$ERROR_PATTERN" | grep -oE '[0-9]+' || true)
```

这样 go test（`FAIL\t1`）、cargo test（`test result: FAILED. 1 passed; 1 failed`）、jest（`Tests: 1 failed`）都可通过环境变量适配，无需改脚本。

**关于"适配器输出契约"**：专家评审 §4.2 另提"正式化适配器输出契约"作为 P3 修法的备选。本方案选择可配正则而非适配器契约，理由：① 正则方案改动最小（~5 行 vs 契约需新文件+新约定+文档）；② 正则方案向后兼容（默认值 = 当前 pytest 行为）；③ 契约方案需定义输出格式规范、适配器发现机制、错误处理——是结构性改动，应归 P3 设计讨论。如果正则方案在实践中不够用，再升级为契约。

**关于专家评审 B3（默认值引号）**：原否认有误——测试用的是无外层双引号形式，而实际代码 `FAIL_PATTERN="${...:-'[0-9]+ failed'}"` 整个在双引号内，内层单引号是字面量，B3 成立。但最终代码已去引号（`"${TEST_FAIL_PATTERN:-[0-9]+ failed}"`），功能正确，影响归零。教训：测试要测被测对象的确切形式（带不带外层双引号是两回事）。

#### P1-9: P4 gate 排除收窄 — 仅排流程产物

> **新增**（采纳第 6 轮专家评审 §3）：当前 P4 gate（check-gate.sh:107）排除 `\.(md|yaml)$`，把配置类 .yaml 交付（CI/k8s/compose manifest）也排除在外，导致纯配置任务走 P4 被 exit 1 误拦。收窄为仅排除 agate 流程产物。

当前（check-gate.sh:107）：
```bash
git diff --cached --name-only | grep -qvE '\.(md|yaml)$|^\.state' && exit 0 || exit 1
```

目标：
```bash
git diff --cached --name-only | grep -qvE '(^|/)P[0-8]-.*\.md$|(^|/)\.state\.yaml$' && exit 0 || exit 1
```

变化：从排除所有 `.md`/`.yaml` → 只排除 agate 流程产物。注意：
- `(^|/)P[0-8]-.*\.md$` 而非 `^P[0-8]-.*\.md$`——`git diff --cached --name-only` 输出相对路径，如 `docs/tasks/T001/P2-review.md`，`^` 锚点会匹配失败
- `(^|/)\.state\.yaml$` 同理——避免误匹配项目源码中的 `xxx.state.yaml`

这样：
- 配置类 .yaml（`ci.yml`、`k8s-deploy.yaml`、`docker-compose.yml`）→ 不被排除 → 被识别为实现产出 → P4 gate 通过 ✅
- 非 agate 流程 .md（`README.md`、`CONTRIBUTING.md`）→ 不被排除 → 被识别为实现产出 → P4 gate 通过 ✅
- agate 流程产物（`docs/tasks/T001/P2-review.md`、`docs/tasks/T001/.state.yaml`）→ 被排除 → 不算实现产出 → 正确
