## 一、概念分层：spec / 方案设计 / 实现导航 / 代码实现

### 问题

当前 agate 的概念边界模糊——P2 混了方案设计和实现导航，"实现计划"层缺位但不该新增阶段。

### 设计决策

**四层分离，不新增阶段——在现有 P0-P4 内澄清边界**：

```
P0-brief.md          = spec 层（PM 视角：做什么 + 风险 + 约束）
P1-requirements.md   = spec 层（工程视角：BDD 验收条件 + 范围声明）
P2-design.md §1-3,5  = 方案设计层（怎么做 + 候选方案权衡 + 影响域 + gate 固化）
P2-design.md §4      = 实现导航（files_to_read——P2 architect 产出的上下文地图）
P4 implementer       = 代码实现层（拿 P2 设计 + P3 测试自主实现）
```

**关键澄清**：

1. **P0 和 P1 都是 spec 层，但视角不同**：P0 是 PM 视角，P1 是工程视角。不合并——视角分离是设计选择。

2. **P2 是方案设计层，不是实现计划层**：P2-design.md 的 `files_to_read` 是"实现导航"——资源地图（列实现时需要读哪些文件 + 为什么），不是步骤计划（不列每步做什么）。

3. **不新增"实现计划"阶段**：P4 implementer 拿 P2 设计 + P3 测试自主实现。如果 P2 设计够明确 + 实现导航够完整，implementer 不需要步骤计划。如果 P2 不够明确——那是 P2 的问题，不是"缺一个实现计划阶段"的问题。

4. **P2 歧义风险兜底**（评审补充）："P2 设计够明确"是一个需要验证的上游断言——agate 在其他环节不信任上游（P6 不信任 P5、provenance 不信任 agent 自报），但在 P2→P4 这一环目前直接信任了 P2。如果 P2 设计有隐含歧义（architect 以为讲清楚了，implementer 理解偏了），没有机制会发现。**轻量兜底**：P4 implementer 在产出里附"实现偏差声明"——如果实现时发现 P2 设计有歧义/缺口而自主做了决策，必须显式标注 `[DESIGN_GAP: xxx]`，供 P7 一致性检查捕获。这不引入步骤脚本，但比"完全信任 P2 明确"多一道网。

5. **复杂任务的可选实现顺序**：如果任务确实复杂（多文件、有依赖顺序），P2-design.md 可含一个"实现顺序"节，但不是强制——简单任务不需要。

|------|------|
| `agate/assets/templates/task-files.md` | P2-design.md §4 "涉及文件清单"改名为"实现导航"并澄清语义 |

| `agate/assets/execution-roles/architect.md` | P7 一致性检查加 `[DESIGN_GAP]` 捕获 |

## 二、P2 不可裁剪（含例外口）

### 问题

当前协议允许 P1 裁剪 P2（条件：`risk_level=low`）。但 P1 analyst 做需求分析不做方案设计——它没有做方案设计就无法预知 P2 architect 会发现哪些 P0/P1 没想到的问题。

T045 是实证：P0/P1 觉得"CSS 改个 display 属性很明确"，但 P2 如果做了就会发现 `overflow:auto + display:block` 的交互问题。

### 设计决策

**P2 不可裁剪，含两个例外口**——镜像 P6 模式（P6 有 `no_behavior_change: true` 例外）：

```
P2 不可裁，除非满足以下任一例外：
  design_trivial: true        — 设计无关紧要的改动
  follows_existing_pattern: true — 照搬已有模式，无新设计决策
```

**例外口 1：`design_trivial: true`**（纯 typo/文案/配置值）
限定条件（全部满足）：
- 单文件改动
- 无逻辑变更（纯 typo / 纯文案 / 纯配置值修改）
- ≤ 5 行改动
- 主 Agent 确认（P1 需声明 `design_trivial: true` + 理由）

**例外口 2：`follows_existing_pattern: true`**（机械但非 typo）
限定条件（全部满足）：
- 照搬项目已有同类实现（如已有 5 个 CRUD 端点，加第 6 个）
- 无新设计决策（方案就是"照抄 X"）
- 附参照文件路径（P1 声明 `follows_existing_pattern: [参照文件]`）
- 主 Agent 确认
- **跳过多方案探索**（照抄任务不需要 2 个候选方案）

**为什么需要两个例外口**：T045 的教训是"看起来简单但有隐藏耦合的任务不该裁 P2"，不是"所有非 typo 任务都必须做完整方案设计"。给已有函数加参数、按已有模式加 CRUD 端点、重命名变量——这些任务方案设计价值接近零，但不符合 design_trivial（多文件 / > 5 行 / 有逻辑变更）。用 follows_existing_pattern 覆盖这一大类，避免"为一个照抄任务硬写 2 个假候选方案"的形式主义。

### 影响评估

| 维度 | 影响 |
|------|------|
| 简单任务 | typo fix 走 `design_trivial: true` 例外口跳 P2，不增加开销 |
| 中等任务 | P2 一遍过（architect 写设计 + 评审通过），如果 P0/P1 真想清楚了开销不大 |
| 现有任务兼容 | 已声明裁剪 P2 的 P1-requirements.md 下次 commit 会被 hook 拦截——过渡期方案：现有裁剪 P2 的任务用独立字段 `legacy_p2_pruned: true` 标记（不复用 `override:` 字段——override 语义是"裁剪声明与执行不一致时的例外说明"，v0.6 迁移不是执行不一致，复用会导致审计时两种含义混淆）|
| 裁剪哲学 | 从"P1 可裁 P2（risk_level=low）"改为"P2 不可裁（含 design_trivial 例外）"——与 P6 同级（含例外口）|

### 改动范围

| 文件 | 改动 |
|------|------|
| `agate/state-machine.md` | 裁剪条件表：P2 从"需 risk_level=low"改为"不可裁（除非 design_trivial: true）" |
| `agate/scripts/check-pruning.sh` | 检查 2：P2 裁剪 → 检查 `design_trivial: true` OR `follows_existing_pattern: true`（含参照文件路径存在性）；检查 7 保持不变（合法裁剪 P2 时仍要求"跳过风险"评估——不是死代码）|
| `agate/WORKFLOW.md` | P2 行说明不可裁 + 例外口 |
| `agate/dispatch-protocol.md` | 门槛表同步 |
| `agate/orchestrator-template.md` | 裁剪说明同步 |

### 下游同步（评审补充）

| 项 | 改动 |
|----|------|
| **P0-brief pruning_tendency** | "激进 — 单文件 typo 修复，直接做"改为"激进 — 单文件 typo 修复，走 design_trivial 例外" |
| **risk_level 字段作用域** | P2 不可裁后 risk_level 只 gate P3 裁剪，文档同步说明 |

## 三、brainstorming 多方案探索（绑例外口）

### 缺口

P2 architect 当前只要求产出一个方案。强制多方案探索是发现"P1 没想到的问题"的有效手段。

### 设计决策

**P2 强制 2-3 个候选方案 + 权衡 + 选择理由——但绑 design_trivial 例外口**：

- `design_trivial: true` 时跳过多方案（typo fix 不需要 2 个方案）
- 其余强制 2-3 个候选方案

不照搬 superpowers 的"人-Agent 逐问对话"——agate 是 subagent 隔离产出，architect 在自己的上下文里独立探索。

### 防线强度标注（评审补充）

**多方案探索是 nudge，不是 barrier**——和 risk_level 声明同级。architect 在隔离上下文里独立产出多方案，没有外部压力保证方案是"真的探索"而非"为满足数量凑的陪衬"。一个赶进度的 architect 可以写一个真方案 + 两个稻草人，权衡表里把稻草人缺点写得很明显，选择理由"自洽"——这能通过 plan-eng-review 的"探索了替代方案 + 理由自洽"检查。

这和 T045 的空 png 充数、纯断言后门是同一类问题：数量/形式要求在自报产出下会被最小成本地满足。方案质量本质上不可机器判定，也不可被隔离 reviewer 独立重推。

**诚实标注**：多方案探索的价值是"强制 architect 至少走一遍'还有别的做法吗'的思考"（nudge），不是"保证方案是最优的"（barrier）。不要高估防护强度——否则会重蹈"以为加了检查就安全了"的覆辙。

### 改动范围

| 文件 | 改动 |
|------|------|
| `agate/assets/templates/task-files.md` | P2-design.md §1 改为"候选方案（至少 2 个 + 权衡 + 选择理由）"——design_trivial 时可只写 1 个 |
| `agate/assets/execution-roles/architect.md` | 加"至少提 2 个候选方案"要求 |
| `agate/assets/review-roles/plan-eng-review.md` | 加"审查是否探索了替代方案 + 选择理由自洽 + 实现就绪度" |

## 四、扩写 investigate.md（superpowers 借鉴）

### 缺口

agate 的 `investigate.md` 评审角色只有 32 行，缺多组件取证插桩、root-cause-tracing 回溯技术。

### 借鉴内容

- 四阶段结构化调试（根因调查 → 模式分析 → 假设测试 → 实现）
- 多组件边界插桩取证（在组件边界 log 输入/输出，定位问题传播路径）
- root-cause-tracing 回溯（从症状沿调用链回溯到根因）

### 关键适配（评审补充）

**"3 次修复失败 → 质疑架构假设"不能靠 subagent 自己生效**——失败计数在 `.state.yaml` 的 `retries[Pn]`，不在 subagent 上下文里。subagent 不知道自己是第几次被派发。

这条规则必须落在**主 Agent 派发逻辑**：
- `dispatch-protocol.md` 重派 prompt 模板加：`len(retries[Pn]) >= 3` 时注入"你已是第 N 次尝试，质疑架构假设而非继续同层试错"
- investigate.md 角色文件只写"若被告知是高次重试，优先质疑架构"

### 改动范围

| 文件 | 改动 |
|------|------|
| `agate/assets/review-roles/investigate.md` | 从 32 行扩到 ~100 行（四阶段 + 取证 + 回溯）|
| `agate/dispatch-protocol.md` | 重派 prompt 模板加"3 次失败质疑架构"注入逻辑 |

## 六、TDD 反合理化话术（评审否决，砍掉）

### 评审否决理由

agate 靠机器 gate 不信任 Agent 自觉（LIMITATIONS.md 局限 3），却借鉴"话术约束自觉"机制——哲学自相矛盾。

- 话术映射到 gate 的是冗余（"删了就是删了"已有 check-tdd-red.sh）
- 话术映射不到的是无效（"太简单不用测"无脚本能查，话术不会可靠改变行为）

### 替代方案（如需补强 TDD）

改为机器 gate：例如"每个改动的源文件须有对应测试文件变动"（coverage-delta gate）——比话术强得多。但这需要项目侧测试框架支持，标记为 Phase 2 候选，不进 v0.6。

**v0.6 不做此项。**

### hook 改动明细

#### check-pruning.sh 检查 2（P2 不可裁 + 例外口）

当前逻辑：
```bash
if ! echo "$PHASES_DECLARED" | grep -qw 'P2'; then
    if [ "$RISK_LEVEL" != "low" ]; then
        ERRORS="${ERRORS}裁剪 P2 需 risk_level=low，实际=${RISK_LEVEL}\n"
    fi
fi
```

改为：
```bash
# 检查 2：P2 不可裁剪（除非 design_trivial 或 follows_existing_pattern）
if ! echo "$PHASES_DECLARED" | grep -qw 'P2'; then
    # 过渡期：legacy_p2_pruned 字段放行
    if grep -qE '^legacy_p2_pruned:\s*true' "$P1_FILE" 2>/dev/null; then
        : # 放行，不报错
    # 例外口 1：design_trivial
    elif grep -qE '^design_trivial:\s*true' "$P1_FILE" 2>/dev/null; then
        : # 放行
    # 例外口 2：follows_existing_pattern（须含参照文件路径）
    elif grep -qE '^follows_existing_pattern:\s*\[.+\]' "$P1_FILE" 2>/dev/null; then
        : # 放行
    else
        ERRORS="${ERRORS}P2 不可裁剪（除非 design_trivial: true 或 follows_existing_pattern: [参照文件]）\n"
    fi
fi
```

#### check-gate.sh P2（多方案探索 nudge）

**⚠️ 现状**：P2 分支当前只有两行——`echo "..." >&2; exit 2`。"追加"在 `exit 2` 后面的代码不可达，必须重写整个 P2 分支。

**⚠️ 变量定义**：`check-gate.sh` 当前只接收 `PHASE` 和 `TASK_DIR`，没有 `P2_FILE`。重写时需在 P2 分支内定义 `P2_FILE="$TASK_DIR/P2-design.md"` 和 `P1_FILE="$TASK_DIR/P1-requirements.md"`。

**⚠️ 跳过分支不可达**：`design_trivial` / `follows_existing_pattern` 声明时 P2 阶段本身被裁剪，`check-gate.sh P2` 不会被调用——不需要在 gate 内部判例外口。多方案检查只在 P2 实际执行时触发。

重写后的 P2 分支（替换现有的 `echo + exit 2`）：

```bash
  P2)
      P2_FILE="$TASK_DIR/P2-design.md"
      # 多方案探索检查（v0.6：nudge 强度）
      # 只在 P2 实际执行时触发（design_trivial/follows_existing_pattern 时 P2 被裁剪，不会走到这里）
      if [ -f "$P2_FILE" ]; then
          CANDIDATE_COUNT=$(grep -cE '^###?\s*候选方案|^###?\s*方案[ABC123]' "$P2_FILE" 2>/dev/null || echo 0)
          CANDIDATE_COUNT=$(echo "$CANDIDATE_COUNT" | tail -1)
          if [ "$CANDIDATE_COUNT" -lt 2 ]; then
              echo "GATE P2: P2-design.md 需至少 2 个候选方案 + 权衡 + 选择理由" >&2
              exit 1
          fi
      fi
      echo "GATE P2: 需从 P2-design.md gate_commands 动态读取，主 Agent 自行判定" >&2
```

#### dispatch-protocol.md 重派 prompt 模板（最后一次重试质疑架构）

重派 prompt 模板加条件注入（伪代码，实际在 dispatch-protocol.md 文档层描述）：

**⚠️ 关键：`retries[Pn] >= 3` 在现有重试上限下永不可达**——`dispatch-protocol.md:647` 重派判定是 `if len(retries[Pn]) < MAX_RETRY`，`>= MAX_RETRY` 时不再生成重派 prompt 而是直接上溯。必须用 `>= MAX_RETRY(Pn) - 1`（即本次已是最后一次允许的重派）：

```
若 len(retries[Pn]) >= MAX_RETRY(Pn) - 1（即本次已是最后一次允许的重派）:
  在重派 prompt 追加：
  "⚠️ 你已是第 {N} 次尝试此阶段（最后一次机会）。前 {N-1} 次尝试均失败。
   请优先质疑架构假设而非继续在同一层面试错。

   若确认架构假设有误，标 [SCOPE+] 触发回 P2 重新设计。"
```

P1/P2/P4（MAX_RETRY=3）在第 3 次派发前提示一次；P3/P5/P6/P7/P8（MAX_RETRY=2）在第 2 次派发前提示一次——每个阶段有且仅有一次触发机会。
