## 一、概念分层：spec / 方案设计 / 实现导航 / 代码实现

### 问题

当前 agate 的概念边界模糊——P2 混了方案设计和实现导航，"实现计划"层缺位但不该新增阶段。

### 设计决策

**四层分离，不新增阶段——在现有 P0-P4 内澄清边界**：

```
P0-brief.md          = spec 层（PM 视角：做什么 + 风险 + 约束）
P1-requirements.md   = spec 层（工程视角：BDD 验收条件 + 范围声明）
P2-design.md §1-3,5  = 方案设计层（怎么做 + 候选方案权衡 + 影响域 + gate 固化）
P2-design.md §4      = 实现导航（files_to_read——P2 architect 产出的上下文地图）
P4 implementer       = 代码实现层（拿 P2 设计 + P3 测试自主实现）
```

**关键澄清**：

1. **P0 和 P1 都是 spec 层，但视角不同**：P0 是 PM 视角，P1 是工程视角。不合并——视角分离是设计选择。

2. **P2 是方案设计层，不是实现计划层**：P2-design.md 的 `files_to_read` 是"实现导航"——资源地图（列实现时需要读哪些文件 + 为什么），不是步骤计划（不列每步做什么）。

3. **不新增"实现计划"阶段**：P4 implementer 拿 P2 设计 + P3 测试自主实现。如果 P2 设计够明确 + 实现导航够完整，implementer 不需要步骤计划。如果 P2 不够明确——那是 P2 的问题，不是"缺一个实现计划阶段"的问题。

4. **复杂任务的可选实现顺序**：如果任务确实复杂（多文件、有依赖顺序），P2-design.md 可含一个"实现顺序"节，但不是强制——简单任务不需要。

### 改动范围

| 文件 | 改动 |
|------|------|
| `agate/assets/templates/task-files.md` | P2-design.md §4 "涉及文件清单"改名为"实现导航"并澄清语义 |
| `agate/assets/execution-roles/architect.md` | 澄清 P2 产出的是"方案设计 + 实现导航"，不是"实现计划" |
| `agate/assets/execution-roles/implementer.md` | 澄清 P4 拿的是"方案设计 + 实现导航"，自主决定实现步骤 |
| `agate/WORKFLOW.md` | 阶段总览表 P2 描述加"方案设计层"标注 |

### 不做的

- 不新增"实现计划"阶段
- 不照搬 superpowers 的 writing-plans 步骤脚本
- 不合并 P0 和 P1

## 二、P2 不可裁剪（含例外口）

### 问题

当前协议允许 P1 裁剪 P2（条件：`risk_level=low`）。但 P1 analyst 做需求分析不做方案设计——它没有做方案设计就无法预知 P2 architect 会发现哪些 P0/P1 没想到的问题。

T045 是实证：P0/P1 觉得"CSS 改个 display 属性很明确"，但 P2 如果做了就会发现 `overflow:auto + display:block` 的交互问题。

### 设计决策

**P2 不可裁剪，含例外口**——镜像 P6 模式（P6 有 `no_behavior_change: true` 例外）：

```
P2 不可裁，除非 design_trivial: true
```

`design_trivial: true` 限定条件（全部满足）：
- 单文件改动
- 无逻辑变更（纯 typo / 纯文案 / 纯配置值修改）
- ≤ 5 行改动
- 主 Agent 确认（P1 需声明 `design_trivial: true` + 理由）

### 影响评估

| 维度 | 影响 |
|------|------|
| 简单任务 | typo fix 走 `design_trivial: true` 例外口跳 P2，不增加开销 |
| 中等任务 | P2 一遍过（architect 写设计 + 评审通过），如果 P0/P1 真想清楚了开销不大 |
| 现有任务兼容 | 已声明裁剪 P2 的 P1-requirements.md 下次 commit 会被 hook 拦截——过渡期方案：现有裁剪 P2 的任务自动注入 `override:` 字段（一次性 migration，主 Agent 在下次 commit 前补 `override: P2 retained (reason: v0.6 迁移)`）|
| 裁剪哲学 | 从"P1 可裁 P2（risk_level=low）"改为"P2 不可裁（含 design_trivial 例外）"——与 P6 同级（含例外口）|

### 改动范围

| 文件 | 改动 |
|------|------|
| `agate/state-machine.md` | 裁剪条件表：P2 从"需 risk_level=low"改为"不可裁（除非 design_trivial: true）" |
| `agate/scripts/check-pruning.sh` | 检查 2：P2 裁剪 → 检查 `design_trivial: true` + 限定条件；检查 7 的 `! grep -qw 'P2'` 分支在 P2 不可裁后恒 false，清理死代码 |
| `agate/WORKFLOW.md` | P2 行说明不可裁 + 例外口 |
| `agate/dispatch-protocol.md` | 门槛表同步 |
| `agate/orchestrator-template.md` | 裁剪说明同步 |

### 下游同步（评审补充）

| 项 | 改动 |
|----|------|
| **P0-brief pruning_tendency** | "激进 — 单文件 typo 修复，直接做"改为"激进 — 单文件 typo 修复，走 design_trivial 例外" |
| **risk_level 字段作用域** | P2 不可裁后 risk_level 只 gate P3 裁剪，文档同步说明 |
| **check-pruning.sh 检查 7** | `! grep -qw 'P2'` 在 P2 不可裁后恒 false，清理死分支 |

## 三、brainstorming 多方案探索（绑例外口）

### 缺口

P2 architect 当前只要求产出一个方案。强制多方案探索是发现"P1 没想到的问题"的有效手段。

### 设计决策

**P2 强制 2-3 个候选方案 + 权衡 + 选择理由——但绑 design_trivial 例外口**：

- `design_trivial: true` 时跳过多方案（typo fix 不需要 2 个方案）
- 其余强制 2-3 个候选方案

不照搬 superpowers 的"人-Agent 逐问对话"——agate 是 subagent 隔离产出，architect 在自己的上下文里独立探索。

### 评审角色审查

plan-eng-review 审查：
- ✅ "是否探索了替代方案 + 选择理由自洽"（可判定）
- ❌ "是否选了最优方案"（reviewer 隔离上下文无法独立重推，不要求）

**额外审查项**（评审补充）：方案是否清晰到 implementer 无需步骤计划即可自主实现；实现导航（files_to_read）是否覆盖实现所需全部上下文。

### 改动范围

| 文件 | 改动 |
|------|------|
| `agate/assets/templates/task-files.md` | P2-design.md §1 改为"候选方案（至少 2 个 + 权衡 + 选择理由）"——design_trivial 时可只写 1 个 |
| `agate/assets/execution-roles/architect.md` | 加"至少提 2 个候选方案"要求 |
| `agate/assets/review-roles/plan-eng-review.md` | 加"审查是否探索了替代方案 + 选择理由自洽 + 实现就绪度" |

## 四、扩写 investigate.md（superpowers 借鉴）

### 缺口

agate 的 `investigate.md` 评审角色只有 32 行，缺多组件取证插桩、root-cause-tracing 回溯技术。

### 借鉴内容

- 四阶段结构化调试（根因调查 → 模式分析 → 假设测试 → 实现）
- 多组件边界插桩取证（在组件边界 log 输入/输出，定位问题传播路径）
- root-cause-tracing 回溯（从症状沿调用链回溯到根因）

### 关键适配（评审补充）

**"3 次修复失败 → 质疑架构假设"不能靠 subagent 自己生效**——失败计数在 `.state.yaml` 的 `retries[Pn]`，不在 subagent 上下文里。subagent 不知道自己是第几次被派发。

这条规则必须落在**主 Agent 派发逻辑**：
- `dispatch-protocol.md` 重派 prompt 模板加：`len(retries[Pn]) >= 3` 时注入"你已是第 N 次尝试，质疑架构假设而非继续同层试错"
- investigate.md 角色文件只写"若被告知是高次重试，优先质疑架构"

### 改动范围

| 文件 | 改动 |
|------|------|
| `agate/assets/review-roles/investigate.md` | 从 32 行扩到 ~100 行（四阶段 + 取证 + 回溯）|
| `agate/dispatch-protocol.md` | 重派 prompt 模板加"3 次失败质疑架构"注入逻辑 |

## 六、TDD 反合理化话术（评审否决，砍掉）

### 评审否决理由

agate 靠机器 gate 不信任 Agent 自觉（LIMITATIONS.md 局限 3），却借鉴"话术约束自觉"机制——哲学自相矛盾。

- 话术映射到 gate 的是冗余（"删了就是删了"已有 check-tdd-red.sh）
- 话术映射不到的是无效（"太简单不用测"无脚本能查，话术不会可靠改变行为）

### 替代方案（如需补强 TDD）

改为机器 gate：例如"每个改动的源文件须有对应测试文件变动"（coverage-delta gate）——比话术强得多。但这需要项目侧测试框架支持，标记为 Phase 2 候选，不进 v0.6。

**v0.6 不做此项。**
