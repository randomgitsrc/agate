## 一、P2 不可裁剪

### 问题

当前协议允许 P1 裁剪 P2（条件：`risk_level=low`）。但 P1 analyst 做的是需求分析不是方案设计——它没有做方案设计就无法预知 P2 architect 会发现哪些 P0/P1 没想到的问题。

T045 是实证：P0/P1 觉得"CSS 改个 display 属性很明确"，但 P2 如果做了就会发现 `overflow:auto + display:block` 的交互问题。P1 裁 P2 省下的时间，远不如 P2 能提前发现问题的价值。

### 设计决策

**P2 不可裁剪**——方案设计 + 评审是必经阶段。

### 影响评估

| 维度 | 影响 |
|------|------|
| 简单任务开销 | P2 一遍过（architect 写设计 + 评审通过），如果 P0/P1 真想清楚了，P2 就是"把想清楚的落盘"，开销不大 |
| 现有任务兼容 | 已声明裁剪 P2 的 P1-requirements.md 下次 commit 会被 hook 拦截——需评估是否要过渡期 |
| 裁剪哲学 | 从"P1 可裁 P2（risk_level=low）"改为"P2 不可裁"——与 P6 同级 |

### 改动范围

| 文件 | 改动 |
|------|------|
| `agate/state-machine.md` | 裁剪条件表：P2 从"需 risk_level=low"改为"不可裁" |
| `agate/scripts/check-pruning.sh` | 检查 2：P2 裁剪 → 无条件 exit 1 |
| `agate/WORKFLOW.md` | P2 行说明不可裁 |
| `agate/dispatch-protocol.md` | 门槛表同步 |
| `agate/orchestrator-template.md` | 裁剪说明同步 |

### 待决问题

- 是否需要例外口（如 P0 显式声明 `design_trivial: true` + 主 Agent 确认）？
- 向后兼容：现有裁剪 P2 的任务是否给过渡期？

## 一·B、概念分层：spec / 方案设计 / 实现计划

### 问题

当前 agate 的概念边界模糊：

| 概念 | 回答什么 | agate 当前在哪 | 问题 |
|------|---------|--------------|------|
| **spec**（需求规格） | 做什么 + 为什么 + 验收条件 | P0-brief + P1-requirements | P0 和 P1 的边界模糊——P0 是"PM 视角的任务简报"，P1 是"需求基线"，但都是 spec 层 |
| **方案设计** | 怎么做 + 影响什么 + 选哪个方案 | P2-design.md | 混了实现计划（files_to_read 是实现导航不是方案设计）|
| **实现计划** | 按什么步骤做 | 无独立产物 | P2 的 files_to_read 是雏形但太薄；P4 implementer 拿 P2 设计自主实现，中间没有步骤计划 |
| **代码实现** | 实际代码 | P4-implementation/ | 清晰 |

### 设计决策

**四层分离，不新增阶段——在现有 P0-P4 内澄清边界**：

```
P0-brief.md     = spec 层（PM 视角：做什么 + 风险 + 约束）
P1-requirements.md = spec 层（工程视角：BDD 验收条件 + 范围声明）
P2-design.md    = 方案设计层（怎么做 + 候选方案权衡 + 影响域 + gate 固化）
P2-design.md §4 = 实现导航（files_to_read——P2 architect 产出的上下文地图）
P4 implementer  = 代码实现层（拿 P2 设计 + P3 测试自主实现）
```

**关键澄清**：

1. **P0 和 P1 都是 spec 层，但视角不同**：P0 是 PM 视角（任务定义 + 风险 + 环境），P1 是工程视角（BDD + 范围 + 裁剪）。不需要合并——两者的视角分离是 agate 的设计选择。

2. **P2 是方案设计层，不是实现计划层**：P2-design.md 的 `files_to_read` 是"实现导航"——P2 architect 作为唯一既读了代码又设计了方案的角色，把上下文地图交给 P4。这不是"步骤计划"（不列每步做什么），是"资源地图"（列实现时需要读哪些文件 + 为什么）。

3. **不新增"实现计划"阶段**：superpowers 的 writing-plans 是精确到每步 2-5 分钟的步骤脚本——agate 不需要这个层。P4 implementer 拿 P2 设计 + P3 测试自主实现，如果 P2 设计够明确 + files_to_read 够完整，implementer 不需要步骤计划。如果 P2 设计不够明确——那是 P2 的问题，不是"缺一个实现计划阶段"的问题。

4. **复杂任务的实现计划**：如果任务确实复杂到需要步骤计划（多文件、多步骤、有依赖顺序），P2-design.md 可以含一个"实现顺序"节（哪些先做、哪些后做、为什么），但不是强制要求——简单任务不需要。

### 改动范围

| 文件 | 改动 |
|------|------|
| `agate/assets/templates/task-files.md` | P2-design.md 结构：§1 改为"候选方案"（配套 brainstorming 借鉴）；§4 files_to_read 改名为"实现导航"并澄清语义；可选加"实现顺序"节 |
| `agate/assets/execution-roles/architect.md` | 澄清 P2 产出的是"方案设计 + 实现导航"，不是"实现计划" |
| `agate/assets/execution-roles/implementer.md` | 澄清 P4 拿的是"方案设计 + 实现导航"，自主决定实现步骤 |
| `agate/WORKFLOW.md` | 阶段总览表 P2 描述加"方案设计层"标注 |

### 不做的

- **不新增"实现计划"阶段**——P0-P8 已经够多，再加一层增加开销但收益有限
- **不照搬 superpowers 的 writing-plans 步骤脚本**——模型不同，agate implementer 是自主实现不是按步执行
- **不合并 P0 和 P1**——视角分离是设计选择

## 二、superpowers 借鉴

### 调研结论

superpowers 与 agate 是**互补关系**：
- superpowers 是单 Agent 行为纪律层（靠话术约束 Agent 自觉）
- agate 是多 Agent 编排结构层（靠机器 gate 不信任 Agent 自觉）
- 重叠区（TDD/验证/评审/调试）里 agate 结构上更强（机器校验 vs 自觉），但 superpowers 在调试技术上更细

### 借鉴项 1（高优先级）：扩写 investigate.md

**缺口**：agate 的 `investigate.md` 评审角色只有 32 行，缺多组件取证插桩、3 次失败质疑架构、root-cause-tracing 回溯技术。superpowers 的 systematic-debugging skill 有完整方法论。

**借鉴内容**：
- 四阶段结构化调试（根因调查 → 模式分析 → 假设测试 → 实现）
- 多组件边界插桩取证（在组件边界 log 输入/输出，定位问题传播路径）
- 3 次修复失败 → 质疑架构假设（不是继续在同一层面试错）
- root-cause-tracing 回溯（从症状沿调用链回溯到根因）

**改动范围**：扩写 `agate/assets/review-roles/investigate.md`（从 32 行扩到 ~100 行）

### 借鉴项 2（中高优先级）：TDD 反合理化话术

**缺口**：agate 的 P3/P4 gate 是机械的（check-tdd-red.sh 验证红灯），但 gate 之间 subagent 仍需自觉。superpowers 的 test-driven-development skill 有完整反合理化话术。

**借鉴内容**：
- 反合理化红旗清单（"太简单不用测" → 简单代码也会崩；"先写代码再补测试" → 删掉重来）
- "删了就是删了"（看到测试失败才算红灯，不是"应该会失败"）
- Red-Green-Refactor 严格执行

**改动范围**：往 `agate/assets/execution-roles/test-designer.md` 和 `implementer.md` 加反合理化段

### 借鉴项 3（中优先级）：模型选择

**缺口**：agate 没提"机械任务用便宜模型，设计任务用强模型"。

**借鉴内容**：
- dispatch-protocol.md 加模型选择建议
- P1/P2 用强模型（需求分析 + 方案设计需要推理能力）
- P4 实现可按复杂度选模型（简单 CRUD 用便宜模型，复杂算法用强模型）
- P6 验收用强模型（需要判断力）

**改动范围**：`agate/dispatch-protocol.md` 加模型选择段

### 借鉴项 4（高优先级）：brainstorming — P2 强制多方案探索

**缺口**：P2 architect 当前只要求产出一个方案，不要求探索多个方案。superpowers 的 brainstorming skill 强制"提 2-3 方案权衡后再设计"——这是发现"P1 没想到的问题"的有效手段。

**与 P2 不可裁剪的配套关系**：P2 不可裁剪的理由是"P2 会发现 P1 没想到的问题"。如果 P2 只产一个方案，发现能力有限；如果 P2 强制探索多个方案，发现能力更强。两者配套——P2 不可裁剪 + P2 强制多方案 = P2 真正发挥方案设计价值。

**借鉴内容**：
- P2-design.md 的"改动方案"节改为"候选方案"：列 2-3 个候选方案 + 各自的权衡（优点/风险/工作量）+ 选择理由
- 不照搬 superpowers 的"人-Agent 逐问对话"模式——agate 是 subagent 隔离产出，architect 在自己的上下文里独立探索
- 评审角色（plan-eng-review）审查方案选择是否合理，不只是审查最终方案

**改动范围**：
- `agate/assets/templates/task-files.md`：P2-design.md 结构的"## 1. 改动方案"改为"## 1. 候选方案（至少 2 个 + 权衡 + 选择理由）"
- `agate/assets/execution-roles/architect.md`：加"至少提 2 个候选方案"要求
- `agate/assets/review-roles/plan-eng-review.md`：加"审查方案选择是否合理"

**代价**：P2 多花几分钟探索替代方案。如果 P0/P1 真想清楚了，第 2-3 个方案很快就能写出"为什么不选它"——开销不大。如果 P0/P1 没想清楚，多方案探索会暴露盲区——这正是 P2 的价值。
