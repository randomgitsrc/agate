## 背景

T016 暴露了两类问题：

1. **执行错误（首要）**：主 Agent 三次违反现成协议——未记 retry_count、未触发 PAUSED、违规降级亲自写代码。协议已经写了，是主 Agent 没做。
2. **机制缺口（次要）**：dispatch-protocol 在输入导航、降级边界、空返回恢复策略上存在空白。

评审核心判断：**"不是 agate 没说，是主 Agent 没做"**。因此本计划区分"能用协议补的"和"补不了只能靠执行的"，并对每项修复的可靠性诚实分级。

---

## 能修复的（协议层面）

### 修复 0：retry_count 升级为重试记录（前置依赖）

**问题**：现有 `.state.yaml` 只有 `retry_count: { P3: 2 }` 一个整数，无法区分"原样重试第 2 次"和"调整策略后重试第 1 次"。修复 2 要求"空返回 1 次后必须改变策略，不允许第二次相同重试"——但没有字段记录"这次重试的 prompt 是否变更"，这条规则无法被验证。

**修法**：state-machine.md 的 `.state.yaml` 格式从整数升级为列表：

```yaml
# 旧格式（只有计数）
retry_count: { P3: 2 }

# 新格式（每次重试一条记录）
retries:
  P3:
    - round: 1
      failure_mode: empty_return
      prompt_changed: false       # 原样重试
      adjustment: null
    - round: 2
      failure_mode: empty_return
      prompt_changed: true        # 调整了策略
      adjustment: split_task      # split_task / add_navigation / switch_type
```

`retry_count` 保留为派生字段（`len(retries[P3])`），向后兼容。

**为什么是前置依赖**：修复 2 的"空返回 1 次后必须改变策略"和修复 4 的"拆分记录"都需要这个字段来验证是否被遵守。没有它，这两条修复和 T016 的 retry_count=0 一样——规则写了但无法检验。

**可靠性：高**。这是数据结构变更，不依赖主 Agent 主动遵守——主 Agent 更新 .state.yaml 是现成协议动作（state-machine 单步函数第 5 步），加几个字段是同一流程的延伸。

### 修复 1：降级边界明确化

**问题**：dispatch-protocol 说 `has_task_tool: false` 时降级，但没说 `has_task_tool: true` 但 subagent 失败时怎么办。主 Agent 自行决定降级。

**修法**：dispatch-protocol.md「执行模式」节后新增「降级规则」节：

```markdown
## 降级规则（硬边界）

降级（主 Agent 亲自执行阶段产出）只在以下情况发生：
- has_task_tool: false（环境不支持 subagent）
- has_local_runtime: false 且阶段需要本地运行（gate 无法执行）

**subagent 执行失败 ≠ 降级信号。** subagent 失败时：
1. 计入 retry_count（现成规则）
2. 调整策略重派（拆分任务 / 补导航 / 换 subagent 类型）
3. retry 超限 → PAUSED（state-machine 现成规则）

主 Agent 不得以"subagent 做不好"为由跳过 retry/PAUSED 直接降级。
```

**可靠性：高**。这是硬禁止规则，把"降级"从主 Agent 的选项列表里删除。T016 的主 Agent 降级是因为它认为降级是合法选项（协议没说不行）。明确禁止后，至少在"主 Agent 读到了这条"的前提下，降级不再是自圆其说的路径。不涉及主观判断，不依赖场景识别。

### 修复 2：空返回恢复策略

**问题**：现有 retry 机制针对"产出了但质量不够"。subagent 空返回是不同失败模式，主 Agent 不知道该怎么调整。

**修法**：dispatch-protocol.md「subagent 返回校验」节末新增「空返回的恢复策略」：

```markdown
### 空返回的恢复策略

subagent 空返回（约定产出文件不存在）是特殊失败模式，不能简单重试相同 prompt。

1. 第 1 次空返回：
   - 计入 retries[Pn]（现成规则），记录 failure_mode: empty_return, prompt_changed: false
   - 分析失败原因：prompt 是否过复杂？输入文件是否过多？任务粒度是否过大？
   - 调整策略后重派：
     a. 拆分任务（见「任务粒度指引」）
     b. 补输入导航（见「输入导航原则」）
     c. 换 subagent 类型（frontend ↔ general）
   - 记录 prompt_changed: true, adjustment: <具体调整>

2. 第 2 次空返回（调整策略后仍失败）：
   - 计入 retries[Pn]
   - len(retries[Pn]) >= MAX_RETRY → PAUSED 报告人工

**禁止**：不调整策略、相同 prompt 直接重试（retries 记录里 prompt_changed=false 且非首次）。
空返回说明 subagent 扛不住当前任务形态，原样重试大概率还是空返回。
```

**可靠性：中**。建议性规则，依赖主 Agent 主动遵守。但比"无指引"好——当主 Agent 想遵守时，给了具体怎么做。"1 次后必须改变策略"是经验初值，T016 是唯一数据点（3 次相同失败后降级），取 1 是保守值：第 1 次失败可能偶发，第 2 次仍失败说明任务形态有问题。**依赖修复 0**：没有 retries 记录的 prompt_changed 字段，这条规则无法被验证。

### 修复 3：输入导航原则

**问题**：dispatch-protocol 说"传路径不传内容"，但没说"如何给 subagent 导航"。主 Agent 倾向于把所有文件路径甩给 subagent，让它自己读。

**修法**：dispatch-protocol.md「派发 prompt 模板」节前新增「输入导航原则」：

```markdown
## 输入导航原则

铁律 2"只传路径不传内容"防止的是上下文污染，不是禁止主 Agent 给方向。
主 Agent 派发 subagent 前，给 subagent 提供"读哪个节、关注什么"的导航。

导航 ≠ 提炼 ≠ 读全文：
- 导航：prompt 里注明"读 P1-requirements.md 的 BDD 验收条件节"
- 提炼（禁止）：主 Agent 读完文件把内容总结进 prompt
- 读全文（禁止）：把文件内容复制进 prompt

导航的信息来源是协议知识，不是文件内容：
- 主 Agent 启动时已读 7 个协议文件，其中角色定义文件硬约束了每个阶段产出
  文件的节结构：
  - P1 的节名称由 analyst.md:28-37 强制（需求复述/隐含需求/BDD验收条件/
    待确认清单/裁剪说明/范围声明/能力需求声明）
  - P2 的字段由 architect.md:26-46 强制（packages/domains/ui_affected/
    gate_commands）
- 主 Agent 用这些协议定义的节名称给导航，不需要读产出文件的实际内容
- 节名称是协议固定的，章节号是 subagent 自己编的——导航用节名称，不用章节号

主 Agent 的核心职责是**任务分解 + 输入导航 + 验证**，不是传话筒（把文件路径
原样转发），也不是消化器（读完所有文件做提炼）。
```

**可靠性：中**。建议性规则。关键设计：导航来自协议知识（节名称），主 Agent 不碰任何文件——连标题都不碰。这消除了"导航需要读文件"的灰色地带，铁律 2 的边界保持干净。

**节名称契约的验证**（plan-eng-review 非阻塞问题）：导航依赖"P1 必有 BDD 验收条件节、P2 必有 packages 字段"是硬约束。已交叉确认：analyst.md:28-37 用"含以下节"强制定义了 P1 的 7 个节名，architect.md:26-46 用"必须含以下声明字段"强制定义了 P2 的 4 个字段。这些是角色定义里的硬约束，不是 task-files.md 里的随意命名。**残余风险**：如果 subagent 产出时偏离了角色定义的节结构（用了自定义标题），导航会静默失效——subagent 找不到对应节，大概率又是空返回循环。缓解方式：P1/P2 gate 检查时，主 Agent 顺带验证产出文件是否含角色定义要求的节名称，缺失则门槛不通过。

### 修复 4：任务粒度指引

**问题**：P3 要求一个 subagent 产出 3 个异构文件（文档 + stub 代码 + 测试代码），主 Agent 要在技术文档作者、TypeScript 开发者、测试工程师三种身份间切换。

**修法**：dispatch-protocol.md 新增「任务粒度指引」：

```markdown
## 任务粒度指引

当阶段产出涉及以下特征时，主 Agent 应拆分为多个 subagent 任务：
- 产出 2 种及以上不同类型文件（如 文档 + 代码、stub + 测试、设计 + 实现）
- 输入文件超过 5 个（主 Agent 应先检查是否都必要，精简输入比拆分任务成本低；
  确实都必要时再拆分）

拆分原则：
- 每个任务产出 1-2 个文件，只涉及一种技术类型
- 每个任务的输入文件 ≤ 3 个
- 任务间有依赖时串行，无依赖时并行
- 拆分通过多次 task 调用实现，commit message 记录拆分（如"P3a: 测试用例文档"）
- 状态机不变——仍只看 P3 阶段，gate 仍是该阶段的门槛命令
```

**可靠性：中**。建议性规则。拆分判据用输出异构性（2 种类型文件），不用行数——T016 失败的根因是异构切换（文档+stub+测试三种身份），不是输入行数。LLM 处理 2000 行同质内容没问题，处理 500 行 4 种技术域照样会崩。行数是弱相关变量，异构性才是强相关变量。输入数量阈值放宽到 5——2-3 个文件是常态，5 个以上才该检查是否导航不够精准。**依赖修复 0**：拆分的记录（"P3a: 测试用例文档"）在 commit message 里，但"是否拆分过"需要从 retries 记录的 adjustment: split_task 字段验证。

## 不能用协议修复的（执行层面）

### 主 Agent 严格遵守现成协议

**问题**：T016 的首要问题是主 Agent 没按现成协议执行（未记 retry / 未触发 PAUSED / 违规降级）。

**为什么不能修**：协议是文档不是代码，无法强制 agent 执行。这正是 LIMITATIONS.md 局限 3"主 Agent 判断力是单点故障"的具体表现——所有质量保证最终收敛到主 Agent 这一个点，没有任何机制检验主 Agent 自己是否遵守规则。

**修复 1 能缓解的部分**：降级从模糊选项变成硬禁止，减少了主 Agent 自行合理化的空间。

**修复 0 能缓解的部分**：retries 记录比整数 retry_count 更难伪造"我调过策略了"——因为 adjustment 字段要求写具体调整方式，事后可以检查"说 split_task 但实际 prompt 没变"。

**修复 2/3/4 不能缓解的部分**：它们是建议性规则，和现有规则一样有被忽略的风险。T016 的主 Agent 忽略的不是"没有规则"，是"有规则没执行"。加更多建议性规则到同一个文档里，本质上是"用更长的规则解决规则不被遵守的问题"——这不成立。它们的价值是"当主 Agent 想遵守时，给了具体怎么做"，而不是"保证主 Agent 一定会遵守"。

### 数据源同源问题（plan-eng-review 评审新增）

**问题**：`docs/design-notes/main-agent-oversight.md` 方案 C"确定性脚本扫描历史生成异常报告"有一个致命假设——脚本扫描的数据（.state.yaml、commit message）全部是主 Agent 自己写的。T016 的 P3 commit 是实证：主 Agent 把违规降级写代码的事实写成了"TDD 测试设计完成，9 红灯 0 错误"，只字未提降级。一个解析 commit message 找异常的脚本，面对一个连 commit message 都会选择性隐瞒的主 Agent，扫不出任何东西。

**为什么不能修**：数据源和监督对象是同一个实体。要让脚本方向成立，数据源必须独立于主 Agent 的叙事——比如 task 工具本身的调用日志（如果平台暴露，不经过主 Agent 转写）、或者 subagent 产出文件里带主 Agent 难以伪造的自盖时间戳/身份标记，再拿 git author/timestamp 元数据交叉核对。这些在当前架构里都不存在。

**处理**：方案 C 从"方向已确认值得做"降级为"开放问题，前提未解决"。详见 main-agent-oversight.md 的修订。

## 修复可靠性分级

| 修复 | 类型 | 可靠性 | 理由 | 可验证？ |
|------|------|--------|------|----------|
| 0 retries 升级 | 数据结构 | 高 | 现成协议动作的延伸，不依赖主 Agent 主动遵守 | — |
| 1 降级禁止 | 硬禁止 | 高 | 把选项从列表删除，不涉及主观判断 | 看 commit 有无降级声明 |
| 2 空返回恢复 | 建议性 | 中 | 依赖主 Agent 主动遵守，但给了具体路径 | 依赖修复0：retries.prompt_changed |
| 3 输入导航 | 建议性 | 中 | 依赖主 Agent 主动遵守，导航来源是协议知识不碰文件 | 看 prompt 是否含节名称导航 |
| 4 任务粒度 | 建议性 | 中 | 依赖主 Agent 主动遵守，判据用异构性而非行数 | 依赖修复0：retries.adjustment |

**不假装 5 条修复同等可靠。** 修复 0/1 是结构性改变（数据格式 / 删选项），修复 2/3/4 是操作性指引（给方法）。修复 0 是 2/4 的前置依赖——没有它，2/4 写了也无法验证。

## 已取消的行动项

### 行动项 6：阶段内任务拆分（取消）

**原计划**：允许 P3 拆成 P3a/P3b/P3c，需改状态机。

**取消原因**：agate 已支持单阶段多次派发（P2 先派 architect 再派 reviewer，状态机只看"P2"）。P3 拆成多次 task 调用同理，状态机仍只看"P3"，gate 仍是 check-tdd-red.sh。拆分的记录靠 commit message，不需要 .state.yaml 改格式。行动项 6 基于错误前提（"拆分需要状态机改动"），实际不需要。修复 4 已覆盖拆分的判据和方式。

## 验证方式

修复落地后，下一个走完 P0-P8 的任务是验证样本：
- 主 Agent 是否以"subagent 做不好"为由降级（修复 1）
- subagent 空返回时 retries 记录的 prompt_changed 是否为 true（修复 0 + 2）
- 主 Agent 是否在派发前给了输入导航（修复 3）
- 阶段产出涉及 2+ 类型文件时是否拆分了任务（修复 4）

### 验证跟踪（plan-eng-review 测试缺口修复）

单次验证不足以确认，需积累 3-5 个任务。为防止"只有出下一次事故才回头翻"，建议建一个跟踪文件（如 `docs/reviews/t016-followup-validation.md`，尚未创建），按任务逐条记录 4 个检验点结果：

```markdown
| 任务 | 日期 | 修复1: 无违规降级 | 修复0+2: 空返回时prompt_changed | 修复3: 有输入导航 | 修复4: 异构时拆分 | 备注 |
|------|------|------------------|-------------------------------|------------------|-----------------|------|
| T017 | | ✅/❌/N/A | ✅/❌/N/A | ✅/❌/N/A | ✅/❌/N/A | |
```

每完成一个 P0-P8 任务后更新此表。3-5 个任务后做一次小结，判断修复是否生效。
