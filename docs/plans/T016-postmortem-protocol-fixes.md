## 背景

T016 暴露了两类问题：

1. **执行错误（首要）**：主 Agent 三次违反现成协议——未记 retry_count、未触发 PAUSED、违规降级亲自写代码。协议已经写了，是主 Agent 没做。
2. **机制缺口（次要）**：dispatch-protocol 在输入导航、降级边界、空返回恢复策略上存在空白。

评审核心判断：**"不是 agate 没说，是主 Agent 没做"**。因此本计划区分"能用协议补的"和"补不了只能靠执行的"，并对每项修复的可靠性诚实分级。

---

## 能修复的（协议层面）

### 修复 1：降级边界明确化

**问题**：dispatch-protocol 说 `has_task_tool: false` 时降级，但没说 `has_task_tool: true` 但 subagent 失败时怎么办。主 Agent 自行决定降级。

**修法**：dispatch-protocol.md「执行模式」节后新增「降级规则」节：

```markdown
## 降级规则（硬边界）

降级（主 Agent 亲自执行阶段产出）只在以下情况发生：
- has_task_tool: false（环境不支持 subagent）
- has_local_runtime: false 且阶段需要本地运行（gate 无法执行）

**subagent 执行失败 ≠ 降级信号。** subagent 失败时：
1. 计入 retry_count（现成规则）
2. 调整策略重派（拆分任务 / 补导航 / 换 subagent 类型）
3. retry 超限 → PAUSED（state-machine 现成规则）

主 Agent 不得以"subagent 做不好"为由跳过 retry/PAUSED 直接降级。
```

**可靠性：高**。这是硬禁止规则，把"降级"从主 Agent 的选项列表里删除。T016 的主 Agent 降级是因为它认为降级是合法选项（协议没说不行）。明确禁止后，至少在"主 Agent 读到了这条"的前提下，降级不再是自圆其说的路径。不涉及主观判断，不依赖场景识别。

### 修复 2：空返回恢复策略

**问题**：现有 retry 机制针对"产出了但质量不够"。subagent 空返回是不同失败模式，主 Agent 不知道该怎么调整。

**修法**：dispatch-protocol.md「subagent 返回校验」节末新增「空返回的恢复策略」：

```markdown
### 空返回的恢复策略

subagent 空返回（约定产出文件不存在）是特殊失败模式，不能简单重试相同 prompt。

1. 第 1 次空返回：
   - 计入 retry_count（现成规则）
   - 分析失败原因：prompt 是否过复杂？输入文件是否过多？任务粒度是否过大？
   - 调整策略后重派：
     a. 拆分任务（见「任务粒度指引」）
     b. 补输入导航（见「输入导航原则」）
     c. 换 subagent 类型（frontend ↔ general）

2. 第 2 次空返回（调整策略后仍失败）：
   - 计入 retry_count
   - retry_count >= MAX_RETRY → PAUSED 报告人工

**禁止**：不调整策略、相同 prompt 直接重试。空返回说明 subagent 扛不住当前任务形态，原样重试大概率还是空返回。
```

**可靠性：中**。建议性规则，依赖主 Agent 主动遵守。但比"无指引"好——当主 Agent 想遵守时，给了具体怎么做。"1 次后必须改变策略"是经验初值，T016 是唯一数据点（3 次相同失败后降级），取 1 是保守值：第 1 次失败可能偶发，第 2 次仍失败说明任务形态有问题。

### 修复 3：输入导航原则

**问题**：dispatch-protocol 说"传路径不传内容"，但没说"如何给 subagent 导航"。主 Agent 倾向于把所有文件路径甩给 subagent，让它自己读。

**修法**：dispatch-protocol.md「派发 prompt 模板」节前新增「输入导航原则」：

```markdown
## 输入导航原则

铁律 2"只传路径不传内容"防止的是上下文污染，不是禁止主 Agent 给方向。
主 Agent 派发 subagent 前，给 subagent 提供"读哪个节、关注什么"的导航。

导航 ≠ 提炼 ≠ 读全文：
- 导航：prompt 里注明"读 P1-requirements.md 的 BDD 验收条件节"
- 提炼（禁止）：主 Agent 读完文件把内容总结进 prompt
- 读全文（禁止）：把文件内容复制进 prompt

导航的信息来源是协议知识，不是文件内容：
- 主 Agent 启动时已读 7 个协议文件，其中 task-files.md 和各角色定义规定了
  每个阶段产出文件的节结构（如 P1 必有"BDD 验收条件"节、P2 必有
  "packages/domains/gate_commands"字段）
- 主 Agent 用这些协议定义的节名称给导航，不需要读产出文件的实际内容
- 节名称是协议固定的，章节号是 subagent 自己编的——导航用节名称，不用章节号

主 Agent 的核心职责是**任务分解 + 输入导航 + 验证**，不是传话筒（把文件路径
原样转发），也不是消化器（读完所有文件做提炼）。
```

**可靠性：中**。建议性规则。关键设计：导航来自协议知识（节名称），主 Agent 不碰任何文件——连标题都不碰。这消除了"导航需要读文件"的灰色地带，铁律 2 的边界保持干净。

### 修复 4：任务粒度指引

**问题**：P3 要求一个 subagent 产出 3 个异构文件（文档 + stub 代码 + 测试代码），主 Agent 要在技术文档作者、TypeScript 开发者、测试工程师三种身份间切换。

**修法**：dispatch-protocol.md 新增「任务粒度指引」：

```markdown
## 任务粒度指引

当阶段产出涉及以下特征时，主 Agent 应拆分为多个 subagent 任务：
- 产出 2 种及以上不同类型文件（如 文档 + 代码、stub + 测试、设计 + 实现）
- 输入文件超过 5 个（主 Agent 应先检查是否都必要，精简输入比拆分任务成本低；
  确实都必要时再拆分）

拆分原则：
- 每个任务产出 1-2 个文件，只涉及一种技术类型
- 每个任务的输入文件 ≤ 3 个
- 任务间有依赖时串行，无依赖时并行
- 拆分通过多次 task 调用实现，commit message 记录拆分（如"P3a: 测试用例文档"）
- 状态机不变——仍只看 P3 阶段，gate 仍是该阶段的门槛命令
```

**可靠性：中**。建议性规则。拆分判据用输出异构性（2 种类型文件），不用行数——T016 失败的根因是异构切换（文档+stub+测试三种身份），不是输入行数。LLM 处理 2000 行同质内容没问题，处理 500 行 4 种技术域照样会崩。行数是弱相关变量，异构性才是强相关变量。输入数量阈值放宽到 5——2-3 个文件是常态，5 个以上才该检查是否导航不够精准。

## 不能用协议修复的（执行层面）

### 主 Agent 严格遵守现成协议

**问题**：T016 的首要问题是主 Agent 没按现成协议执行（未记 retry / 未触发 PAUSED / 违规降级）。

**为什么不能修**：协议是文档不是代码，无法强制 agent 执行。这正是 LIMITATIONS.md 局限 3"主 Agent 判断力是单点故障"的具体表现——所有质量保证最终收敛到主 Agent 这一个点，没有任何机制检验主 Agent 自己是否遵守规则。

**修复 1 能缓解的部分**：降级从模糊选项变成硬禁止，减少了主 Agent 自行合理化的空间。

**修复 2/3/4 不能缓解的部分**：它们是建议性规则，和现有规则一样有被忽略的风险。T016 的主 Agent 忽略的不是"没有规则"，是"有规则没执行"。加更多建议性规则到同一个文档里，本质上是"用更长的规则解决规则不被遵守的问题"——这不成立。它们的价值是"当主 Agent 想遵守时，给了具体怎么做"，而不是"保证主 Agent 一定会遵守"。

## 修复可靠性分级

| 修复 | 类型 | 可靠性 | 理由 |
|------|------|--------|------|
| 1 降级禁止 | 硬禁止 | 高 | 把选项从列表删除，不涉及主观判断 |
| 2 空返回恢复 | 建议性 | 中 | 依赖主 Agent 主动遵守，但给了具体路径 |
| 3 输入导航 | 建议性 | 中 | 依赖主 Agent 主动遵守，导航来源是协议知识不碰文件 |
| 4 任务粒度 | 建议性 | 中 | 依赖主 Agent 主动遵守，判据用异构性而非行数 |

**不假装 4 条修复同等可靠。** 修复 1 是结构性改变（删选项），修复 2/3/4 是操作性指引（给方法）。如果主 Agent 只读一条，应该是修复 1。

## 已取消的行动项

### 行动项 6：阶段内任务拆分（取消）

**原计划**：允许 P3 拆成 P3a/P3b/P3c，需改状态机。

**取消原因**：agate 已支持单阶段多次派发（P2 先派 architect 再派 reviewer，状态机只看"P2"）。P3 拆成多次 task 调用同理，状态机仍只看"P3"，gate 仍是 check-tdd-red.sh。拆分的记录靠 commit message，不需要 .state.yaml 改格式。行动项 6 基于错误前提（"拆分需要状态机改动"），实际不需要。修复 4 已覆盖拆分的判据和方式。

## 验证方式

修复落地后，下一个走完 P0-P8 的任务是验证样本：
- 主 Agent 是否以"subagent 做不好"为由降级（修复 1）
- subagent 空返回时是否调整了策略（修复 2）
- 主 Agent 是否在派发前给了输入导航（修复 3）
- 阶段产出涉及 2+ 类型文件时是否拆分了任务（修复 4）

单次验证不足以确认，需积累 3-5 个任务。
