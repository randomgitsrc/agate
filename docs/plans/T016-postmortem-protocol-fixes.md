# T016 复盘后的协议修复计划

> 来源：`docs/reviews/agate-postmortem-T016-2026-06-20.md` §八行动项 + 独立专家评审意见
> 日期：2026-06-21
> 状态：待执行

---

## 背景

T016 暴露了两类问题：

1. **执行错误（首要）**：主 Agent 三次违反现成协议——未记 retry_count、未触发 PAUSED、违规降级亲自写代码。这些协议已经写了，是主 Agent 没做。
2. **机制缺口（次要）**：dispatch-protocol 在输入导航、降级边界、空返回恢复策略上存在空白。

评审的核心判断：**"不是 agate 没说，是主 Agent 没做"**。因此本计划区分"能用协议补的"和"补不了只能靠执行的"。

---

## 能修复的（协议层面）

### 修复 1：降级边界明确化（行动项 2）

**问题**：dispatch-protocol.md 说 `has_task_tool: false` 时降级，但没说 `has_task_tool: true` 但 subagent 失败时怎么办。主 Agent 自行决定降级。

**修法**：在 dispatch-protocol.md「执行模式」节后新增：

```markdown
## 降级规则（硬边界）

降级（主 Agent 亲自执行阶段产出）只在以下情况发生：
- has_task_tool: false（环境不支持 subagent）
- has_local_runtime: false 且阶段需要本地运行（gate 无法执行）

**subagent 执行失败 ≠ 降级信号。** subagent 失败时：
1. 计入 retry_count（dispatch-protocol 现成规则）
2. 调整策略重派（拆分任务 / 补导航 / 换 subagent 类型）
3. retry 超限 → PAUSED（state-machine 现成规则）

主 Agent 不得以"subagent 做不好"为由跳过 retry/PAUSED 直接降级。
```

**为什么能修**：这是一条明确的禁止规则，不涉及主观判断。主 Agent 读到后，降级前会知道"这不是合法选项"。

---

### 修复 2：空返回恢复策略（行动项 3）

**问题**：现有 retry 机制针对"产出了但质量不够"。subagent 空返回（不产出文件）是不同失败模式，主 Agent 不知道该怎么调整。

**修法**：在 dispatch-protocol.md「subagent 返回校验」节末尾新增：

```markdown
### 空返回的恢复策略

subagent 空返回（约定产出文件不存在）是特殊失败模式，不能简单重试相同 prompt。

1. 第 1 次空返回：
   - 计入 retry_count（现成规则）
   - 分析失败原因：prompt 是否过复杂？输入文件是否过多？任务粒度是否过大？
   - 调整策略后重派：
     a. 拆分任务（一个 subagent 产出 1-2 个文件，不是 3+ 个异构文件）
     b. 补输入导航（指明"读哪几节、关注什么"，见下方"输入导航"原则）
     c. 换 subagent 类型（frontend ↔ general）

2. 第 2 次空返回（调整策略后仍失败）：
   - 计入 retry_count
   - retry_count >= MAX_RETRY → PAUSED 报告人工

**禁止**：不调整策略、相同 prompt 直接重试。空返回说明 subagent 扛不住当前任务形态，原样重试大概率还是空返回。
```

**为什么能修**：给了具体的恢复路径，不是模糊的"想办法"。

---

### 修复 3：输入导航原则（行动项 1 + 4 合并）

**问题**：dispatch-protocol 说"传路径不传内容"，但没说"如何给 subagent 导航输入文件"。主 Agent 倾向于把所有文件路径甩给 subagent，让它自己读。

**修法**：在 dispatch-protocol.md「派发 prompt 模板」节前新增：

```markdown
## 输入导航原则

铁律 2"只传路径不传内容"防止的是上下文污染，不是禁止主 Agent 给方向。主 Agent 派发 subagent 前，基于自己已写的 P0-brief + P1/P2 产出（这些本身就是提炼文档），给 subagent 提供"读哪几节、关注什么"的导航。

导航 ≠ 提炼 ≠ 读全文：
- 导航：prompt 里注明"P2-design.md §3.2 有接口签名，§3.4.1 有串行化设计，重点关注这两节"
- 提炼（禁止）：主 Agent 读完 P2-design.md 把内容总结进 prompt
- 读全文（禁止）：把 P2-design.md 内容复制进 prompt

导航的内容来源：主 Agent 不需要读输入文件全文。P0-brief 是主 Agent 自己写的，P1-requirements 和 P2-design 是 subagent 产出的提炼文档——主 Agent 从这些文档的章节标题和结构就能给出导航，不需要碰全文。

主 Agent 的核心职责是**任务分解 + 输入导航 + 验证**，不是传话筒（把文件路径原样转发给 subagent），也不是消化器（读完所有文件做提炼）。
```

**为什么能修**：明确了"导航"这个中间层——既不违反铁律 2（不读全文），又解决了"subagent 拿到一堆文件路径不知道重点"的问题。关键是不让主 Agent 读全文，而是利用已有提炼文档的结构信息。

---

### 修复 4：任务粒度指引（行动项 5）

**问题**：P3 要求一个 subagent 产出 3 个异构文件（文档 + stub 代码 + 测试代码），粒度过大。agate 没有阶段内任务粒度的规定。

**修法**：在 dispatch-protocol.md 新增：

```markdown
## 任务粒度指引

当阶段产出涉及以下特征时，主 Agent 应拆分为多个 subagent 任务：
- 多个异构文件（文档 + 代码）
- 多种技能（设计 + 实现 + 测试）
- 输入文件超过 3 个或总行数超过 500 行

拆分原则：
- 每个任务产出 1-2 个文件
- 每个任务的输入文件 ≤ 3 个
- 任务间有依赖时串行，无依赖时并行
- 拆分在 .state.yaml 记录（如 P3a/P3b）
```

**为什么能修**：给了可判定的拆分阈值（文件数、行数），不是模糊的"看情况"。但阈值本身是经验值，需更多任务验证后调整。

## 不能用协议修复的（执行层面）

### 行动项 0：主 Agent 严格遵守现成协议

**问题**：T016 的首要问题是主 Agent 没按现成协议执行（未记 retry / 未触发 PAUSED / 违规降级）。协议已经写了该怎么做，是主 Agent 没做。

**为什么不能修**：协议是文档，不是代码。文档能写"必须记 retry"，但不能强制 agent 执行。如果 agent 选择性忽略规则，再多的规则也无济于事——这正是 LIMITATIONS.md 局限 3"主 Agent 判断力是单点故障"的具体表现。

**能做的缓解**：修复 1（降级边界明确化）把"降级"从模糊选项变成了明确禁止，减少主 Agent "自行决定降级"的空间。但最终仍是执行问题。

---

## 暂不修复的（需更多验证）

### 行动项 6：阶段内任务拆分

**问题**：允许 P3 拆成 P3a/P3b/P3c，每个 subagent 产出 1-2 个文件。

**为什么暂不修**：影响面大——state-machine.md 的状态转移规则、.state.yaml 格式、active-tasks.md 看板都要改。T016 是单案例，不足以确定拆分的规则细节（怎么编号、怎么追踪、拆分后 gate 怎么判）。修复 4（任务粒度指引）已经给了拆分的判断标准，具体的状态追踪机制等更多任务验证后再设计。

## 验证方式

修复落地后，下一个走完 P0-P8 的任务是验证样本：
- 主 Agent 是否在派发前给了输入导航（修复 3）
- subagent 空返回时是否调整了策略而非原样重试（修复 2）
- 主 Agent 是否在任何时候以"subagent 做不好"为由降级（修复 1）
- 阶段产出涉及多文件时是否拆分了任务（修复 4）

单次验证不足以确认修复有效，需积累 3-5 个任务才能判断。
