# v0.13.1 适用边界强化 + gate 补漏

> 来源：PeekView 项目两份复盘（T058 + v0.9.1），提取对 agate 通用规则的改进。

## 0. 问题陈述

### 0.1 "直接做"判断边界模糊

WORKFLOW.md §适用边界说"微任务直接做，不走 agate"，但"微任务"定义仅举例（typo、文案、单行配置、debug 后的精确修复），无判断标准。

v0.9.1 复盘证明：**判断"简单"本身就是风险最高的决策**。Bug #3 看似"点击无反应"（简单 bug），实际涉及三个机制交叉（sticky header + absolute 定位 + click-outside），修复引入新 bug，形成挤牙膏模式。如果走 agate，P1 会识别为架构级改动，P6 不会漏掉交互测试。

当前风险矩阵（P2.13）只看"复杂度 × 风险"两个维度，缺少**改动性质**维度——声明性改动 vs 行为逻辑改动 vs 机制交叉，性质不同，流程需求不同。

### 0.2 P8 gate 缺 tag 存在性检查

T058 §3.4：T056 bump 了版本但 git tag 缺失，P8 gate 没拦住。当前 check-gate.sh P8 检查 version 文件变更和 CHANGELOG，但不检查 `git tag -l "v{version}"`。

### 0.3 P3 测试设计缺"前置状态"覆盖提示

T058 §3.5：BDD-05 badge bug 穿透 P3 和 P5，因为测试只覆盖 When/Then 触发的交互路径，没覆盖 Given 条件隐含的前置状态（页面首次加载时 badge 展示已有数据）。

### 0.4 orchestrator-log 纪律不足

T058 §3.8：17 行日志覆盖 10 次派发和 4 次 gate 失败，关键事件零记录。当前规则"想写就写"太宽松。

## 1. 设计原则

1. **通用性**：规则不绑定特定技术栈（不写"CSS/JS/组件"），适用于 Python/Vue、Java/Spring、Rust/CLI、嵌入式 C 等任何项目
2. **最小侵入**：不改 gate 脚本核心逻辑，只补漏和加提示
3. **不阻断"直接做"**：给出判断标准，不是禁止"直接做"——微任务仍然可以直接做，但判断有据可依
4. **渐进采纳**：新规则以 WARNING/提示为主，不新增硬拦截

## 2. 修补清单

### P1 — "直接做"判断标准（WORKFLOW.md + orchestrator-template.md）

**现状**：WORKFLOW.md L118 "微任务（typo、文案、单行配置、debug 后的精确修复）| 直接做"——举例式，无判断规则。

**改为**：分层判断——先看改动性质决定流程类型，再看影响范围和风险等级决定裁剪程度。改动性质是入口判断，不是矩阵维度。

```
第一步：改动性质（决定流程类型）
- 声明性改动（不改变程序运行时控制流，只改变声明值）→ 可直接做
  判断方法：改前和改后，程序的控制流是否相同？相同→声明性，不同→行为逻辑
  示例：样式值、配置常量、文案 typo、连接字符串
- 行为逻辑改动（条件分支、状态转换、数据处理）→ 至少走裁剪 agate
- 机制交叉（≥2 个子系统交互、时序依赖、跨层影响）→ 必须走完整 agate

第二步：影响范围（调整裁剪程度）
- 单点（单文件/单模块内）→ 按性质判断
- 跨模块且有语义关联 → 性质升一级（声明性→裁剪 agate，行为逻辑→完整 agate）
- 跨模块但无语义关联（如两个文件各改一个 typo）→ 按单点处理

覆盖规则：高风险（安全/数据/权限）任务不论改动性质，至少走裁剪 agate
```

**具体改动**：

1. WORKFLOW.md §适用边界：替换 L118 的举例式定义为上述分层判断
2. WORKFLOW.md §风险矩阵（L161-171）：保留现有矩阵，在矩阵前加"改动性质"入口判断，矩阵本身只决定裁剪程度
3. orchestrator-template.md 关键不变量：增加"机制交叉改动必须走 agate"
4. "直接做"最低要求（P2.14）补充：commit message 须声明改动性质（声明性/行为逻辑/机制交叉）+ 为什么安全

**不动**：
- 不改 gate 脚本——"直接做"时 agate 不介入，无法在 gate 层拦截
- 不改裁剪规则——P2/P4/P5/P6 不可裁等 v0.13.0 结论不变
- 不做三维矩阵——12 格太复杂，主 Agent 实际决策时不会逐格查表

### P2 — P8 gate 增加 tag 存在性检查（check-gate.sh）

**现状**：check-gate.sh P8 检查 version 文件变更 + CHANGELOG 变更，不检查 tag。

**改为**：P8 gate 增加 tag 存在性检查。tag 通常在 gate 通过后才打，所以此检查为 **WARNING 不阻断**——提醒"打 tag 后再推进到 READY"。

**具体改动**：

1. check-gate.sh P8 分支：在 version+CHANGELOG 检查后，从 CHANGELOG 暂存 diff 提取版本号（正则匹配 `[version]` 或 `[vX.Y.Z]`），检查 `git tag -l "${VERSION_TAG_PREFIX}${version}"`（`VERSION_TAG_PREFIX` 环境变量，默认 `v`），无 tag → WARNING（exit 2 而非 exit 1）
2. state-machine.md P8→READY 转移条件：补充"tag 存在"为推荐条件

**不动**：
- 不阻断——tag 时机因项目而异（有的项目 gate 通过后手动打 tag）
- 不改 P8 release 流程——tag 创建仍是 P8/P8 后的步骤

### P3 — P3 卡片增加"前置状态"覆盖提示（P3-tdd.md）

**现状**：P3-tdd.md 无关于 Given 条件隐含的前置状态的提示。

**改为**：在 P3 卡片"常见错误"节增加一条提示（1 行 + WORKFLOW.md 引用）：

> 测试设计应覆盖 BDD Given 隐含的前置状态，不只覆盖 When/Then 路径（详见 WORKFLOW.md §P3 测试设计指导）。

WORKFLOW.md §P3 测试设计指导（新增小节）放完整说明：

> Given 不仅是"数据准备"，也是"系统处于某种状态"——这个状态本身需要被验证。例如：UI 项目验证页面首次加载时展示已有数据；后端项目验证服务启动后初始状态正确；嵌入式项目验证上电后外设处于已知状态。测试设计容易只覆盖 When/Then 触发的交互路径，忽略 Given 隐含的初始化路径。

**不动**：
- 不写死具体场景——用跨领域举例说明通用原则
- 不加 gate 检查——这是设计指导，不是可机器判定的条件

### P4 — orchestrator-log 最低纪律（orchestrator-template.md + state-machine.md）

**现状**：orchestrator-template.md §主 Agent 分阶段落盘规则为"想写就写"。

**改为**：将"想写就写"改为**最低纪律 + 自由追加**：

```
规则：
- 以下事件应追加至少一行：gate 失败、subagent 失败/空返回、流程决策（PAUSED/回退/跳阶）、用户叫停
- 其他事件自由追加，仅追加不编辑不整理
- 不写思考过程、不写文件内容摘要、不写 subagent 返回原文——只写决策和下一步
- 任务从 DONE 重新激活 → 清空后重建；active/PAUSED 恢复 → 追加
```

**具体改动**：

1. orchestrator-template.md §主 Agent 分阶段落盘：替换规则（"必须"→"应"，与 agate 其他不可机器判定的规则一致）
2. state-machine.md 单步函数：在 gate 失败处理步骤中加"追加 orchestrator-log"

**不动**：
- 不加 gate 检查——log 内容不可机器判定
- 不改 log 格式——保持自由格式，只加最低纪律
- 不用"必须"——不可执行的"必须"比"应"更弱，因为"必须"暗示有检查机制

### P5 — self-gate 审查清单增加 A7（设计原则一致性）

**现状**：self-gate 审查清单 A1-A6 全部是内部一致性审查（文档↔脚本、脚本↔测试、变更↔传播），缺少"变更是否符合 agate 自身声明的设计原则"的检查。

v0.9.1 复盘的教训是"方向判断失误"——A1-A6 能保证"改对了"，但不能保证"该不该这样改"。

**改为**：protocol-alignment-review.md 审查清单增加 A7：

| # | 审查项 | 说明 |
|---|--------|------|
| A7 | 设计原则一致性 | 变更是否符合 agate 声明的设计原则？审查员从变更涉及的协议文件中提炼该领域声明的设计原则，判断本次变更是否违反。结论只有 ALIGNED 或 NEEDS_HUMAN_REVIEW（设计原则是指导性的，不是可机器判定的硬规则，不存在 MISALIGNED） |

**具体改动**：

1. protocol-alignment-review.md 审查清单表：增加 A7 行
2. SELF-GATE.md 派发模板审查清单：增加 A7
3. protocol-alignment-review.md 输出格式：汇总表增加 A7 行
4. protocol-alignment-review.md 闭环规则：A7 只有 ALIGNED/NEEDS_HUMAN_REVIEW，无 MISALIGNED

**不动**：
- 不改 A1-A6——现有审查项完整
- A7 不引入 MISALIGNED——设计原则是指导性的，违反原则需人工裁决而非强制修复

## 3. 不做的事

| 考虑过但不做 | 理由 |
|------------|------|
| "直接做"时加测试覆盖约束 | "直接做"= 跳过 agate，agate 无法约束；约束应写在项目 AGENTS.md，不是通用协议 |
| gate 脚本拦截"直接做" | "直接做"时不走 agate 流程，无 .state.yaml，gate 不触发 |
| P3 加 gate 检查"前置状态覆盖" | 不可机器判定，属于设计指导 |
| P8 tag 检查硬拦截 | tag 时机因项目而异，硬拦截会误伤 |
| 三维矩阵（性质×范围×风险） | 12 格太复杂，主 Agent 不会逐格查表；分层判断更实用 |
