# v0.14.0 ADR + 术语表 + 设计原则正式化

## 0. 问题陈述

### 0.1 设计原则未正式声明

v0.13.1 新增 A7（设计原则一致性）审查项，但 agate 协议本身没有声明设计原则。A7 让审查员"从变更涉及的协议文件中提炼设计原则"——无据可依，因人而异，形同虚设。

设计原则散落在各处作为行为规则存在，但没有被抽象为"为什么这样设计"的原则声明。例如"主 Agent 不写产出"是行为规则，背后的设计原则是"隔离性"——但"隔离性"从未被正式命名和记录。

### 0.2 术语无统一入口

agate 术语散落各处：gate、裁剪、机制交叉、声明性改动、BDD、NEED_CONFIRM、SCOPE+……新读者（或审查 subagent）必须通读多个文件才能理解术语含义。v0.13.1 新增的"声明性改动""行为逻辑改动""机制交叉"定义只在 WORKFLOW.md 一个角落，容易被忽视或误解。

### 0.3 架构决策无正式记录

agate 的核心架构决策（为什么用双层角色、为什么 gate 门槛必须机器可判定、为什么 subagent 独立上下文）散落在 LIMITATIONS.md、WORKFLOW.md、dispatch-protocol.md 的叙述中。这些决策的**理由**和**取舍**没有被结构化记录——当有人提议"让主 Agent 也能写产出"时，没有 ADR 可以引用来说明"为什么不允许"。

## 1. 设计原则

1. **增量引入**：新增文件类型，不搬迁现有内容——现有内容留在原处，ADR/Glossary 引用原处
2. **最小集合**：ADR 只记录核心架构决策（<10 条），Glossary 只记录有歧义或跨文件使用的术语
3. **A7 锚定**：ADR 正式化后，A7 审查从"提炼原则"改为"引用 ADR"
4. **不破坏现有结构**：不删不改现有文件中的内容，只新增引用关系

## 2. 修补清单

### P1 — ADR 文件 + 初始记录

**新增**：`agate/adr.md`，所有架构决策按编号排列在同一文件中（agate 核心决策 <10 条，单文件更紧凑，审查员一次读完不需跨文件跳转）。

**初始 ADR 清单**（从现有协议中提炼，非新发明）：

| ADR | 标题 | 来源 | 核心决策 |
|-----|------|------|---------|
| 001 | 隔离性：主 Agent 不写产出 | WORKFLOW.md "主 Agent 不亲自写代码或文档" | 主 Agent 只做四件事（读状态/派发/验 gate/更新状态），不写阶段产出。理由：自审不可信（T005/T006/T019 三个独立案例证明主 Agent 遇困难时倾向自行解决而非触发安全网） |
| 002 | 可判定性：gate 门槛机器可判定 | WORKFLOW.md "门槛机器可判定" | gate 通过/不通过由脚本 exit code 决定，不依赖主 Agent 自我报告或主观判断。理由：主 Agent 的"通过"不可信（C7 规则 / T058 P5 verifier 事件） |
| 003 | 最小约定：不绑定技术栈 | WORKFLOW.md 适用边界 | agate 不硬编码测试框架/语言/部署方式，只定义流程骨架。理由：不同项目技术栈差异大，硬编码会限制适用范围 |
| 004 | 安全网分层：hook 兜底，主动验主流程 | state-machine.md "hook 是兜底，主动验是主流程" | 多层防线（主 Agent 主动验 → pre-commit hook → CI backstop），任何单层失效不致命。理由：T019 证明单层防线会被绕过 |
| 005 | 改动性质决定流程：声明性/行为逻辑/机制交叉 | WORKFLOW.md §改动性质判断（v0.13.1 新增） | 判断"直接做"还是走 agate 的入口是改动性质而非任务规模。理由：v0.9.1 复盘证明"简单"判断本身是最高风险决策 |
| 006 | 双层角色：执行角色 + 评审角色 | role-system.md | 执行和评审由不同角色完成，评审角色 agent≠main。理由：自审不可信（ADR-001 的推论），独立评审提供外部视角 |

**ADR 格式**（参考 ADR 社区规范，简化）：

```markdown
# ADR-{NNN}: {标题}

## 状态

{已接受 / 已废弃 / 已取代 by ADR-XXX}

## 语境

{什么情况下需要做这个决策？}

## 决策

{我们决定了什么？}

## 理由

{为什么这样决定？考虑了哪些替代方案？}

## 后果

{这个决策带来了什么影响？}
```

## 3. 不做的事

| 考虑过但不做 | 理由 |
|------------|------|
| 把现有协议中的设计原则搬迁到 ADR | ADR 引用原处，不搬迁——避免两处维护同一内容 |
| ADR 覆盖所有历史决策 | 只记录核心架构决策（<10 条），细节决策留在现有协议中 |
| Glossary 覆盖所有术语 | 只记录有歧义或跨文件使用的术语，不记自明术语 |
| 改现有协议文件引用 Glossary | 现有文件的术语定义留在原处，Glossary 是补充不是替代 |
| 为每个 v0.x 版本的变更写 ADR | ADR 只记录架构决策，不记录版本变更日志 |
