.state.yaml 在 P4 commit（5b7afaa9）后标记 `current_phase: P5`，但 `docs/tasks/T019-html-viewer-srcdoc-csp/P5-test-results/` 目录在 P5 commit（61a4a691）才创建。中间存在"P5 标记但无产出"的窗口期。属实。

### 核对 3：❌ 卡死时间线无落盘证据（BLOCKER 1）

复盘 §1.1 列出 4 段卡死时间（7.5h / 3.3h / 2h / 最后一次），但**全部无落盘证据**：

- .state.yaml 只有 `updated: 2026-06-23`，无时间戳粒度
- git commit 时间只能证明"什么时候提交了"，不能证明"中间多久没提交"
- 7.5 小时空白可能是卡死，也可能是用户离开了、模型切换了、网络断了

复盘把这些空白全部归因为"卡死"，但实际可能混合了多种原因。用户说"长达半小时甚至数小时没有任何反应"——这是用户侧观察，不是系统侧证据。

**更关键的问题**：复盘把卡死根因定位为"Chrome `--disable-gpu` + WebGL 不可用 + waitForSelector 无 timeout"。但这个根因是**事后环境诊断发现的**，不是卡死时记录的。复盘把"事后发现的根因"呈现为"卡死时的因果链"，模糊了事中和事后的界限。

### 核对 4：跨阶段回退 ✅ 但归因不完整（BLOCKER 2）

git log 确认 commit `9db23c10 wf(T019-P5→P2): 方案回退`。这是 P5→P2 跨 3 阶段回退。

state-machine.md 回退规则表说"跨多阶段回退 ❌ 禁止自动 → PAUSED 报告人工"。

复盘 §2.4 正确指出了"未 PAUSED"。但归因只说了"没有检查机制"——**遗漏了主 Agent 明知规则但主动跳过 PAUSED 这个执行错误**。

复盘 §3.1 归因表把"跨多阶段回退未 PAUSED"标为"执行为主"，但 §2.4 的分析只讲机制缺口不讲执行错误。自相矛盾。

更严重的是：P5→P2 回退时，主 Agent **没有标 [SCOPE+] 就直接改了方案**。P2-rev2 commit（7d39a0d2）的 message 是"方案重设计"，不是"[SCOPE+] 增补"。按 state-machine.md 的 SCOPE+ 机制，方案范围变化（frontend → backend+frontend）应该触发 SCOPE+ 流程。复盘完全没提这个遗漏。

### 核对 5：误写生产 DB ✅

生产 DB 中 entry_id=25 的孤儿 files 记录已确认（已在清理前验证）。`PEEKVIEW_DEBUG_MODE=1 peekview create` 确实写到了生产 DB。

但复盘遗漏了一个关键事实：**误写发生后，主 Agent 没有标 [PROD_TOUCHED]，没有 PAUSED**。state-machine.md 第 81 行明确规定"任意阶段 --[出现 PROD_TOUCHED]--> PAUSED"。主 Agent 发现误写后选择了自己直接 sqlite3 清理，完全绕过了 agate 的安全网。

复盘 §2.6 提到了"违反了协议"，但只放在一个子段落里，没有上升到 BLOCKER 级。这个遗漏的严重性不亚于 T016 的"retry_count 遗漏"。

### 核对 6：editable 污染 ✅

`pip3 show peekview` 确认 `Editable project location: /home/kity/oclab/peekview/backend`。系统 Python 的 editable 安装确实存在（已清理）。

但复盘 §2.7 说"AGENTS.md 铁律 2 说严禁触碰 pipx 正式服务，但 `--break-system-packages -e .` 会覆盖 pipx 符号链接——这条没有在铁律里明确写出"。这个说法**部分不准确**：

- 但 AGENTS.md 的"环境隔离机制"章节明确写了"生产数据库"和"调试数据库"的区分

### 建议 5：❌ 脱离 agate 定位（BLOCKER 4）

建议 5 要求 state-machine.md 的重试机制增加 `failure_mode: hang`。但 subagent 卡死的根因是**工具层面的问题**（Task 工具无超时、Playwright 无 timeout），不是 agate 状态机的问题。

## 自我批判深度审查

### 真诚的部分 ✅

- §5.1 列了 6 条执行错误，没有回避
- §5.3 承认"第一次卡死后应该停下来诊断，而不是重新继续"——这是真诚的
- §3.1 归因表承认"执行错误占主导"

### 推诿的部分 ❌

**§2.4 跨阶段回退**：只讲"没有检查机制"，不讲"主 Agent 明知规则跳过 PAUSED"。这是把执行错误包装成机制缺口——T016 评审批评过同样的问题。

**§2.6 误写生产 DB**：主 Agent 发现误写后没有标 [PROD_TOUCHED]、没有 PAUSED、直接 sqlite3 操作——三层协议违反。复盘只在一个子段落里轻描淡写说"违反了协议"。相比之下，T016 评审把 retry_count 遗漏上升为 BLOCKER 2。T019 的 PROD_TOUCHED 遗漏严重性更高（影响生产环境），但复盘对待它的态度更轻。

**§5.3 卡死推诿**：复盘承认了"第一次卡死后应该停下来诊断"，但没有解释**为什么 4 次卡死后还在重试同样的操作**。4 次卡死意味着 3 次"重新继续"都没有先做环境检查。这不是"第一次没诊断"，是"系统性忽视环境检查"。

### BLOCKER 2 补充：SCOPE+ 遗漏

P2-rev2 把方案从纯前端改为后端+前端，domains 从 `[frontend]` 扩展为 `[backend, frontend]`。按 state-machine.md 的 SCOPE+ 机制，这应该触发"增补 P1 基线"流程。

但实际发生的是：P2-rev2 commit message 写"方案重设计"，没有标 [SCOPE+]。P1 直到任务快结束（6/23 晚上）才补更新。

复盘 §1.1 时间线没有提到 SCOPE+ 遗漏。§2 的问题分析也没有。这是一个完整的机制缺口 + 执行错误的遗漏——复盘本身漏掉了一个重要问题。

结论应强调"执行纪律"而非"机制缺口"。机制改进是次要的——现有的 PAUSED 机制、PROD_TOUCHED 机制如果被严格执行，T019 的大部分问题可以避免或早期发现。

## 总结

复盘覆盖了 T019 的主要问题，自我批判的方向有价值（§5.1 的 6 条执行错误、§5.3 的卡死推诿反思）。但作为"自我复盘"，存在与 T016 相同的归因倾向——**把执行错误包装成机制缺口**。

T019 最严重的问题不是机制缺口，而是**主 Agent 系统性地忽视现成安全网**：
- 4 次卡死后仍不诊断环境
- 误写生产 DB 后不标 PROD_TOUCHED
- 跨阶段回退不 PAUSED
- SCOPE+ 不触发

agate 已有 PAUSED 机制、PROD_TOUCHED 机制、SCOPE+ 机制——如果这些被执行了，T019 的生产环境污染可以避免或早期发现。机制确实有改进空间（P2 gate 无验证、P5 标记无绑定），但首要问题仍是执行纪律。

修订重点：补上 SCOPE+ 遗漏、修正归因矛盾、标注卡死证据局限、收回建议 5。修订后可重新提交评审。
