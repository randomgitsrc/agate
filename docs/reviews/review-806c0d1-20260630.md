## 二、需要关注的问题

### 2.1 hardening-roadmap 是设计文档，但缺少「当前状态」标记

路线图文件头部写了 `状态：设计文档（待评审）`，但正文里 Phase 1 的 7 个子项（P1.1-P1.7）没有任何「已完成 / 进行中 / 待启动」的状态标记。读者无法区分「这是已存在的能力」和「这是待实现的能力」。

例如：P1.4（.gate-result.json）、P1.5（READY 收尾检查脚本化）在路线图里是「新增项」，但 check-gate.sh 已经有部分 P8 检查了。P1.3（CI backstop）在路线图里写成「待实现」，但 `.github/workflows/protocol-consistency.yml` 已有 CI，只是没有 gate backstop 的部分。

**建议**：在路线图表格里加一列「状态」，区分「已有」「部分实现」「待实现」，避免读者把规划当现实。

### 2.2 PeekView 实战评审的 S2（非 agate 任务质量盲区）尚无裁决

实战评审提出 5 个结构性问题（S1-S5）和 4 个执行层问题（E1-E4），路线图 v2 纳入了部分（S1→P1.6 CHANGELOG 检查、S3→P2.10 P2 评审派发强制、S4→P2.7/P2.8 风险等级 + 裁剪条件），但 **S2（非 agate 任务是质量盲区）尚无明确处理**。

S2 的核心问题是：「不走 agate」等于「没有质量保障」，对安全修复尤其危险。评审建议的「复杂度 × 风险」矩阵和「直接做」最低要求（commit message 声明改了什么 + 为什么安全）是有价值的，但路线图里没有对应条目。

**建议**：对 S2 做明确裁决——采纳、修改采纳、或有原因地拒绝。「微任务直接做」这条规则的风险边界需要在协议层写清楚，不能只靠工程师判断。

### 2.3 Phase 2 最关键的子项（P2.1 P6 验收独立化）复杂度被低估

路线图把 P2.1（verifier subagent 独立执行 BDD 验收，主 Agent 不写 P6-acceptance.md）列为 Phase 2 的首项，但这是整个路线图里实现难度最高的一条。

难点不是「如何让 verifier subagent commit」，而是：

1. **主 Agent 如何在不写结论的情况下「读别人的结论判断 gate」**——当前协议里主 Agent 是 P6 的执行者，改成「只读」需要重写 P6 的派发逻辑和 state-machine 转移规则。
2. **verifier subagent 的 dispatch prompt 怎么设计才能防止主 Agent 通过 prompt 预设结论**——路线图写了「prompt 必须包含独立验证，不接受主 Agent 预判结论」，但这条规则本身还是靠 Agent 自觉遵守的——主 Agent 可以在 dispatch-context.md 里放「预判」内容。
3. **git author 验证的实际可操作性**——如何在 Claude Code 里让 subagent 用不同 git author 提交？这依赖平台能力，可能需要 Phase 3 的平台支持才能真正做到。

**建议**：在 P2.1 下补一个「已知难点」节，明确「verifier subagent 独立 commit」的平台依赖，以及在平台不支持时的降级方案（如用文件 mtime 或内容哈希做替代独立性验证）。避免 Phase 2 启动时发现 P2.1 实际上依赖 Phase 3。

### 2.4 裁剪条件阈值的来源需要更显眼的警示

路线图第 8 节「风险 8.6」已承认：「阈值来自 PeekView 一个项目，可能不适用其他项目。」但这个警示在第 8 节，而阈值的具体定义在第 5 节（P2.8）——读者很容易直接用这些阈值，不读到风险说明。

**建议**：在 P2.8 的阈值表格紧接着加一行注：`⚠️ 以下阈值来自 PeekView 实战数据，非理论推导，各项目应根据实际情况调整（见第 8.6 节）`。把警示放在阈值旁边而不是远在第 8 节。

## 三、对路线图的战略判断

### Phase 1（pre-commit hook）是正确的立即行动

性价比最高，不改协议，落地后直接消除「gate 被跳过」和「gate 结果被伪造」两个最高频的失效模式。7 个子项里 P1.1 + P1.3 + P1.4 是核心骨架，其余可以后续迭代加入。

**建议立即启动的最小实现**：
- P1.1：pre-commit-gate.sh，触发条件是 .state.yaml phase 变更或阶段产出文件变更，跑 check-gate.sh，结果写 .gate-result.json
- P1.3：CI workflow 追加 gate backstop（在现有 protocol-consistency.yml 里加一个 job）
- P1.4：.gate-result.json + .gate-history.jsonl 格式定义

P1.6（CHANGELOG 检查）和 P1.7（P6 证据格式）可以在骨架验证后追加。

### Phase 2 建议拆成两批

Phase 2 的 12 个子项混合了「确定性逻辑可实现」（P2.3/P2.4/P2.5/P2.6 状态一致性强制，可以和 Phase 1 hook 一起实现）和「需要协议改动 + 评审」（P2.1 P6 独立化、P2.7-P2.12 流程选择硬约束）。

把 Phase 2 当成一个整体会让「何时开始 Phase 2」变成难题——实际上 P2.3-P2.6 可以在 Phase 1 hook 建好之后立刻追加，不需要等 P2.1。

**建议**：
- Phase 2A（状态一致性强制，P2.3-P2.6）：和 Phase 1 捆绑，在 hook 里实现
- Phase 2B（P6 独立化 + 流程选择硬约束，P2.1/P2.7-P2.12）：单独评审 + 实现，需先解决平台依赖问题

### Phase 3 的规范文档可以现在写

## 五、总体评估

**agate 在本轮完成了一次重要的认知升级**：从「主 Agent 遇到困难时会走捷径」到「主 Agent 出厂设置就是少走步骤」。这个认知升级直接影响防护策略的覆盖范围——从「加固困难场景」变成「所有依赖自觉的环节都需要硬约束」。

hardening-roadmap 把这个认知转化成了一个清晰的执行路径。三层替换（执行 → 产出 → 选择 → 存储）逻辑完整，Phase 独立可落地。

**当前最需要做的三件事，按优先级：**

1. **Phase 1 最小实现**（pre-commit hook 骨架 + CI backstop）：这是整个路线图的基础设施，先把「gate 不被跳过」这一条实现，其他改进都在这个基础上叠加。
2. **S2 裁决**（非 agate 任务质量盲区）：PeekView 实战评审里这条最接近「安全漏洞」——高风险小改动走「直接做」路径，等于没有质量保障。需要在协议层明确底线。
3. **P2.1 平台依赖澄清**（verifier subagent 独立 commit 的可操作性）：在动手实现 Phase 2 之前，先搞清楚 Claude Code / OpenCode 是否支持「subagent 用不同 git author 提交」，如果不支持，降级方案是什么。

**agate 在 v0.4.0 已经是一个可用于真实项目的协议。hardening-roadmap 的落地是从「可用」走向「可信」的关键路径。**
