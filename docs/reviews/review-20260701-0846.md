## 总体判断

好消息是：每一处都可以用小改动修好，且不影响 v1-v3 已经站住的核心设计决策（P2 不可裁 + 双例外口、DESIGN_GAP 兜底、nudge/barrier 诚实标注）。建议 v5 集中修复第十节，不需要重新论证前九节。

## 一、🔴 Critical："3 次失败质疑架构"的注入条件永远不会触发

### 计划怎么说

第十节 `dispatch-protocol.md` 重派 prompt 模板：

> 若 `len(retries[Pn]) >= 3`：在重派 prompt 追加质疑架构假设的提示

### 对照现状

`state-machine.md` 的重试上限表（第 428 行起）：

| 阶段 | MAX_RETRY |
|------|-----------|
| P1/P2/P4 | **3** |
| P3/P5/P6/P7/P8 | **2** |

`dispatch-protocol.md` 第 647 行的重派判定逻辑（这是唯一决定"要不要再派发一次、要不要重新生成 prompt"的地方）：

```
if len(retries[Pn]) < MAX_RETRY:
    带着失败原因重新派发同阶段 subagent   ← 只有这个分支会生成"重派 prompt"
else:
    触发 L2 上溯（不再重派）
```

逐步推演 P1/P2/P4（MAX_RETRY 最大值 = 3，其余阶段更小）：

| 事件 | retries[Pn] | 判定 `< 3`？ | 结果 |
|------|-------------|-------------|------|
| 第 1 次尝试失败 | 1 | 是 | 生成重派 prompt（第 2 次尝试）|
| 第 2 次尝试失败 | 2 | 是 | 生成重派 prompt（第 3 次尝试）|
| 第 3 次尝试失败 | 3 | **否** | **不再生成重派 prompt，直接上溯** |

**"生成重派 prompt"这个动作，只会在 `retries[Pn] ∈ {0,1,2}` 时发生。`retries[Pn] >= 3` 的那一刻，系统已经放弃重派、转去上溯了，根本不会再走到"构造重派 prompt"这一步。** 对 MAX_RETRY=2 的五个阶段（P3/P5/P6/P7/P8）情况更极端——重派 prompt 只在 `retries[Pn] ∈ {0,1}` 时生成，`>= 3` 从未有机会成立。

也就是说：**按第十节给出的伪代码实现，这条"3 次失败→质疑架构假设"的规则在任何阶段、任何场景下都不会被触发。** 这不是边界情况覆盖不全，是这条规则在当前的重试上限设定下没有可达路径。

### 建议

条件需要改成"即将用尽重试预算时注入"，而不是"超过重试预算后再注入"，例如：

```
若 len(retries[Pn]) >= MAX_RETRY(Pn) - 1（即本次已是最后一次允许的重派）:
    注入质疑架构假设提示
```

这样 P1/P2/P4 会在第 3 次派发前提示一次，P3/P5/P6/P7/P8 会在第 2 次派发前提示一次——每个阶段都有且仅有一次触发机会，语义上也更合理（"最后一次机会了，换个思路"，而不是"已经没有下一次了才提醒"）。

## 二、🔴 Critical：`check-gate.sh` P2 多方案检查——变量未定义 + 读错文件 + 分支逻辑不可达

第十节给出的代码：

```bash
if ! grep -qE 'design_trivial:\s*true|follows_existing_pattern:' "$P2_FILE" 2>/dev/null; then
    CANDIDATE_COUNT=$(grep -cE '^###?\s*候选方案|^###?\s*方案[ABC123]' "$P2_FILE" 2>/dev/null || echo 0)
    ...
fi
```

三个问题，同一段代码里：

**1. `$P2_FILE` 未定义。** 现有 `check-gate.sh`（第 14-15 行）只接收 `PHASE` 和 `TASK_DIR` 两个参数，脚本全文没有任何地方定义过 `P2_FILE` 变量（对比 `check-p6-provenance.sh` 第 142 行有显式 `P2_FILE="$TASK_DIR/P2-design.md"`）。照抄这段代码会直接因未绑定变量在 `2>/dev/null` 掩盖下静默判定为"文件不存在"，行为不可预期。

**2. 即便补上 `P2_FILE="$TASK_DIR/P2-design.md"`，字段也读错了文件。** `design_trivial` / `follows_existing_pattern` 是 P1 analyst 在 `P1-requirements.md` 里声明的字段——这是计划自己在第二节写的（"P1 需声明 `design_trivial: true`"），也是 `check-pruning.sh` 检查 2 的实际读取来源（`$P1_FILE`）。但第十节的这段 `check-gate.sh` 代码去查 `$P2_FILE` 里有没有这两个字段。两处对同一对字段的读取来源不一致，其中一处必然是错的（应为 `$P1_FILE`）。

**3. 更根本的问题：这个"跳过检查"分支逻辑上永远不会被执行到。** 按第二节的设计，`design_trivial: true` 或 `follows_existing_pattern: [...]` 一旦声明，**P2 阶段本身就被裁剪**（P1 phases 声明里不含 P2，`check-pruning.sh` 直接放行不裁剪），也就是说这类任务根本不会产出 `P2-design.md`，`check-gate.sh P2` 根本不会被调用。**"P2 被执行 + design_trivial 同时为真"这个组合在设计上不存在**——所以无论把这两个字段放在 P1 还是 P2 文件里查，这段"跳过检查"的分支都是死代码：它想跳过的场景，本来就不会走到这段代码。

### 建议

- 补上 `P2_FILE="$TASK_DIR/P2-design.md"` 定义。
- 删除这段代码里对 `design_trivial`/`follows_existing_pattern` 的跳过判断——它是多余的，因为触发这两个例外口时 P2 阶段整体不会执行，`check-gate.sh P2` 不会被调用，不需要在 gate 内部再判一次。
- 如果保留跳过判断是出于"防御性编程"（万一 P2 被误执行），至少把读取文件改成 `$P1_FILE`，并在注释里说明这是防御性分支、正常路径不会触发。

## 三、🟠 Important：`check-pruning.sh` 检查 7"清理死代码"的论断是错的，且这次提交并未兑现

计划在第二节"改动范围"和"下游同步"两处都写：

> 检查 7 的 `! grep -qw 'P2'` 分支在 P2 不可裁后**恒 false**，清理死代码

### 对照现状（`check-pruning.sh` 第 94-98 行）

```bash
# 检查 7：裁剪理由必须含"跳过风险"评估
if ! echo "$PHASES_DECLARED" | grep -qw 'P2' || ! echo "$PHASES_DECLARED" | grep -qw 'P3' \
   || ! echo "$PHASES_DECLARED" | grep -qw 'P7' || ! echo "$PHASES_DECLARED" | grep -qw 'P8'; then
    if ! grep -qE '跳过风险:' "$P1_FILE" 2>/dev/null; then
        ERRORS="${ERRORS}裁剪声明缺'跳过风险:'评估...\n"
    fi
fi
```

**这个论断和计划自己在第二节建立的设计直接矛盾**："P2 不可裁"不等于"P2 永远不会被裁剪"——恰恰相反，第二节引入 `design_trivial` / `follows_existing_pattern` / `legacy_p2_pruned` 三个口子，就是为了让 P2 在满足条件时**依然可以被合法裁剪**。只要这三个口子中任何一个被使用，`PHASES_DECLARED` 里就不含 P2，检查 7 里 `! grep -qw 'P2'` 这个子句就为真——它和 v0.6 之前一样，是活跃、且必要的（合法裁剪 P2 时仍要求 P1 写"跳过风险"评估，这本身是合理的 nudge，不应该被移除）。**"恒 false"这个判断只有在"P2 从此再也不能被裁剪"的前提下才成立，但这正好是计划自己否定的前提。**

值得注意的是：这次 v4 提交虽然做出了"要清理死分支"的承诺（写进了改动范围表），但第十节给出的 `check-pruning.sh` 检查 2 具体 diff 里**根本没有触碰检查 7**——承诺和第十节的落地代码本身就对不上，只是这次巧合地"没做错事"（因为没删）。

### 建议

删掉这条"死代码"论断（改动范围表和下游同步表里各一处），检查 7 保持不变。

## 四、🟠 Important：DESIGN_GAP 是硬阻断，但没有"确认后解除"的出口——会激励隐瞒而非诚实上报

第十节 `check-gate.sh P7` 新增检查：

```bash
DESIGN_GAP_COUNT=$(grep -cE '\[DESIGN_GAP' "$TASK_DIR/P7-consistency.md" 2>/dev/null || echo 0)
if [ "$DESIGN_GAP_COUNT" -gt 0 ]; then
    echo "GATE P7: ... 需主 Agent 审查" >&2
    exit 1
fi
```

只要 `P7-consistency.md` 里出现过 `[DESIGN_GAP]` 标记，P7 gate 就无条件 `exit 1`（不通过）。问题是：**这份计划没有说清楚"主 Agent 审查完之后，怎么让 gate 变成通过"。**

对照协议里已经存在的同类模式——`[SCOPE+]` / `[SCOPE_RESOLVED]`（`dispatch-protocol.md` 第 630 行附近、`check-scope-resolved.sh`）：产出里出现 `[SCOPE+]` 不会让 gate 直接失败，而是要求主 Agent 在 `P1-requirements.md` 里补一条 `[SCOPE_RESOLVED: 来源文件]` 标记，`check-scope-resolved.sh` 检查的是"每个 SCOPE+ 是否都配对了 SCOPE_RESOLVED"，而不是"有没有 SCOPE+"。这是一套完整的"上报 → 审查 → 标记已处理 → gate 检查配对"闭环。

DESIGN_GAP 目前只抄了"上报"和"gate 拦截"，没有抄"标记已处理"这一环。实际后果：

- implementer 诚实标了 `[DESIGN_GAP: xxx]` → P7 gate 永远 exit 1，任务卡死，除非有人手动改文件删掉这个标记（删了就等于抹掉了审查记录，和协议"审计留痕"的一贯哲学冲突）。
- 唯一能让 gate 通过又不需要手工改文件的路径，是 implementer **一开始就不写 `[DESIGN_GAP]`**——直接在遇到歧义时自己拍板、不留痕迹。

这正好和计划第一节引入 DESIGN_GAP 的初衷相反：第一节说这是为了给"P2→P4 之间过度信任上游"打一道补丁、鼓励 implementer 暴露自己做的隐性决策。**没有解除机制的硬阻断，会训练 implementer 学会不报，而不是学会报。**

### 建议

复用 SCOPE+/SCOPE_RESOLVED 的模式，给 DESIGN_GAP 配一个"已审查"标记，例如：

```
主 Agent 审查后，在 P7-consistency.md 追加：
[DESIGN_GAP_REVIEWED: 主 Agent 已确认该决策合理 / 已打回 P2 补充设计]
```

`check-gate.sh P7` 检查的是"未配对 REVIEWED 标记的 DESIGN_GAP 数量"，而不是"DESIGN_GAP 数量"本身：

```bash
UNREVIEWED=$(( $(grep -cE '\[DESIGN_GAP:' P7-consistency.md) - $(grep -cE '\[DESIGN_GAP_REVIEWED' P7-consistency.md) ))
```

## 五、🟡 Minor：两处"追加"的表述低估了实际改动量

第十节对 `check-gate.sh` 的 P2、P7 改动都描述为"在现有检查后追加"。但对照现状：

- P2 分支现状（第 21-23 行）就两行代码，直接 `exit 2`——"追加"在这之后的任何代码都是不可达的死代码，必须先删掉这个提前 `exit`，把多方案检查插进逻辑前面，再在末尾保留原有的"需主 Agent 自行判定"提示 + `exit 2`。
- P7 分支现状（第 50-59 行）是 `if/else` 两个分支各自 `exit 0`/`exit 1` 收尾——"追加"同样不成立，需要把 DESIGN_GAP 判断插到 `if/else` 之前，或者把整个分支的判定逻辑合并重写。

这不影响设计本身是否合理，但会误导实施者以为这是"在文件末尾加几行"的低风险改动，实际是要重写这两个 `case` 分支的控制流。建议第十节的代码片段直接给出重构后的完整分支，而不是"现有代码 + 追加"的两段式呈现，避免实施时的返工。

## 六、核实无误、可以直接采纳的部分

- **P2 裁剪例外口测试表**（design_trivial / follows_existing_pattern / legacy_p2_pruned 五个场景）：与 `check-pruning.sh` 检查 2 的实际改动代码逐条对应，正则边界（`follows_existing_pattern: []` 因 `\[.+\]` 要求至少一个字符而正确判 fail）验证无误，可直接作为验收用例。
- **P2 裁剪逻辑改动本身**（检查 2 的三段 if-elif 结构）：语法和分支覆盖正确，`legacy_p2_pruned` 优先于两个例外口的顺序合理（过渡期数据不需要重新满足新条件）。
- **实施顺序拆分**（七步独立 commit）：延续了 v3 已确认合理的"小步可回退"节奏，第十节把每步对应到具体 hook 改动，可追溯性比 v3 更好。

## 八、总评

v1-v3 把这份计划的**方向**打磨得很扎实——P2 不可裁的核心论证、双例外口的边界、DESIGN_GAP 作为"P2→P4 信任缺口"的兜底思路，都经得起推敲，v3 评审提出的四项问题也被认真吸收。v4 想往前迈一步，把"设计决策"落到"可以直接照着改代码"的颗粒度，这个方向也是对的。

但第十节暴露的问题指向同一个根因：**第十节的代码是"照着协议的意图重新写了一遍应该怎么做"，而不是"打开对应脚本文件、在真实控制流上做最小改动"**。`$P2_FILE` 未定义、检查 7 死代码论断、"追加"到已经 `exit` 的分支之后——这三处如果作者在写第十节之前实际打开过 `check-gate.sh` 和 `check-pruning.sh` 通读一遍，本可以避免。`retries[Pn]>=3` 的不可达问题更隐蔽一些，需要把 `state-machine.md` 的重试上限表和 `dispatch-protocol.md` 的重派判定逻辑放在一起推演才能发现，但这恰恰是"实施前评审"应该做的事——这也是这份文档存在的意义。

**建议**：这五项修复体量都不大（一个阈值表达式、一个变量定义+文件路径修正、删掉一段错误论断、加一个配对标记、重排两段代码），不需要重新走 v1-v3 的方向性论证。可以在 v5 里一次性修完，修完后第十节就真的具备"可直接实施"的质量了。
