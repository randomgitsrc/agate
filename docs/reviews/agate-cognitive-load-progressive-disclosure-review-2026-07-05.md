# 对「渐进披露重构」的专家评审

## 总体评价

方向对（减少信息量），但把问题过度归因于认知过载，忽视了行为层和激励层的问题。核心假设未经检验，有几个结构性风险点。

---

## 攻击点 1：核心假设不可靠

**假设**：Agent 信息量从 2900 行降到 80 行 → 错误率下降。

**但 T046 的根因不是信息过载，是激励不对称**。retrospective 自己说的：gate 格式是硬约束（不通过 = commit 不了），功能验证是软约束（不验证也能 commit）。Agent 选了满足硬约束。这不是"信息太多了不知道要做什么"，这是"知道要做什么但选了更容易通过的那条路"。

把信息从 2900 行减到 80 行，没有改变激励结构。Agent 还是可以用松散的验收标准通过 gate——卡片再简洁，也没说"你必须打开浏览器看图片渲染了没有"。

**卡片的"常见错误"节能部分缓解**——写上"不要用 DOM 属性替代视觉验证"。但这还是软提醒，和当前 dispatch-protocol.md 里的 P2 最小验证规则一样——**写了但 Agent 过载时不读（或读了不执行）**。缩小信息量让 Agent 更容易看到这条提醒，但看到 ≠ 执行。

结论：**卡片解决的是"不知道规则"的问题，不是"知道但不执行"的问题。** 两者都需要解决，但不能用卡片单独应对后者。

---

## 攻击点 2：自包含 = 近视

P4 implementer 读到 P4 卡片：前置条件、派发模板、产出规格、gate 规则、推进条件。**全都不包含"你实现的这个东西，P6 验收时 vision-helper 要能验证"。**

当前模式虽然是信息过载，但 Agent 在 P4 时确实知道"后面还有 P6，P6 要截图验证"——因为 state-machine.md 和 dispatch-protocol.md 都在上下文里。Agent 可能因为过载而忽视，但至少信息是存在的。

卡片模式：Agent 在 P4 时看到的 P4 卡片，没有 P6 的任何相关信息。Agent 可以实现一个"技术上正确但视觉上无法验证"的东西——因为 P4 卡片不要求它考虑 P6 的可验证性。

**这不是理论问题**：T046 的 implementer buildPathMap 逻辑正确（38 个测试全绿），但整个实现依赖于后端 Content-Type 这个前提——而 P4 卡片不会提到"你需要验证后端 Content-Type"。当前 protocol 也不提——这是 dispatch-protocol.md 的 P2 最小验证规则应该覆盖的——但至少 Agent 见过 P2 规则。去掉全文上下文后，Agent 完全不知道这个前置条件的存在。

**需要的**：每张卡片有一个"下游影响"小节——这一阶段产出如果不能被某下游阶段验证，会产生什么问题。例如 P4 卡片写：`P6 验收依赖你的实现路径——确保路径重写后的 URL 对应的端点行为已验证`。

## 攻击点 3：导航链只覆盖快乐路径

```
orchestrator-template → P0 卡片 → P1 卡片 → P2 卡片 → ...
```

每一步都是："读完执行 → 末端指向下一张卡片"。这是线性快乐路径。

**中断恢复**：Agent 被压缩，重启，读 orchestrator-template，查 .state.yaml → phase=P3 → 读 P3 卡片。但卡片是给"首次进入"设计的——重试 P3 时 Agent 不需要重新派 test-designer（测试用例已经有了），它需要的是修复上一次的失败项。"如果是重试"小节在卡片上（§6.3 设计），但 Agent 从被压缩中恢复时，会读到完整的卡片——包括首次流程和重试流程——需要自己从上下文中区分。

**gate 失败回退**：P4 gate 失败，Agent 要修 P4。这时该重读 P4 卡片（首次模式），但卡片没有"gate 失败时的处理流程"小节。Agent 需要自己判断"我应该只修 gate 报的那个问题，还是要重新派 implementer"。

**裁剪跳阶**：P1 声明跳过 P3、P7。Agent 在 P2 完成 → P2 卡片说"去读 P3 卡片" → P3 卡片第一条说"如果被裁剪，确认后读下一张" → 去读 P4 卡片。三张卡片的跳转，路径正确但脆弱——Agent 可能在跳转链里丢失"我已经做了 P2、正在去 P4"的意识。

**需要**：mapping 表不只是"当前阶段 → 读哪张卡片"，还挂一个"当前状态 → 读哪个模式的卡片片段"。同一张卡片可以有两三个模式入口：`## 首次`、`## 重试`、`## 裁剪跳阶`。

## 攻击点 4：维护成本被低估

你说维护成本"CHECK 9 扩展为卡片-协议一致性检查"就解决了。但这比锚点检查复杂得多：

- 锚点检查是**结构对结构**：check-gate.sh P2 分支用到的 grep 关键词必须在锚点表里出现 → 可以机械化验证
- 卡片检查是**语义对语义**：P4 卡片说"派 review 角色"，review-mapping.md C8 表说"frontend → design-review, backend → review" → 怎么机械化验证卡片和 C8 表一致？

前者可以做正向正则检查。后者本质上是"卡片是否正确提炼了协议文件的内容"，而 **"正确提炼"是语义判断，不是结构性检查**。

**可以做的**：检查卡片引用了哪些协议文件的哪些段，确认那些段确实存在（CHECK 2 的事情）。但不能检查"卡片是否遗漏了那个文件里另一条相关规则"。

**长期风险**：卡片和协议文件会漂移。半年后，协议文件被维护者改过，但卡片没同步——因为一致性检查不够强。Agent 按旧卡片执行，gate 按新脚本检查，两者不一致时 Agent 撞墙且不理解为什么。

**可能的缓解**：卡片**不内联规则**，而是**引用协议文件的精确行范围**作为权威源。卡片退化为"路由表"——告诉 Agent 什么规则在什么文件的什么位置。这样卡片本身不包含规则内容，只是索引。Agent 按卡片索引去读协议文件的精确片段（不是整个文件），规避双重维护问题。

但这又回到"要读外部文件"。权衡点：卡片内联省一次文件读取但引入维护复杂性，卡片引用规避免维护成本但保留了一次外部读取。哪种更优取决于卡片和协议文件的**变更频率**——如果 agate 规则稳定期长（半年不改一次），内联更安全；如果还在快速迭代期（每月都改），引用更安全。agate 当前是后者。

## 攻击点 5：问题归因可能错了

review 把 T046 的问题归因为"规则密度 > 认知容量"。但从 retrospective 的因果链看：

```
P2 未验证 /content 端点 Content-Type   ← 设计遗漏，不是过载
  → P4 只跑单元测试                    ← 实现范围窄，不是过载
    → P5 gate_commands 缺 E2E          ← P2 设计遗留，不是过载
      → P6 被 gate 格式带偏            ← 激励不对称，不是过载
        → vision 报告被反驳             ← 行为模式，不是过载
```

这 5 层只有第一层（P2 缺评审）可以部分归因于"忘了查 C8 表"。其余四层和认知过载的关系很弱——更像是**激励结构倾斜**（gating 格式优先于功能验证）和 **Agent 行为模式**（反驳否定证据、指标替代功能）。

2800 行的认知负担可能加速了这些偏差，但不能单独解释它们。一个信息充足的 Agent 也可能做出同样的选择——因为它选了"更容易通过 commit"的路，不是因为"不知道还有更好的路"。

**一个反例**：T046 的根因定位阶段（章节 162-177 行），Agent 花了约 15 分钟反复验证图片——这不是认知资源不足。Agent 有充足的认知资源去跑 Playwright、解析 PNG 字节、调 canvas API——它选择了相信程序化指标而不相信 vision-helper 的结论。**这不是信息不充分，是判断标准错误。**

## 攻击点 6：Phase Cards 是新概念，需要自己的 gate

卡片引入了一个新的风险面：**Agent 按错误的卡片执行。** 如果卡片里写错了（比如 P4 卡片把 review 角色写成了 architect），Agent 会忠实地做错。

当前没有机制检查"Agent 读到的卡片是否和协议一致"。协议协议一致性检查是 CHECK 1-9——这些都不覆盖卡片。

**需要**：卡片自身的 gate（CHECK 10-class），或者至少卡片文件作为 CHECK 2 的检查对象（引用完整性）。这不是大工作量，但必须纳入范围。

## 正面部分

1. **减少信息量的基本方向是对的**。当前 2900 行确实超过 Agent 的有效处理量。降到 80 行能显著减少"查不到规则"的失误。
2. **映射表 + 指针链的导航设计简洁有效**。不需要 Agent 自己判断该读什么——或者通过 .state.yaml 的 phase 字段，或者通过卡片末尾的显式指针，都是可判定的。
3. **裁剪/重试/SCOPE+ 的非标准情况处理是必要的设计考虑**。card 需要"非快乐路径"，你已经在 §6.2-6.4 考虑了。
4. **540 行的实施量是合理的**。不做 MVP 是正确的——只做一张卡片解决不了问题。
5. **self-gate 触发补、卡片-文件一致性检查的识别是对的**——§6.8 的拆分（触发漏/一致漏/完备漏）很精确。

## 建议修改

| # | 建议 | 理由 |
|---|------|------|
| 1 | 每张卡片加"下游影响"小节 | 防止 P4 近视——Agent 在实现时不知道后续阶段需要什么 |
| 2 | 导航链加状态参数：不只是"phase"，还有"retry/skip/fail" | 同一张卡片的不同入口——Agent 不需要从卡片全文里自己区分模式 |
| 3 | 明确卡片维护策略：内联 vs 引用 | agate 当前快速迭代期，可能引用+精确行范围比内联更安全 |
| 4 | 扩展 CHECK 覆盖卡片文件 | 作为 CHECK 2 的对象 + CHECK 新增项（至少引用完整性、不能有现有文件未涵盖的凭空规则） |
| 5 | 降低对认知过载归因的单变量解释 | §一-§二强调过载是唯一根因，忽略激励和行为因素。允许多变量解释会提高分析的诚实性 |
| 6 | 实施前先判定"规则稳定期" | 如果 agate 进维护期（规则不再频繁变化），内联卡片更安全；如果还在迭代期（每两周改一次），引用+行范围更安全 |
