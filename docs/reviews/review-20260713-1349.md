## 2. 该评审自身的问题（须修）

### 2.1（须修 · 内部矛盾）B-4 违反该文自己的 §0.5 核心论点

§0.5 是这份评审最有价值的部分——它论证："arms race 永远输……A11 在 1 小时内发现 5 条规则的 5 个绕过路径，加第 6 条不会让第 6 次绕过更难"，并把 A11 归类为"**不靠 hook 拦截，靠'修改即审计'的 observability**"、"LLM 行为问题（agate 不该解决）"。

但 §4.4 的 **B-4 恰恰提议给 check-p6-format.sh 加一条针对 A11 字面篡改的检测规则**（检测 `PASS_COUNT`/`FAIL_COUNT` 字面）。这就是：

1. **它自己定义为必输的那类**——检测特定字面的规则，对手换写法（`PASS-COUNT`/`passcount`/重排）即绕过，正是"第 6 条规则"

**修法**：B-4 应从"立即修的 B 类补丁"里删除，并入 A 类第 2 项（audit/observability 重设计）——即"修改即审计"应记录 P6-acceptance.md 的**变更历史**（谁在 gate 通过后改了判定行），而不是猜测某个字面。这才与 §0.5 一致。保留 B-4 会让整份评审在"反对枚举补丁"上自我拆台。

## 4. 追加专项审计（应用户提问）：agate 有没有硬编码 / 语言锁定？是否只能用于 Python/Vue 类项目？

**直接结论：核心工作流不锁语言，agate 不是"只能 Python/Vue"。但确有两处真实的目标语言耦合会硌到非 Python/JS 项目，另有一层常被误当成"语言锁"的工具运行时依赖。都可解，且是有界的客观修复（无对手、非军备竞赛）。**

审计的关键是把两件被 T053 评审 H 类混在一起的事分开：**"跑 agate 工具需要什么" ≠ "被管理项目是什么语言"**。

### 4.1 三层，分清各是什么

**第 1 层 — 工具运行时依赖（bash + git + python3 + pyyaml）：不是语言锁**

8 个 `.sh` 脚本内联 `python3 -c`（实测确认），但它们用 python3 **只是当 YAML/JSON 解析器**，解析的是 agate **自己的元数据**——`.state.yaml`、markdown frontmatter、gate-result JSON，**不是被管理项目的源码**。一个 Rust/Go/C++ 项目照样能被 agate 管，只要跑 agate 的机器上有 bash+git+python3。这跟"用 eslint 得先装 Node"同类——是工具依赖，不是把你的项目锁成 JS。**应文档化，不应称之为 Python/Vue 锁。**（T053 评审 H1/H2/H4 归"不修只文档化"，这一层它归得对。）

**第 2 层 — P3 TDD 红灯（check-tdd-red.sh）：pytest 是铺好的路，他语言要自写适配器（真摩擦，非硬锁）**

脚本头部**明确自我定位**："本脚本是 pytest 的参考实现。agate 是通用协议，不绑定特定技术栈。非 Python 项目应提供自己的 TDD 红灯检查脚本"，并写了**输出契约**（摘要行含 `N failed`/`M error`、import 错误标记、退出码）+ TEST_RUNNER/PROJECT_MODULE 环境变量 + 适配器路径。错误分类其实**已多语言**：`ImportError|ModuleNotFoundError|Cannot find module|ClassNotFoundException|NoClassDefFoundError|unresolved import`（Python/Node/Java/Rust）。

但即便设了 TEST_RUNNER，仍残留两处 pytest 味：
- `$RUNNER -q`——`-q` 标志硬编码（jest 用 `--silent`、go test 无等价）＝ T053 的 M5
- 摘要解析 `grep -oE '[0-9]+ failed'` / `'[0-9]+ error'` 吃的是 pytest 的汇总格式；go/cargo/jest 的汇总词不同 → 匹配不到就 FAILED=0/ERRORS=0 落到兜底 exit 1

所以 P3 **不是硬锁**（有契约+适配器逃生门），但"开箱即用"只对 pytest 成立，其他语言要么运行器碰巧输出 pytest 式汇总、要么手写适配器。这是真实摩擦。

> **这里我修正 T053 评审一处过于乐观的判断**：它 §5 把 G2 归为"不修——已有 TEST_RUNNER，项目自己 export 即可"。但**光 export TEST_RUNNER 不够**——`-q` 标志和汇总格式解析仍是 pytest 专属，非 pytest 运行器即使 export 了也会误判。P3 需要 `TEST_RUNNER_FLAGS` + 汇总正则可配（或正式化适配器返回约定），不是"export 即可"。这一条应从"不修"上移到"可修"。

**第 3 层 — P8 版本文件检测（check-gate.sh:183）：唯一的真硬锁**

`git diff --cached --stat | grep -qiE 'version|__version__|package.json|Cargo.toml|pyproject.toml|go.mod|pom.xml|gemspec|csproj'`，不匹配 → RC=1 → **`exit 1` 硬拦**（实测确认）。覆盖 Python/JS/Rust/Go/Java/Ruby/C#，但 **Elixir(mix.exs)/Gradle(build.gradle)/Haskell(.cabal)/Dart(pubspec.yaml)/PHP(composer.json)/Swift(Package.swift) 全部缺席** → 这些生态的项目走到 P8 发布会被硬拦。这是唯一名副其实的"语言锁"。

**第 4 层 — 模板措辞（M3）**：P0/P2 模板样例写 `npm install`/`pytest`。占位符，影响小，改词即可。

### 4.2 能解决吗？能——有界、客观、不引发军备竞赛

这些全是**配置面**，没有"对手绕过"的概念（跟 §2.1 B-4 那种检测补丁性质完全不同），所以修它们不违反 §0.5 的军备竞赛论。

| 锁点 | 修法 | 成本 |
|------|------|------|
| **P8 版本检测**（硬锁）| 首选：把不匹配从 `exit 1` 降为 **WARNING**（版本文件路径本就"因项目而异"，脚本自己注释也这么说）。更佳：**从 P2-design.md `packages` 字段读项目声明的版本文件路径**再校验——协议已要求 packages 四字段、且 check-gate.sh:172/196 已注明"应从 P2 packages 读路径"，数据现成，只是 gate 没用它，还在瞎猜。备选：`AGATE_VERSION_FILES` 环境变量覆盖。| 首选 ~5 行；数据驱动版 ~20 行（需确认 packages 子 schema 是否含版本路径——存疑，未逐字段核）|
| **P3 TDD 红灯**（摩擦）| 加 `TEST_RUNNER_FLAGS`（默认 `-q`）；汇总计数正则可配 or 正式化"适配器输出契约"；多语言 import 标记保留。把"pytest 铺路"升级为"任意运行器+适配器铺路"。| ~10–15 行 + 文档 |

| **模板措辞** | 样例改中性/多语言并列 | 文档 |

### 4.3 对我上一层判定的影响

这不推翻 §3 的 approve-with-revisions，而是给 T053 评审的 H/M 清单补两条更锋利的结论：

1. **T053 评审把 H 类整体归"不修只文档化"是过粗的**。运行时依赖（python3/git/bash/GHA）归"文档化"对；但 **P3(H3) 和 P8(M6) 是目标语言耦合、会真的弄坏非 Python/JS 项目，应"修"不是"只文档化"**。M6 它已放"可修"（对），但 P3 它放了"不修"（§5 G2）——应上移。
2. **"platform-agnostic"这个立身宣称目前是"P0-P2/P4-P7 成立、P3 半成立、P8 不成立"**。要让宣称名副其实，P8 降级/数据驱动 + P3 加 flags/契约 是必要项——两处加起来 ~20–30 行 + 文档，不是重构。

**一句话回用户**：不是"只能 Python/Vue"——核心流程语言无关；真正的语言锁只有 P8 版本检测一处（硬拦非主流生态），P3 是 pytest 铺路+他语言自写适配器的摩擦。两者都能低成本、无军备竞赛地解决，且 P8 的数据（版本文件路径）协议里已有、只是没被 gate 用起来。
