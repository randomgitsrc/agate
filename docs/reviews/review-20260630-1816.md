## 🔴 阻断级：P2.10 的 git author 区分在 agate 架构里不成立

**这是整个 Phase 2B 最关键的问题，且它与 roadmap 自己的 P2.1「已知难点」是同一个未解决的平台依赖。**

### 实测证据

`check-p2-review.sh`（Task 2）的核心逻辑：
```bash
REVIEW_AUTHOR=$(git log -1 --format='%an' -- "$P2_REVIEW_REL")
ORCHESTRATOR_NAME=$(git config user.name)
if [ "$REVIEW_AUTHOR" = "$ORCHESTRATOR_NAME" ]; then
    echo "...P2 评审 author 是主 Agent，需独立 subagent" >&2
    exit 1
fi
```

我实测模拟了 Claude Code 的真实场景：
```bash
# 主 Agent commit P2-design.md
$ git config user.name "Claude"; git commit P2-design.md
P2-design.md author: Claude

# subagent 产出 P2-review.md，commit
$ git commit P2-review.md
P2-review.md author: Claude   ← 仍然是 Claude
```

**结论**：在 Claude Code 里，主 Agent 和 subagent **共享同一个 git config user.name**。subagent 不会自动用不同 git identity 提交。因此 `REVIEW_AUTHOR == ORCHESTRATOR_NAME` 恒成立，`check-p2-review.sh` 对所有 high risk 任务**永远返回 exit 1**——high risk 任务永远无法 commit。

### 双重矛盾

这个问题不止是「检查永远失败」，它撞上了两堵墙：

**墙 1 — 与 agate 核心原则直接冲突。** `git-integration.md` 明确规定「只有主 Agent 提交，subagent 不 commit」，理由是「subagent 自己 commit 会产生混乱历史，且不知道全局状态」。但 P2.10 要求「P2-review.md 的 git author 必须是 subagent（非主 Agent）」——**这要求 subagent 自己 commit，与核心原则不可同时成立。**

**墙 2 — roadmap 自己已经承认这是未解决的平台依赖。** roadmap P2.1「已知难点」明确写了：「git author 验证的实际可操作性——如何在 Claude Code 里让 subagent 用不同 git author 提交？这依赖平台能力，可能需要 Phase 3 的平台支持才能真正做到。」**P2.10 的实现假装这个难点已经解决了，但它没有。**

### 这意味着什么

P2.10 在当前架构下有三个出路，但都不是「改脚本」能解决的：

1. **降级为非 author 的独立性证据**：不靠 git author（平台不支持区分），改靠其他难伪造的信号——比如要求 P2-review.md 的 Header 含 subagent 自报的 `reviewer_run_id`，且与主 Agent 的产出时间戳交叉核对。这是 roadmap P2.1「已知难点」里提的降级方案（mtime / 内容哈希），应该用在这里。
2. **协议显式要求 subagent 改 git config 后 commit**：但这违反「commit 只由主 Agent 做」，需要先修改核心原则——代价很大，且会引入 roadmap 自己警告的「混乱提交历史」。
3. **承认 P2.10 依赖 Phase 3 平台支持，本阶段不实现**：和 P2.1 一样标记为「待平台支持」，从 Phase 2B 移除。

**建议**：选 1 或 3。P2.10 不应该在「git author 可区分」这个未验证的假设上实现。如果坚持实现，high risk 任务会全部卡死——这比 P2.6 的「误拦正常流程」更严重，因为它是 100% 拦截。

**这个问题与之前 P2.6 移除是同一类**：P2.6 想用 hook 验证「full run vs partial run」（hook 观察不到），被正确移除；P2.10 想用 git author 验证「评审独立性」（平台不提供区分），同样应该被重新设计或移除。

## 🟠 高优先级：check-pruning 的 BDD 提取格式与协议明文矛盾

**位置**：`check-pruning.sh`（Task 1）BDD 计数
```python
m = re.search(r'## 3\. BDD 验收条件(.*?)(?=## 4\.|## 5\.|\Z)', text, re.S)
items = re.findall(r'^- (?:AC|BDD-)\d+', m.group(1), re.M)
```

**问题**：这段代码假设 P1 的 BDD 条目是 `- AC1` 或 `- BDD-1` 这种固定编号格式，且在 `## 3. BDD 验收条件` 标题下。但 `dispatch-protocol.md` L570 明确规定：

> P1→P2 gate：含 ≥1 条 BDD 条件（**BDD 编号格式不固定，按实际格式 grep**）

协议**明文承认 BDD 编号格式不固定**——这正是 P1 gate 不硬编码 BDD 正则的原因（T025 v2 复核结论）。check-pruning 却用固定正则 `(?:AC|BDD-)\d+` 提取 BDD 数，如果真实任务的 BDD 用了别的格式（比如 `- 场景1`、`- Given...`、`1.` 数字列表），`BDD_COUNT` 会是 0。

**影响**：BDD_COUNT 错误会让「裁剪 P2 需 BDD ≤ 10」这个检查失效——BDD 实际有 15 条但提取成 0，会让本该被拦的过度裁剪通过。这是「检查存在但不生效」，比没有检查更危险（给人虚假的安全感）。

**修法**：要么先在协议里正式约定 P1 BDD 必须用固定格式（`- AC\d+`），再实现 check-pruning；要么 check-pruning 的 BDD 计数改为「按实际格式 grep」的宽松匹配——但宽松匹配很难可靠计数。**更现实的方案**：BDD ≤ 10 这个量化条件本身可能不适合 hook 验证（依赖不固定格式的计数），考虑把「裁剪 P2 的 BDD 数限制」从 hook 移到主 Agent 流程层（像 P2.6 那样），hook 只验证 risk_level=low 这个确定性条件。

## 🟡 其他观察

### 复盘触发（P2.12）设计合理

`check-retrospective.sh` 只提醒不中止（exit 0 always），检测三种异常模式（重试超限、SCOPE+、override）。这个设计正确——复盘是「应该做」不是「必须做才能 commit」，用 WARNING 而非 ERROR 是对的。唯一小问题：它扫 `P*.md`（M2 在 check-scope-resolved 改成了 `*.md`，但 check-retrospective 的 SCOPE+ 检测还是 `P*.md`），两处应该一致。

### SCOPE+ 追踪（P2.11）设计可靠

`check-scope-resolved.sh` 的逻辑清晰：产出含 `[SCOPE+]` → P1 必须有 `[SCOPE_RESOLVED]`。M2 修复后扫 `*.md` 覆盖了 dispatch-context.md 等非 P 前缀文件。这个检查是确定性的（grep 标记），不依赖不固定格式，可靠。唯一弱点：它只检查「有没有 SCOPE_RESOLVED 标记」，不检查「标记的内容是否真的对应那个 SCOPE+」——但这是语义问题，hook 做不到，可接受。

### 风险矩阵 + override 字段（Task 6/7）方向正确

把「复杂度 × 风险」二维矩阵写进 WORKFLOW.md、risk_level 字段写进 P1 模板，这是把 S2/S4 裁决落地的正确做法。override 字段（裁剪声明与执行不一致时回写）也是合理的审计机制。这部分协议改动没有问题。

## 总评

**这是 Phase 2B 第一次触及「流程选择硬约束」，方向正确——把裁剪/评审/SCOPE+/复盘从主 Agent 自觉变成机器验证，正是 PeekView 实战评审揭示的最大盲区（主 Agent 默认少走步骤）。Task 1/3/4/6/7 的大部分设计是站得住的。**

**但 P2.10 暴露了一个反复出现的模式**：agate 想验证「评审的独立性」，但「独立性」在当前平台上没有可靠的机器信号——git author 不区分主 Agent 和 subagent，而让 subagent 独立 commit 又违反核心原则。这和 P2.1（P6 验收独立化）是**同一个未解决的根问题**：T026 的「主 Agent 编造结果」和 P2.10 的「主 Agent 自审」都源于「主 Agent 既是执行者又是验证者」，而要打破这个循环需要真正独立的数据源——这正是 roadmap P2.1 标为「平台依赖、待 Phase 3」的东西。

**P2.10 不应该在 Phase 2B 假装这个依赖已解决。** 要么用降级方案（自报 run_id + 时间戳交叉核对，不依赖 git author），要么诚实地标记为「待平台支持」并移出本阶段——就像 P2.6 被正确移除那样。**「知道哪个检查在当前架构下做不了」和「把能做的做对」同样重要。**

最该先定的：C1 的设计方向。在它定下来之前，Task 2 不应该按当前计划实现——否则 high risk 任务会全部卡死，且修复方向是推翻重做而非补丁。
