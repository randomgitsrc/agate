## 二、✅ 已验证：P8 gate chicken-and-egg bug 修复（`7f4648d`）

这是这次 maintainer 自己发现的新 bug（不是我上轮提的），修复思路和之前 P4/P7 的同款 bug 一致：`git diff HEAD~1` 在 pre-commit 阶段永远看不到暂存区的变更，因为此时 commit 还没创建。

**没有只信 commit message，实际用真实 git 仓库复现了修复前后的行为差异：**

```bash
# 构造一个真实的 P8 场景：package.json + CHANGELOG.md 已经 git add 到暂存区，但还没 commit
$ git status --short
M  CHANGELOG.md
A  docs/tasks/T001/P8-release.md
M  package.json

# 用旧版本逻辑（git diff HEAD~1）跑同一个场景
$ bash old-check-gate-p8.sh docs/tasks/T001
exit=1
GATE P8: HEAD~1 无 version 文件变更
GATE P8: CHANGELOG.md 无变更
   ← 完全合规的暂存区变更被误判为不合规

# 用当前（已修复）版本（git diff --cached）跑同一个场景
$ bash agate/scripts/check-gate.sh P8 docs/tasks/T001
exit=2
GATE P8: 脚本化检查通过。仍需主 Agent：①②③④
   ← 正确识别暂存区变更，放行到下一步
```

**确认这是一个真实、严重的 bug**：修复前，P8 gate 在任何正常的 pre-commit 流程下都会对着完全合规的暂存区内容误报"无 version/CHANGELOG 变更"，必然 `exit 1`——意味着修复前的每一次 P8 阶段 commit 理论上都过不了这道 gate。**修复生效，且验证方式（真实 git 仓库 + 真实 staged 内容）比读 diff 更有说服力。**

## 四、⚠️ 需要明确决策：DESIGN_GAP 静默放过漏洞，这次选择"记录"而不是"关闭"

上一轮评审复现过这个漏洞：`implementer.md` 要求 DESIGN_GAP 标记写在 `P4-implementation.md`，但 `check-gate.sh P7` 只扫描 `P7-consistency.md`，如果 architect 忘了转抄，gate 会静默放过。maintainer 当时的修复是在 `architect.md` 里加了一句强制转抄的文字指令（纯文本约束，非 hook 强制）。

这次的测试计划里，**这个已知漏洞被直接写成了一条测试用例**：

```
### R2：v0.6 DESIGN_GAP 配对（cf6cd80）
@test "DESIGN_GAP + REVIEWED 配对可解除" { ... }
@test "DESIGN_GAP 不在 P7-consistency.md → 静默放过（已知风险）" { ... }
```

**这不是错误，是一个明确的取舍**：与其假装漏洞不存在，不如写一个测试把当前行为（静默放过）锁定下来、标注"已知风险"，至少保证以后改代码时不会在不知情的情况下让这个行为"变得更糟"或者"意外变好又意外变坏"。这种"诚实记录局限而不是假装修复"的态度，和 `LIMITATIONS.md` 一直以来的风格是一致的。

**但这里有个问题值得摆出来**：上一轮评审已经给出了一个成本很低、能把这个"自写文件 gate（可伪造）"升级为"部分外部产出核对（较难伪造）"的具体方案——`check-gate.sh P7` 同时扫描 `P4-implementation.md` 和 `P7-consistency.md`，交叉核对两边的 `[DESIGN_GAP:` 数量，数量对不上就报错，不再依赖 architect"记得转抄"。这个改动量很小（多一次 grep + 一次数值比较），而且不违反"机器可判定优先于自写文件信任"这条协议贯穿始终的核心哲学。

**目前测试计划选择了"写测试记录风险"而不是"实施建议的核对方案"，这是可以接受的优先级选择（先把测试基础设施搭起来，再逐个关闭已知风险），但建议在 `R2` 这条测试的注释里明确写一句"这是待关闭的已知风险，不是设计如此"，避免几个月后有人看到这条测试通过，误以为这个场景"已经妥善处理"——**"静默放过"被测试覆盖，容易被误读成"这个行为经过测试、是预期内的"，而不是"这个行为是个洞，测试只是先记录下来"。这个区分对协议自己反复强调的"诚实标注 nudge vs barrier"原则很重要。

## 五、其他小观察

- **附录 A 的"总测试用例数估计"公式本身写法也值得改进**：`17+27+11+15+6+9+8+5+4+8+1 = 111` 这种把 11 个数字用加号连起来手算的写法，本身就是这次出错的直接原因之一——人工加总 11 个两位数，中间任何一个数字改了都要重新手算一遍，出错概率高。建议这张表格改成"自动生成"（比如 `tests/README.md` 的覆盖度表用脚本从 `.bats` 文件的 `@test` 数量统计生成，而不是手写死数字），这本来就是这份计划自己在"进度追踪"一节提到的方向，只是这次的静态版本没有先做到。
- **十二节"待评审问题"的 5 个选项都给出了推荐项 + 理由**，没有丢给读者自己判断，这个习惯很好，符合"给决策依据而不是甩问题"的评审礼仪。

## 七、总评

这轮的两个代码修复（YAML 缩进、P8 chicken-and-egg）都是真实存在的问题，也都用真实场景验证了确实修好了——没有发现新的实现层 bug。测试计划本身的方向和大部分设计细节经得起抽查，写这份计划的人显然是打开脚本读过、认真推演过 CI 环境差异的，不是照抄套路模板。

**唯一的硬伤是自报的数字**——而且这个硬伤本身就是一个有意思的巧合：这份测试计划的核心目标之一就是让协议"从靠人工数分支覆盖率，变成机器自动统计"，但计划文档自己汇报覆盖率数字的方式，恰恰还是最原始的"人工数表格、手动加总"，于是就在自己身上复现了一次它想解决的那类问题。这不影响方向判断，但建议在正式进入 Phase A 实施之前，先把这几个数字核对一遍——用几行脚本比手工数表格可靠，而且以后每次加测试用例，回头跑一下同一个脚本就能知道文档是否又漂移了。
