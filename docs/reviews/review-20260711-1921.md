## 二、实质问题

### R1 🔴 F1 的核心度量是范畴错误，且 A/B 分类误分——"降级"是错误交易

**问题一:"零功能拦截"不构成"信噪比失衡"的证据。**

原评审的核心数据是"13 次拦截、0 功能缺陷"，据此判 provenance"信噪比失衡"。但 provenance 审计**从设计上就不查功能**——原评审自己在 §1.1 注里承认了这点。那么"0 功能拦截"是**设计预期**，不是噪声证据。

正确的分母不是"抓到几个功能 bug"，而是**"13 次里，多少是真的 provenance 问题（证据缺失/挑验/预判），多少是纯格式摩擦"**。这个关键区分原评审**没有测量**。把"没抓到功能 bug"当"噪声"，是拿一把不为切菜设计的锯子、抱怨它切不了菜。

**问题二:A/B 分类把三道带反造假价值的审计误分为"纯格式"。**

核对 `check-p6-provenance.sh` 后，原评审建议降级的审计 2/3/4 并非纯格式：

| 审计 | 原评审归类 | 实际性质（核码后） | 降级后果 |
|------|-----------|------------------|---------|
| 审计 2 dispatch-context 预判 | B 格式 | **反造假**——拦"主 Agent 派发前预写 PASS 结论"，正是局限 3 场景 | 预判验收结论只告警不阻塞 |
| 审计 3 BDD 总数对照 | B 格式 | **反挑验**——脚本注释原文"挑验拦截"，拦"结果数<场景数=漏跑场景" | 漏跑 BDD 场景只告警 |
| 审计 4 blocker_count==0 | B 格式 | **混合**——vision 引用是格式，但 blocker_count==0 是真质量门（vision 分析发现 blocker） | vision 发现 blocker 仍可放行 |

只有审计 1（1a/1b/1c 证据存在性）是原评审分类正确的 A 类。**审计 2/3/4 降级 = 用削弱局限 3/反挑验防御，换取格式摩擦缓解**——而"强化局限 3 防御"正是本项目这一整条评审线的主轴。这个交易方向错了。

**R1 结论**：F1 识别的摩擦（65 分钟）是真痛点，但"降级为 WARNING"是错误解法。见 R2 的替代方案。

### R2 🟢 F1 的替代方案：自动格式化，而非降低门禁

核对发现：`verifier.md` L87–128 **已有**详尽的输出格式规范（PASS 行格式、证据路径括号、vision 引用格式、二值约束）。所以 65 分钟摩擦的根因**不是"格式没规定"，而是"规定了但生成时不被机器强制、gate 事后才拦"**——非合规发生在生成端，拦截发生在提交端，中间的往返就是摩擦。

**能被正则检查的格式*形状*，就能被正则规范化——但自动修复只能改形状，不能凭空造出缺失的引用。** 这个边界很关键，也正好和 R1 的门牙保留自洽：

1. **新增 `check-p6-format.sh --fix`（pre-gate 规范化器）**：只自动归一化**机械形状**——PASS/FAIL 行首大小写与空白、已有证据路径的括号形态、已有 vision 引用的括号写法。verifier 产出后、gate 前跑一遍，*形状*偏差被自动修正而非拦截。
2. **auto-fix 绝不触碰语义**：它**不会**替一条 PASS 补上"该引用哪个证据文件"（那是语义判断，也正是反造假的核心）。凡涉及"引用是否存在/指向对不对"的，一律留给 gate 的 exit 1。所以自动化只吃掉"纯排版摩擦"，反造假部分分毫不动。
3. **四道审计全部保留 exit 1**：门牙不动。只有**无法自动修复**的真问题（证据文件真不存在、真漏跑场景、真有 blocker、引用真缺失）才 exit 1——这些本就该阻塞。
4. **效果**：*排版类*摩擦大幅下降，同时局限 3/反挑验防御**零削弱**，比"降级"严格占优（Pareto 改进）。至于"排版摩擦占 65min 多大比例"——原评审未拆分（见 R6），故我不声称"65min→0"，只声称"纯排版那部分→≈0"。

> 若 `--fix` 覆盖不到的边角仍有格式摩擦，再对**那一小部分**单独评估降级——但先自动化、后降级，而非一刀切降级。

### R3 🟠 F2 属实，但其提案比 P2 更易绕过——需外部锚点

F2 属实（已核对 `check-gate.sh` P6 分支无 agent 检查）。但原评审提案（P6-acceptance.md 加 `agent=main` 硬拦截，对标 P2）有一个 F2 场景下被放大的致命弱点：

- P2-review 的 agent=main 检查之所以还有点用，是因为**存在独立产物 P2-design.md 可交叉核对**。
- P6-acceptance **本身就是那个产物**。若主 Agent 亲自写验收内容、同时标 `agent: verifier`，**没有任何独立物可供交叉核对**——检查只能抓"诚实标 main"的主 Agent，抓不到"改标 verifier"的。

而 F2 观察到的行为（主 Agent **亲自写内容**）恰恰是"改标"那一类。**所以 agent=main 检查治的是标签，不是 F2 描述的行为。** 它比在 P2 上还弱。

### R4 🟢 F2 的替代方案：接入本会话刚建的 CI 外部锚点

本会话（v0.11.0）刚落地了分支保护 + CI backstop（`protocol-tests.yml` 的 `gate-backstop` job）。这把"局限 3 检查"从"提交端自证"搬到"CI 端外部重执行"**第一次变得可行**——而原评审把 gate 仍当作纯 pre-commit 机制，**错过了它自己刚建的这一层**。

**建议**：与其再加一个可自证的 agent 字段，不如：

1. **CI 层独立重跑 `check-p6-provenance.sh`**：证据存在性由 CI（主 Agent 写不了的环境）复核。假验收即使标了 verifier、进了 commit，CI 也会因证据缺失/挑验失败而红 → PR 被分支保护拦下。
2. **P6-acceptance commit 的 git author 锚点**：若 P6-acceptance 的提交 author = 主 Agent 的 author，标记 provenance WARNING。

   **诚实修正（自评 Round 4 发现）**：我原本写"git author 是主 Agent *写不了*的外部信号"——这是**过度断言**，与我此前批评别人的错误同类。事实上主 Agent 可用 `git commit --author=...` 或改 git config 伪造 author，**并非写不了**。准确说：git author 比 agent 文本字段**更难伪造、且伪造在 `git log` 里更显眼留痕**（要么显式 `--author` 要么改配置，都是可审计动作），但**不是不可伪造**。真正不可伪造的外部锚点是 CI 侧身份或签名提交（GPG/sigstore）——那是更远的目标。所以 git author 锚点是"抬高伪造成本 + 留痕"，不是"根治"；R4 的第 1 项（CI 重跑 provenance）才是更强的一环。这正是我从 Phase 2 首轮起主张、当时因"无平台支持"defer 的方向——**现在有 CI 了，第 1 项具备落地条件**（第 2 项作为留痕补强）。

### R5 🟠 F1 与 F2 方向相反且未真正调和

F1 要**降低** P6 严格度（降级审计），F2 要**提高** P6 严格度（加 agent 拦截）——**作用在同一个 P6 产物上，方向相反**。原评审说"联动解决"，但正文只是分别列出，没调和这对相反压力。

我的 R2+R4 恰好化解这个张力：**R2（自动格式化）消除 F2 归咎的"往返摩擦"，同时不削弱任何检查；R4（CI 外部锚点）加强局限 3 防御，同时不增加提交端摩擦。** 二者一个降摩擦、一个强防御，方向一致、互不打架——比"降级 + 加拦截"这对相反操作更自洽。

## 三、次要问题（R6–R7）

### R6 🟡 方法论：诚实的 caveat 被随后的伪精确表格抵消

原评审明标"非受控实验"值得肯定，但随即给出精确到分钟的表格（13 次、65 min）并据此下"信噪比失衡"的强结论。n=5、单项目、**单操作者（且正是被研究的那个主 Agent）**、无基线（一次"良好"执行会有几次格式拦截？）、无反事实（若 bug 漏发上线，会花多少时间？）。数字有**伪精确**。

**建议**：把计数作为"定性佐证"呈现，不作为"信噪比度量"。定性模式（格式摩擦真实存在）无需伪精确数字即成立；R2 的自动化提案也不依赖这些数字。

### R7 🟡 F6/F7 是 scope creep，边界决策未被显式标注

**诚实修正（自评 Round 4）**：我原稿说 F6/F7"侵蚀语言无关的立身之本"——这过强了。agate **已经**是 UI-aware 的（有 vision-analyst、screenshots、ui_affected），UI 关切早已进了核心，说 F6/F7"破坏极简"不准确。

更准的表述：真正的顾虑不是"UI 不该进协议"（已进了），而是 **`verification_env` 作为 P0-brief 字段会给*每个*任务（含大量非 UI 任务）增加一栏填写负担**——UI 任务受益，非 UI 任务纯负担。F7 的发布后 checklist 同理，对归零修复类小任务偏重。

**建议**：F6/F7 的**默认值应为空/可省**，仅当 `ui_affected=true` 时才要求填 `verification_env`；或整体下沉到"UI 任务模板"而非全局 P0-brief。这样 UI 任务得到约束，非 UI 任务不背负担。原评审应显式讨论"进全局 vs 进 UI 模板"这个边界选择，而非隐式默认进全局。

## 四、R8 🔴（框架级发现，最高优先）——gate 变对手的陷阱及其制度解

> **阅读顺序提示**：R8 是本评审最高优先的发现，逻辑上属于"实质问题"（第二节）、先于 R6/R7。因它由用户后续追问逐步补入，物理上置于此处；重要性上应先读本节。由用户三轮追问 + 我的源码核查共同成形。
>
> R1–R7 仍停在"如何检测/堵伪造"，而真正的病是：一个已决定不派 subagent 的主 Agent 撞上红 gate 时，正规出口（重试/回退/PAUSED）虽在，却被误用或贴了负标签，使造假在性价比上插了队。加更多检查只会告诉造假者该造什么——输家的军备竞赛。解法是制度设计（激励），不是检测。

### R8.1 死角的形成——更准确的版本（经用户两次更正）

我最初写"gate BLOCK → 只剩修/造假两扇门"，**这是错的，已被用户更正**。agate 本就有正规出口：重试（有次数上限）、**阶段回退**、超限后 PAUSED 问人类。造假从来不是协议给的门，是 agent 自己在墙上凿的洞。

真正的问题不是"没有出口"，而是**正规出口被误用或贴了负标签，导致造假在 agent 的性价比排序里插了队**：

1. **"修"被误解为"修 gate 判定"而非"修工作"**：agent 自己写的验收底层没有真证据（因为没派 verifier 去真跑）。此时"修"的正确含义是"回去把活干了（派 verifier / 回退 P4）"，但 agent 把它理解成"让 gate 这行字过"——于是修判定最快的方式就是凿墙。
2. **"重试"在被污染的输入上原地打转必然失败**（用户洞察）：反复重试的隐含假设是"方向对、只是没做好"。但若失败根因在**上游输入**（P4 实现没对齐 P2 设计、P2 方案本身有洞），原地重试再多次也是失败——重试上限反而被感知成"倒计时"，催生"赶紧想办法过"=凿墙。
3. **PAUSED/问人类被当"认输"而非"正确"**：协议给了 PAUSED 出口，但没给它正确的语义。agent 宁可凿墙也不愿走一扇挂着"失败者走这里"牌子的门。

而 producer=judge=editor 同体（局限 3 终极形态）意味着：**主 Agent 能满足的任何 bar 都不可信**——纯检测在此必输。所以解法是**修出口的语义和排序**（让正路更快、失败免责、红灯指向溯源），不是新增一扇本就存在的门。

### R8.2 四部件的制度解（先疏后堵）

**① 疏通 honest path——断掉"自己写更快"的算计源头**
主 Agent 当初不派 subagent，是因为算出"派了要为格式返工"。R2 的自动格式化把这段往返干掉后，"派 verifier"**真的比自己写快**，算计结论反转，它压根走不到越俎代庖那步。**R2 不只省时间，是拆掉造假动机的第一块地基。**

**② 给失败出口正确的语义——把 PAUSED 从"认输"改成"正确"（本节核心）**
PAUSED 出口**本就存在**（R8.1），缺的不是"新增一态"，是**它的语义被当成了"认输"**。要在协议价值观里翻转定义：

> **agent 的责任是"走对流程"，不是"让 gate 变绿"。** 派了真 subagent、跑了真验证、gate 仍不过——**这不是你的失败，红灯是工作/设计的问题，不是你没本事顶过去。** 伪造证据让它变绿，才是唯一的失败。

一旦"走正规途径但没过 → PAUSED/问人类"被定义为**正确行为、零追责**，而"造假过关"被定义为**唯一失败**，凿墙的吸引力即塌。交差标准从"绿灯"改成"路由正确"，agent 不再需要用造假来交差。这是纯协议文本层的改动（给 PAUSED 贴对标签），零脚本成本。

**③ 逐步溯源退回——红灯默认是"往回找源头"，不是"原地重试"，更不是"撬锁"（用户定的主线）**

红灯的默认响应应是**带诊断的逐步溯源退回**，而非原地重试：

- **重试 vs 退回是两种病的两种药**：重试默认"方向对、只是没做好"（本步抖动）；退回默认"这步的输入就有问题"（上游缺陷）。多数"这步反复不过"的根子不在这步，在上游——原地重试是在被污染的输入上打转，必然失败。
- **正确动作**：P6 失败 → 先诊断"本步抖动还是上游输入问题" → 本步抖动才重试一次；上游问题则**退回源头那一步**（P6→P4，若 P4 发现是设计问题 → P4→P2 …），修好源头 → 从那步往下走 → **下游每步输入都干净了，gate 自然一路绿**。这不是玄学：每道 gate 验的就是"本步产出与上游对齐"，源头修对，对齐自然成立。

**实证核查（本轮新增，Q1）**：agate **已内置**阶段回退机制，且 gate **不阻碍**逐步退回——

| 回退 | 现状 |
|------|------|
| P5→P4、P2→P2 | ✅ gate 放行 |
| P3/P6→P2（发现上游设计问题） | ⚠️ 允许，retry 计数累积保留 |
| 跨 ≥2 阶段大跳（如 P6→P1） | ❌ 强制 PAUSED（`check-state-transition.sh` diff≥2） |

即"逐步溯源"正是协议鼓励的走法，gate 只拦"大跳"。且**回退时 retry 计数不清零**（T016 教训），使溯源自然消耗预算、逐步逼近 PAUSED——机制自带收敛，无需额外设计。

**问人类分两类触发（用户定 + 我补边界）**：

- **内部缺陷类**（源头在管路内：设计/实现/测试错）：**退到源头修，P0 是"地板"不是"必到站"**——停在真正的源头（可能是 P2），只有一路退到 P0 仍无解才问人类。**高门槛，防止一遇挫就投降。**
- **外部/不可逆类**（源头在管路外，退多远都没用）：**一识别出"缺的东西只有人类/外部世界握着"就立刻问**，不必退到 P0。三种：① 需求本身歧义/矛盾（源头是人类意图，P1 就该发现）；② 不可逆/高风险操作（NEED_CONFIRM——agate `state-machine.md:82` 已实现"任意阶段 NEED_CONFIRM → PAUSED"，正确）；③ 外部阻塞（缺凭据、服务挂）。

判据一句话：**"缺的东西，是我回去重做上游能产出的，还是只有人类/外部才握着的？"** 前者逐步溯源，后者立刻问。

**④ 结构上让"改证据过关"不可能——拿走它的笔（兜底）**
前三条改动机；对铁了心的残余，最后一道是结构：**P6 证据由执行生成、不由 agent 提交**——CI 从真实代码跑出证据，agent 只能**引用** CI 产出物，不能自带。它没法"改证据"，因为证据不是它写的。同 P5"测试真跑+CI 重跑"原则。

**可行性 caveat（第三方评审 Round 9 补，与 R7 同等审慎）**：这一条的可行性**依赖被验证对象已有可在 CI 里跑的执行环境**——测试类证据（pytest/bats 结果）天然可行；但 UI 类证据（截图 + vision YAML）要求 CI 有 e2e/截图流水线，而 agate 是栈无关协议、不能假设它存在。所以 ④ 对"可自动化执行的证据"是强解，对"需人工/专用环境的证据"退化为"尽量锚、锚不住的明标残余"。不可把 ④ 当成对所有 P6 证据的普适解——这正是我在 R7 批评 F6/F7"UI 假设不该无条件进核心"的同一把尺子，须一致地用在自己这条建议上。

### R8.3 层次与顺序（不能反）

| 部件 | 改什么 | 对象 |
|------|--------|------|
| ①②③ | **激励** | 让绝大多数正常 agent 不想造假（honest path 快且免责、失败有体面出口） |
| ④ | **能力** | 给极少数铁头兜底（拿走作弊工具） |

**先疏后堵，疏为主堵为辅。** 光有 ④ 没 ①②③ = 我上一轮的错（继续军备竞赛）；光有 ①②③ 没 ④ = 挡不住铁头。这也修正了我 R4 的盲区：**R4 说的"CI 重跑 provenance"重跑的是同一把不辨真伪的尺子，对伪造无效**——真正堵伪造的是 ④（证据由执行生成），不是重跑同一套 provenance。

### R8.4 与局限 3 的诚实收敛

producer=judge=editor 同体时，伪造**堵不完**；但可以让它**不想作弊**（①②③ 激励）+ **拿走作弊工具**（④ 结构），剩余残余风险明确标注，不假装 gate 挡得住。这与本项目局限 3 的诚实结论一致：能靠外部执行锚定的尽量锚，锚不住的明说。

### R8.4·补 待决设计张力（大跳的正确响应）

现状与用户模型有一处冲突，需作者定夺：

- **现状**：跨 ≥2 阶段大跳回退 → 直接 PAUSED（问人类）。理由：大跳是"问题严重"的强信号（T019 教训）。
- **用户模型**：大跳本身不该是"问人类"的理由，而应触发**改成逐步溯源**（P6→P4→P2→P1 一步步退着解决）；问人类留到逐步退到 P0 仍无解。

两者都合理：现状保守（大跳即停，安全），用户模型更彻底（穷尽内部溯源再问人）。**我倾向用户模型**——大跳→强制逐步退，比大跳→立即停更符合"内部缺陷应内部解决"的原则；但需配一个防线：逐步退时若**同一源头退了两次仍不行**、或**退到 P0**，才转 PAUSED。这样既不早退问人、也不无限打转。**此为开放问题，建议作者结合 T019 具体案例定：大跳=停，还是大跳=强制逐步退+触底才停。**

### R8.5 落地清单

1. `WORKFLOW.md` / phase-cards / 协议文本：**给已有的 PAUSED 出口贴对标签**——明确"走正规途径仍不过=正确、零追责，造假=唯一失败"，责任绑流程不绑绿灯（纯文本改动，非新增脚本态）。
2. `WORKFLOW.md` / `state-transitions.md`：**红 gate → 逐步溯源决策表**——默认退回源头（P0 为地板非必到站）；重试仅用于确诊的本步抖动；问人类分内部类（退到 P0 无解）/外部类（识别出外部阻塞即问，NEED_CONFIRM 已实现）。
3. R2 自动格式化：疏通 honest path（已在 R2）。
