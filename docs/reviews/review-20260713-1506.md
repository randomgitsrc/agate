## 2. Blocking（P1 规格层，实测证据）

### B1 — P1-1 机制错 + 抵触"P2 不可裁剪" + 新开一个绕过口（最严重）

方案 P1-1 目标代码：缺 P2-review.md 时，除非 `.state.yaml` 命中 `grep -qiE 'P2.*裁剪|P2.*trimmed|P2.*skip'` 才放行。三重问题：

1. **查错文件/机制**：真实裁剪声明在 **P1-requirements.md 的 `phases:` 字段**（check-pruning.sh 提取 PHASES_DECLARED），不是 `.state.yaml` 文本。方案自己也标了"需确认字段名"——确认结果是：此机制不存在。
2. **抵触协议硬规则**：check-pruning.sh:46 明写 **"P2 不可裁剪（无例外口）"**，`design_trivial`/`follows_existing_pattern` 只能**简化**（1 个候选方案）**不可省略**。所以"P2 被裁剪"是协议禁止的状态——为一个不可裁的阶段造裁剪逃生门，逻辑上就错了。
3. **新开绕过口（把要修的洞修出一个新洞）**：`P2.*skip` 这种松正则扫的是 `.state.yaml` 状态日志——里面任何一行状态文本（如 "P2 skipped dispatch"）都可能命中，于是本该强制的 P2 review 被状态日志文本豁免。这正是我在 PR#6 F1 标过的"松 grep 扫全文=绕过"同类，方案在这里复现了。

**正确修法（反而更简单）**：P2 既不可裁，P2-review.md 就应**无条件必需**，直接镜像 P1 分支：
```bash
P2_REVIEW="$TASK_DIR/P2-review.md"
if [ ! -f "$P2_REVIEW" ]; then
    echo "GATE P2: P2-review.md 不存在——P2 不可裁剪，方案设计必经独立评审" >&2
    exit 1
fi
# 之后接现有 status/agent 检查（已存在）
```
且 **P2 与 P4 必须分开处理**——check-pruning 只把 P2/P6/（高风险)P3 列为不可裁，**P4 未列**，即 P4 可能合法可裁。所以 P4-review 的"缺失是否放行"应键于 P1 `phases:` 是否声明裁剪 P4，而非 .state.yaml 文本，也不能"P4 分支同理"照搬 P2 的无条件必需。

### B2 — P1-2 片段在最常见情形（0 个标记）直接崩溃，且没照抄旁边现成的正确写法

方案 P1-2：
```bash
need_confirm_count=$(grep -cE '\[NEED_CONFIRM\]' "$P1_FILE" 2>/dev/null || echo 0 | tail -1)
if [ "$need_confirm_count" -gt 0 ]; then ...
```
实测（0 个标记时）：`grep -c` 无匹配会**打印 "0" 并退出 1** → `|| echo 0 | tail -1` 再打印 "0" → 捕获值 = `0\n0` → `[ "0\n0" -gt 0 ]` 报 **`[: Illegal number: 0\n0`**，`set -euo pipefail` 下中断 gate。而 0 个 NEED_CONFIRM 恰是正常任务的常态。

讽刺的是 P1-2 号称"对称 P6"，而 **P6 同一文件里就是正确写法**，应逐字照抄：
```bash
NC=$(grep -cE '\[NEED_CONFIRM\]' "$P1_FILE" 2>/dev/null || echo 0)
NC=$(echo "$NC" | tail -1)     # 关键：独立一行归一 "0\n0"→"0"
if [ "$NC" -ne 0 ]; then ... ; fi
```

### B3 — P1-5 可配正则默认值带字面引号，连 pytest 默认路径都匹配不到（引入回归）

方案 P1-5：
```bash
FAIL_PATTERN="${TEST_FAIL_PATTERN:-'[0-9]+ failed'}"
FAILED=$(echo "$output" | grep -oE "$FAIL_PATTERN" | ...)
```
实测：`${:-...}` 默认值里的单引号是**字面字符**，`FAIL_PATTERN` 实际等于 `'[0-9]+ failed'`（含引号）→ `grep -oE "'[0-9]+ failed'"` 对 "3 failed" **匹配不到**。即：该"可配"改动在没设环境变量时**破坏了 pytest 原本能用的默认解析**——是回归，不是增强。

**修法**：去掉内层引号 `"${TEST_FAIL_PATTERN:-[0-9]+ failed}"`（ERROR_PATTERN 同）。

### B4 — `write_gate_result "FAIL"/"WARNING"` 在 check-gate.sh 里不存在（系统性伪代码）

P1-1/P1-2/P1-6 片段都在调 `write_gate_result "FAIL" "..."` / `"WARNING"`，但 check-gate.sh **没有这个函数**（也未 source 任何提供它的库）——真实惯用法是 `echo "GATE ..." >&2; exit N`。照抄这些片段会 `command not found`。此外 P1-6 依赖一个 "WARNING" 非阻断档位，但当前 gate 结果模型是否支持 WARNING（vs 只有 exit 码）**未经验证**——若不支持，"降级 WARNING"无处落地。

**修法**：要么在 check-gate.sh 定义 `write_gate_result`（并明确 WARNING=不设 RC、继续）并全仓统一；要么把所有片段改回真实的 `echo >&2 / RC=1 / exit` 惯用法（推荐后者，改动小且与现有 17 处一致）。
