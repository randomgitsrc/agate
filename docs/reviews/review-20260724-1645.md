## 需修复项

### 1. [BLOCKING] P2.43：P6 并行验收缺"多包产出如何合并为单一 P6-acceptance.md"的规则

**问题**：

计划里 P6 并行的全部文字是：

> P6-acceptance.md 同理新增按包拆分并行指引，基础设施隔离要求与 P5 一致（验收端口、截图目录 P6-evidence/{pkg}/ 独立）

以及并行安全表里的：

> P6 验收 | 多包各验各的 BDD | 无 | 有——验收端口/截图目录 | ⚠️ 需隔离

这暗示"P6 并行"和"P5 并行"是同一类问题（只需要基础设施隔离），但我实际核对了现有 gate 脚本，发现两者本质不同：

**P5 是 external-output-gate**（`P5-verification.md:74` 原文："主 Agent 验证的是 verifier subagent 的产出（P5-test-results/），而非自己跑的命令结果"）。`check-state-transition.sh:150` 对 P5 只做**目录存在性检查**（`[ ! -d "$TASK_DIR/P5-test-results" ]`），不关心目录内部结构——所以 `P5-test-results/{pkg}/` 这种嵌套天然兼容，不需要额外合并逻辑。

**P6 是 self-authored gate**（`P6-acceptance.md:92` 原文明确写"P6 是 self-authored gate"），而且我核对了全部 4 个相关脚本，全部硬编码单一文件名，不是目录级判定：

```
agate/scripts/check-gate.sh:171        grep -ciE '^\s*- (PASS|FAIL)' "$TASK_DIR/P6-acceptance.md"
agate/scripts/check-p6-format.sh:20-21  basename_check=$(basename "$FILE"); if [[ "$basename_check" != P6-acceptance.md ]]; then exit 0
agate/scripts/check-p6-provenance.sh:36 只在 P6-acceptance.md 存在时运行
agate/scripts/agate-archive-stale-outputs.sh:14  P6) echo "P6-acceptance.md" ;;
```

`check-p6-provenance.sh` 的 BDD 总数审计（v0.20.0 新加的审计 3）比对的是**这一个文件**里 `#### BDD-NN` 覆盖数与 P1 总数是否一致。如果多包 verifier 并行"各验各的 BDD"、各自产出自己的结果，却没有明确谁在什么时候把它们汇总进同一个 `P6-acceptance.md`，会出现两种真实后果之一：

1. 实现时各包各写一个 `P6-acceptance-{pkg}.md`（类似 P4 的 `P4-implementation/{pkg}/` 模式）——但现有 4 个 P6 脚本都不认这个文件名，gate 直接跳过或报错（`check-p6-format.sh` exit 0 静默放过、`check-p6-provenance.sh` 因文件不存在直接不运行），验收结果实际上完全脱离了 gate 校验。
2. 主 Agent 手动把多个并行 verifier 的结果誊抄拼接进一份 `P6-acceptance.md`——这本身就是"主 Agent 转抄 subagent 自撰内容"，如果拼接时漏掉/错位几条 BDD，provenance 审计的计数比对不会发现（它只看总数对不对，不看是否来自正确的包），恰好复现了 P7 阶段"architect 忘记转抄 DESIGN_GAP"那类问题（`check-gate.sh:224` 专门为此加了 P4/P7 交叉核对，P6 目前没有对应机制）。

**对比佐证**：P4 并行早就有了同样的"多包产出"问题，而现有代码已经给出了范式——`check-gate.sh:224` 的 P7 阶段 DESIGN_GAP 交叉核对同时兼容单文件和目录两种模式：
```bash
grep -rh '\[DESIGN_GAP:' "$TASK_DIR"/P4-implementation.md "$TASK_DIR"/P4-implementation/ 2>/dev/null
```
这说明协议维护者之前扩展 P4 支持多包产出时，是同步改了下游读取脚本的。P2.43 的 P4 部分因为有这层已有基础设施，风险确实低（我认可"P4 需要额外约束但基本安全"的判断）；但 P6 目前完全没有这层基础设施，计划文本却把两者一并归为"同理"处理，掩盖了这个差异。

**建议修复方向**（供实施前明确，不代表这是唯一方案）：
- 方案 A：P6 阶段不真正并行验收产出文件本身——各包 verifier 只并行**跑验证**（写 `P6-evidence/{pkg}/` 证据），但最终由**一个**汇总 subagent（或主 Agent）把所有包的 BDD 结果整合进唯一的 `P6-acceptance.md`，并且这一步需要有交叉核对机制（类似 P7 的 P4/P7 DESIGN_GAP 计数比对），确认"每包声称验的 BDD 数之和 = P1 总 BDD 数，且无重复/遗漏"
- 方案 B：扩展 `check-p6-format.sh` / `check-p6-provenance.sh` / `agate-archive-stale-outputs.sh` 支持 `P6-acceptance/{pkg}.md` 目录模式（像 P4 那样），但这样至少要新增 3 个脚本改动 + 对应 bats 测试，会让 P2.43 从"纯文档改动"升级为"脚本改动"，测试计划（当前写的是"P2.43：文档改动，无新增 bats"）需要同步更新

无论选哪个方向，这个决策需要**在计划里写清楚**，不能留到实现阶段由 implementer 临场发挥——P6 是这个协议里被加固次数最多的阶段，不应该是唯一一个"并行怎么合并"没写清楚就进入实现的阶段。
