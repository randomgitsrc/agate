## 2. Blocking（这轮新引入）

### B1(新) — P1-2 用了未定义的 `$P1_FILE`，在 P1 分支 set -u 崩溃

P1-2 片段：
```bash
NC=$(grep -cE '\[NEED_CONFIRM\]' "$P1_FILE" 2>/dev/null || echo 0)
```
实测源码：`P1_FILE` 在 check-gate.sh **只在第 54 行赋值，而第 54 行在 P2 分支内**；case 语句前只定义了 `PHASE`/`TASK_DIR`。P1 分支（21-46 行）**没有 `P1_FILE`**。脚本头是 `set -euo pipefail` → 在 P1 分支引用 `"$P1_FILE"` 会 **`P1_FILE: unbound variable` 直接崩溃**。

这与上一轮同类：片段没在真实分支上下文里跑过。讽刺的是 P1-1 的 **P4 片段自己定义了** `P1_FILE="$TASK_DIR/P1-requirements.md"`，P1-2 片段却漏了——两处不一致。

**修法**：NEED_CONFIRM 查的是需求文件，在 grep 前显式定义：
```bash
P1_FILE="$TASK_DIR/P1-requirements.md"
NC=$(grep -cE '\[NEED_CONFIRM\]' "$P1_FILE" 2>/dev/null || echo 0)
NC=$(echo "$NC" | tail -1)
if [ "$NC" -ne 0 ]; then
    echo "GATE P1: $NC 个未解决的 NEED_CONFIRM 项" >&2
    exit 1
fi
```

### 次要 — P1-2 插入点歧义（"P1 分支末尾"与终止 `exit 2 ;;` 冲突）

P1 分支的末尾就是第 45-46 行 `echo "..."; exit 2 ;;`（approved+agent≠main+锚点 → 交主 Agent 自判）。"在 P1 分支末尾加"若照字面加在 `exit 2 ;;` 之后 → 死代码/语法破坏。NEED_CONFIRM 检查必须插在 **BDD 锚点检查（44 行）之后、终止 `echo+exit 2`（45 行）之前**。规格应写明这个位置，别用"末尾"。

（语义上也对：一个 approved 的 P1 若仍有遗留 NEED_CONFIRM，应 exit 1 拦下，而不是走到 exit 2 交主 Agent。）

## 3. B3 争议裁决：我原判成立，但你的最终代码已顺带修对（功能 moot）

方案"否认 B3"，理由是"bash `${:-'value'}` 中引号是 shell 语法会被剥掉，实测确认"。我在 **bash 5.2.21** 下对两种形式对拍：

| 形式 | 结果 |
|------|------|
| `X="${VAR:-'[0-9]+ failed'}"`（**方案原始代码**：整个 `${}` 在双引号内）| `X` = `'[0-9]+ failed'`（**引号未剥掉**）→ grep 对 "3 failed" **匹配不到** |
| `X=${VAR:-'[0-9]+ failed'}`（**无外层双引号**）| `X` = `[0-9]+ failed`（引号被剥）→ 匹配到 |

关键区别：`${VAR:-word}` 里 `word` 的**引号移除只在展开不处于双引号内时发生**。方案的原始代码 `FAIL_PATTERN="${...:-'...'}"` 整个在双引号里，所以内层单引号是**字面量**——**B3 成立**。"实测确认引号被剥"这个测试，用的是**无外层双引号的形式**，那不是方案的实际代码形式；换句话说，测的不是被测对象。

**但**——方案最终 P1-5 代码"为清晰仍去掉内层引号"，`FAIL_PATTERN="${TEST_FAIL_PATTERN:-[0-9]+ failed}"`，实测（形式C）匹配正常。所以**P1-5 功能上已正确，无残留 bug**。B3 的实际影响归零。

我不纠结这个记分——结论对就行。但把这条讲清楚有价值：**测试要测被测对象的确切形式**（带不带外层双引号是两回事）。这刀两面都割——正是这个原因，我每条都对着片段的确切写法实跑，也正因如此这轮又抓到了 P1-2 的 `$P1_FILE`。
