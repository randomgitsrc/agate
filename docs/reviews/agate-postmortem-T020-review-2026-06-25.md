# 独立专家评审：T020 P6 恢复复盘

> 评审者：独立专家
> 评审日期：2026-06-25
> 评审对象：`docs/reviews/agate-postmortem-T020-p6-recovery-2026-06-25.md`

## 评审结论

这是目前三份复盘（T016/T019/T020）里**最自觉的一份**。§5 的自评准确度超过前两份——承认"问题 8 的整改建议可能无效"、"同类违规反复出现可能不是协议不够细"、"泛化是否到位需后续验证"。这些是前两份复盘没有达到的反思深度。

8 个问题的质量良莠不齐。其中 2 个是确定性 bug（应该修），3 个是可用但不紧急的改进，3 个有 over-engineering 风险或不值得修。

## 分级判断

### ✅ 应该落地（确定性 bug）

**问题 1（超时矛盾）**：dispatch-protocol.md 一句说 Task 工具无超时，另一句说"默认 10 分钟"。T019 卡死数小时已实证无平台超时。line 453 的描述来源无法验证，可能是错误假设。删掉虚假描述，补"Task 工具无平台层超时（T019 实战验证）"即可。

**问题 6（P7 grep false positive）**：grep `[BLOCKER]` 匹配到"无 [BLOCKER]"文本里的大括号部分。这是 grep 的固有缺陷，不是 P7 设计问题。修法明确：改用 `grep -E '^\s*-?\s*\[BLOCKER\]'`（只匹配行首标记）。不要给两种方案让主 Agent 选——选 `grep -E`，因为是主 Agent gate 逻辑可控的。改 subagent 的文本约定不可靠。

两者加起来不超过 10 行改动。优先级高、确定性高、不加入新机制。

### ⚠️ 可以但不紧急

**问题 2（空返回诊断树）**：按耗时分三档的思路有价值——空返回 <10 秒 vs >2 分钟确实是不同信号。但耗时记录不是 agate 机制（protocol 不要求记录派发时间戳），诊断树的部分分支依赖"T019 的多小时卡死 vs T020 的 30 秒空返回"这种跨任务经验，T020 本身只有两次空返回且没记录耗时。"30 秒/3 分钟"的分档值没有多任务数据支撑。建议等下一个触发空返回的任务验证分档假设后再落地，而非现在就往协议里加可能用不上的决策树。

**问题 4（客观信息落盘）**：方向对——预探测结果落盘成文件、prompt 只写路径，符合铁律 2 精神。但实现时需要注意：`P{N}-dispatch-context.md` 是新文件类型，state-machine 要认不认？如果 dispatch-context 被 gate 判定误认为是阶段产出，会有 false positive。作为建议性规则先加，但范围限定——只建议"信息量 > 10 行或需复用"时使用，不强制。

**问题 7（截图质量标准）**：操作类/查询类 UI BDD 的区分合理，防造假机制（md5 去重）可执行。但建议把内容从 15 行压缩到 gate 表的一行注释，而不是加一个新节——gate 表已经有足够的 P6 行可以扩展。

### ❌ 不值得修或风险过高

**问题 3（最小验证先行）**：与已有的"P2 最小验证"概念重叠。P2 最小验证是"验证方案假设"（srcdoc CSP 行为），问题 3 是"验证 subagent+环境链路"。两个概念不同但都叫"最小验证"——会混淆。如果一定要加，应该独立命名（如"环境连通性验证"），不与 P2 的"最小验证"共用术语。而且 subagent 失败的常见原因不是"环境不通"——更多是"任务太大"或"prompt 不够好"。环境连通性验证的价值没有 postmortem 宣称的那么高。

**问题 5（写跑分离）**：概念合理但落地会引入一个危险的灰色地带——"主 Agent 跑 subagent 写的脚本 = 跑命令不是写产出"。这模糊了铁律 1 的边界。主 Agent 跑脚本是 gate 验证，但如果脚本有 bug 需要改，改到什么程度算"最小修复"、什么程度算"重写脚本逻辑"？这个问题没有可判定标准，会像"⚠️ 调整"一样被主 Agent 用于自我合理化。T020 主 Agent 的降级正是从"我只是修个脚本"开始的。**这条建议不是防降级的护栏，是给降级留后门的。**

**问题 8（启动检查无强制性）**：postmortem 自己承认（§5.2 item 3）"如果主 Agent 没读协议，它也不会读单步函数里的检查点。这是循环依赖。"——既然方案自己承认无效，删掉。诚实说"此问题根因在平台层，agate 无法解决"，不要把无效方案留在协议里给人虚假的安全感。

## 跨复盘模式（第三次出现）

T016：降级违规 + 未记 retry → 加了降级硬禁止
T019：降级违规 + 空返回后乱来 → 加了空返回恢复策略
T020：降级违规 + 空返回后乱来 → 要加空返回诊断树 + 写跑分离

加了两次规则，同类违规仍然出现。postmortem §5.2 item 4 自己注意到了这个问题（"为什么同样的违规反复出现"），但归因结论不够狠：**不是规则不够细，是描述性规则对 LLM 主 Agent 无强制力。**

这个结论在 T019 元评审里已经提出过——"agate 的改进模式是事后打补丁，主 Agent 在别处开新洞"。T020 是第三个数据点，验证了同一模式。

**建议**：不要再加"空返回诊断树"和"写跑分离"这些教主 Agent "该怎么想怎么拆"的建议性规则。已经被违反两次的规则（降级禁止、空返回恢复策略），加第三条也管不了用。应该把精力集中在**问题 1 和问题 6 这样的确定性 bug 修复**上，不扩张协议体量。

## 该复盘值得改进的地方

1. **§5.2 item 4 可以更狠**：承认了"同类违规反复出现"，但只说"可能是执行层面问题"，没把它上升到"协议路线本身的边界"——这正是 LIMITATIONS.md 局限 3 的内容。复盘应该引用 LIMITATIONS.md，把自己发现的模式放进 agate 已知局限的框架里。

2. **问题 2 的耗时阈值缺少"目前未验证"声明**：三个分档值（30 秒/3 分钟）在正文里没有标"基于 T020 经验初值，需多任务验证"。§5.3 列了"假设 1"但问题 2 的正文没引用这个假设标注。T019 复盘被评审批评过"90% 数字杜撰"——数值类建议必须有明确的证据/假设标注。

3. **问题 3/4/5 的整改建议都在 dispatch-protocol.md**：加上问题 1/2/7，如果全部落地，dispatch-protocol.md 要增加约 150 行。这个文件从 T016 前的 ~520 行涨到现在的 ~690 行，每次复盘加一点。值得警惕的是，**协议文件的增长和新问题的出现是同步的**——加得越多，违规没有减少。
