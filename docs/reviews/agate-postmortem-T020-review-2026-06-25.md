# 独立专家评审：T020 P6 恢复复盘（修订版）

> 评审者：独立专家
> 评审日期：2026-06-25
> 评审对象：`docs/reviews/agate-postmortem-T020-p6-recovery-2026-06-25.md`
> 修订说明：初版评审过于保守，否定了多个有真实价值的改进。本版重新评估每个问题，区分"合规规则"（加了也不一定遵守）和"能力规则"（给了主 Agent 现在没有的工具）。

## 评审结论

这是三份复盘里最自觉的。§5 自评准确度超过 T016/T019——承认"问题 8 的方案可能无效""同类违规反复出现可能不是协议不够细""假设 1-4 全部未验证"。这些反思是真实的，不是走过场。

8 个问题中：2 个确定性 bug 应该立即修，4 个是**能力补充**（给了主 Agent 现在没有的操作工具，不是合规规则），2 个不值得修。核心判断转变：**"加规则解决不了合规问题"这个论点只对合规规则成立，对能力规则不成立。** 空返回诊断树、客观信息落盘、写跑分离——这些不是"要求主 Agent 做什么"，是"给主 Agent 新的做法选项"。主 Agent 没有这些工具时只能降级，有了才可能不降。

## 逐条评估

### ✅ 立即修（确定性 bug，不涉及新机制）

**问题 1（超时矛盾）**：dispatch-protocol.md:313 说"Task 工具本身无超时参数"，:453 说"task 工具设 generous timeout（默认 10 分钟）"。T019 卡死数小时已实证无平台超时。line 453 来源无法验证。删掉错误描述，改为引用 line 313 的正确结论。

**问题 6（P7 grep false positive）**：grep `[BLOCKER]` 匹配到"无 [BLOCKER]"文本。改用 `grep -E '^\s*-?\s*\[BLOCKER\]'`（只匹配行首标记）。不要给两个方案让主 Agent 选——选 `grep -E`，因为是主 Agent gate 逻辑可控的。改 subagent 文本约定不可靠。

### ✅ 落地（能力补充，给了主 Agent 现在没有的工具）

**问题 2（空返回诊断树）**：初版评审说"阈值未验证所以不加"——这个逻辑不对。诊断树的价值不在具体阈值，在于**给了主 Agent 一个分析框架**。当前协议只说"分析失败原因"，主 Agent 不知道从哪分析。诊断树说"先看耗时、再看任务复杂度"——这是操作性指引，不是合规规则。

阈值（30s/3min）标注为"经验初值，需多任务验证"即可，T020 自己在 §5.3 假设 1 已经标注了。不因为阈值没验证就连框架都不加——那是因噎废食。

落地建议：加诊断树框架，阈值标注经验初值。

**问题 4（客观信息落盘）**：这是本复盘最有价值的建议。主 Agent 派发前查证的客观事实（环境状态、URL、选择器、参照文件）目前只能写进 prompt，违反铁律 2 精神且不可复用。落盘成 `P{N}-dispatch-context.md`，prompt 只给路径——完全符合铁律 2，且解决了 T020 中 50 行 prompt 膨胀的实际问题。

初版评审说"可以但不紧急"——这不对。这不是优化，是**铁律 2 的补全**。当前"只传路径不传内容"只覆盖了阶段产出文件，没覆盖主 Agent 自己查证的客观信息。这个缺口从 T016 就存在，T020 只是第一次被显式提出。

落地建议：加「客观信息落盘」子节，>10 行或需复用时落盘。不强制（信息量小时可写 prompt）。

**问题 5（写跑分离）**：初版评审说"给降级留后门"——这个判断太粗暴了。复盘的建议明确区分了三个层次：
- 主 Agent 跑 subagent 写的脚本 = 跑命令（gate 验证，合法）
- 主 Agent 改参数/超时/选择器 = 最小修复（仍属于跑命令）
- 主 Agent 重写脚本逻辑 = 降级（违规）

这个区分是可操作的：改常量值（timeout、selector、URL）是最小修复，改控制流（if/else 结构、循环逻辑、数据处理）是重写。界限清晰。

写跑分离解决的是真实问题：subagent 既写又跑又调试，几轮试错后上下文窗口满了 → 空返回。拆开后 subagent 只写不跑（上下文不被试错占满），主 Agent 只跑不写（gate 验证，不降级）。

落地建议：加写跑分离模式，明确"改常量=最小修复，改控制流=降级"的界限。

**问题 7（截图质量标准）**：md5 去重是可执行的防造假机制，操作类 vs 查询类 UI BDD 的区分合理。T019 和 T020 都出现了截图问题。初版说"压缩到 gate 表注释"太轻视了——截图造假直接影响 P6 gate 可信度。

落地建议：加截图质量标准。操作类 BDD 截图必须互不相同，查询类 BDD 可不截图。

### ⚪ 不落地

**问题 3（最小验证先行）**：概念合理但与问题 2 的诊断树有功能重叠——诊断树的"< 30 秒空返回"分支本质上就是"最小验证先行"的隐含版。如果问题 2 落地，问题 3 可以暂不加，等更多数据验证"环境连通性验证"是否真的是空返回的高频原因（T020 的两次空返回更可能是任务太大，不是环境不通）。

**问题 8（启动检查强制性）**：复盘自己在 §5.2 item 3 承认"如果主 Agent 没读协议，它也不会读单步函数里的检查点。这是循环依赖。"——方案自己承认无效。诚实记录为平台层限制，不加无效方案。

## 跨复盘模式

T016→T019→T020，三次降级违规。初版评审的元判断（"加规则解决不了合规问题"）方向对，但推论错了。

正确的推论是：**合规规则（"必须走 PAUSED""不得降级"）加了不保证被遵守，这是 LIMITATIONS.md 局限 3 的内容。但能力规则（"空返回时怎么诊断""预探测结果怎么落盘""脚本怎么拆写跑"）不是合规规则——它们是给主 Agent 操作工具，让主 Agent 在遇到困难时有路可走，而不是只有降级一条路。**

T020 的降级根因之一是：主 Agent 空返回后不知道该怎么调整，协议只说"调整策略重派"但没说怎么调整。诊断树、客观信息落盘、写跑分离——这三个补的正是"怎么调整"的能力缺口。不是更多禁止规则，是更多操作选项。

## 对复盘本身的质量评价

§5 自评是这份复盘最好的部分。特别是：
- §5.2 item 4 指出"同类违规反复出现"——这是跨复盘模式识别，前两份复盘没做到
- §5.2 item 3 承认问题 8 方案可能无效——诚实，不是装作有解
- §5.3 列了 4 个未验证假设——自律，不把假设当结论

不足：
1. 问题 2 的阈值没有在正文里标注"经验初值"——§5.3 假设 1 标了但问题 2 正文没引用
2. 问题 5 的"最小修复 vs 重写逻辑"界限没有给出可判定标准——补上"改常量 vs 改控制流"即可
