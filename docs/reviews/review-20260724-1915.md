## 需修复项（非阻塞）

### 1. [NEEDS_FIX] agate-extract-context.sh 对 P4/P5/P6 的按包并行目录结构无感知，P6 会静默给出误导性的"0"

这正是我在计划评审 R3 阶段提过的 SUGGESTION（"提取脚本 P5/P6 行未明确是否递归扫描包级子目录"），当时标为 SUGGESTION 是因为"实施时会被测试暴露"；`aec5219` 的修复记录里也写了"R3-2 提取脚本 P5/P6 递归扫描 | SUGGESTION | 实施时 bats 测试覆盖单文件和按包子目录两种布局"——但实际交付的 EC.10-13 测试**全部只覆盖了单文件（串行）布局**，没有一个测试用了 P2.43 新引入的 `P4-implementation/{pkg}/`、`P5-test-results/{pkg}/` 目录结构。我手动构造了并行包场景验证：

```
$ mkdir -p /tmp/pkgtest/P4-implementation/pkg-a /tmp/pkgtest/P5-test-results/pkg-a
$ echo "implementation_dir: pkg-a/src" > /tmp/pkgtest/P4-implementation/pkg-a/notes.md
$ echo "  failed: 2" > /tmp/pkgtest/P5-test-results/pkg-a/unit.md

$ bash agate-extract-context.sh P5 /tmp/pkgtest
（空输出——implementation_dir 完全没提取到）

$ bash agate-extract-context.sh P6 /tmp/pkgtest
- P5 failed 参考: 0（仅供参考，gate 以主 Agent 实跑为准）
```

P5 的情况是"提取不到就不输出"，还算诚实；**P6 的情况更麻烦**——真实 failed 数是 2（在 `P5-test-results/pkg-a/unit.md` 里），但脚本硬编码只读 `P5-test-results/unit.md`（不存在），`grep -c` 找不到匹配就返回 0，于是打印出"**0**"——这不是"没有数据"，而是"看起来像是查过、确认为 0"的假象。虽然标注了"仅供参考，gate 以主 Agent 实跑为准"，把真正的把关责任放在了主 Agent 亲自跑 gate 上（所以不构成阻塞级风险），但一个数字上的假阴性（真实 2 → 显示 0）比"完全没有这一行"更容易误导匆忙看一眼 dispatch-context 的主 Agent。

**建议**：P5/P6 提取逻辑改成同时检查 `P4-implementation.md`/`P4-implementation/*/` 和 `P5-test-results/unit.md`/`P5-test-results/*/unit.md` 两种路径（类似 `check-gate.sh:224` 里 P4/P7 DESIGN_GAP 交叉核对已经在用的 `grep -rh ... file dir/ 2>/dev/null` 双路径模式），且并行时应该对每个包分别统计再求和/合并，而不是只读某一个包的文件。这个不阻塞——因为只在"P2.43 并行"和"P2.42 提取"两个新功能被同一个任务同时用到时才会触发，而且有"仅供参考"的免责声明兜底——但既然计划里明确承诺了要测两种布局，这个承诺目前没有兑现，建议下一个小版本补上。

`grep -rn "P4-review" agate/scripts/*.sh` 无结果——`check-gate.sh` 里没有任何针对 `P4-review.md` 的 agent 字段检查（P1/P2 有，P4 没有）。这不是 v0.22 引入的新问题（P4 review 汇总机制本来就存在，只是这次补充了更详细的步骤说明），修复项 1 的措辞调整后也已经如实反映了这一点（"与 P2 评审同规则"，没有再声称 P4 自己有硬拦截）。但既然文档现在明确写了"必须非 main"，这条目前完全是纪律层面的自觉遵守，没有任何机制兜底——建议记在 `docs/hardening-roadmap.md`（本次 diff 里已经在动这个文件）里作为一个待评估项，而不是现在就要求补 gate 脚本。

### P6 的 failed 提取假设的"  failed: N"格式没有在别处被确立为标准

`P5-verification.md` 的 N5 残余风险缓解措施用的签名检测是 `grep -cE '^(PASSED|FAILED|passed|failed|ok|not ok)'`（原始 test runner 输出格式），而提取脚本假设的是 `^\s*failed:`（人工标注的字段格式）。两者不冲突（可以同一份 unit.md 里既有 test runner 原始输出用于签名检测，也有一行人工标注用于快速参考），但目前协议文档里没有任何地方明确规定 unit.md 必须同时包含这两种格式。这不影响本次评审结论（历史遗留的格式不确定性，非 v0.22 引入），只是顺带记录，供以后统一 unit.md 格式规范时参考。
