## 2. Blocking — P1-1 应从 v0.13.0 撤下（不是再改键，是撤下）

第 4 轮我说 P1-1 的 P4-review 应"键于 C8 域触发"。这轮我把 exit 2 的语义查到底，结论更进一步：**P1-1 整条（P2 与 P4 的无条件必需）应从 v0.13.0 撤下，有效内核并入 P3-1（C8 机制化）。**

三段证据链：

**(a) C8 表里 P2/P4 评审是域触发，无 catch-all。** review-mapping.md 全表：P4 后评审只有 backend/frontend/mcp/security 四域触发；唯一的"任意|high"兜底插在 **P2**（产 P2-review），不在 P4。P2 后评审只有 frontend / 任意-high / 业务方向不明 触发。所以：
- domain=data/infra/devops/cli/算法 等**表外任务**→ 不触发 P4 评审 → 无 P4-review.md
- backend+low+业务方向明确 → 不触发 P2 评审 → 无 P2-review.md

**(b) 当前缺失时是 exit 2，而 exit 2 不阻断提交。** 实测 check-gate P2 分支：缺 P2-review.md → fall-through 到 exit 2。pre-commit-gate.sh:236 对 `GATE_EXIT=2` 只打印"需主 Agent 手动判断"**不 exit** → 提交放行。所以现状是"软执行点"（主 Agent 自判），不是硬拦、也不是静默 0。

**(c) P1-1 无条件 exit 1 会误拦 (a) 里的表外任务，且与本方案 P1-3 自相矛盾。** P1-3（同方案）加的警告明写"表内触发是**最低要求**"——即承认存在表外任务；而 P1-1 却要求**所有**任务都有 P2/P4-review.md。一个 data 迁移或 infra 任务（合法软件任务、无 C8 P4 评审映射）会被 P1-1 `exit 1` 误拦，但协议从没给它映射过评审。P1-1 与 P1-3 直接打架。

**结论**：
- P1-1 无条件必需 = 错，**撤下**。
- 但 (b) 说明它追的东西有真内核：C8 **映射了**评审的任务，仍可能靠 exit 2 软放行而跳过评审——这与 agate"不信主 Agent"的立身主题一致，是真软点。
- 正确修法 = **当且仅当 C8 对该任务 domains/risk 映射了评审时，硬要求对应 review 文件存在**。这需要 gate 解析 domains 再查映射表——正是 **P3-1（C8 机制化）** 的活。
- 所以：**P1-1 从 v0.13.0 P1 撤下，"域触发硬要求评审"并入 P3-1。**

> **自我更正（承前）**：我第 2 轮 B-1 把 exit 2 说成"绕过"、并建议无条件必需——(b) 证实 exit 2 确是软点（B-1 内核对），但无条件必需的修法错（会误拦表外任务）。第 4 轮我改口"键于域触发"，方向对但仍留在 v0.13.0；这轮定论是**撤到 P3-1**，因为域触发判定本身就是 C8 机制化，不该拆成半截塞进 quick-fix。
