## 一、🟠 Important：`retries[Pn] >= 3` 的错误阈值在文档里仍有两处未同步

### 已修好的地方（第十节，第 355-366 行）

```
⚠️ 关键：retries[Pn] >= 3 在现有重试上限下永不可达 ...
必须用 >= MAX_RETRY(Pn) - 1（即本次已是最后一次允许的重派）
```

我重新推演了一遍：P1/P2/P4（MAX_RETRY=3）在 `retries=2` 时构造第 3 次派发的 prompt，`>= MAX_RETRY-1` = `>=2` 恰好在这一刻成立，且只成立这一次；P3/P5/P6/P7/P8（MAX_RETRY=2）在 `retries=1` 时构造第 2 次派发的 prompt，`>=1` 同样恰好成立一次。**修复后的公式在数学上是对的**，每个阶段都有且仅有一次触发窗口。

### 没修的地方

**第四节（"扩写 investigate.md"，第 174 行）**——这条规则最初提出的地方：

```
- `dispatch-protocol.md` 重派 prompt 模板加：`len(retries[Pn]) >= 3` 时注入"你已是第 N 次尝试..."
```

**第十节的汇总表（第 259 行）**：

```
| investigate.md 扩写 | ... | "3 次失败质疑架构"落 dispatch-protocol.md 重派 prompt 模板（`len(retries[Pn]) >= 3` 时注入...）| 步骤 4 |
```

这两处都还是原始的、已经被证明"永不可达"的 `>= 3`，和第十节第 358 行给出的正确公式 `>= MAX_RETRY(Pn) - 1` 直接矛盾。

**影响**：第四节在文档里排位更早，叙述上更像是这条规则的"权威定义"，第十节的 hook 明细更像是"实施细节展开"。如果实施者（人或 subagent）只读第四节就去改 `dispatch-protocol.md`，会照抄回错误的 `>= 3`，把这一轮刚修好的 bug 重新引入；只有完整读到第十节才会发现真正该用的公式。两处表述不一致本身就是一个需要人工排查"到底以哪个为准"的成本。

### 建议

全文搜索 `>= 3` 和 `retries[Pn]`，把第 174 行、第 259 行统一改成 `>= MAX_RETRY(Pn) - 1`（或直接改成"临近该阶段重试上限时"这种不写死数字的表述，从根上避免以后阶段的 MAX_RETRY 值调整时又要全文搜一遍）。顺带一提：第 171、182、243 行"3 次失败"作为小节标题/简称多次出现——这只是个名字，不影响执行逻辑，但既然实际触发时机因阶段而异（2 次或 3 次），继续叫"3 次失败"容易和刚修好的公式产生认知冲突，改成"临近重试上限质疑架构"之类的中性说法会更准确，不强制要求现在改。

## 三、P7 分支代码风格的一个次要建议（不算 bug，实测行为正确）

第十节给出的 P7 重写代码用了 `! grep ... && ! grep ... && { ... } && exit 0 || exit 1` 这种链式写法。我逐条测试过，在当前 `set -euo pipefail` 环境下行为是正确的（`set -e` 不会在 `&&`/`||` 链的中间项上提前退出，整条链最终总会落到某个显式 `exit`）。

但这种写法比原来的 `if/else` 更难读、更难在以后追加第三个检查项（比如未来再加一类需要拦截的标记）时不出错——每加一条都要在脑内重新推演一遍 `set -e` 语义。原来 P4/P7 之所以能被评为"外部产出 gate，不可伪造"里最简单可靠的一类，很大程度上是因为控制流足够直白。建议这里改回显式的 `if ... elif ... else ... fi`，哪怕多写几行：

```bash
P7)
    P7_FILE="$TASK_DIR/P7-consistency.md"
    BLOCKERS=$(grep -cE '^\s*-?\s*\[BLOCKER\]' "$P7_FILE" 2>/dev/null || echo 0)
    DEVCRIT=$(grep -cE '^\s*-?\s*\[DEVIATION-CRITICAL\]' "$P7_FILE" 2>/dev/null || echo 0)
    if [ "$BLOCKERS" -gt 0 ] || [ "$DEVCRIT" -gt 0 ]; then
        echo "GATE P7: BLOCKER=$BLOCKERS, DEVIATION-CRITICAL=$DEVCRIT" >&2
        exit 1
    fi
    DESIGN_GAP_COUNT=$(grep -cE '\[DESIGN_GAP:' "$P7_FILE" 2>/dev/null || echo 0)
    DESIGN_GAP_REVIEWED=$(grep -cE '\[DESIGN_GAP_REVIEWED' "$P7_FILE" 2>/dev/null || echo 0)
    UNREVIEWED=$((DESIGN_GAP_COUNT - DESIGN_GAP_REVIEWED))
    if [ "$UNREVIEWED" -gt 0 ]; then
        echo "GATE P7: 有 ${UNREVIEWED} 条 [DESIGN_GAP] 未配对 [DESIGN_GAP_REVIEWED]" >&2
        exit 1
    fi
    exit 0 ;;
```

同样跑了一遍测试表，行为和链式写法完全一致，可读性更高。这一条纯属风格建议，不阻塞实施。
