**一致性脚本（P3-1）正常运作，0 ERROR。** 18 个 WARNING 全部是叙事文件里的良性引用（旧路径、提议中的未来文件），无一是协议文件的结构问题。脚本在守住不该破的边界。

**improvement-guide 的拒绝决策判断准确。** 「阶段状态文件 `.running/.done/.failed` 是 subagent 自我报告」这条拒绝理由是协议核心原则（C7 规则）的正确应用。`subagent-context-mechanism.md` 提取的平台机制调研（OpenCode 压缩摘要注入 vs Claude Code sidechain transcript）是有价值的事实性文档，值得保留。

### P-2（高价值）：`check-gate.sh` —— 把 gate 判定从「推演」变为「脚本」

**这是当前架构下能做的最大效率提升。**

T025 测出 gate 判定平均冗余比 30:1。T025 的 gate 命令精确化（已落地）把「主 Agent 读三个文件推演」改成「主 Agent 知道跑哪条命令」，把冗余比从 ~30:1 降到 ~10:1。但主 Agent 还是要自己跑命令、解析输出——下一步是把「跑命令 + 判定 exit code」也封装成脚本，主 Agent 只看 `check-gate.sh P6 Txxx` 的 exit 0 / 1，彻底不需要解析命令输出。

**实现思路：**

```bash
#!/usr/bin/env bash
# check-gate.sh PHASE TASK_DIR
# exit 0 = gate 通过; exit 1 = gate 未通过; exit 2 = 判定依赖语义（主 Agent 自判）
#
# 可脚本化的 gate（exit 0/1）：P3 / P4 / P6 / P7
# 需动态读取 P2 gate_commands 的 gate（主 Agent 调用）：P2 / P5 / P8
# 含语义判断的 gate（exit 2）：P1（BDD 格式不固定）/ P2（四字段内容正确性）

PHASE=$1
TASK_DIR=$2

case "$PHASE" in
  P3) # TDD 红灯验证
      exec scripts/check-tdd-red.sh ;;
  P4) # commit 存在性
      git log --oneline -1 | grep -qE 'P4|wf\(T[0-9]+-P4\)' && exit 0 || exit 1 ;;
  P6) # BDD 全量 PASS
      TOTAL=$(grep -cE '^\s*- (PASS|FAIL)' "$TASK_DIR/P6-acceptance.md")
      FAIL=$(grep -cE '^\s*- FAIL\b' "$TASK_DIR/P6-acceptance.md")
      NC=$(grep -cE '\[NEED_CONFIRM\]' "$TASK_DIR/P6-acceptance.md")
      [ "$FAIL" -eq 0 ] && [ "$NC" -eq 0 ] && [ "$TOTAL" -gt 0 ] && exit 0 || exit 1 ;;
  P7) # BLOCKER + DEVIATION-CRITICAL 均为零
      BLOCKERS=$(grep -cE '^\s*-?\s*\[BLOCKER\]' "$TASK_DIR/P7-consistency.md")
      DEVCRIT=$(grep -cE '^\s*-?\s*\[DEVIATION-CRITICAL\]' "$TASK_DIR/P7-consistency.md")
      [ "$BLOCKERS" -eq 0 ] && [ "$DEVCRIT" -eq 0 ] && exit 0 || exit 1 ;;
  P1|P2|P5|P8)
      echo "GATE $PHASE: 含动态 gate_commands 或语义判断，主 Agent 自行判定（见 state-machine.md 步骤5）" >&2
      exit 2 ;;
  *)  echo "未知阶段: $PHASE" >&2; exit 2 ;;
esac
```

**价值说明：**
- P3/P4/P6/P7 是纯机器可判定的，覆盖了 8 个 gate 中 4 个
- P1/P2/P5/P8 仍需主 Agent 处理（P1 BDD 格式不固定是 T025 v2 复核已验证的结论，不能硬编码正则），脚本返回 exit 2 让主 Agent 知道该自判而非出错
- 与 `check-tdd-red.sh` 同一设计风格，易于维护
- 接入 CI 后可作为回归测试的一部分（对测试任务的 P6 文件跑 check-gate.sh P6）

### P-4（战略）：P3-3 独立数据源 PoC

这是评审里列的最高战略价值项，也是迄今最没有进展的一项。局限 3（主 Agent 单点故障）被 T016 / T019 / T020 三次事故反复验证，但协议对它的处理方式一直是「承认」而不是「缓解」。

`main-agent-oversight.md` 已论证了唯一可行方向：让 subagent 产出带主 Agent 难以伪造的元数据（`agent_run_id` + 时间戳），用 git author/committer timestamp 交叉核对。

**最小 PoC：**
1. 在 `dispatch-prompt.md` 的派发模板里追加一行：要求 subagent 在产出文件 Header 写入 `agent_run_id: $(date +%s%N | sha256sum | head -c8)`
2. 写一个 `scripts/check-agent-integrity.sh`：对比产出文件 Header 里的时间戳与 git commit timestamp，差值 > 阈值（如 10 分钟）时报告
3. 在一次真实任务里跑这个 PoC，收集数据

这不会根治单点故障，但会把「T016 那种主 Agent 把违规降级写成合规总结」变成「有独立数据源可交叉验证」，把局限 3 从「无解」推进到「可监测」。

## 五、一致性脚本 WARNING 趋势评估

| 时间点 | WARNING 数 | 说明 |
|--------|-----------|------|
| P3-1 落地时（初始状态） | 4 | analyst YAML、3处叙事文件引用 |
| T022 plan 合入后 | 13 | T022 文档带进来的示例 YAML 和计划引用 |
| T025 gate-opt 合入后 | 18 | T025 文档带进来的路径引用（含已否决计划） |

**趋势：叙事文件（docs/reviews / docs/plans）每增加一批就带进来 ~5 个 WARNING，大部分是良性的（已否决计划里的旧路径、提议中的未来文件）。**

目前 18 个 WARNING 全部合理，0 ERROR，CI 绿色——这是健康状态。但如果半年内 WARNING 涨到 50+，「仅有 N 个 WARNING」的提示就会失去意义（噪音淹没真正需要注意的边界案例）。

**建议：** 当 WARNING 超过 30 个时，做一次「WARNING 清理」——不是修脚本，而是在叙事文件里把引用了不存在文件的地方加一行注释「此路径为提议中的文件，尚未落地」或直接删除已否决计划里的路径引用。这样 WARNING 保持在「值得注意」的数量级，不变成背景噪音。

## 六、总评

**agate 当前的问题不是设计不够好，而是执行层有债务。** 协议核心（gate 可判定性、状态机落盘、上下文隔离、执行/评审分离）经过 T016-T025 的真实任务验证，已经是有实测支撑的成熟设计。

**最该做的事按顺序：**

1. **T022 债务清还**（P-1）：plan 已写好，5 项完全未动，35 分钟内可以全部落地，然后把 plan status 改为已落地。这是最快能改善协议完整性的动作。
2. **`check-gate.sh`**（P-2）：把 P3/P4/P6/P7 的 gate 判定脚本化，是 T025 gate 精确化的自然延伸，把冗余比从 ~10:1 继续压缩。
3. **plan 状态管理**（P-3）：养成「顺带覆盖时更新对应 plan」的习惯，或把协议改进项纳入 active-tasks.md 跟踪。
4. **P3-3 独立数据源 PoC**（P-4）：战略性的，没有截止日期，但应该开始——哪怕只是在 dispatch-prompt 里加一行 agent_run_id 要求。

**当前不需要做的事：** skill 化（结构性不适配，已决策）、并行执行（串行是已知短板但不是当前瓶颈，T025 的 19 次派发在串行下完成了完整 P0-P7）、协议全面重写（核心设计稳定，局部迭代即可）。
