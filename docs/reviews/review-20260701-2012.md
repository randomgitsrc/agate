## 五、🔴 WARN 锚点分类表的 3 条"✅ 已加"里有 2 条是假锚点——CHECK 9 会给出假 PASS

这是这轮最重要的发现，用真实脚本逐条验证过。

### WARN #3（md5 去重）——✅ 真锚点，能正确工作

```bash
$ grep -n "md5\|去重" agate/scripts/check-p6-evidence.sh
（无输出）
```

关键词 `md5`/`去重` 在 `check-p6-evidence.sh` 里完全不存在。CHECK 9 会正确报 WARN，准确反映"文档声称 hook 强制 md5 去重，但脚本没实现"。**这个锚点是有效的。**

### WARN #7（P6 裁剪跳过风险）——❌ 假锚点，会假 PASS

锚点定义：`check-pruning.sh` 中查找 `P6` + `跳过风险`。

```bash
$ grep -n "P6" agate/scripts/check-pruning.sh
62:# 检查 3：裁剪 P6 的条件（不可裁，除非 no_behavior_change）
63:if ! echo "$PHASES_DECLARED" | grep -qw 'P6'; then
65:        ERRORS="...P6 不可裁剪..."

$ grep -n "跳过风险" agate/scripts/check-pruning.sh
103:# 检查 7：裁剪理由必须含"跳过风险"评估
105:    if ! grep -qE '跳过风险:' "$P1_FILE"
106:        ERRORS="...裁剪声明缺'跳过风险:'评估..."
```

两个关键词都在脚本里——CHECK 9 会报 PASS。**但真实问题是：检查 7 的条件表达式（第 104 行）列出了 P2/P3/P7/P8，唯独没有 P6**：

```bash
if ! echo "$PHASES_DECLARED" | grep -qw 'P2' || ... 'P3' || ... 'P7' || ... 'P8'; then
```

`P6` 存在于脚本的另一个位置（检查 3，第 62-65 行），不在检查 7 的条件里。**关键词存在 ≠ 关键词出现在正确的位置。** CHECK 9 说"对齐了"，实际上 WARN #7 的问题原封不动地继续存在。

### WARN #10（P2 候选方案权衡+选择理由）——❌ 假锚点，会假 PASS

锚点定义：`check-gate.sh` 中查找 `权衡` / `选择理由`。

```bash
$ grep -n "权衡\|选择理由" agate/scripts/check-gate.sh
29:  echo "GATE P2: P2-design.md 需至少 2 个候选方案 + 权衡 + 选择理由（v0.6 多方案探索）" >&2
```

两个关键词都在——CHECK 9 会报 PASS。**但它们只出现在错误消息的文本字符串里**（`echo "... + 权衡 + 选择理由 ..."`），不在检查逻辑里。脚本实际只检查 `CANDIDATE_COUNT < 2`，完全不验证 P2-design.md 里有没有"权衡"或"选择理由"节。

**关键词出现在 echo 的消息文本里 ≠ 关键词被用作检查条件。** CHECK 9 说"对齐了"，实际上 WARN #10 的问题原封不动地继续存在。

### 影响

WARN 分类表的"结论"写：

> 8 条 WARN 中 3 条可加锚点（#3/#7/#10），5 条只能靠 LLM 层。

**实测后应更正为：8 条 WARN 中 1 条可加锚点（#3），7 条只能靠 LLM 层。** #7 和 #10 需要从"✅ 已加"改回"❌ 只靠 LLM"，锚点表里对应的两行也需要删除——它们不是没用，是**有害**的：给维护者一个"CHECK 9 覆盖了这两条"的虚假安全感。

这同时说明 CHECK 9 §2.4 "局限性"一节描述的问题，实际发生频率比计划预估的要高——不只是"文档说 MAX_RETRY=2 脚本写 MAX_RETRY=3"这种值不匹配才会假 PASS，**关键词出现在错误消息文本里而不在检查逻辑里**这种更隐蔽的场景也会假 PASS。建议在 §2.4 补充这个具体的假阳性类型，帮助后续维护者评估 CHECK 9 PASS 的可信度。

## 七、§7.4 回归测试仍然描述了需要 LLM 执行的场景

§7.4 写：

> 模拟审计发现的 2 个 ERROR 场景：
> - MAX_RETRY 文档说 2 脚本写 3 → 审查报告标 MISALIGNED

这两条如果要写成 `.bats` 自动化测试，可以做的是**结构对比**（从 `state-machine.md` 提取各阶段 MAX_RETRY 值，从 `check-state-transition.sh` 提取 `MAX_RETRY=` 赋值，比较是否一致）——这不需要 LLM。但当前描述写的是"审查报告标 MISALIGNED"，暗示需要 LLM 产出报告再检查报告内容——和 §7.2 已经做的"自动化 vs 人工"拆分不一致。

建议把 §7.4 的描述改成结构对比的方式（提取→比较→不一致则 FAIL），或者明确标注"这两条也是人工验收，不是 bats 自动化"。
