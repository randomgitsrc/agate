## 三、新 finding（K1-K3）

### K1 🟡 流程图 P1 缺 trivial 路径分支

v9 流程图 P1 段（mermaid L8-L19）画的是正常路径：analyst → requirements-review → review approved? → gate。但 N4/M1 的 trivial 路径（P1_simplified:true → analyst 自检 → gate 查自检节锚点）在流程图中**完全没有体现**。

读者看流程图会以为 P1 必须走 review subagent——这与 v6 方案的 N4 可裁性矛盾。流程图是主 Agent 读 phase-card 时最直观的参考，缺 trivial 分支会导致实施 U1 时遗漏。

**建议**：在 P1 段加一个条件分支节点——`P1_simplified?` → 是 → analyst 自检 → gate 查自检节 → 否 → 正常 review 路径。不需要画得很复杂，一个菱形判断 + 两条路径即可。

### K2 🟡 ⑨ P8 subagent 化后 READY 收尾的"亲自做"边界需更精确

方案 ⑨ 说"主 Agent 只亲自做 P0-brief + P8 READY 收尾检查"。但当前 P8-release.md（L8-L13）的步骤 1-5 全是"主 Agent 写/做"——subagent 化后，步骤 1-4 由 releaser subagent 做，步骤 6（READY 收尾检查）由主 Agent 做。

方案 ⑨ 修改清单第 3 项说"P8-release.md 改为 subagent 派发模式，READY 收尾仍为主 Agent"，但**没有明确列出主 Agent 亲自做的 READY 收尾检查项与 releaser subagent 做的发布准备项的边界**。

当前 state-machine.md L131-157 的 READY 收尾检查清单（状态与版本 / 测试环境已清理 / 开发环境已还原 / 生产环境无残留）是主 Agent 亲自做的——这没问题。但 P8-release.md 的"临时资源清单"（L31）是 releaser 写的、主 Agent 据此做清理——这个交接关系在方案中没有显式说明。

**建议**：⑨ 修改清单第 3 项补充一句——"releaser subagent 产出 P8-release.md（含临时资源清单），主 Agent 据此执行 READY 收尾检查（清理临时资源 + 环境确认）。临时资源清单是 releaser→主 Agent 的交接文件，不是主 Agent 自己写的。"

### K3 🟢 ⑩ retry 预算与 ③ 回退的交互需落地时注意

⑩ 说"review 迭代和 gate 重试共享同一阶段 retry 预算"。③ 说回退时"修完后从目标往下逐阶段重跑"。

**潜在交互**：P6 gate 失败 → 诊断 → 退回 P4 → P4 修完 → P5 重跑 → P6 重跑。此时 P6 的 retry 预算是否清零？按 ⑩ 的"共享预算"规则，P6 的 retry 计数不清零（state-machine.md 已有"按阶段独立记录，不因进入新阶段而清零"的规则）。但 ③ 的回退是"诊断→跳转"，不是"同阶段重试"——回退后重跑 P6 是因为 P4 修了东西，P6 的失败根因已变。

方案没有显式说明"回退后重跑目标阶段时，目标阶段的 retry 计数是否保留"。按 state-machine.md 现有规则（L553-561），retry 计数不清零——这是对的（防止借回退绕过重试上限）。但 ⑩ 的"review 迭代消耗预算 → 剩余预算给 gate 重试"在回退场景下可能让 P6 的预算很紧（回退前已消耗 + 回退后重跑又消耗）。

**不需要改方案**——state-machine.md 的 recovery_bonus 机制（L239）已覆盖此场景（PAUSED 恢复后该阶段获得额外 1 次重试）。但落地 U1/U4 时需确认：回退（非 PAUSED 恢复）是否也触发 recovery_bonus？当前方案 ③ 的 diff=1 回退（如 P5→P4）不经过 PAUSED，不触发 recovery_bonus——如果 P6 回退到 P4 后重跑 P6 又失败，P6 的 retry 可能已耗尽。

**提醒**：落地 U4（③回退机制）时，在 state-machine.md 或 dispatch-protocol.md 中显式说明——回退后重跑目标阶段时，retry 计数保留（不清零），且不额外给 recovery_bonus（回退不是 PAUSED 恢复）。如果重跑后 retry 耗尽 → 走正常 PAUSED 路径 → 恢复时才给 recovery_bonus。这与现有规则一致，只需显式写出避免歧义。
