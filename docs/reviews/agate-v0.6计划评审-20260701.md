## 总体判断

**这是 agate 迄今战略层次最高的一份计划——它标志着协议从"在 P6 层堵验收造假"转向"在 P2 层堵假设错误"，治的是 T045 那条 7 环节故障链的病根（P2 假设错误从源头没被拦）。计划的成熟度很高：已过一轮评审，自我否决了 TDD 话术（哲学冲突）、把模型选择降级为项目层、把"3 次失败质疑架构"正确落到主 Agent 侧。这些判断我逐条核实，全部站得住。**

**我的二次评审聚焦在三个还没充分论证的真实张力：一是 P2 不可裁的"默认强制"可能对一大类中间任务过度收费（例外口 design_trivial 太窄）；二是"不新增实现计划阶段"这个哲学选择的风险没有被评估；三是多方案探索的"2-3 个候选"在 subagent 隔离下可能退化为形式合规。这三点不否定计划方向，但需要在实施前想清楚。**

## 一、P2 不可裁 — 方向对，但"默认强制"的成本被低估

### 核心论证成立

计划的核心论证是：**P1 analyst 做需求分析不做方案设计，无法预知 P2 会发现哪些 P0/P1 没想到的问题。** T045 是铁证——P0/P1 觉得"CSS 改个 display 很明确"，但 P2 如果做了就会发现 `overflow:auto + display:block` 的交互问题。这个论证扎实：**裁剪 P2 = 在没做方案设计的情况下断言"不需要方案设计"，这是循环论证。**

镜像 P6 模式（`no_behavior_change: true` 例外）在机制一致性上也对。我核实了 P6 的例外口——脚本只查 `no_behavior_change: true` 字段存在性，是纯 self-declaration。所以 P2 用 `design_trivial: true` 自报，**不是新引入自报，是沿用 P6 已确立的模式**，一致性成立。

### 但例外口 design_trivial 太窄，会对一类中间任务过度收费

`design_trivial: true` 的四个限定条件（全部满足）：单文件 + 无逻辑变更 + ≤5 行 + 主 Agent 确认。这个口子**只放过 typo/文案/配置值修改**。

问题在于：**存在一大类"方案确实显而易见、但不满足 design_trivial"的任务。** 例如：
- 给一个已有函数加一个参数（多文件：函数定义 + 3 个调用点，> 5 行，但方案毫无悬念）
- 按已有模式加一个 CRUD 端点（有现成的 5 个同类端点做模板，方案就是"照抄第 6 个"）
- 重命名一个变量/字段（多文件，可能 > 5 行，但零设计决策）

这些任务的方案设计价值接近零（照已有模式做），但因为不满足 design_trivial，强制走完整 P2（architect 写 2-3 个候选方案 + 评审）。**这是把"防 T045 假设错误"的成本，摊到了一批根本没有假设风险的机械任务上。**

**T045 的教训是"看起来简单但有隐藏耦合的任务不该裁 P2"，不是"所有非 typo 任务都必须做完整方案设计"。** 这两者之间有一大片地带，计划用一个极窄的例外口把它们全划进"强制 P2"。

**建议**：给 design_trivial 增加一个"已有模式"维度，或引入第二个例外口。比如 `follows_existing_pattern: true`（声明"本任务照搬已有的 X 实现，无新设计决策"，附参照文件路径）——它仍是 self-declaration（和 design_trivial 同级），但覆盖了"机械但非 typo"这一大类。否则 P2 不可裁会在实践中制造大量"为一个照抄任务硬写 2 个假候选方案"的形式主义——这恰恰是计划第三节想避免的。

### 过渡期方案有一个未说清的风险

计划说现有裁剪 P2 的任务"自动注入 override 字段（一次性 migration）"。但 `override` 字段的语义是"裁剪声明与执行不一致时的例外说明"（P2.9），把它用于"v0.6 迁移"是**语义挪用**——override 本来标记的是"我声明裁了但实际有产出"，现在被用来标记"我按旧规则裁了 P2"。这会让 override 字段承载两种不同含义，未来审计时无法区分"真的是执行不一致的例外"和"只是 v0.6 迁移遗留"。

**建议**：迁移用独立字段（如 `legacy_p2_pruned: true`）而非复用 override，保持 override 语义单一。

## 二、"不新增实现计划阶段" — 一个未评估风险的哲学选择

计划第一节坚定主张：**P4 implementer 拿 P2 设计 + P3 测试自主实现，不需要步骤计划。如果 P2 不够明确，那是 P2 的问题，不是缺一个实现计划阶段的问题。**

这个立场逻辑自洽，也和 superpowers 划清了界限（superpowers 写步骤脚本，agate 写设计契约）。**但它有一个未被评估的风险：它把"实现的正确性"完全押在"P2 设计够明确"这个前提上，而 P2 设计够不够明确，本身没有客观 gate。**

对比一下 agate 已经建立的其他防线——它们都不信任"上游做对了"这个前提：
- P6 不信任"P5 测过了"→ 独立验收
- provenance 不信任"agent 说验了"→ 客观证据
- P2.9 不信任"声明裁了"→ 对比文件系统

唯独在 P2→P4 这一环，计划选择了**信任"P2 设计够明确"**。如果 P2 设计有隐含歧义（architect 以为讲清楚了，implementer 理解偏了），没有任何机制会发现——因为没有"实现计划"这一层让 implementer 先复述理解、再由 P2 review 确认"implementer 理解对了"。

**这不是说必须加实现计划阶段**——计划的担忧（避免 superpowers 式步骤脚本的僵化）是合理的。但"P2 设计够明确"是一个和"P5 测过了""agent 验了"同类的、需要验证的上游断言，而计划直接信任了它。

**建议**：至少承认这个风险，并考虑一个轻量兜底——比如 P4 implementer 在产出里附一句"实现偏差声明"：如果实现时发现 P2 设计有歧义/缺口而自主做了决策，必须显式标注（`[DESIGN_GAP: xxx]`），供 P7 一致性检查捕获。这比"完全信任 P2 明确"多一道网，又不引入 superpowers 式的步骤脚本。计划第三节的评审补充项"方案是否清晰到 implementer 无需步骤计划即可自主实现"是好的，但那是 P2 评审时的**主观预判**，不是实现后的**客观回填**——两者都需要。

## 三、多方案探索 — 在 subagent 隔离下可能退化为形式合规

计划第三节要求 P2 强制 2-3 个候选方案 + 权衡 + 选择理由，plan-eng-review 审查"是否探索了替代方案 + 选择理由自洽"。

**这个方向对**（多方案是发现隐藏问题的有效手段），**评审边界也划得准**（审"是否探索了"可判定，审"是否选了最优"不可判定，正确地不要求后者）。

**但有一个 subagent 隔离带来的退化风险**：architect 在自己的上下文里独立产出多方案，没有外部压力保证这些方案是"真的探索"而非"为满足数量凑的陪衬"。一个赶进度的 architect 完全可以写一个真方案 + 两个明显更差的稻草人方案，权衡表里把稻草人的缺点写得很明显，选择理由"自洽"——这完全能通过 plan-eng-review 的"探索了替代方案 + 理由自洽"检查。

**这和 T045 的空 png 充数、纯断言后门是同一类问题**：一个"数量/形式"要求，在自报产出下会被最小成本地满足。plan-eng-review 的"理由自洽"恰恰**奖励**了稻草人方案——稻草人越明显，选择理由越"自洽"。

**这个问题可能无解**（方案质量本质上不可机器判定，也不可被隔离 reviewer 独立重推——计划自己承认了这点）。但应该诚实标注：**多方案要求是"制造探索的形式义务"（nudge），不是"保证真的探索了"（barrier）**——就像计划在裁剪层区分 nudge/barrier 那样。当前计划把它写得像一个有效防线，实际上它和 risk_level 声明同级，是 nudge。

**建议**：明确标注多方案探索为 nudge 类，价值是"强制 architect 至少走一遍'还有别的做法吗'的思考"，而非"保证方案是最优的"。不要高估它的防护强度——否则会重蹈"以为加了检查就安全了"的覆辙。

## 四、做得对的地方（核实确认）

**TDD 话术自我否决 — 哲学一致性典范。** 计划用"agate 靠机器 gate 不信任 Agent 自觉"这条核心原则，否决了 superpowers 的 TDD 反合理化话术（"话术约束自觉"与 agate 哲学冲突）。**这是整份计划最漂亮的一处**——它用 agate 自己的第一性原则否决了一个诱人的 borrow 项，而且论证完全成立（话术能映射到 gate 的是冗余，映射不到的是无效）。这条原则我一路评审都在强调，计划自己内化并用它做决策，说明协议的核心哲学已经稳固到能自我约束演进方向。

**"3 次失败质疑架构"落主 Agent 侧 — 技术判断正确。** 我核实了：失败计数在 `.state.yaml` 的 `retries[Pn]`，subagent 隔离上下文确实不知道自己第几次被派发。所以这条规则必须落在主 Agent 的重派 prompt 注入逻辑，investigate.md 角色文件只能"若被告知是高次重试则质疑架构"。这个拆分精准——它正确识别了"哪些信息在 subagent 上下文里、哪些不在"，这是 agate 上下文隔离架构的核心约束。

**模型选择降级为项目层 — 边界划得对。** "设计用强模型、机械用便宜模型"是对的洞察，但模型可用性是环境层不是协议层，放进 `executor_env` 可选字段而非 dispatch-protocol 本体，保持协议模型无关。符合 agate"不硬编码环境特定假设"的一贯原则。

**superpowers 借鉴取舍 — 逐条成立。** 第八节"不借鉴的"六项，每项理由我都核实了：worktree（主干 commit vs 独立分支模型冲突）、writing-plans（设计契约 vs 步骤脚本）、always-on 触发（agate 显式 opt-in）——都是基于两个工具的架构差异做的判断，不是拍脑袋。把 superpowers 定位为"单 Agent 行为纪律层，与 agate 编排结构层正交互补"是准确的——这和我之前对两者的对比结论一致。

**实施顺序 — 拆分合理。** 六个独立 commit，概念分层先行（纯文档零风险铺路），P2 不可裁 + 下游同步捆绑（避免死分支残留），各自可独立验证独立回退。符合 agate 一贯的"小步可回退"节奏。

## 六、总评

**v0.6 是 agate 从"事后堵漏"走向"源头预防"的转折点，方向完全正确。** T045 的病根是 P2 假设错误从源头逃逸，v0.6 的 P2 不可裁直击这个根——这比在 P6 层加第十个证据检查有价值得多，因为它把防线往上游移了四个阶段。计划的哲学一致性（用自己的原则否决 TDD 话术）、技术精准度（3 次失败落主 Agent 侧）、边界感（模型选择降级）都是高水准的。

**我的三个 🟠/🟡 都指向同一个更深的模式**：**agate 每加一条"强制上游做某事"的规则，都要区分它是 barrier（客观可验证）还是 nudge（自报可绕过），并诚实标注强度。** P2 不可裁本身是 barrier（phase 声明客观可查），但它的例外口 design_trivial、多方案探索、P2 设计"够明确"——这三个都是 nudge 或未验证的上游信任。计划在裁剪层已经建立了 nudge/barrier 的区分意识（T045 R3/R5），但在 v0.6 的新规则上还没有一致地应用这个区分。

**最该做的**：问题 1（补 follows_existing_pattern 例外口）——否则 P2 不可裁会在落地后制造大量形式主义反弹，损害协议的实用性可信度。P2 不可裁的价值在于拦住"看似简单实则有耦合"的任务，不在于逼所有任务都做方案设计。把例外口开得准，比开得窄更重要——开太窄的代价不是"更安全"，而是"用户开始绕过整个协议"，那才是最大的风险。
