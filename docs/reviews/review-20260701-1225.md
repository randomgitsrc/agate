## 二、🔴 根因 1（占 13/17）：`printf '%s'` 让 check-pruning.sh / check-retrospective.sh 的详细错误信息全部丢失

### 定位过程

`P2.1` 这种最简单的用例都失败，先手工复现：

```bash
$ echo "phases: [P0, P1, P2]" > /tmp/p21/P1-requirements.md
$ bash check-pruning.sh /tmp/p21
exit=1
stderr: "GATE PRUNING: 裁剪条件不满足：\n"
   ← 只有这一行头部，具体是哪条检查失败完全没有输出
```

`bash -x` 跟踪发现，脚本执行到 `printf '%s' "$ERRORS" | while IFS= read -r line; do ...; done` 这一句时，`while` 循环只做了一次 `read -r line` 就直接跳到 `exit 1`，**循环体一次都没执行**（没有任何 `echo "  - ..."` 被跑过）。

用 `od` 直接验证字节流，确认了根因：

```bash
$ printf "%s" "$ERRORS" | od -c | grep -c $'\n'
0     ← printf %s 完全没有产出任何真实换行符

$ printf "%b" "$ERRORS"
第一行错误
第二行错误    ← %b 正确把字面 \n 转成真实换行
```

**`%s` 是"原样输出参数，不解释任何转义序列"，而 `$ERRORS` 里的换行是构造时手写的字面 `\n`（两个字符，不是真实换行符）——脚本从一开始就依赖 `\n` 会被 printf 解释成真实换行。用 `%s` 之后这个前提不成立了，输出变成一整坨没有换行符的文本，喂给 `while IFS= read -r line` 时，`read` 在读不到任何换行符的情况下立即因为 EOF 失败（bash 的 `read` 语义：读到 EOF 且没遇到换行符时返回失败，即使读到了部分内容），`while` 循环条件第一次就失败，循环体完全不执行。**

### 这不是历史遗留 bug，是这次评审自己引入的回归

```bash
$ git blame -L 137,140 agate/scripts/check-pruning.sh
a75585f0 ... 137) if [ -n "$ERRORS" ]; then
a75585f0 ... 138)     echo "GATE PRUNING: ..." >&2
dc2713e2 ... 139)     printf '%s' "$ERRORS" | while IFS= read -r line; do   ← 就是这次评审改的
a75585f0 ... 140)         [ -n "$line" ] && echo "  - $line" >&2
```

`dc2713e` 的 commit message 里写着 "M3: `printf "$VAR"` → `printf '%s' "$VAR"` (SC2059)"——SC2059 是 shellcheck 的一条真实警告（"不要把变量当 printf 格式串，防止内容里意外含 `%` 导致解析错误"），**这个警告本身合理，修的方向也对，但选错了替换符**。同样的改动被复制到了 `check-retrospective.sh` 的 `printf '%s' "$WARNINGS"`（第 51 行），带来完全同样的 bug——这就是 RT.2/RT.3/RT.4 三个失败的根因。加上 P2.1/P2.2/P2.3d/P2.4/P2.5/P2.5b/P2.6a/P2.6c/P2.7/P2.8/P2.9（11 个）+ R4.1/R3.1（2 个，走的是同一段 check-pruning.sh 错误输出逻辑）= 11+3+2 = **16 个失败**都是这一个根因（P2.6a 还叠加了第三个独立问题，见下文），**check-gate.sh 的 P6/P7/P8 分支完全没受影响**，因为那些分支用的是独立的 `echo` 直接输出，没有走 `printf | while read` 这条路径。

### 验证修复方案

`%b` 既能解释 `\n`，又不会把 `$ERRORS` 当格式串使用（`%` 后面必须紧跟合法转换符才会被解释，字面 `%s`、`%d` 这种"看起来像格式符但其实是用户数据"的内容会被原样保留，不会引发 SC2059 担心的问题）：

```bash
$ printf "%b" "$DANGEROUS_INPUT_含字面%s%d"
用户输入包含 %s %d 这种格式符    ← 原样保留，没有被误解析，安全性没有退步
```

把 `check-pruning.sh:139` 和 `check-retrospective.sh:51` 的 `printf '%s'` 改成 `printf '%b'`，重新造一遍 `full-task/.state.yaml`（见下一节）后，**完整套件从 135/152 提升到 151/152**，只剩 P2.6a 这一个独立问题（见第四节）。改完记得把 `%b` 而不是退回 `%s`/`"$VAR"` 直接用——退回 `printf "$VAR"` 又会重新触发 SC2059。

## 四、🟠 根因 3（占 1/17）：P2.6a 测试用例自己的构造错误，不是脚本的 bug

排除前两个根因后，`P2.6a` 单独还是失败。这次问题不在 `check-pruning.sh`，在测试用例本身：

```bash
# tests/unit/check-pruning.bats 里 P2.6a 的构造顺序
for i in 1 2 3 4 5 6; do echo "file $i" > "$repo/src_$i.py"; done
git_commit "$repo" "init"        # ← 先把 6 个源文件提交了
cp -r "$dir" "$repo/task"
git -C "$repo" add src_*.py      # ← 再对已提交、内容未变的文件执行 git add
```

`check-pruning.sh` 检查 5 用的是 `git diff --cached --name-only` 来统计"这次改动涉及几个源文件"。**问题是：6 个源文件在 `git_commit` 那一步已经进了 HEAD，内容此后完全没变过，再对它们执行 `git add` 不会产生任何暂存区差异**——`git diff --cached` 天然只显示"和 HEAD 相比有实质变化"的文件，对一个内容与 HEAD 完全一致的已跟踪文件重复 `add`，diff 结果是空的。

实测验证：

```bash
# 复现 P2.6a 原始顺序（先 commit 再 add 同内容）
$ git diff --cached --name-only | wc -l
0     ← 确认没有暂存差异，check-pruning.sh 判定 SOURCE_FILE_COUNT=0，自然不会触发拦截

# 改成"先 add，不预先 commit"（模拟"这次改动新增了 6 个文件"的真实场景）
$ git add src_*.py && git diff --cached --name-only
src_1.py src_2.py src_3.py src_4.py src_5.py src_6.py
$ bash check-pruning.sh task
exit=1
GATE PRUNING: 裁剪条件不满足：
  - 裁剪 P7 需源码文件数 ≤ 5，实际=6      ← 脚本本身逻辑完全正确
```

**`check-pruning.sh` 检查 5 的逻辑没有问题**，是测试夹具的构造顺序反了。修复方式：把 `git_commit "$repo" "init"` 挪到创建 6 个源文件之前（先 commit 一个空仓库或者只有 README 的初始状态，再创建源文件、`git add`，不 commit），这样 `git diff --cached` 才能看到这 6 个文件是"这次改动新增的"。

**这次的教训比较微妙**：`dc2713e` 这条提交本身是一次"实施评审"（3 个 Critical + 8 个 Important + 4 个 Minor），修复的很多问题都是真实、必要的（比如 C2/C3 的 Python shell 注入修复方向完全正确）。但 M3 这一条——为了消除 shellcheck 的 SC2059 警告——在两个脚本里做了同一处替换，而这个替换悄悄破坏了一个脚本行为上依赖已久的隐式前提（`\n` 会被解释成真实换行）。**这类"为了满足静态检查工具、做了看起来无害的等价替换，实际改变了运行时语义"的问题，恰恰是只能靠真的跑一遍测试才能抓到的——单纯读 diff 或者信任 shellcheck 通过是看不出来的**，这也是这次坚持"能实测就实测"而不是停留在读 commit message 的价值所在：如果只看 commit message 里"152/152 全过"就采信，这个 bug 会在下一次真正遇到裁剪失败场景时才被发现，而那时候 Agent 只会看到一句没有任何细节的"裁剪条件不满足"，debug 成本会高很多。
