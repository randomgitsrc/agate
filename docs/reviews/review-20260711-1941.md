# 专家评审：P6 Gate 制度设计方案（v1）

> 结论：**needs-revision**。骨架与纪律对，诚实标注质量高；但有 2 个 blocking 级问题，其中 B1 使 ③ 的核心机制在当前代码下落不了地。所有判断已实核源码。

---

## 一、总评

方案把上一轮评审的 R8 制度设计忠实落成 7 部件，几处做得好：

1. **"先疏后堵"骨架正确**，且明确 ①②③ 激励 / ④⑤ 结构 / ⑥⑦ 卫生的层次与顺序。
2. **"不做的事"是好纪律**：明确拒绝 provenance 降级、拒绝 agent=main 硬拦截、拒绝 verification_env 无条件进全局——都对（除 B2 那处自相矛盾）。
3. **诚实标注质量高**：①②③ 标为 L0 指导非 L3 硬拦截、需实证验证；④ 分测试类（强解）/UI 类（条件解）；R8.4·补 留为开放问题。这种"把不确定的标成不确定"的克制，正是本项目一贯的高质量特征。

但有 **2 个 blocking + 2 个次要**。

---

## 二、Blocking 问题

### B1 🔴 ③ 的核心机制与"承诺不碰的脚本"直接冲突，且实证表有硬错误

**先纠正我自己**：这个错的源头是我上一轮 R8③ 的表——我把"P6→P2"标为"⚠️允许"，方案继承并"优化"成了"diff=1 单步放行"。两者都错。实核 `check-state-transition.sh` 检查 1：

```bash
diff=$((old_num - new_num))
if [ "$diff" -ge 2 ]; then
    echo "回退跳变 P${old_num}→P${new_num}，强制 PAUSED"; exit 1
fi
```

`diff = old_num − new_num`，据此：

| 回退 | 真实 diff | 真实结果 | 方案③表所标 |
|------|----------|---------|------------|
| P6→P5、P5→P4、P3→P2 | 1 | ✅ 放行 | ✅（对） |
| **P6→P4** | **2** | **❌ 强制 PAUSED** | ✅ 放行 diff=1（**错**） |
| **P6→P2** | **4** | **❌ 强制 PAUSED** | ✅ 放行 diff=1（**错**） |

**这不是笔误，是机制层面的冲突**：

- 方案 ③ 的整个决策表（第 165-166 行）把"P6 失败 → 退回 P4"、"退回 P2"当作**默认正确动作**。
- 但当前脚本对**任何 ≥2 阶段的直接回退强制 PAUSED**——即 ③ 想设为默认的那条路，恰恰被脚本拦成 PAUSED。
- 而方案 ③（第 160 行）明确承诺**"不改变 check-state-transition.sh 的脚本逻辑"**。

→ **矛盾闭环**：③ 想要的默认机制（多阶段溯源退回），与它承诺不碰的那个脚本（强制拦多阶段回退），无法共存。方案 ③ 作为"纯 L0 文本改动"落不了地。

**这也顺带证明 R8.4·补 不是"开放问题"，而是已被代码决定**：当前设计就是"大跳（≥2）→ PAUSED"。方案想要的"默认逐步溯源到源头"若指**直接跳到诊断出的源头**（P6→P2），必须**改检查 1**——这与 ③"不改脚本"承诺冲突。

**两个修法，二选一（都要先修正实证表）**：

- **方案 A（顺代码，推荐，最省）**：把 ③ 的"逐步溯源"明确为**字面意义的一次退一阶**（P6→P5→P4→…，每步 diff=1 均放行）。**直接跳 ≥2 阶不是选项**——若 agent 确信源头在 2+ 阶之外，那恰恰就是"PAUSE 问人类"的信号（符合当前代码）。这样 ③ 真能保持"不改脚本"，且与现有设计自洽。代价：溯源要一阶一阶走，略笨但安全。
- **方案 B（改代码）**：若坚持允许"直接跳到诊断源头"，则必须改 `check-state-transition.sh` 检查 1（放宽 diff≥2 拦截、或改为"带诊断的多阶回退白名单"）。那 ③ 就**不再是纯 L0 文本改动**，需升级为脚本改动 + bats，并明确它替代/修改 T019 教训的那条规则——需谨慎论证为何 T019 的拦截可以放宽。

无论哪个，**③ 的实证表必须先改对**：P6→P4 是 diff=2、P6→P2 是 diff=4，均被现行脚本强制 PAUSED。

### B2 🟠 ④ 中期方案与"不做的事"自相矛盾

- **④ 落地路径·中期**（第 240 行）："`ci-gate-backstop.py` 增加 P6 证据复核——CI 环境**独立重跑 provenance**"。
- **不做的事**（第 353 行）："**CI 重跑同一套 provenance**——重跑同一把不辨真伪的尺子，对伪造无效……真正堵伪造的是 ④ 证据由执行生成"。

→ ④ 中期提议的正是"不做的事"明令不做的东西。**直接自相矛盾。**

**修法**：④ 中期应改为**"CI 独立重新生成证据"**（CI 跑测试→产出结果文件），而非"重跑 provenance"。堵伪造靠"证据由执行**生成**"（拿走它的笔），不靠"重跑**审计**"（同一把不辨真伪的尺子）。把"重跑 provenance"从 ④ 中期删掉，与短期/长期的"证据由执行生成"口径统一。

## 三、次要问题

### B3 🟡 ① 的 auto-fix 示例 3-4 越过了它自己画的"只碰形状"线

方案原则正确（只碰形状、绝不触语义），但**它自己的示例 3-4 违反了这条线**：

- 示例 3「`b01.png → (b01.png)`」、示例 4「`vision: b01.yaml → (vision: b01.yaml)`」——给**裸路径**补括号，需要先**判断"这个 token 是证据路径"**。但描述文本里也可能出现文件名（verifier.md 规范：`- PASS B01: 描述 (path)`，描述部分可含任意词）。"哪个 token 是路径"是**语义判断**，不是纯形状。
- 真正安全的只有示例 1（PASS/FAIL 行首大小写）、示例 2（行首空白）。

**修法**：`--fix` 只做示例 1-2 两类无歧义形状归一；"裸路径没加括号"归入 gate 关注（本就近似"引用格式不对"，可 exit 1 或列 WARNING），不由 auto-fix 猜着包。否则 ① 会犯它自己警告的"触语义"错。

### B4 🟡 ⑥ 正则：问题被部分夸大 + 提案 snippet 丢了 heading 锚点

实核 `check-gate.sh:29` 现状正则：`^###?\s*(候选方案|方案\s*[ABC123abc一二三四五])`。

- **现状已含 `候选方案` 这个 alternative**——`### 候选方案 Alpha` 本就匹配。所以"对多词方案名不友好"被部分夸大：只要用 `候选方案` 前缀就已工作，问题仅限 `### 方案 Alpha` 这种写法。
- 方案提案的 snippet（第 295 行）`候选方案|方案\s*[A-Za-z一二三四五]|...` **丢了 `^###?\s*` 行首锚点**。若照字面实现，会匹配行内任意位置的"候选方案"，放得过松。

**修法**：保留 `^###?\s*` 锚点；正则改动的收益说明改为"支持 `方案 <多词名>` 写法"，而非"现状完全不支持多词"（现状用 `候选方案` 前缀已支持）。
