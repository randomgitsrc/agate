## 需修复项

### 1. [NEEDS_FIX] P6 的 failed 求和逻辑在 flat（串行）场景下重复计数一倍

新实现：

```bash
failed="$(grep -rh '^\s*failed:' "$task_dir/P5-test-results/unit.md" "$task_dir/P5-test-results/" 2>/dev/null | grep -oE '[0-9]+' | paste -sd+ 2>/dev/null | bc 2>/dev/null || echo 0 | tail -1)"
```

问题在于**同时**把 `"$task_dir/P5-test-results/unit.md"`（显式文件路径）和 `"$task_dir/P5-test-results/"`（目录，`-r` 递归）两个参数传给了同一个 `grep`。当 `unit.md` 就直接放在 `P5-test-results/` 目录下（也就是最常见的单包/串行场景——这本来就是这次修复要保留兼容的"原始布局"），这个文件会被**同一条 grep 命令匹配两次**：一次因为它被显式列为参数，一次因为它也在被递归扫描的目录里。`-h` 抑制了文件名前缀，两次匹配的内容看起来完全一样，但 `paste -sd+ | bc` 会把它们当成两个独立的数字相加。

实测复现（用 EC.7 的原始数据，failed 真实值是 1）：

```
$ echo "  failed: 1" > P5-test-results/unit.md
$ bash agate-extract-context.sh P6 <task_dir>
- P5 failed 参考: 2（仅供参考，gate 以主 Agent 实跑为准）    ← 真实值 1，显示成 2
```

**为什么新增的 EC.15 没抓住这个问题**：EC.15 只测了两个包子目录（pkg-a/pkg-b）的场景，没有覆盖"flat unit.md 直接在 P5-test-results/ 下"这个最基础的场景。而**修改前就存在**的 EC.7（用的正是 flat 布局）虽然测了 flat 场景，但它的断言只检查了 `*"仅供参考"*` 这个字符串是否出现，**没有断言具体数值**（`grep -n "EC.7" -A 12` 核对过，确实没有 `*"P5 failed 参考: 1"*` 这类断言）——所以这个回归在现有测试套件里是"漏网"的，420→422 的测试数字增长掩盖了这个问题，而不是证明了它。

**根因**：这个双路径写法本身是不必要的。`grep -r` 对一个目录递归扫描，天然就会扫到直接放在这个目录下的文件（不需要额外把 `dir/unit.md` 再单独列一遍）。实测验证：**只传目录、不重复传 flat 文件路径**，两种布局都能正确处理，没有重复计数：

```bash
$ grep -rh '^\s*failed:' P5-test-results/ 2>/dev/null   # flat 场景，只传目录
  failed: 3          # 正确，没有翻倍

$ grep -rh '^\s*failed:' P5-test-results/ 2>/dev/null   # 并行场景，只传目录
  failed: 2
  failed: 1          # 两个包各自的行都在，递归天然覆盖
```

**建议修复**：把 `grep -rh '...' "$task_dir/P5-test-results/unit.md" "$task_dir/P5-test-results/"` 改成只传目录一个参数 `grep -rh '...' "$task_dir/P5-test-results/" 2>/dev/null`（P4/P5 那边如果也用了这种"文件+目录都传"的写法，建议一并检查——不过我核对过 P5 的 `implementation_dir` 提取用的是 `P4-implementation.md`（无扩展名前的文件）和 `P4-implementation/`（目录），两者名字不同、不存在嵌套关系，不会重复计数，这个问题只出在 P6，因为 `unit.md` 恰好直接嵌套在被递归扫描的同名目录下）。

同时建议把 EC.7 和 EC.15 的断言都加上具体数值检查（`*"P5 failed 参考: 1"*` / `*"P5 failed 参考: 3"*`），而不只是检查免责声明字符串——这次回归能溜过测试，本质上就是断言强度不够。

**严重性判断**：不阻塞——这个字段本来就标注"仅供参考，gate 以主 Agent 实跑为准"，不影响任何 gate 判定。但比上一轮的问题更值得优先处理：上一轮是"并行时读不到"（新功能的边界情况），这一轮是"最常见的默认场景下数值是错的"（老功能被新改动带出的回归），影响面反而更大——几乎每一次 P6 verifier 派发都会带着这个翻倍的数字。

## 复核方法说明

- 未仅凭"422 bats / 0 fail"通过就采信修复质量，而是重新读了两处 diff 的具体 grep/管道逻辑，对"文件路径 + 同名父目录同时传给 grep -r"这种写法保持怀疑（这是个常见的 shell 脚本陷阱），手动构造了 flat 场景复现
- 复现用的正是本次改动之前就存在的 EC.7 测试数据（failed: 1），说明这不是我编的极端案例，而是这个协议自己既有测试场景就能触发的问题，只是那条测试断言写得不够严格没测出来
- 顺带确认了 P5 的 `implementation_dir` 双路径提取不存在同样问题（文件名和目录名不同，没有嵌套关系）
