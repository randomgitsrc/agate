# 对「渐进披露重构评审」的再评审

## 总体：评审的框架比内容更有问题

评审提出了 6 个攻击点，2 个过度攻击（似是而非），2 个部分有效，2 个真正有效。但评审本身的框架——把认知过载和行为偏差分开讨论——掩盖了两者的交互效应。

---

## 驳攻击点 1：认知过载和激励不对称不是独立的

评审说"T046 的根因是激励不对称，不是认知过载"。这是假二分法。

激励不对称是**为什么 Agent 选了 pass gate 这条路**。认知过载是**为什么 Agent 没看到另一条路**。两者是协同致因：

- Agent 不做过载时：有能力通盘考虑"gate 会查什么 + 功能应该验证什么"→ 两个都能做
- Agent 过载时：只能看到"哪些东西会 block 我"→ gate 是硬阻塞 → 优先处理

认知过载是激励不对称变得致命的**条件**。如果 Agent 认知资源充足，"gate 硬 vs 功能软"的优先级差不会产生错误——Agent 两个都能照顾到。

**评审的错误**：把协同因素当作独立替代解释来评断。

---

## 驳攻击点 2：自包含导致近视——设计问题，不是结构问题

评审说"P4 卡片没有 P6 信息 → Agent 近视"。这是对的，但原因不是"自包含"这个设计——而是卡片到底写什么。P4 卡片完全可以在"下游影响"小节写"P6 验收需要 vision-helper 验证图片渲染——确认你的实现路径的端点能返回正确的 Content-Type"。

**这不是卡片的理论缺陷，是卡片内容的完备性问题**。建议 1（加"下游影响"节）已经解决了它。把这个当作对"自包含"方向的反驳，是过度推理。

---

## 驳攻击点 3：导航链只覆盖快乐路径——对，但当前更糟

评审说卡片导航链缺少"重试/gate 失败/裁剪跳阶"的模式区分。这是对的。

但当前系统（8 文件全读）根本没有模式区分——Agent 自己从 state-machine.md、dispatch-protocol.md、WORKFLOW.md 三个文件里推导"我现在是该首次进入还是重试"。卡片至少可以通过"首次/重试/裁剪"节来显式区分——这是改进，不是引入新问题。

**评审的错误**：把"卡片没做到完美"等同于"卡片比现状差"。实际上卡片在这一点上比现状强。

---

## 驳攻击点 4：维护成本——评审高估了迭代频率

评审说"agate 当前在快速迭代期，每月都改，引用更安全"。实际上，看最近的 git log：

```
v0.6 → v0.7: 6 个 commits，2 周
v0.7 → v0.8: 4 个 commits，1 周
v0.8 → 当前: 修复 + 小迭代，频率明显下降
```

agate 已经过了密集规则增发期——v0.6 解决了双方案探索，v0.7 解决了裁剪规范化，v0.8 当前是稳定和修补阶段。之后的改动更可能是"修复"而非"新增规则"。

**内联模式在这种频率下是安全的**——规则稳定期，卡片只需要偶尔同步。维护成本真实但被评审高估了。

但评审有一个子观点是对的：**卡片-协议一致性检查不能像 CHECK 9 那样机械化。** "卡片遗漏了某个规则"是语义缺口，不是结构性缺口。解决方案不是放弃内联，而是退化到更弱的检查——至少检查"卡片引用的协议文件段确实存在"和"卡片声明的规则类别在协议文件里有对应内容"（不是逐条比，是分类比）。

---

## 驳攻击点 5：问题归因——评审低估了认知过载的因果占比

评审说 T046 的 5 层因果链只有 1 层可归因过载。重新审视：

| 层 | 做了什么 | 过载是原因吗？ |
|----|---------|-------------|
| P2 未验证 Content-Type | 最小验证规则存在但未触发 | **是**——规则在 dispatch-protocol.md 中间位置，过载时不会主动回查 |
| P4 只跑单元测试 | 实现范围窄 | **部分**——P2 写 gate_commands.P5 只列 pytest（P2 也是过载状态），P4 Agent 只跑声明的命令 |
| P5 gate_commands 缺 E2E | P2 遗留 | 这是 P2 的因果链延续，不是独立事件 |
| P6 被 gate 格式带偏 | 花 2h 凑格式 | **是**——过载时优先处理硬阻塞（gate 格式），功能验证被排到后面后就被"2h 已经花掉了"的心理账户锁死 |
| vision 被反驳 | 行为模式 | **不太是**——反复验证花 15 分钟说明认知资源充足，是不信 tool 的结论 |

所以是 2.5-3 层，不是 1 层。认知过载和激励不对称的交互才是完整解释。

---

## 部分接受：攻击点 4 的核心关切

"内联 vs 引用"的权衡是真实的。评审建议根据 agate 迭代频率选择策略——这个视角有价值。结论应该是：

- 当前（v0.8 稳定期）→ 内联安全
- 如果以后进入规则增发期 → 退化为引用+行范围

不是二选一，是按阶段切换。

---

## 接受：攻击点 6

卡片需要被协议一致性检查覆盖。这不是复杂工作量——至少作为 CHECK 2 的检查对象（引用完整性），至多增加 CHECK 10 类的卡片-文件分类一致性（检查卡片声明的每条规则类别在对应文件里是否有内容）。

---

## 评审的模式问题

1. **过度二分化**：把认知过载和行为偏差当作互斥解释来对抗，掩盖了它们的协同效应
2. **完美标准**：用"卡片也解决不了某些问题"来论证"卡片不解决问题"，忽略"卡片比现状强"的增量价值
3. **攻击点 2/3 对卡片做最坏假设**：假设卡片会写得不好，且忽略可以通过卡片内容设计来解决

**但这不是说评审没有价值**。它指出了三个真实缺口：
- 下游影响信息需要在卡片间传递（不是放弃自包含，是补信息）
- 导航需要模式参数（首次/重试/裁剪——不是放弃卡片链，是丰富它）
- 维护策略需要根据迭代频率调整（不是放弃内联，是有条件内联）

---

## 综合判断

| 评审的 6 个攻击点 | 有效程度 | 结论 |
|-----------------|---------|------|
| 1. 核心假设不可靠 | 2/5 | 假二分法。认知过载是激励不对称致命的**条件** |
| 2. 自包含=近视 | 2/5 | 设计问题不是结构问题。加"下游影响"节即可修复 |
| 3. 导航链只覆盖快乐路径 | 3/5 | 对，但卡片比现状强。需要加模式参数 |
| 4. 维护成本被低估 | 4/5 | 核心关切有效，但高估了迭代频率。内联安全，有条件退化为引用 |
| 5. 问题归因错了 | 2/5 | 低估了认知过载在因果链中的占比 |
| 6. 卡片需要自己的 gate | 4/5 | 真实需求，作为 CHECK 扩展即可 |

**评审指出的有效问题**有一个共同主题：**卡片需要更丰富的结构**——不只是快乐路径的首进模式，而是能处理 gate 失败、重试、裁剪跳阶、考虑下游影响。这不是废弃卡片方向的原因，是完善卡片结构的方向。
