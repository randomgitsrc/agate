## 1. 真正的根因（更正后）

既然 4 红落地时 CI 已在跑 bats，那么真正的问题不是「CI 不跑 bats」，而是：

> **CI 是「事后检测」（detective），不是「事前阻断」（preventive）。红 CI 不能阻止红 commit 落地 main。**

**证据链**：
- 仓库全程**直推 main**：`git log --merges` = **0 个 merge commit**，历史完全线性，无 PR。
- 无分支保护线索：无 CODEOWNERS，无 required-status-check 配置。
- GitHub Actions 的 `on: push` 是在 push **之后**触发——red commit 已在 main 上，CI 只是随后把徽章变红。

**因此真实事故链是**：
1. self-gate 评审假 ✓ → 主 Agent **相信**是绿 → 直推 main（3053d14）
2. protocol-tests.yml 的 CI **随后**跑 bats → 变红（4 个 not ok）
3. 但因直推 + 无分支保护，**红 commit 已落地并停留在 main**，直到外部评审（我）手动实跑才被发现

真正让「假绿灯进主干且停留」的，不是缺 CI-bats，而是**缺「红 CI 阻断落地」的强制**。缺口 2（A4 看不跑）和缺口 3（评审自著）依然成立且是假 ✓ 的**产生**源；但假 ✓ 得以**落地并停留**的根因是分支保护缺位。

## 附二：可执行修复补丁

### B-1. 清理冗余 job（建议 #4）

确切 job 名比对：

| 功能 | protocol-tests.yml | protocol-consistency.yml | 结论 |
|------|--------------------|--------------------------|------|
| 全量 bats | `bats` | `bats-self-gate` | **重复** |
| 一致性检查 | `consistency` | `check` | **重复** |
| shellcheck | `shellcheck` | —— | 仅 tests.yml |
| gate backstop | —— | `gate-backstop` | **仅 consistency.yml（唯一新增）** |

**推荐做法：把唯一有效的 `gate-backstop` 并入 protocol-tests.yml，删除整个 protocol-consistency.yml**（单一 workflow 作为真相源，避免两份文件漂移）。

在 `protocol-tests.yml` 末尾（`consistency` job 之后）追加：

```yaml
  gate-backstop:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with:
          python-version: '3.10'
      - name: Install PyYAML
        run: pip install pyyaml
      - name: Run gate backstop check
        run: python3 agate/scripts/ci-gate-backstop.py
```

然后删除冗余文件：

```bash
git rm .github/workflows/protocol-consistency.yml
git add .github/workflows/protocol-tests.yml
git commit -m "ci: 合并 gate-backstop 入 protocol-tests，删除冗余 protocol-consistency workflow"
```

清理后 CI 的 4 个 check：`bats`、`shellcheck`、`consistency`、`gate-backstop`——无重复。

> 若不想动 protocol-tests.yml，最小改动替代方案：只从 `protocol-consistency.yml` 删掉 `check` 和 `bats-self-gate` 两个 job、保留 `gate-backstop`。但会留下两份 workflow，不如合并干净。

### B-2. 分支保护：让红 CI 长出牙齿（建议 #1，唯一根治项）

这是把 CI 从「事后检测」变「事前阻断」的关键。**注意：这是 GitHub 仓库服务端设置，不在代码里，需仓库 admin 操作。**

**方式一：GitHub 网页 UI**

Settings → Branches → Add branch protection rule
- Branch name pattern：`main`
- ☑ Require a pull request before merging（关闭直推 main，强制走 PR）
- ☑ Require status checks to pass before merging
  - ☑ Require branches to be up to date before merging（对应 API 的 `strict`）
  - 在搜索框逐个加入必须通过的 check：`bats`、`shellcheck`、`consistency`、`gate-backstop`
- ☑ Do not allow bypassing the above settings（对应 `enforce_admins`，防 admin 自己绕过）

**方式二：`gh` CLI（可脚本化、可复现）**

```bash
cat > /tmp/main-protection.json <<'EOF'
{
  "required_status_checks": {
    "strict": true,
    "contexts": ["bats", "shellcheck", "consistency", "gate-backstop"]
  },
  "enforce_admins": true,
  "required_pull_request_reviews": {
    "required_approving_review_count": 0
  },
  "restrictions": null
}
EOF

gh api -X PUT repos/randomgitsrc/agate/branches/main/protection \
  -H "Accept: application/vnd.github+json" \
  --input /tmp/main-protection.json
```

字段说明：
- `contexts`：必须通过的 check 名 = **job 名**。清理后为 `bats / shellcheck / consistency / gate-backstop`（若未做 B-1 清理，则用现有名：`bats / shellcheck / consistency / check / gate-backstop / bats-self-gate`）。
- `strict: true`：分支必须先 rebase/merge 到最新才能合入，防"绿在旧基线、合入后其实红"。
- `enforce_admins: true`：admin 也不能绕过——**直接关闭「我是作者所以直推」这条路**，这正是本次假 ✓ 得以落地的通道。
- `required_pull_request_reviews.required_approving_review_count: 0`：不强制人工 review 数（单人项目可为 0），但**配合 "Require a pull request" 仍会强制走 PR**，从而让 required status checks 有机会拦截。

**⚠️ check 名核对**：GitHub 的 status check context 名取决于 job 名（有时显示为 `workflow / job`）。首次配置前，先在 Settings → Branches 的 required-checks 搜索框里确认自动补全出的确切名称，或跑一次 Actions 后从最近一次 run 的 check 列表里抄准。

**效果**：配置后，即便 self-gate 评审再次假 ✓、作者再次误以为绿而发起合入——只要 bats CI 红，**PR 无法合入 main**。假 ✓ 仍可能产生（缺口 2/3 未除），但**永远进不了主干**。这才是上一轮我真正想要的效果，手段应为「给已有 CI 上分支保护」，而非「再加一份 CI」。

### B-3. 兜底：本地 pre-push hook（建议 #3）

分支保护是服务端强制；本地再加一道，挡「忘了跑就 push」：

```bash
cat > .git/hooks/pre-push <<'EOF'
#!/usr/bin/env bash
echo "pre-push: 运行 agate 自门禁 bats..."
if ! bats agate/tests/sanity.bats agate/tests/unit/ agate/tests/regression/ agate/tests/integration/; then
    echo "❌ bats 未全绿，拒绝 push（如确需绕过：git push --no-verify）" >&2
    exit 1
fi
EOF
chmod +x .git/hooks/pre-push
```

局限：`--no-verify` 可绕过、且 hook 不随仓库分发（需各人自装或用 `core.hooksPath` + 提交的 hook 目录）。因此它是**兜底**，不替代 B-2 的服务端强制。
