## 三、新 finding（N1-N6）

### N1 🔴 12 部件一次性大更新，blast radius 过大，应拆分为独立可回滚单元

方案从 7 部件长到 12，新增的 ⑧⑨⑩⑪⑫ **全是核心工作流的结构性改动**（新 P1 评审角色、P5/P7/P8 subagent 化、新落盘文件、dispatch-context 改结构）。方案把它们打包进"Phase 0"一次落地。

**问题**：这是 agate 史上最大的一次协议变更。12 个部件耦合在一次更新里——任何一个有隐藏缺陷，都和其余 11 个纠缠，难定位、难回滚。尤其 ⑨（P5/P7/P8 subagent 化）是**编排哲学的转变**（主 Agent 从"亲自跑部分阶段"变为"只做 P0+READY"），单独就该有独立验证周期。

**建议**：按**可独立 landable 的单元**拆分，每个单元独立评审+bats+可回滚，而非一个 mega-PR：
- **U1（结构对称）**：⑧ P1 评审 + ⑩ 迭代循环（⑩是⑧的必要补充）
- **U2（编排哲学）**：⑨ P5/P7/P8 subagent 化——**单独一个 PR**，因为它动摇"主 Agent 亲自"的既有设计，风险最高
- **U3（上下文机制）**：⑪ dispatch-context 扩展 + ⑫ 诊断落盘（⑫依赖⑪的冻结约束）
- **U4（激励层）**：①②③（先疏）
- **U5（结构+卫生）**：④⑤⑥⑦

每个 U 落地后跑全套件 + self-gate + 走 PR（分支保护已启用）。这样 blast radius 可控，且符合 agate 自己"小步可回滚"的精神。

### N2 🟠 ⑪ × 审计2 集成风险：诊断若用 `- FAIL` 行首会误触预判审计（须约束格式）

实核 `check-p6-provenance.sh` 审计2：它 grep `^\s*- (PASS|FAIL)\b` 于 P6-dispatch-context.md，命中即 exit 1（判为"验收结论预判"）。

**风险**：⑪ 扩展 dispatch-context + ⑫ 的回退诊断含"失败项 B03 / gate 结果：FAIL=3"。**若诊断或 dispatch-context 的回退诊断节用 `- FAIL B03` 这种行首格式列失败项，会被审计2 判为预判、exit 1**——而这是**合法的事后诊断**，不是预判。审计2 分不出"事后诊断真实失败"与"事前预判结论"，两者都是 `- FAIL` 行。

方案说"审计2 需适配"（第378行）但**没给出具体约束**。

**建议（具体）**：
1. **明确禁止** gate-diagnosis.md 和 dispatch-context 回退诊断节使用 `^\s*- (PASS|FAIL)` 行首格式；诊断列失败项改用 `失败项：B03, B07`（内联，非列表行首）或 `- 失败BDD: B03`（前缀不同于裸 PASS/FAIL）。方案示例里的 `- 失败项：...` 恰好安全（`失败项`不匹配 `(PASS|FAIL)\b`），但须写成**强制约束**而非碰巧。
2. dispatch-context 只放 gate-diagnosis 的**路径引用**，绝不 inline 诊断内容——方案已这么说（第365行），但要和 N2.1 的格式禁令一起，作为审计2 适配的明确规格写进 ⑪/⑫。
3. 补 `check-p6-provenance.bats` 用例：dispatch-context 含"回退诊断节引用路径"时审计2 放行、含 `- FAIL` 行首时 exit 1——固化这条边界。

### N3 🟠 ⑨ P7 subagent 化改善了 producer=judge，但 gate 仍读自写文件——需实质校验（非 D2）

⑨ 把 P7 从"主 Agent 亲自做"改为"派 consistency-reviewer subagent"。**这实际上改善了 producer=judge**：当前主 Agent 既做一致性检查又自己判 gate（同体）；改为 subagent 产出、主 Agent 验 gate 后，producer≠judge。方向是对的。

**但**：P7 gate 读的是 consistency-reviewer **自写的** P7-consistency.md 里的 BLOCKER=0——这是"自写文件 gate"，subagent 可以不做检查直接写 BLOCKER=0（F5 假完成的翻版）。

**自评修正（Round 4）**：我原稿说"挂 ⑤ D2 兜底"——**这不准确**。核实 ⑤ D2 = "grep 确认**代码文件**改动"（dispatch-protocol.md 的"subagent 返回校验"节）。对 **review/report 类 subagent**（requirements-review、consistency-reviewer），产物就是 review 文件本身，文件**确实写了**，D2 grep 不到"是否真审查了"。所以 D2 治的是"执行 subagent 没改代码却报已改"，治不了"review subagent 没审查却报 approved/BLOCKER=0"。

**准确的建议**：review 类 subagent 需要一层**不同于 D2 的实质校验**——review 结论必须**引用具体产物锚点**才可信：
- requirements-review 的 approved 必须引用具体 BDD 编号 + 覆盖维度清单（不是裸"approved"）；
- consistency-reviewer 的 BLOCKER=0 必须附 DESIGN_GAP 配对项 + 跨文件引用清单（证明真做了交叉检查）。
- 主 Agent 验 gate 时（或 gate 脚本）检查 review 文件**含这些锚点**，而非只查 `status: approved`。这是"自写文件 gate"的通用加固：**要求自证附带不可空手写出的具体锚点**。方案应为 ⑧⑨ 的 review 产出补这条"实质锚点"要求，并考虑是否加进 check-gate 的 P1/P7 分支（轻量正则查锚点存在性）。

（注：执行类 subagent——analyst/architect/implementer/verifier——的假完成仍由 ⑤ D2 grep 覆盖，那条不变。）

### N4 🟡 ⑧ P1 评审叠加 P2/P6 不可裁 → 微任务负担，未讨论 P1 评审可裁性

⑧ 给 P1 加强制 requirements-review。叠加此前已定的 P2 不可裁、P6 不可裁，一个 trivial 任务现在要走：P0-brief + P1写 + **P1评审** + P2写 + P2评审 + P3 + P4 + P5 + P6 + P7 + P8。**仪式成本显著上升**，正是这个项目一直警惕的"微任务不划算"。

方案没讨论 P1 评审**是否可裁**。P2 当初留了 `design_trivial`（1 候选方案但不可省）的泄压阀；P1 评审是否也该有类似的"trivial 时可简化"（如 analyst 自检替代独立 review，或明确标注可裁条件）？

**建议**：为 ⑧ 补一节"可裁性"——明确 P1 评审在什么条件下可简化/可省（如 `follows_existing_pattern` 或纯文档任务），与 P2 的 `design_trivial` 对称。否则微任务的编排开销会劝退使用者。

### N5 🟡 ⑨ P5 伪造结果有"commit→push"窗口污染下游

⑨ 把 P5 改为 verifier subagent 跑测试。方案的"外部产出 gate"论证是**成立的**：P5 结果来自 test runner exit code，CI backstop push 后重跑真实测试（=④"证据由执行生成"，非"重跑同尺子"），能暴露伪造。这与 ④ 一致，我认可。

**但有一个方案只半承认的残余**：伪造的 P5 结果在 **commit→push→CI 之间**仍会流向下游——P6 verifier 会"复用 P5-test-results 避免重复跑"（verifier.md 输入已写）。即一个任务的 P6/P7/P8 可能在本地跑在**伪造的 P5** 上，直到 push 时 CI 才拦。分支保护保证伪造进不了 main，但**本地任务链会被污染、白跑**。

**建议**：⑨ 补一句残余风险标注 + 缓解——主 Agent 按 ⑤ D2 对 P5-test-results 做最小校验（grep test runner 的真实输出签名，如 pytest 的 `passed`/`failed` 汇总行、bats 的 `ok/not ok`），不信裸 `failed=0`。把"commit→push 窗口 + 下游污染"如实写入残余风险，别只说"commit前无法100%验证"。

### N6 🟢 流程图小问题（clarity）

1. **P6 双 gate 节**（P6CK → G6）：P6CK 含 provenance、G6 是"BDD总数对照 主Agent核实"，两个 gate 节的分工没解释，读者易困惑。建议合并或标注"P6CK=脚本审计层，G6=主Agent核实层"。
2. **SCOPE+ × ⑧ 交互未画**：SCOPE+ 定向回补 P1（第120行），但回补后是否重触 ⑧ 的 P1 评审？未示。建议补一条边或注明。
3. 其余流程图质量很高，回退分支/PAUSED恢复/诊断落盘都画全了，diff 标注与 ③ 一致。
