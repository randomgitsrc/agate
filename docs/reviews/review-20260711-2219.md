## 二、新 blocking：M1 🔴 check-gate.sh P1 与 trivial 路径未接上（G2.5 同类）

**这是 N4 修复时引入的新 gap。** 两处规格在不同小节，未接上：

- **⑧ 修改清单第 4 项**（plan L145）：`check-gate.sh` P1 分支**无条件**增加"`P1-review.md status: approved AND agent≠main`"检查。
- **N4**（plan L134）：`requirement_trivial` 时"**无需 P1-review.md**"，analyst 自检写在 P1-requirements.md。

**冲突**：check-gate.sh P1 无条件要求 P1-review.md，但 trivial 任务不产出该文件 → **照写实现，每个 requirement_trivial 任务都会在 P1 gate 因缺 P1-review.md 而 exit 1**。这与 G2.5 同类——gate 分支没处理一条合法的替代路径，对合法输入给出错误的阻塞结果。

注意：`check-pruning.sh`（N4 提到的"可简化"）与 `check-gate.sh`（实际 gate）是**两个脚本**。check-pruning 校验的是"裁剪声明是否合法"，不改变 check-gate 的 pass/fail。所以即使 check-pruning 放行 P1_simplified，check-gate P1 仍会因缺 P1-review.md 失败。

**修法（必须写进 ⑧ 的 gate 逻辑，不能靠"显然意图"——G2.5 教训正是"注释说一套代码做一套"）**：

check-gate.sh P1 分支改为**条件分支**：

```
读 P1-requirements.md 的 P1_simplified 字段：
  IF P1_simplified == true:
      # trivial 路径：不要求 P1-review.md
      要求 P1-requirements.md 含自检节（P1_simplified:true + requirement_trivial:<理由>
        + BDD 编号 + 覆盖维度自检），缺则 exit 1
      （trivial 条件本身的合法性由 check-pruning.sh 校验）
  ELSE:
      # 正常路径
      要求 P1-review.md status:approved + agent≠main + 含 BDD-/B[0-9] 锚点，缺则 exit 1
```

并补 bats：`unit/check-gate.bats` 增加——
- P1_simplified:true + 自检节含锚点 → exit 0（不要求 P1-review.md）
- P1_simplified:true + 自检节缺锚点 → exit 1
- 正常任务缺 P1-review.md → exit 1
- 正常任务 P1-review.md agent=main → exit 1

**这条不修，requirement_trivial（N4 的泄压阀）会把每个微任务卡死在 P1 gate——恰好是 N4 想缓解的微任务场景，反被 gate 拦死。**

## 三、次要

### M2 🟡 ⑩ 迭代循环表未标注 trivial 例外

⑩ 的 do→review 循环表把 P1 列为"analyst 写 → requirements-review 循环"，是**正常任务**的情形。N4 的 requirement_trivial 路径无独立 review（analyst 自检），此时无 review 循环。⑩ 表应加一列或一句注明"P1 review 循环仅正常任务；requirement_trivial 时为 analyst 自检，无循环"，与 N4 一致，避免读者以为 P1 review 循环无条件适用。
