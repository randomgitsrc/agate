# 生产环境隔离机制的来历

> 日期：2026-06-16（事故发生于 2026-06-13，本文档为事后提炼）
> 状态：已决策，已落地
> 落地位置：`dispatch-protocol.md`「环境隔离（强制，所有阶段适用）」一节、`[PROD_TOUCHED]` 标记机制
> 原始事故记录（项目特定细节）：PeekView 仓库 `docs/reviews/workflow-v4-postmortem-T005-T006-2026-06-13.md`（非本仓）

## 为什么 dispatch-protocol.md 里有这条强制规则

```
- 调试/验证必须使用 P0-brief 的 debug_env 声明的测试环境，严禁直接操作生产环境
- 开发全程不应接触生产环境；若意外接触，立即停止并标注 [PROD_TOUCHED] 报告主 Agent
```

这条规则不是预先设计出来的，是一次真实事故换来的。PeekView 项目用 agate 前身（workflow-v4）执行两个任务时，发生了生产数据库被测试数据污染、随后清理操作又误删了用户真实数据的连续事故。

## 通用教训（剥离 PeekView 项目细节后）

**直接原因**：subagent 在 P4 阶段调试代码时，没有显式指定测试环境路径，代码按默认配置直接连接到了生产环境的数据存储。这是一个具体的、技术性的疏漏——任何项目只要存在"代码默认配置指向生产环境，需要显式参数才能切换到测试环境"这种模式，都有同样的风险。

**更深层的流程原因，这才是真正通用、值得记录的部分**：

1. **P4 派发 prompt 没有显式要求 subagent 使用隔离环境**。subagent 不知道项目有标准的测试/调试环境，就可能走默认路径。
2. **gate 判定范围太窄**。P5 gate 只验证了"测试结果对不对"（pytest exit 0），没有验证"过程是否触碰了不该触碰的东西"。一个测试可以全绿，同时生产环境已经被污染了——这两者不冲突，gate 没有覆盖到这一层。
3. **裁剪判断不够保守**。涉及多端改动（API+Service+CLI+Client 四层）的任务被裁剪掉了验收阶段（P6），而验收阶段恰恰是有机会发现"哪里不对劲"的最后一道关卡。主 Agent 接受了这个裁剪建议，没有自己重新判断这个任务是否真的"小"。
4. **主 Agent 的产出校验太浅**。只检查了"文件存在 + Header 合法"，没有检查改动本身是否涉及环境安全风险（比如改动的代码路径是否可能触碰真实数据）。

## 为什么这次事故指向了"主 Agent 单点故障"这个更大的问题

仔细看上面四条原因，1、2、4 都是可以用具体规则修补的（已经修补，见 dispatch-protocol.md 的环境隔离条款）。但第 3 条——"主 Agent 接受了一个不该被接受的裁剪建议"——修补方式只能是"要求主 Agent 以后更谨慎"，这不是规则补丁能解决的，因为最终判断权始终在主 Agent 手上，没有任何外部约束力。

这正是 `main-agent-oversight.md` 讨论的问题的真实案例：**具体的、已知的风险可以用强制信号堵住（`[PROD_TOUCHED]` 机制），但"主 Agent 的判断力本身是否可靠"这个更根本的问题，这次事故的整改并没有触及，只是让同一个单点"下次更小心"。**

## 给新项目接入 agate 的启示

如果你的项目存在"测试环境和生产环境共享配置入口、需要显式参数才能区分"这种模式（这种模式比想象中常见），在 P0-brief 的 `env_constraints.debug_env` 字段务必写清楚具体的隔离方式（端口、数据目录、环境变量），并在 P4 派发 prompt 里显式引用这个约束，不要假设 subagent 会自己想到。涉及多端改动的任务，裁剪验收阶段（P6）前先问自己：如果这个任务出问题，P6 是不是本来能发现？如果答案是"是"，就不该裁剪。
