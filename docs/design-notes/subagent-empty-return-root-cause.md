# Subagent 空返回根因分析与缓解

> 日期：2026-06-25
> 状态：已验证（steps 假设被证伪，落盘假设已验证有效）

## 问题

主 Agent 派发 general subagent 执行复杂任务（如 10 文件交叉评审），subagent 返回空 `<task_result></task_result>`。中间过程不可见（LIMITATIONS.md 局限 4），无法判断原因。

## 验证实验（2026-06-25）

| 实验 | 任务 | 配置 | 结果 |
|------|------|------|------|
| 1 | 5 文件交叉评审 | 无 steps | ✅ 完整返回 |
| 2 | 10 文件交叉评审 | 无 steps | ❌ 空返回 |
| 3 | 10 文件交叉评审 | steps: 30 | ❌ 空返回 |
| 4 | 10 文件交叉评审 | steps: 15 | ❌ 空返回 |
| 5 | 5 文件 + 落盘指令 | 无 steps | ✅ 完整返回 + 进度文件 |
| 6 | 10 文件 + 落盘指令 | 无 steps | ✅ 完整返回 + 进度文件（114 行） |

### 结论

**`steps` 假设被证伪**：steps:30 和 steps:15 都不能解决空返回。到 steps 上限时，subagent 仍然返回空。根因不是迭代次数。

**落盘指令已验证有效**：同样是 10 文件任务，加"每读完一个文件就写进度到 /tmp/opencode/xxx-progress.md"指令后，subagent 产出了完整的 10 文件交叉评审报告（含 7 个问题发现）+ 114 行进度文件。

**为什么落盘指令有效**（推测）：
- 落盘指令改变了 subagent 的工作模式——从"读完全部再写"变成"读一个写一个"
- 每个 read → write 循环是独立的小步骤，subagent 不会一次性读 10 个文件把上下文占满
- 写入进度文件本身也是"输出"，可能帮助模型保持上下文活跃（不是只输入不输出）
- 即使最终返回空，进度文件也有中间产物

**实际根因推测**：不是"上下文窗口被输入占满"（1M context 足够装 10 个文件），更可能是**模型在处理超长上下文 + 复杂推理任务时的"放弃行为"**——当任务过于复杂、模型不确定如何组织回复时，它倾向于返回空而非部分结果。落盘指令通过把大任务拆成"读一个→写一个"的小循环，降低了每次推理的复杂度。

## 可落地的缓解

### 缓解 1（已验证有效）：prompt 里要求"分阶段落盘"

在派发 prompt 里加：
```
每读完一个文件，立即把关键发现追加写入 /tmp/opencode/{task_id}-progress.md（追加模式）。
即使最终无法产出完整报告，progress 文件也能让主 Agent 知道你做了什么。
```

**验证结果**：10 文件任务从空返回变为完整返回 + 114 行进度文件。

**对 agate 的影响**：dispatch-prompt 模板可以加这条要求。这不是 agate 协议层的规则，是派发 prompt 层的最佳实践。

### 缓解 2（未验证）：`question` 改为 `allow`

```json
{
  "agent": {
    "general": {
      "permission": {
        "question": "allow"
      }
    }
  }
}
```

未验证——subagent 的 question 机制是否支持 subagent→主 Agent 双向通信不明。

### 缓解 3（已证伪）：`steps` 上限

`steps: 30` 和 `steps: 15` 都不能解决空返回。到 steps 上限时 subagent 仍返回空。不推荐。

## 不可落地的（需 OpenCode 平台支持）

| 能力 | 当前状态 | 需要的 |
|------|---------|-------|
| subagent 活动信号 | 不可观测 | Task 工具暴露 subagent 的工具调用日志/输出流 |
| 基于活动的智能超时 | 不存在 | 有活动续期，无活动终止 |
| subagent 上下文使用量 | 不可观测 | 暴露 subagent 当前 context 使用率 |

这些是 LIMITATIONS.md 局限 4 的"根本解决"，超出 agate 协议范围。

## 对 agate 的影响

1. **dispatch-prompt 模板加"分阶段落盘"要求**——已验证有效，是最直接的操作性缓解
2. **LIMITATIONS.md 局限 4 补充**——"`steps` 配置不能缓解空返回（已证伪），prompt 层的'分阶段落盘'指令有效（已验证）"
3. **dispatch-protocol.md 空返回恢复策略**——补充"落盘指令可预防空返回"
