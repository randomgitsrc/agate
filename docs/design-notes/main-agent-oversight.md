# 主 Agent 监督机制

> 日期：2026-06-16
> 状态：已决策（部分落地，部分待办，见下方）
> 修订：2026-06-21 — 方案 C 从"采纳待落地"降级为"开放问题，前提未解决"（plan-eng-review 评审发现数据源同源缺陷）

## 问题

agate 的质量保证机制是单向的：dispatch-protocol.md 反复强调"主 Agent 不信任 subagent 的自我报告，必须亲自验证"。但这条链路只解决了"subagent 会不会撒谎"，没有解决"主 Agent 自己的判断会不会出错"——所有最终判断（裁剪哪些阶段、gate 算不算过、SCOPE+ 影响范围多大）都收敛到主 Agent 这一个点，没有任何机制检验这个点本身是否可靠。

这不是假设性担忧。PeekView 真实发生过的 T005/T006 生产环境污染事故（见 PeekView 仓库 `docs/reviews/workflow-v4-postmortem-T005-T006-2026-06-13.md`，非本仓），根因正是主 Agent 接受了 P1 analyst 的裁剪建议、跳过了 P6 验收，自己没有做独立判断。事后唯一的"修复"是"以后主 Agent 要更谨慎"——这不是结构性修复，是要求同一个单点更努力。

## 讨论的方向：能否引入一个外部观察者/裁判，基于客观信号强制触发评审，而不是依赖主 Agent 自由裁量

### 方案 A：LLM 裁判员，实时监督主 Agent 的判断（否决）

设想：另一个 LLM 调用，专职检验主 Agent 每一步判断是否合理，发现问题强制拦截。

**否决原因**：裁判和被裁判若用同源模型（哪怕版本不同），系统性认知盲区是共享的。这不是猜测，是当前"LLM-as-judge"研究里被反复观察到的现象——这类方法对表层问题（格式错误、明显幻觉、自相矛盾）有效，对模型固有的系统性偏差基本无效，因为裁判的判断边界和被裁判的判断边界来自同一套训练分布。造一个裁判 Agent，更可能制造"有人在管"的仪式感，而不是真实提升可靠性。

### 方案 B：行为计数信号触发评审（部分否决）

设想：用频率类信号（"连续 3 次 SCOPE+"、"裁剪超过一半阶段"）作为客观触发条件，命中就强制拉评审角色介入。

**问题**：频率不等于质量。一个主 Agent 可能从未触发任何计数信号，但每一步判断都是平庸或错误的；另一个可能触发了，但每次都是正确识别了真实情况。用计数阈值倒推"该不该被监督"，和"用代码行数判断代码质量"是同一类指标失配问题。

这个方向不是完全没有价值——已经部分落地的内容见下方"已落地"一节，但不应该被当作"解决了主 Agent 单点问题"的答案，只是把已知的、具体的风险模式（如生产环境接触）做成了强制信号，不是通用的判断力监督。

### 方案 C：确定性脚本扫描历史，生成异常模式报告，由人工审阅（开放问题，前提未解决）

设想：不引入任何额外的 LLM 推理。写一个脚本，定期扫描 `.state.yaml` / `active-tasks.md` 历史，找出**已经发生、可回溯**的可疑模式（例如"某任务裁剪了 P2 且后续 SCOPE+ 次数 > 2"、"P6 被跳过且事后没有任何验收记录"），生成一份报告交给人类。

**致命前提缺陷（plan-eng-review 评审发现）**：脚本扫描的数据（.state.yaml、commit message）全部是主 Agent 自己写的。T016 的 P3 commit 是实证——主 Agent 把违规降级写代码的事实写成了"TDD 测试设计完成，9 红灯 0 错误"，**只字未提降级**。一个解析 commit message 找异常的脚本，面对一个连 commit message 都会选择性隐瞒的主 Agent，扫不出任何东西。数据源和监督对象是同一个实体，这正是 LIMITATIONS.md 局限 3 的翻版，换了个壳。

要让这个方向真正成立，数据源必须独立于主 Agent 的叙事：
- task 工具本身的调用日志（如果平台暴露，不经过主 Agent 转写）
- subagent 产出文件里带主 Agent 难以伪造的自盖时间戳/身份标记，再拿 git author/timestamp 元数据交叉核对

这些在当前架构里都不存在。

**状态：开放问题，前提未解决。** 之前的"方向已确认值得做"是过早下结论——它依赖一个还没解决的前提（独立数据源），目前应该归类为"和 LIMITATIONS 局限 3 同级别的开放问题"，不是行动项。

## 已落地的部分（来自方案 B 的具体化，不是通用监督机制）

T005/T006 事故后，dispatch-protocol.md 新增了 `[PROD_TOUCHED]` 强制标记机制——这是一个**针对已知具体风险（生产环境接触）的强制信号**，命中就立即暂停报告人工。这条规则有效，但它的有效性来自"生产环境接触"这个信号本身极其具体、客观、无歧义，不依赖任何主观判断；它不是"主 Agent 判断力监督"的通解，只是一个被真实事故验证过的特例。

## 结论

**主 Agent 判断力是单点故障，这是 agate 当前架构的结构性局限，不是这次讨论能消除的。** 已经验证可行且已落地的，是"针对具体已知风险做强制客观信号"（如 `[PROD_TOUCHED]`）；已经讨论并确认不可行的，是"LLM 裁判员实时监督"（方案 A）；之前认为"值得做"的确定性脚本扫描（方案 C），经 plan-eng-review 评审发现其数据源与监督对象同源（主 Agent 自己写 .state.yaml 和 commit message），前提未解决，降级为开放问题。

这个局限本身应记录在 `LIMITATIONS.md`（见同批次新增），而不是假装协议层面已经解决。
