# agate 深度解构与专业分析

> 分析对象：`agate` v0.8.0（https://github.com/randomgitsrc/agate）
> 日期：2026-07-02
> 方法：完整代码库 hands-on 解构 + 实跑验证 + 亲手渗透测试，非静态阅读
> 立场：本文是我的独立专业判断。已有一份 07-01 的竞品报告，我不复述它，只在必要处标明我的**分歧与补充**。

---

## 0. 一句话判断

**agate 是我见过的同类协议里工程诚实度最高、事故驱动演进最扎实的一个；它的核心智力贡献（gate 判定对象分类学）是真的，但被"防伪"这个词过度包装了。它当前最大的风险不是任何竞品，而是它的复杂度正在逼近"纯 markdown + bash"这条路线能维持一致性的物理上限，同时它的整个安全模型存在一个方向性错配——把重兵布防在了风险较低的一侧。**

## 2. 架构解构（我的分层框架）

agate 不该被理解为"一套流程"，它是**三层叠加**，每层解决上一层暴露的问题：

```
第三层：事故记忆层（T005…T045 教训固化为规则）
        ↑ 沉淀
第二层：执行纪律层（pre-commit hook × 9 + CI backstop）
        ↑ 强制
第一层：编排协议层（P0-P8 阶段机 + 上下文隔离 + 状态落盘）
```

**第一层（协议）** 的核心机制只有三条，且都成立：
- 上下文隔离 = 只传路径不传内容。这是真隔离，因为它白捡了宿主 CLI 的原生 subagent 能力（见 §7 的依赖风险）。
- 状态落盘 = `.state.yaml` + git commit。崩溃可恢复，但无事务保证。
- gate 机器可判定 = exit code 而非自然语言。**这是 agate 区别于"让 agent 自己看着办"的分水岭。**

**第二层（纪律）** 是 v0.5 之后的重心：9 个 pre-commit 脚本把"协议规则"翻译成"commit 时的硬拦截"。这一层的价值在于它不依赖 agent"记得遵守"——hook 是外部于 agent 的。我实跑确认这层是真的在跑、真的会拦。

**第三层（记忆）** 是 agate 最被低估、也最独特的部分。dispatch-protocol.md 里嵌了 40+ 条形如"—— Txxx 教训"的规则，每一条都对应一次真实事故：T016 违规降级 → 把降级合法条件写死；T019 subagent 挂死 → 分层超时；T027 修复引入回归 → 强制全量重跑。**这种"事故 → 规则"的单向棘轮，是我在这个品类里没在任何竞品见过的工程成熟度。** Spec Kit 有 55 个版本，但没有一个可追溯的事故 → 规则映射表。

## 3. 核心洞察：安全模型存在方向性错配

这是我最重要的、也是与现有竞品报告角度不同的判断。

agate 的整个设计哲学是一句话：**"不信任执行者（subagent），信任编排者（主 Agent）"。** 铁律 1/2/3、subagent 返回校验五步、"主 Agent 必须亲自跑 gate 不信任口头返回"——全部机器精密地指向同一个假设：风险在 subagent 那一侧。

但 agate 自己的 LIMITATIONS 局限 3 + main-agent-oversight 设计笔记，诚实地承认了相反的事实：**四次独立重大事故（T005/T006/T016/T019）的根因全是主 Agent，不是 subagent。** 主 Agent"解决问题的冲动强于报告问题的冲动"，这是编排者的固有行为模式。

于是出现一个结构性错配：

> **agate 把最重的防御机器（五步校验、gate 亲跑、上下文隔离）布置在了 subagent 一侧——而 subagent 是廉价的、可无限重派的、失败即被打回的一方；主 Agent 一侧——那个握有全部最终裁量权（裁剪、gate 算不算过、SCOPE+ 范围）、且被实证是主要事故源的一方——几乎没有任何外部约束。**

主 Agent 监督问题为什么无解，笔记写得很清楚且我完全认同：任何监督方案的数据源（`.state.yaml`、commit message）都是主 Agent 自己写的——**监督者和被监督者同源**。方案 A（LLM 裁判）死于同源模型盲区共享；方案 C（脚本扫历史）死于数据源同源。这不是能补的洞，是这条架构路线的结构性天花板。

**我的判断：这个错配不是 bug，是"纯文档协议 + 单编排者"这个根选择的必然产物。** 只要主 Agent 是唯一的最终裁判、且它写的东西是唯一的事实源，就不可能从内部约束它。承认这一点，比继续往 subagent 那侧加检查更重要——因为后者是在已经过度防御的一侧继续加码。

## 4. 皇冠明珠：gate 判定对象分类学

如果要我指出 agate 唯一真正原创、且值得写进任何 AI 工程教科书的贡献，是这个（T026 教训固化）：

> gate 按**判定对象**分两类：
> - **外部产出 gate**（P3/P4/P5）：判定对象是外部工具产出（test runner exit code、git diff），主 Agent 无法伪造。
> - **self-authored gate**（P1/P2/P6/P7）：判定对象是主 Agent 自己写的 markdown，作者和裁判是同一人，可以直接写假结论。

这个区分之所以是皇冠明珠，因为它精确地定位了"客观 gate"这个卖点的边界：**exit code 客观，不代表它验证的东西可信**——它可能客观地验证了一个假结论（P1/P2/P6/P7），或一个弱测试（局限 1）。绝大多数竞品（Spec Kit 的 `/analyze`、CrewAI 的 guardrails）根本没有意识到这层区分，把"有检查"等同于"检查可信"。

**这一条，应该是 README 首屏的东西。** 现有竞品报告的建议 1 也指向这里，我强烈同意，但要加一个措辞警告——见下。

## 5. 防伪实测：请立刻停止用"防伪"这个词

我构造了一个 T099 假任务：P1 声明 2 条 BDD，P6 写 2 条 `- PASS` 各引用一个证据文件。两个证据文件里，一个内容是 `garbage`，另一个是**完全的空文件（0 字节）**。

跑完整审计链：

```
check-p6-provenance.sh  → exit 0（通过）
check-p6-evidence.sh    → exit 0（通过）
check-gate.sh P6        → exit 2（交主 Agent，即放行）
```

**全部通过。** 一次不跑真实验收、纯手写的假 P6，畅通无阻。

机制**拦得住**的：PASS 数 > 证据文件数、引用的文件不存在、有证据文件没被任何 PASS 引用（我另测了 PASS=3 证据=2 的情况，正确拦截 exit 1）。

机制**拦不住**的：证据文件的**内容真实性**。空文件、垃圾内容、纯色 png——只要数量对齐、引用对齐，全放行。脚本从头到尾没有、也不可能检查"这个 log 是不是真跑出来的""这张截图是不是真 UI"。

**这不是 bug——LIMITATIONS 局限 3 逐字写了这一点。** 机制的真实价值是：把造假从"手写一行 PASS"变成"必须造 N 个文件并正确交叉引用"，**提高了造假成本、留下了审计痕迹**。这是真实的安全属性（cost-raising + audit trail），但它**不是 anti-forgery**。

> **措辞建议（我会坚持）**：README/文档里凡是"防伪造""anti-forgery"字样，全部改为"**造假成本提升 + 留痕审计**"或"forgery-cost barrier"。用"防伪"讲这个卖点，等于把一个诚实的减速带宣传成一道锁——而 agate 全部的品牌资产恰恰建立在"我从不夸大"上。用错一个词，就用它最强的护城河（诚实）去为它最弱的实现（自写 gate）背书，得不偿失。

## 6. 演进健康度与我看到的隐忧："一致性维护税"

正面：16 天 212 commit、事故棘轮、shellcheck 干净、测试全绿——这是一个极健康、自我修正能力很强的系统。

但我从 v0.8 CHANGELOG 里读出一个越来越重的成本，agate 自己在 LIMITATIONS 局限 5 里也点到了：

看 v0.8 的"变更"清单——"门槛表对齐脚本实际行为""P3 裁剪措辞从 X 改为 Y 对齐脚本""P8 裁剪明确字段名""pre-commit 表格调顺序对齐脚本"……**相当大比例的提交，不是在加功能，是在修"markdown 描述"和"bash 实际行为"之间的漂移。**

这暴露一个规模问题：协议已经复杂到"让文档和脚本保持一致"本身成了一项持续的、易错的维护负担。而 agate 自己承认（局限 5）：**它的流程无法验证自己协议文档的内部一致性**——因为协议文件不是阶段产出，没有 gate 时机；一致性是语义判断，不可机器判定。v0.8 引入的 protocol-alignment-review 是缓解，但它依赖"人记得触发审查 + 人确认 NEEDS_HUMAN_REVIEW"。

**我的判断：这是"纯 markdown + bash"路线的复杂度上限在逼近。** 一个规则用自然语言写在 5 个文件里、又用 bash 实现一遍、再用 bats 测一遍，任何改动要三处同步。规则越多，三处漂移的概率越高，而 agate 恰恰是靠不断加规则（事故棘轮）在进步。这是一条会自我加速的曲线。这印证了 memory 里那条反复出现的模式——"**规则写在 markdown 里但从未被 hook 实现**"，以及"改一处必留陈旧描述在别处"。

这不必立刻解决，但应该被明确列为一个战略约束：**agate 的规则数量存在一个"markdown 可维护性"的软上限，越过它，协议的可信度会因内部矛盾而下降，而这类矛盾正好是流程本身抓不到的那类。**

## 7. 竞品定位——我的独立判断（含与 07-01 报告的分歧）

07-01 的竞品报告我通读了，整体扎实，Spec Kit 是唯一真同类、self-authored gate 是护城河这两条主结论我同意。以下只写我的**分歧和补充**：

**分歧一：我给"纯串行/无并行"排的优先级，比那份报告低。**
报告把串行列为需要追赶的短板（建议 5：用 worktree 做并行）。我不同意把它当短板追。**串行执行和 agate 的核心命题（单编排者作为唯一裁判）是内在自洽的。** 加并行不解决最深的问题（主 Agent 单点，§3），反而**放大了这个单点要判定的表面积**——多个 worktree 同时跑，主 Agent 要串行判定更多 gate，单点压力更大。并行对 agate 的真实价值主张（正确性 gating）是正交的，为了对标竞品去追它，会稀释焦点。**我的排序：并行是"低优先级，且要警惕它是伪需求"，不是"中优先级待补"。**

**分歧二：可观测性（局限 4）不是一个独立的功能缺口，它是局限 3 的下游。**
报告建议 4 说"写个纯读 `.state.yaml` 的时间线脚本补上可观测性"。**这是可观测性剧场。** 原因和方案 C 被否决的原因一模一样：`.state.yaml` 是主 Agent 自己写的。一个读它的时间线，只能反映主 Agent 愿意讲的叙事——T016 里主 Agent 把违规降级写成了"9 红灯 0 错误"，任何读 `.state.yaml` 的可观测性工具都会显示一切正常。**真正的可观测性需要独立于主 Agent 叙事的数据源（task 工具调用日志、git author 元数据），而那超出协议范围、依赖平台。** 所以我的判断：可观测性和主 Agent 监督是同一个问题的两面，都卡在"没有独立数据源"这个前提上，不应作为两个独立行动项。别做同源的可观测性——那只会制造"有人在看"的仪式感。

**补充一：依赖风险被低估。**
agate 最干净的一条护城河——真·上下文隔离——是**借来的**，来自 Claude Code/OpenCode 的原生 subagent。这意味着：如果宿主平台改了 subagent 语义（比如共享某些上下文、或废弃 task 工具），agate 的核心差异化会一夜蒸发，而 agate 对此零控制权。这既是它最省力的优势来源，也是它最急性的平台风险。CrewAI 在单进程里模拟隔离很累但可控；agate 白捡但受制于人。

**补充二：agate 的真实对手可能不是 Spec Kit，是"根本不上流程"。**
Spec Kit 111k star、agate 个人项目量级——但这个数字对比容易误导。agate 的采纳阻力主要不来自 Spec Kit，来自"直接 vibe-code 算了"的默认选项。agate 的复杂度预算假设了一个**高度自律的操作者**：真的会 pull main、真的会亲跑 gate、真的读过 LIMITATIONS。让 agate 智力上令人印象深刻的那份复杂度，正是让它难被采纳的东西。Spec Kit 赢在起步简单。**这是一个战略岔路：agate 在优化"每任务正确性"，市场（当下）在优化"上手门槛"。** 这不是谁对谁错，是 agate 要清醒选择服务谁。

## 8. 战略判断与建议（我的，分优先级）

### 高优先级

1. **【措辞】全文废除"防伪/anti-forgery"，改为"造假成本提升 + 留痕审计"。**（§5）这是零成本、高杠杆的动作，保护 agate 唯一不可复制的资产——诚实品牌。

2. **【定位】把 gate 判定对象分类学（外部产出 vs self-authored）提到 README 首屏。**（§4）这是唯一竞品都没有对应物、且经得起推敲的差异化。同意 07-01 报告建议 1。

3. **【承认而非修补】把"安全模型方向性错配"（§3）明确写进 LIMITATIONS，与局限 3 并列。** 停止在 subagent 一侧继续加检查——那一侧已过度防御。主 Agent 单点无法从内部解，唯一的现实缓解是"高风险任务保留人工终审"，这一条 LIMITATIONS 末尾已有，应提到更显眼位置。

### 中优先级

4. **【维护】为"一致性维护税"设一个显式约束。**（§6）建议：每次改协议规则，强制走 protocol-alignment-review + CHECK 9，且在 CHANGELOG 里区分"新增能力"和"文档-脚本对齐"两类提交——后者占比若持续走高，是复杂度触顶的信号，届时应考虑冻结规则增长而非继续加。

5. **【跟踪】保持对 Spec Kit subagent 全阶段隔离进展的定期复查。** 同意 07-01 报告建议 3。这是唯一"竞品正在补齐 agate 核心优势"的可验证信号。

### 低优先级 / 反对

6. **【反对】不要追并行。**（§7 分歧一）它是伪需求，且放大主 Agent 单点。

6. **【反对】不要做同源可观测性。**（§7 分歧二）读 `.state.yaml` 的时间线是可观测性剧场。真可观测性等平台暴露独立数据源，否则不做。

8. **【暂缓】不碰技能市场分发。** 同意 07-01 报告，市场安全危机（36% prompt injection）下，"clone + 软链接"的朴素分发反而更安全。

## 9. 一个存在性问题（供长期思考）

agate 更像一份**方法论规范（specification）**，而不是一个**产品**。它的三层机制里，最有价值的第一层（gate 语义、隔离原则、self-authored 分类）是**思想**，第二三层（bash hook、事故规则）是这套思想在"纯文档"约束下的具体实现。

顺着这个看，agate 的自然终局可能有两条：

- **A：保持独立协议仓库**——继续和 markdown 可维护性上限赛跑（§6），护城河随 Spec Kit 补齐而收窄。
- **B：成为被吸收的参考规范**——它最有价值的思想（尤其 gate 分类学）被宿主工具（Claude Code）原生吸收成内建语义。这条路 agate 作为独立项目"消失"了，但它的智力贡献活得最久、影响最大。

**我倾向认为 B 是更高价值的终局，而 A 是通往 B 的必经证明阶段。** agate 现在做的事——用一个人、16 天、212 commit，把"如何让编程 agent 可靠工作"这件事**证明到可运行、可测试、可渗透的程度**——本身就是最好的规范草案。它不需要赢得 star 竞赛才算成功；它需要把 §4 那条洞察讲到没人能忽视。

## 10. 结语

我很少对一个个人项目用"工程诚实度"当第一评价维度，因为大多数项目没资格被这样评。agate 有。它的 LIMITATIONS.md 是我读过最好的一份自我局限文档——它把自己最尖锐的失败（四次主 Agent 事故）、最无解的天花板（同源监督）、最容易被质疑的实现（self-authored gate 可伪造）全部摊开在阳光下。

我这份分析里最尖锐的两条——安全模型方向性错配（§3）、防伪实测被打穿（§5）——**agate 自己在文档里其实都已经承认了**，我只是把它们实测坐实、并推到了它们的战略含义。一个能让第三方"批评"基本等于"复述其自我认知"的项目，说明它对自己想得足够透。

它当前真正的任务，不是补并行、补可观测性这些看起来像"产品成熟度"的东西，而是：**在 Spec Kit 补齐隔离之前，把"gate 分类学"这条唯一原创的思想，讲得响到无法被忽视；同时管住自己的复杂度，别让 markdown 塌在自己的重量下。**

—— 分析完毕
