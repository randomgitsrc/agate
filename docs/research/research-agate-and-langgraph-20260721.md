# agate 与 LangGraph：定位、替代边界与战略判断

> 日期：2026-07-21
> 状态：讨论纪要（非协议变更提案，不改变 agate 现有设计）——已通过独立评审（APPROVED，无需修改）
> 关联：`agate/LIMITATIONS.md`

## 问题背景

行业里"多 Agent 编排"这个词被 LangGraph、CrewAI、AutoGen/AG2、Google ADK、OpenAI Agents SDK 等一批**代码框架**占据了主流叙事。这些框架 2025-2026 年密集迭代（LangGraph 2025年10月 v1.0 GA、2026 Q2 加入 per-node timeout 和 v2 流式 API），共同特征是：图/状态机由代码编译执行，转移强制、不可绕过。这带来一个自然的问题：agate 这种"纯文档协议，靠主 Agent 读 markdown 自觉遵守"的路线，是否已经被这条技术趋势淘汰？

本文档的结论：**没有，但边界需要讲清楚，不能用"我们也在做编排"这种模糊说法回避真实的能力差距。**

## 一、二者不在同一个问题层面

| | agate | LangGraph |
|---|---|---|
| 解决的问题 | 让一个**已存在**的通用编码 Agent（Claude Code/OpenCode）在给人类写软件时保持工程纪律 | 构建一个**可部署的**多 Agent 应用的运行时基础设施 |
| 用户视角 | "我要让 Agent 帮我开发软件时别乱来" | "我要开发一个 Agent 产品" |
| 交付物 | 一套约定 + 脚本，挂在已有 Agent 会话上 | 一个需要部署运行的图执行程序 |
| 状态载体 | `.state.yaml` + git commit，人眼直接可读可改 | checkpointer（SQLite/Postgres/Redis），序列化对象 |
| 接入成本 | clone + 软链接 + 一个 markdown 文件 | 需要运行时环境、依赖安装、部署或长驻进程 |

这不是"谁更先进"的比较，是两个不同市场的两个不同工具。可以类比：agate 更像是给一个已经存在的团队引入的"研发流程规范"，LangGraph 更像是"一套新系统的技术栈选型"。前者约束的是行为过程，后者构建的是执行基础设施。

## 二、LangGraph 真实解决的问题（不是营销话术的部分）

有两点是 agate 结构上做不到、LangGraph 结构上能做到的，应当承认：

1. **节点级可观测性**：per-node timeout（`run_timeout`/`idle_timeout`）+ v2 流式输出，能直接判断"节点在跑但慢"还是"节点已经卡死"。agate 只能靠"分阶段落盘"这种间接手段猜测 subagent 状态，这正是 `LIMITATIONS.md` 局限4的内容，且明确写着"根本解决需要平台支持，超出 agate 协议范围"。
2. **状态转移的强制力**：图编译后不可变，不合法转移在结构上不存在，不依赖执行者"自觉遵守"。agate 的 `state-machine.md` 靠主 Agent 读文档自觉执行，理论上存在被绕过的可能（尽管 gate 脚本对客观可判定的部分做了代码强制，见下节）。

## 三、LangGraph **没有**解决、也解决不了的问题

这是最容易被忽略、但对 agate 而言是核心的一点：**图的确定性只覆盖编排层，不覆盖节点内部的判断质量。**

- 一个 LangGraph 节点内部调用 LLM 做代码评审时，判断是否可靠，仍然完全取决于那次 LLM 调用本身——图保证的是"这个节点一定会被执行一次"，不保证"这个节点判断得对"。
- agate `LIMITATIONS.md` 讨论的局限1（测试质量上限）、局限2（角色隔离是伪独立视角）、局限3（主 Agent 判断力单点故障，producer=judge=editor 同一实体）——这三条在 LangGraph 的图节点内部**原样存在**，因为节点内部依然是 LLM 在做判断，跟外面套什么编排框架无关。
- 换句话说：LangGraph 解决的是"确定性"问题（编排层的可重复、可追溯），没有解决"可信度"问题（判断内容本身是否正确）。这两个问题经常被混为一谈，但对 agate 想解决的场景来说，后者才是核心痛点。

## 四、agate 自己其实已经在走"能编译就编译"这条路

一个容易被忽略的事实：agate 并不是纯粹"靠信任"运作的。23 个 gate 脚本（`check-gate.sh`、`check-state-transition.sh`、`check-p6-provenance.sh` 等）本身就是**代码强制**的部分——它们覆盖的是客观可判定的转移条件（exit code、git diff、文件存在性、格式匹配）。真正留给"文档 + LLM 判断"的，只是无法被编译的语义部分（P2 设计是否合理、P6 验收结论是否站得住）。

这意味着 v0.11→v0.15 的演进方向（dispatch-context 单一信息源重构、P6 gate 证据-结论对应审计、CI backstop）本质上是同一条路径的延续：**把能编译的判断持续挪进脚本，把不能编译的部分留给角色隔离和人工复核，并诚实标注哪部分是"缓解"哪部分是"根治"**。这条路径本身没有走到尽头，只是需要持续投入，而不是等一次"文档转代码框架"的范式跳跃。

## 五、如果真的用 LangGraph 重新实现 agate 会发生什么

不是移植，是换了一个产品类别，代价是结构性的：

- **零基础设施原则会被放弃**——用户需要运行时环境和持久化服务，这与 agate README 强调的"不需要 Python/数据库/部署服务"直接冲突。
- **平台无关性会消失**——agate 现在能挂在任何"能读 markdown、能跑 bash"的编码 Agent 上；LangGraph 版本会是一个独立应用，节点内部需要重新实现或包装一套等价于 Claude Code 自身 agentic loop 的东西（文件编辑、工具调用迭代），这是巨大的额外工程量，且大概率做不到平台原生实现的完成度。
- **可审计性会下降**——理解一次失败需要理解 checkpointer schema、图编译产物、节点间状态传递，对"只是想让编码 Agent 靠谱一点"的用户是不成比例的认知负担。
- **换来的仅仅是**：节点级超时/可观测性（解决局限4）、结构上不可绕过的状态转移。
- **换不来的是**：局限1/2/3 一条都不会被解决，因为这是"用 LLM 做判断节点"这件事本身的属性，不是编排框架的属性。

## 六、更现实的中间路线（记录，不作为当前提案）

比"全部重写"更值得考虑的方向：**LangGraph 只作为编排的执行后端，节点不自己调 LLM API，而是派发一次 Claude Code 会话作为节点的执行体**——图负责状态持久化、超时控制、失败路由到恢复节点，实际写代码的仍是 Claude Code。这样能拿到 LangGraph 的真实优势（节点级超时、结构化不可绕过的转移），同时不需要重新发明 Claude Code 本身。

但这依然是一个**需要独立开发和运维的新项目**，不是免费升级，而是拿"零基础设施、平台无关"这条核心设计原则去换"更强的编排确定性 + 更好的可观测性"——这笔交换值不值，取决于目标用户群体是否需要生产级、多租户、高可观测性的编排基础设施。对 agate 现在服务的场景（个人/小团队用编码 Agent 干活时要一层轻量纪律），这笔代价目前判断不划算。

## 七、决策框架：什么信号出现时应该重新评估这个判断

本文档的结论不是一次性定论，以下信号出现时应重新评估：

1. 编码 Agent 平台（Claude Code/OpenCode 等）原生提供了等价于 agate 派发+gate+状态追踪的内置能力——此时 agate 的价值会收缩到"SWE 特定的 BDD 纪律 + gate 脚本库"，是否还需要独立协议层需要重新判断。
2. agate 的实际使用场景演变为需要多租户、跨会话持久编排、生产级审计——这已经超出"给单个编码 Agent 会话套纪律"的原始定位，届时"中间路线"（第六节）值得重新考虑为正式提案。
3. 局限4（subagent 不可观测）造成的实际损失（如反复出现的空返回、无法诊断的卡死）在使用中被证实是高频、高成本问题，而非理论风险——此时需要重新权衡是否值得为解决这一个局限而承担第五节列出的全部代价。

## 结论

"零基础设施、平台无关"这个设计原则对 agate 现在的定位是合理的，理由是场景匹配（给已有编码 Agent 套纪律层，而非构建 Agent 产品）、生命周期对冲（框架迭代快，协议层应更稳定）、可审计性（人人可读的 markdown+脚本 vs 图执行内部机制）。这个原则的真实代价是局限4（不可观测性），这个代价已经被诚实记录，不是被回避的盲区。LangGraph 类框架解决的是编排确定性问题，不解决 agate 真正难的判断可信度问题——因此"转向 LangGraph"不是"agate 问题的解法"，而是"换一个不同的问题去解决"，这笔交换目前不建议做。
