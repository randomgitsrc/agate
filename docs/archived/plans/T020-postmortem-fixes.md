## 实施清单

### 1. 问题 1：删超时矛盾描述（bug fix）

**现状**：dispatch-protocol.md:313 说"Task 工具本身无超时参数"，:453 说"task 工具设 generous timeout（默认 10 分钟）"。T019 卡死数小时已实证无平台超时。

**为什么修**：line 453 的"默认 10 分钟"来源无法验证，T019 实战已证伪。两句直接矛盾，误导主 Agent 对空返回的判断方向。

**动作**：dispatch-protocol.md:453 删掉"默认 10 分钟"错误描述一句，改为引用 line 313 的正确结论。保留该节其他内容（如 subagent 内部脚本硬超时机制、进展标记等）。

### 2. 问题 6：P7 grep false positive（bug fix）

**现状**：共 **3 处**用 `grep -qF '[BLOCKER]'`，匹配到"无 [BLOCKER]"文本会 false positive。

**为什么修**：T020 实战中 P7-consistency.md 结论写"一致性检查通过，无 [BLOCKER]"，grep 误判为失败，主 Agent 不得不改文本规避。

**动作**：3 处改为 `grep -E '^\s*-?\s*\[BLOCKER\]'`（只匹配行首标记）：
- state-machine.md:120（转移规则）
- state-machine.md:294（单步函数）
- dispatch-protocol.md:402（gate 表 P7 行）

### 3. 问题 4：客观信息落盘（能力补充）

**现状**：主 Agent 派发前查证的客观事实（环境状态、URL、选择器）只能写进 prompt，违反铁律 2 精神且不可复用。T020 中主 Agent 把约 50 行预探测结果全写进 prompt。

**为什么修**：这是铁律 2 的补全，不是优化。当前"只传路径不传内容"只覆盖了阶段产出文件，没覆盖主 Agent 自己查证的客观信息。这个缺口从 T016 就存在，T020 第一次被显式提出。

**动作**：dispatch-protocol.md「输入导航原则」节后新增「客观信息落盘」子节。>10 行或需复用时落盘成 `P{N}-dispatch-context.md`，prompt 只给路径。不强制（信息量小时可写 prompt）。

### 4. 问题 5：写跑分离模式（能力补充）

**现状**：subagent 既写又跑又调试，几轮试错后上下文窗口满 → 空返回。T020 主 Agent 因此降级亲自写脚本。

**为什么修**：这是能力规则不是合规规则——给了主 Agent "怎么拆"的操作工具，不是"不准降级"的禁止规则。主 Agent 没有这个工具时只有降级一条路，有了才可能不降。

**动作**：dispatch-protocol.md「长时操作 subagent 派发策略」节增加「写脚本与跑脚本分离」子节。界限明确：
- 改常量值（timeout/selector/URL）= 最小修复（合法，属于跑命令）
- 改控制流（if/else/循环/数据处理）= 重写（降级，违规）

### 5. 问题 7：截图质量标准（gate 完善）

**现状**：P6 gate 要求"UI 条件须截图"但无质量标准。T020 出现多张纯查询类 BDD 截图 md5 完全相同。

**为什么修**：截图造假直接影响 P6 gate 可信度。无标准时主 Agent 可把"截图相同合理"当借口通过 gate。

**动作**：dispatch-protocol.md gate 表 P6 行补充截图质量标准。操作类 BDD 截图必须互不相同（md5 去重），查询类 BDD 可不截图（断言值是唯一证据）。

### 6. 问题 2：空返回诊断降级为已知限制

**现状**：耗时三档诊断树基于虚构场景（"<30秒空返回"未实际发生），耗时是弱相关变量——同样耗时，应对策略可能相反。

**为什么修**：基于不存在的场景定阈值比没有阈值更危险——主 Agent 会机械套用错误结论。正确的态度是做不到就不假装能做到。

**动作**：
- dispatch-protocol.md「空返回的恢复策略」节：撤回耗时三档表述，改为间接缓解三条（记录耗时作参考 + 查中间产物 + 从任务本身分析）。标注"活动检测"为平台层缺口。
- LIMITATIONS.md 新增局限 4，内容草稿：

```markdown
## 局限 4：subagent 活动不可观测

Task 工具返回最终结果，不暴露中间过程。主 Agent 无法判断 subagent 是否在干活（在推理/调工具/输出）。

导致：
- 无法区分"subagent 卡死"和"subagent 在干活但慢"
- 空返回时无法诊断是"没启动"还是"跑了一阵放弃"

间接缓解（agate 协议层，有限但诚实）：
- 派发时要求 subagent 进度落盘（空返回的 subagent 可能也不做）
- 主 Agent 记录派发耗时作参考（弱信号）
- 空返回后检查中间产物文件判断 subagent 是否动过

根本解决（需平台支持，超出 agate 协议范围）：
- Task 工具暴露 subagent 活动信号（工具调用日志、输出流、心跳）
- Task 工具支持智能超时：有活动续期，无活动终止
```
