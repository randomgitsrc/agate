## 落地动作

### 动作 1：修 P0 模板事实错误（#7）

**文件**：`dispatch-protocol.md` L162-172

**问题**：P0-brief YAML 模板缺 `executor_env` 段，但 L177-179 自查清单要求 5 字段含 executor_env 四子项。两个模板不一致（dispatch-protocol 是 4 字段，task-files 是 5 字段）。

**修法**：在 dispatch-protocol.md L162-172 的 YAML 块中补 `executor_env` 段，与 task-files.md:83-87 对齐。

```yaml
executor_env:
  platform: {opencode | claude-code | codex | claude-project}
  has_task_tool: true
  has_local_runtime: true
  network: {full | restricted}
```

**验证**：修后 dispatch-protocol.md 的 P0 模板与 task-files.md 的 P0 模板字段完全一致，自查清单 5 项与模板 5 项一一对应。

### 动作 2：加 P0/P1 职责边界三层指引（#5 + #9，采纳二轮评审微调）

**文件**：`dispatch-protocol.md` 步骤 0（P0-brief 介绍之后、P1 派发之前）

**问题**：P0 已有"用户决策"类内容（实战中主 Agent 普遍扩展），P1 analyst 是否重写这些内容——无指引。

**修法**：加三层指引（引用/形式化/补全），不做逐字段映射：

```markdown
### P0 / P1 职责边界

P0 是"决策记忆"（PM 视角），P1 是"需求基线"（analyst 视角）。

P1 读 P0 作为输入，遵循三层处理：
- **引用**：P0 已有的决策记忆内容，P1 直接引用（不重写）
- **形式化**：P0 的验收基线，P1 转化为 BDD Given/When/Then 格式（仅改格式，不改内容）
- **补全**：P0 没覆盖的隐含需求、待确认清单、能力需求，由 P1 独立产出
```

**为什么不做逐字段映射**：
- 逐字段映射是单个项目的实践模式，不是通用规则
- 不同项目的 P0 扩展章节不同，硬编码映射会偏向特定项目
- 协议只管"原则"（P0 是记忆，P1 是基线），不管"格式"（每个字段怎么映射）

**验证**：修后 dispatch-protocol.md 有显式的 P0/P1 边界声明，P1 analyst 能据此判断"哪些从 P0 继承、哪些自己产出"。

### 动作 3："一句话"加拆分引导（#6，拒绝二轮评审的字数判据）

**文件**：`dispatch-protocol.md` L164/176，`task-files.md` L76

**问题**：评审者用 T027（130 字"一句话"）论证约束不实用。但 T027 需要 3 个后端端点 + 前端管理面板 + 分享机制——这是一个 epic 不是 task。写不出一句话 = 任务太大该拆，不是放宽措辞。

**二轮评审异议**：要求加"≤60 字 / ≤2 改动端"判据。

**拒绝字数判据的理由**：
- agate 已有「任务粒度指引」（dispatch-protocol.md L324-338），用"输出异构性"而非字数做判据，T016 验证过这是强相关变量
- 字数是弱相关变量（dispatch-protocol.md L329 已明确说"行数是弱相关变量，异构性才是强相关变量"）
- 加"≤60 字 / ≤2 改动端"是另一个 spec bloat
- T020/T021 的 task 字段 200+ 字没人拆是实践偏离协议，不是协议该迁就的信号

**修法**：保留"一句话"约束，加一句引导（不加字数判据）：

```diff
- task: {一句话描述这个任务是什么}
+ task: {一句话描述这个任务是什么}
+   （若一句话无法概括，考虑拆分为多个任务——见「任务粒度指引」）
```

自查清单 L176 同理：

```diff
- task：是否是工程视角的一句话描述（不是产品语言的模糊表述）
+ task：是否是工程视角的一句话描述。若写不出一句话 → 任务太大，拆分
```

task-files.md L76：

```diff
- task: "一句话描述任务（工程视角，不是产品语言）"
+ task: "一句话描述任务（工程视角）。若写不出一句话 → 任务太大，考虑拆分"
```

**验证**：措辞仍强调"一句话"，但加了"写不出 = 该拆"的引导信号，不加字数判据。

### 动作 4：加扩展章节提示性参考类别（#8，采纳二轮评审微调）

**文件**：`task-files.md` P0-brief 模板节末尾

**问题**：评审者建议 agate 提供官方扩展模板（user_decisions / coordination 等）。原方案拒绝并提供官方模板、显式声明"扩展自定"。二轮评审认为"从'不标准化'直接跳到'显式禁止提及'，有中间地带"。

**采纳中间地带的理由**：
- "提示性参考类别"（5 行清单）≠ "维护具体模板"（20-50 行格式）
- 前者成本低、后者成本高
- 新项目主 Agent 确实需要知道"实战常见哪些扩展类别"，从零摸索有重复踩坑成本
- 与"不维护具体模板"不矛盾——参考类别是方向提示，不是格式规范

**修法**：在 P0 模板末尾加"提示性参考 + 明确不强制 + 不维护模板"：

```markdown
## 扩展章节（项目自定）

5 字段是 agate 协议要求的最小集。项目可根据需要扩展。

**常见扩展类别（实战中验证有效，仅作参考不强制）**：
- `user_decisions`：PM 视角记录已与用户确认的关键决策
- `coordination`：与其他任务的依赖和时序约束
- `验收基线`：PM 视角的可量化验收条件
- 其他按需扩展

注意：
- 这些是参考类别而非模板。具体格式和内容由项目决定。
- agate 不维护具体模板——避免偏向单一项目实践。
- 项目可自由选择不用上述任何类别。
```

**验证**：修后 task-files.md 提供方向性参考但不强制、不维护模板。
