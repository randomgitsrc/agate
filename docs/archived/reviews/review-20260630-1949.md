## 总体判断

**这是整个 hardening 项目最有分量的一份设计——它正面尝试解决 agate 从 main-agent-oversight.md / 局限 3 起就一直没解决的根问题：主 Agent 既是执行者又是验证者。方向值得肯定，「先用降级方案、平台支持后升级」的思路也对。**

**但方案的核心安全论证有一个根本漏洞：它声称的「多重交叉验证」实际只有一层真正能硬拦截，而这一层（agent 字段检查）恰恰建立在 self-reported 数据上——这正是 main-agent-oversight.md 早已论证过「不能作为独立验证源」的东西。结果是：方案要解决的头号威胁（T026 模式：主 Agent 跳过 verifier 自写 P6）在新方案下仍可绕过，四层检查全部放行，只留一个 CI WARNING。**

这不是说方案没价值——它确实提高了「不知不觉绕过」的门槛。但它不应该宣称把 self-authored 风险从「无法检测」降为「可检测且成本高于合规」，并据此去掉 `⚠️ self-authored` 标记。**那个标记应该保留，因为风险的本质没有改变。**

---

## 一、核心漏洞：四层验证只有一层硬拦截，且建立在自报数据上

### 方案的安全模型

文档第 2.3 节声称四层交叉验证，伪造成本 > 合规成本。我逐层分析了实际效力：

| 层 | 机制 | 实际拦截效力 |
|----|------|------------|
| 层 1：agent 字段检查 | P6-acceptance.md 的 `agent:` 必须 == verifier | **唯一硬拦截层，但字段是主 Agent 自填的** |
| 层 2：phase 序列对照 | .state.yaml 当前 phase >= P6 | 主 Agent 正常走 P5→P6，序列本就合法，**不构成额外门槛** |
| 层 3：dispatch-context 审计 | P6-dispatch-context.md 不含 PASS/FAIL | 只在「误导 subagent」场景有用；**主 Agent 不派 subagent 时此文件可不存在，检查跳过** |
| 层 4：git blame | 单 author → CI WARNING | **不拦截，仅警告** |

### T026 攻击在新方案下的实际走向

文档把 T026（主 Agent 编造 11/16 条 BDD 结果）列为头号威胁。我推演了它在新方案下的完整路径：

```
主 Agent 正常走到 P6（phase 序列合法）
  → 自己写 P6-acceptance.md（不派 verifier subagent）
  → Header 填 agent: verifier（自报，一行字）
  → 不创建 P6-dispatch-context.md

层 1：agent == verifier        → ✓ 放行（字段是自己填的）
层 2：phase 已到 P6            → ✓ 放行（确实在 P6 阶段）
层 3：dispatch-context 不存在  → ✓ 跳过（for 循环 [ -f ] 不成立）
层 4：单 author                → CI WARNING（不拦截）
```

**结论：T026 模式四层全部放行，pre-commit 阶段零拦截，只在 CI 留一个 WARNING。** 而 CI WARNING 是「单 author」——但主 Agent 自写 + 自己 commit 本来就是单 author，这个 WARNING 对所有「主 Agent 亲自写的文件」都会触发，噪音极高，等于没有信号。

### 为什么这是根本问题

`agent: verifier` 是**主 Agent 自己填进文件的字符串**。这与被 main-agent-oversight.md 方案 C 否决的 `.gate-result.json` 自报、被 P2.6 否决的「last_fix_phase 自报」是**完全同一类**——self-reported 数据不能验证产出者身份，因为填字段的人和被验证的人是同一个。

文档第 6.1 节自己承认了这一点（「主 Agent 可以伪造 agent 字段」），但缓解理由是「需同时确保 phase 序列经过 P6」。我的分析表明：**phase 序列经过 P6 是正常流程的必然结果，不是额外障碍**——主 Agent 走到 P6 写 P6-acceptance.md，phase 当然是 P6。层 2 根本没有提高伪造成本。

所以文档第 2.3 节「伪造成本分析」表里写的「跳过 verifier 自己写 P6 = 需同时改 agent 字段 + .state.yaml phase 序列 = 2 处」是**错误的**：phase 序列不需要「改」，它本来就是合法的。实际伪造成本是「改 1 处（agent 字段填 verifier）」，而且这 1 处就是写文件时顺手填的一行，零额外成本。

## 二、实现层缺口：层 2 在脚本里根本没有代码

即使接受层 2 有价值（实际没有），它在 `check-agent-provenance.sh` 里**也没有实现**：

- 脚本签名收了 `STATE_FILE="${2:-}"` 参数，注释「用于 phase 序列交叉验证」
- 但脚本正文从头到尾**没有任何读取 STATE_FILE、检查 phase 的代码**
- 四个检查（P6 agent / P2 agent / dispatch-context / agent 字段存在性）全部不涉及 phase 序列

文档第 2.3 节说「这一层已有 check-state-transition.sh 保证，只需在 check-agent-provenance.sh 里确认 phase >= P6」——但脚本里这个「确认」不存在。这是设计文档自己描述的机制与给出的脚本实现不一致。

**即便补上 phase >= P6 检查也无意义**——如前所述，正常走到 P6 的主 Agent phase 本来就 >= P6。

---

## 三、这个方案真正能拦住什么

公平地说，方案不是毫无价值。它能拦住的是：

1. **subagent 漏填 agent 字段**（层 4 的存在性检查）—— 有用，但这是格式校验，不是安全机制
2. **主 Agent 在 dispatch-context 里预写 PASS/FAIL 结论**（层 3）—— 真正有效的一层，能拦住「误导 subagent」场景，因为它检查的是客观文件内容
3. **risk=medium 时主 Agent 自审 P2**（层 1 的 medium 分支 exit 2 警告）—— 软提醒，有价值

**层 3 是整个方案里唯一真正基于「客观、非自报数据」的检查**——dispatch-context.md 里有没有 `- PASS` 是事实，不是主 Agent 对自己身份的声明。如果方案聚焦在层 3 + 扩展类似的「客观内容检查」，会比现在这个「以 agent 字段为核心」的设计更扎实。

## 四、建议

### 不要去掉 `⚠️ self-authored` 标记

文档计划在 dispatch-protocol.md / state-machine.md / WORKFLOW.md 去掉 P6 的 `⚠️ self-authored` 标记，理由是「provenance 检查把风险从无法检测降为可检测」。**但分析表明风险本质没变**——核心威胁仍可绕过。去掉标记会给读者「这个问题已解决」的虚假印象，比保留标记更危险。建议：保留标记，在旁边注明「已有 provenance 部分缓解（dispatch-context 审计），但 agent 字段自报仍可伪造，根治待 Phase 3 平台支持」。

### 重新定位方案：从「身份验证」到「行为审计」

agent 字段做不到「验证产出者身份」（自报数据的固有局限）。但方案里的层 3（dispatch-context 审计）指向一个更可行的方向——**审计客观行为痕迹，而非验证自报身份**。建议把重心从层 1（agent 字段）转移到「扩展客观审计」：

- dispatch-context 审计（已有，层 3）—— 保留并加强
- 时间戳交叉：P6-acceptance.md 的内容写入时间 vs 验证脚本/截图的生成时间（main-agent-oversight.md 早就提的方向）—— 截图 mtime、pytest 运行日志这些是外部工具产物，不是主 Agent 自报
- 证据-结论对应：P6-evidence/ 里的截图数量/内容 vs P6-acceptance.md 的 PASS 条目数（已部分在 check-p6-evidence.sh）

这些都基于「主 Agent 难以伪造的外部产物」，比 agent 字段可靠。

### 诚实标注降级方案的强度

方案第 4 节表格说 P2.1「主 Agent 不派 verifier → agent=main → gate 不通过」。这只在「主 Agent 老实填 agent=main」时成立。应该改为：「主 Agent 不派 verifier 且如实标注 → 拦截；主 Agent 不派 verifier 但谎填 agent=verifier → 仅 dispatch-context 审计 + CI WARNING 可能发现」。让读者知道这是「防君子不防小人」的软约束，不是硬保证。

### agent 字段仍值得加，但定位要对

加 agent 字段本身是好的——它对**诚实但疏忽**的主 Agent 有用（提醒「这个文件该由谁写」），也为 Phase 3（git author 可用时）做了协议铺垫。只是不能把它当安全边界。定位成「协作规范 + Phase 3 铺垫」，而非「self-authored 风险的解决方案」。

## 五、P2.9 override 修复（同提交）— 实测通过

`check-pruning.sh` 的 P2.9 override 检查从死代码变成真正实现，我实测了三个场景：

| 场景 | 结果 |
|------|------|
| 声明裁剪 P3 但有 P3-tests.md，无 override | ✅ 拦截 |
| 同上 + 加 override 字段 | ✅ 放行 |
| 声明裁剪 P3 且确实无产出 | ✅ 放行 |

这个修复是对的，而且**它恰好是 provenance 方案该学习的范本**：P2.9 验证的是「P1 声明 vs 文件系统实际产出」——文件系统状态是客观事实，不是主 Agent 自报。这正是 provenance 方案缺的东西。provenance 想验证「谁写的」（自报，不可靠），P2.9 验证「实际有没有」（客观，可靠）。**方向对比鲜明。**

## 六、总评

**这份设计的价值在于：它是 agate 第一次系统性地尝试给「主 Agent 单点故障」一个工程解，而不只是在 LIMITATIONS.md 里记一笔。这个尝试本身值得肯定。**

**但它的核心机制（agent 字段）选错了支点**——把安全性建立在主 Agent 自报的字段上，重蹈了 main-agent-oversight.md 已经否决过的 self-reported 覆辙。方案精巧的「四层交叉验证」论证经不起对抗性推演：三层是软的或空的，唯一硬拦截的一层可被一行自报字段绕过。

**最关键的建议：承认 agent 字段是「协作规范 + Phase 3 铺垫」而非「安全边界」，保留 self-authored 标记，把真正的防护重心放在客观行为审计（dispatch-context 内容、外部产物时间戳、证据-结论对应）上。** 这些方向 agate 已经有零散实现（check-p6-evidence、P2.9 override、层 3），把它们系统化，比给每个产出文件加一个可伪造的 agent 字段更有价值。

**一句话**：这个方案会让诚实的主 Agent 更规范，但不会让不诚实的主 Agent 被拦住——而后者正是 T026 暴露的、方案声称要解决的问题。在它实现并去掉 self-authored 标记之前，值得先把这个根本问题谈清楚。
