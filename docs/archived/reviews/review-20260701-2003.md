# agate-self-gate 设计意见

> 对象：`docs/plans/agate-self-gate-2026-07-01.md`
> 定位：不重复上一份评审已经说过的 4 个问题（循环依赖/CHECK 8 缺测试/CHECK 8-9 重叠/WARN #7 取舍），聚焦计划本身的设计层面——锚点表完备性、触发范围、测试计划可行性、闭环机制的盲区

---

## 一、CHECK 9 锚点表覆盖了 8 个脚本，漏了 3 个

锚点表列出了 13 条锚点，涉及的脚本：

```
check-pruning.sh          ✓（5 条锚点）
check-state-transition.sh ✓（2 条）
check-gate.sh             ✓（1 条）
check-p6-evidence.sh      ✓（1 条）
check-p6-provenance.sh    ✓（1 条）
check-scope-resolved.sh   ✓（1 条）
check-retrospective.sh    ✓（1 条）
pre-commit-gate.sh        ✓（1 条）
---
check-changelog.sh        ✗ 无锚点
check-state-yaml.sh       ✗ 无锚点
check-tdd-red.sh          ✗ 无锚点
```

这 3 个都是真实的 gate 脚本，背后都有协议规则：

- **check-changelog.sh**：P8 gate 的一部分，检查 CHANGELOG 变更。`state-machine.md` 第 398 行有对应的转移规则描述，`dispatch-protocol.md` 的门槛表也列了"CHANGELOG 须包含任务 ID"。如果以后改了 P8 的 CHANGELOG 要求但没改脚本，CHECK 9 不会发现。
- **check-state-yaml.sh**：验证 `.state.yaml` 结构（task_id 格式、phase 合法值、retries 类型），`state-machine.md` 第 450 行附近有"每任务独立状态文件"一节定义了这些字段的合法值。
- **check-tdd-red.sh**：P3 TDD 红灯检查，有 8 个分支（exit 0/1/2/3），`dispatch-protocol.md` 门槛表有对应的"P3: scripts/check-tdd-red.sh exit 0"声明。

**如果只覆盖 8/11 个脚本，CHECK 9 的价值会被高估**——锚点表声称做了"协议规则 vs 脚本实现的结构兜底"，但实际有 3 个脚本完全不在监控范围内，出了语义偏差也不会被报出来。

### 建议

补齐这 3 个脚本的锚点，至少各加 1 条关键词核对：

| 锚点 | 文档位置 | 脚本 | 关键词 |
|------|---------|------|--------|
| P8 CHANGELOG | dispatch-protocol.md 门槛表 | check-changelog.sh | `CHANGELOG` / `task_id` |
| state.yaml 结构 | state-machine.md 状态文件节 | check-state-yaml.sh | `task_id` / `phase` / `retries` |
| TDD 红灯 | dispatch-protocol.md 门槛表 | check-tdd-red.sh | `pytest` / `ImportError` |

## 二、审计发现的 WARN 里有 3 条指向的规则，锚点表也没覆盖

这是一个更微妙的遗漏——计划自己的附录 A 里，WARN #4、#6、#10 分别指出了 P3 UI 用例检查、P4 gate 路径偏差、P2 候选方案"权衡+选择理由" vs 脚本只查数量这三处语义偏差。但 CHECK 9 的锚点表里没有任何一条能覆盖这三处。

也就是说：**计划已经审计出了这些语义偏差，但计划自己设计的 CHECK 9 不会在以后再出现同类问题时报出来**——即使 CHECK 9 全部锚点都 PASS，这 3 类偏差也不会被结构层拦截，只能完全依赖 LLM 审查层。

这不是说 CHECK 9 的思路有问题——关键词存在性检查天然覆盖不了"关键词存在但语义不同"的场景（计划自己在 §2.4 诚实地写了这个局限）。但如果审计里已经发现了一类偏差，**而且能为这类偏差设计出新的结构锚点**（比如 P2 候选方案的锚点可以是"check-gate.sh 里同时出现 `候选方案` 和 `权衡` 两个关键词"），就值得加进去——不是为了替代 LLM 审查，是为了在 LLM 漏审时多一层兜底。

### 建议

通读一遍 WARN #3-#10，逐条判断"是否能设计出关键词级的结构锚点"。能的就加入锚点表，不能的（比如 #5"文档说 low、脚本禁 high"——关键词层面无法区分这两者的语义差异）就明确标注"此项只能靠 LLM 审查层覆盖，CHECK 9 无法结构兜底"。这个分类本身就是有价值的——让维护者知道 CHECK 9 的盲区在哪里，而不是以为"锚点表全绿 = 协议和脚本一定对齐"。

## 三、触发范围漏了角色文件和模板文件

§3.1 的触发条件：

> - `agate/scripts/*.sh` 有改动
> - `agate/WORKFLOW.md` / `state-machine.md` / `dispatch-protocol.md` / `git-integration.md` / `platform-notes.md` / `LIMITATIONS.md` 有改动
> - `agate/scripts/check-protocol-consistency.py` 有改动

没有列出的：

- `agate/assets/execution-roles/*.md`（analyst/architect/implementer/verifier/...）
- `agate/assets/review-roles/*.md`（9 个评审角色）
- `agate/assets/templates/*.md`（task-files.md 等模板）

但 v0.6 的实施经验告诉我们：**这三类文件改动引发语义偏差的概率不低于协议文档本身。** 具体例子：

- DESIGN_GAP 功能同时改了 `implementer.md`（加偏差声明要求）、`architect.md`（加转抄要求）和 `check-gate.sh`（加 P7 核对）——如果只改了脚本没改角色文件，`check-gate.sh` 加了检查但 implementer 不知道要写 `[DESIGN_GAP:]`，gate 会拦住所有任务
- `task-files.md` 的 YAML 模板格式是 Agent 生成真实任务文件时的参照——v0.6 的 `executor_env` 缩进 bug 就出在这个模板文件里，而且 pre-commit 没拦住（因为不在 pre-commit 的触发范围内）

如果触发范围不覆盖这三类文件，以后改角色文件或模板时不会触发 LLM 审查，和"改脚本但没改配套文档"是一样的盲区。

### 建议

触发条件加上：
- `agate/assets/execution-roles/*.md`
- `agate/assets/review-roles/*.md`
- `agate/assets/templates/*.md`

或者更简洁地写成 `agate/**/*.md`（agate 目录下任何 `.md` 文件改动都触发），再加一条白名单排除"纯文档修订不影响语义的文件"（比如 `CHANGELOG.md`）。

## 四、§7.2 的 LLM 集成测试——3 个用例写成了 `.bats` 文件，但实际上不可自动化

§7.2 声称要写 `tests/integration/protocol-alignment-review.bats`，里面 3 个用例：

| # | 用例 | 预期 |
|---|------|------|
| 1 | 改 scripts/*.sh 后派发 review subagent | 产出 `agate-alignment-review-*.md` |
| 2 | 审查报告含 A1-A6 六项 | 每项有结论 |
| 3 | MISALIGNED 项有差异描述 + 建议方向 | 不为空 |

**这 3 个用例都需要真实调用 LLM 才能产出审查报告——不可能在 bats 里自动执行。** 和 §3.1 自己写的"LLM 判断不可复现，不能作为 CI gate"直接矛盾。

bats 能测的是"审查流程的脚手架"，不是"LLM 审查的输出质量"。可自动化的部分：

- 角色文件 `protocol-alignment-review.md` 存在且含必需字段（role_id/type/phases/agent）
- 派发 prompt 模板可以被 `dispatch-protocol.md` 的模板引擎正确解析（如果有的话）
- 触发条件检测：给定一组变更文件路径，判断是否命中触发条件（纯逻辑，不需要 LLM）

不可自动化、只能作为人工检查清单的部分：

- 审查报告的结构完整性（A1-A6 六项都有结论）
- MISALIGNED 项的内容质量（差异描述+建议方向）
- 对真实变更的审查准确性

### 建议

把 §7.2 拆成两部分：
1. **可自动化的部分**写成 bats 测试（角色文件存在性、触发条件逻辑）
2. **不可自动化的部分**写成一份人工验收清单（checklist），放在 `protocol-alignment-review.md` 角色文件的末尾或者单独一份文档里，标注"每次使用后人工核对"

§7.3 的回归测试同理——"审查报告标 MISALIGNED"这个预期需要 LLM 实际产出报告才能验证。

## 五、NEEDS_HUMAN_REVIEW 的闭环缺"已确认"标记

§3.6 闭环规则：

> | NEEDS_HUMAN_REVIEW | 标记到审查报告，人工确认后可 commit（附确认理由）|

这里"人工确认"的记录方式没有定义。对比协议里已有的同类机制：

- `[DESIGN_GAP]` → `[DESIGN_GAP_REVIEWED: ...]`，`check-gate.sh P7` 可以机器核对配对数量
- `[SCOPE+]` → `[SCOPE_RESOLVED: ...]`，`check-scope-resolved.sh` 可以机器核对

NEEDS_HUMAN_REVIEW 现在缺这个"已确认"标记——审查报告写了 3 条 NEEDS_HUMAN_REVIEW，人工看了其中 2 条就 commit 了，没有任何机制能发现第 3 条被漏过。

这个问题的严重度取决于实际使用频率——如果 NEEDS_HUMAN_REVIEW 很少出现（大多数项都能判 ALIGNED/MISALIGNED），风险不大；如果经常出现，就会变成一个"看起来有闭环、实际靠人记忆"的软约束。

### 建议

定义一个标记格式，比如：

```
| A3 | 一致性连锁 | NEEDS_HUMAN_REVIEW |
    → [HUMAN_CONFIRMED: 2026-07-01 确认：这是有意的设计取舍，P3 裁剪条件文档将在 v0.9 统一措辞]
```

主 Agent commit 前检查：审查报告里每条 NEEDS_HUMAN_REVIEW 下面都有一条 `[HUMAN_CONFIRMED: ...]`。这不需要脚本自动化（LLM 审查本身就不是脚本化的），但给人工确认提供一个可追溯的落盘格式，以后翻审查报告时能看到"谁确认了、理由是什么"。

## 六、§3.3 派发模板里的"审查范围"是固定列表，但实际审查范围应该随变更内容动态变化

§3.3 的 prompt 模板：

```
## 审查范围
读以下文件全文：
- {变更的协议文件}
- {变更的脚本}
- agate/state-machine.md（裁剪表、重试表、转移规则——权威规则源）
- agate/dispatch-protocol.md（gate 表、门槛表——检查项声明源）
```

`state-machine.md` 和 `dispatch-protocol.md` 是固定要读的两份"权威源"——这个直觉是对的（几乎所有协议规则的权威定义都在这两份文件里）。但实际经验表明，**有些语义偏差的"另一半"在角色文件或模板里，不在这两份权威源里**：

- DESIGN_GAP 的"implementer 要写什么标记"定义在 `implementer.md`，不在 `state-machine.md`
- P2 候选方案的模板标题格式定义在 `task-files.md`，gate 正则写在 `check-gate.sh`——两者对不对得上，需要同时看这两个文件

如果 subagent 只读变更文件 + `state-machine.md` + `dispatch-protocol.md`，遇到上面这类场景时会缺少"另一半"的上下文，可能误判或漏判。

### 建议

prompt 模板里加一句指引：

```
- 如果变更涉及 gate 检查逻辑（check-gate.sh），同时读对应的角色文件
  （implementer.md / architect.md / verifier.md）确认角色侧描述是否一致
- 如果变更涉及文件格式/字段（check-pruning.sh / check-state-yaml.sh），
  同时读 task-files.md 确认模板是否一致
```

不需要把所有文件列进固定列表——让 subagent 根据变更内容自行判断还需要读哪些配套文件，但给一个"常见的配对关系"提示，降低漏读的概率。

## 七、汇总

| # | 问题 | 性质 | 建议 |
|---|------|------|------|
| 1 | 锚点表漏了 3 个 gate 脚本 | 覆盖度缺口 | 补 check-changelog/check-state-yaml/check-tdd-red 的锚点 |
| 2 | 审计 WARN 里 3 条指向的规则不在锚点表里 | 盲区未标注 | 逐条判断能否加锚点；不能的标注"只靠 LLM 层覆盖" |
| 3 | 触发范围漏了角色文件和模板文件 | 触发盲区 | 加 `agate/assets/**/*.md` |
| 4 | §7.2 的 3 个 LLM 集成测试写成了 .bats 但不可自动化 | 测试计划矛盾 | 拆成"可自动化 bats"+"人工验收清单" |
| 5 | NEEDS_HUMAN_REVIEW 缺"已确认"标记 | 闭环缺口 | 定义 `[HUMAN_CONFIRMED: ...]` 格式 |
| 6 | 派发模板的审查范围固定列表偏窄 | subagent 上下文不足 | 加"常见配对关系"提示，引导 subagent 读配套角色/模板文件 |

以上 6 条都不影响核心设计方向（Layer 0 + Layer 1 双层叠加），全部是"方向对了、边界往外扩一圈"级别的改进。
