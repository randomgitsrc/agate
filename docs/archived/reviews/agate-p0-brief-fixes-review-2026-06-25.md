### ⚠️ #5 P0/P1 职责边界 — 部分接受，措辞需微调

**修复方案动作 2**：✅ 修，但**简化**为 2-3 句原则性指引（拒绝逐字段映射）：

```markdown
P0 是"决策记忆"（PM 视角），P1 是"需求基线"（analyst 视角）。
P1 读 P0 作为输入，**形式化并扩展**——不重写 P0 的决策内容，
而是把 PM 的判断转化为 BDD 等可验证形式，并补充 P0 没有的隐含需求识别。
```

**评估**：⚠️ **方向对，措辞有歧义**。

**接受的部分**：
- ✅ 拒绝逐字段映射（避免偏向 peekview 实践）—— 我原 review 的"逐字段映射"确实过僵
- ✅ "P0 是记忆、P1 是基线" 的对比准确
- ✅ "原则管方向、格式管实践" 的边界清楚

**异议的部分**：
- ⚠️ "形式化并扩展" 措辞**有歧义**：
  - "形式化" 听起来像"重新写为 BDD 格式"——但这算"重写"吗？
  - "扩展" 听起来像"补充新内容"——但"补充"什么？P0 没有的？
  - **实操层面**：P1 analyst 拿到 P0 后，到底**重写** user_decisions 段还是**引用**？
  - 这正是 review #5 想消除的歧义

**具体建议**：

修复方案"形式化并扩展" 改为三层指引：

```markdown
### P0 / P1 职责边界

P0 是"决策记忆"（PM 视角），P1 是"需求基线"（analyst 视角）。

P1 读 P0 作为输入，遵循三层处理：
- **引用**：P0 已有的 user_decisions / 协调依赖 / 验收基线等内容，P1 直接引用（不重写）
- **形式化**：P0 的验收基线段，P1 转化为 BDD Given/When/Then 格式（仅改格式，不改内容）
- **补全**：P0 没覆盖的隐含需求、待确认清单、能力需求，由 P1 独立产出
```

**为什么这样改**：
- "形式化" 单独不够清晰，加"仅改格式，不改内容" 限定范围
- "扩展" 替换为"补全"（P0 没有的）
- 三层关系清楚：**引用 → 形式化 → 补全**，P1 不重写 P0 已有内容

**优先级**：🟡 中（影响每个任务的 P1 阶段产出清晰度）

### ✅ #9 合并到 #5 — 完全接受

### ⚠️ #6 "一句话"硬约束 — 部分接受，边界需明确

**修复方案动作 3**：❌ 拒绝放宽措辞，**保留"一句话"约束 + 加"该拆"引导**：

```diff
- task: {一句话描述这个任务是什么}
+ task: {一句话描述这个任务是什么}
+   （若一句话无法概括，考虑拆分为多个任务——见「任务粒度指引」）
```

**评估**：⚠️ **方向对，缺边界**。

**接受的部分**：
- ✅ "写不出 = 该拆" 是有价值的元判断信号
- ✅ 不放宽措辞避免"消除该拆信号"
- ✅ T027 130 字确实是个 epic，可拆为 3 个子任务

**异议的部分**：
- ⚠️ **没说"该拆"的边界条件**：
  - 多少字 = "一句话"？30 字？60 字？100 字？
  - 多少改动端 = "复杂任务"？2 个？3 个？
  - 修复方案说"任务粒度指引"但没给具体指引
- ⚠️ **实战与规则自相矛盾**：
  - T020 实战 task 字段是 1 段 200+ 字，没人拆
  - T021 实战也是 1 段长描述 + 类比说明，没人拆
  - 修复方案加"该拆"信号后，**T020/T021 算不算违反？**
- ⚠️ "该拆"信号**有提示但无判据**，主 Agent 实际操作时仍会模糊

**具体建议**：

修复方案"该拆"引导应明确边界条件：

```markdown
task: {一句话描述这个任务是什么}
   任务粒度指引：
   - 简单任务（≤2 个改动端，task ≤60 字）→ 保持"一句话"
   - 复杂任务（≥3 个改动端，task 难收敛）→ 拆分多个 task，或 P1 BDD 阶段细分
   - P0 brief 的"详细"段可展开完整改动清单（不与 task 字段冲突）
```

**为什么这样改**：
- "≤60 字" 给明确字数边界（避免 130 字还说"一句话"）
- "≤2 / ≥3 改动端" 给改动端数量判据
- 允许"P0 详细段展开"——这正是 peekview 等项目的实际做法（T020-T027 都有"task 一句话 + 详细段"）

**优先级**：🟡 中（避免落地后 T020/T021 等已成 brief 都不符合"一句话"约束）

### ❌ #8 扩展章节官方模板 — 异议，修复方案走极端

**修复方案动作 4**：❌ 拒绝官方模板，**显式声明"扩展是项目自定"**：

```markdown
## 扩展章节（项目自定）
以上 5 字段是 agate 协议要求的最小集。项目可根据自身需要扩展章节
（如 user_decisions / coordination / 验收基线等），格式和内容自定。
agate 不提供也不维护扩展模板——扩展是项目实践，不是协议规范。
```

**评估**：❌ **走极端，有中间地带**。

**修复方案的合理之处**：
- ✅ "peekview 的扩展模式是项目特有" 我同意
- ✅ "标准化会偏向单一项目实践" 我同意
- ✅ "违反'文档协议，非代码框架'原则" 我同意
- ✅ "扩展是项目实践不是协议规范" 部分同意

**我的异议**：
- ❌ 修复方案**从"不标准化扩展模板"直接跳到"显式禁止 agate 提及扩展"**——这是非黑即白
- ❌ **有中间地带**：
  - **协议层**：不规定"用什么章节"（避免偏向）✅
  - **提示层**：可列"实战常见类别"（user_decisions / coordination / 验收基线），**不规定格式**、**不强制使用**、**明确"项目可不用"**
- ❌ 修复方案说"agate 不提供也不维护扩展模板"——**这个排除太绝对**
  - 维护"提示性类别清单" ≠ "维护具体模板"
  - 前者是 5 行提示，后者是详细格式模板（20-50 行）
  - 工作量差异巨大

**实战影响**：

修复方案做法 = "协议层完全不提及扩展" → 新项目主 Agent 不知道"实战常见哪些扩展类别"，每次都要从零摸索（5+ 个项目踩过同样的坑）
我的建议 = "协议层提'常见类别'但不强制" → 新项目主 Agent 知道"实战用过什么"，但仍可自由选择

**具体建议**：

修复方案改为"提示性参考 + 明确不强制 + 不维护模板"：

```markdown
## 扩展章节（项目自定）

5 字段是 agate 协议要求的最小集。项目可根据需要扩展。

**常见扩展类别（实战中验证有效，**仅作参考不强制**）**：
- `user_decisions`：PM 视角记录已与用户确认的关键决策
- `coordination`：与其他任务的依赖和时序约束
- `验收基线`：PM 视角的可量化验收条件
- 其他按需扩展

注意：
- 这些是**参考类别**而非模板。具体格式和内容由项目决定。
- agate 不维护具体模板——避免偏向单一项目实践。
- 项目可自由选择不用上述任何类别。
```

**为什么这样改**：
- 协议层提供"提示性参考"（低成本）——新项目主 Agent 知道"实战用过什么"
- 不规定格式（避免偏向）
- 明确"不强制"（保留项目自由）
- 明确"不维护模板"（避免维护负担）

**优先级**：🟠 中（修复方案走极端，新项目会缺指引）

## 跨项模式识别

### 模式 1：修复方案的"原则 vs 实践"边界清楚

修复方案严格区分"协议"和"项目实践"：
- 协议：5 字段、阶段链、gate 命令（必须遵守）
- 实践：扩展章节、具体格式（项目自定）

这是 agate 设计的核心精神，修复方案**坚守**得很好。

### 模式 2：3 处异议的共同点

我的 3 处异议（#5 / #6 / #8）都是"修复方案走极端"的问题：
- #5：把 review 的"逐字段映射"（过僵）推到"原则性指引"（过简）

**根因**：修复方案**矫枉过正**——从"过 X 端"直接跳到"过 Y 端"，没有"中间地带"。
