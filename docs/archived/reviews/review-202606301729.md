## 二、🔴 新发现：P4 gate 鸡生蛋悖论（阻断级）

**此问题未被计划评审、维护者自评审捕获，因为它只在真实 commit 流程里暴露。**

**位置**：`scripts/check-gate.sh` P4 分支
```bash
P4)
    git log --oneline -5 | grep -qE 'P4|wf\(T[0-9]+-P4\)' && exit 0 || exit 1 ;;
```

**问题**：P4 gate 检查「最近 5 条 commit message 是否含 P4 字样」。但 pre-commit hook 在 commit **创建之前**运行——你正在提交的这条 P4 commit 此刻**还没进入 git log**。如果这是该任务的第一条 P4 commit，`git log` 里查不到任何 P4 字样，gate 返回 exit 1，commit 被拦截。**结果：永远无法创建第一条 P4 commit，P4 阶段彻底卡死。**

**实测证据：**
```bash
# 全新任务，git log 无 P4 字样
$ git commit -m "wf(T001-P4): implement feature"
GATE P4: 未通过
git log 未新增  ← 即使 message 含 P4，pre-commit 时这条还没进 log
```

**根因**：P4 gate 的设计语义（「P4 阶段必须有实现 commit」）放在 pre-commit 阶段是**时序错位的**——pre-commit 验证的是「即将发生的 commit」，而 P4 gate 验证的是「已经发生的 commit」。这两个时态在 hook 上下文里冲突。

**这个 bug 的隐蔽性**：端到端测试（`aa2b471`）通过，是因为测试用的 commit message 恰好不含触发 P4 gate 的 phase，或者测试的 phase 停在 P2（返回 exit 2 放行）。只有真正走到「P4 阶段 + 第一条 commit」才会触发。

**修法（三选一）：**

- **A（推荐）**：P4 gate 在 pre-commit 阶段改为**检查暂存内容**而非 git log——即「本次 commit 是否包含代码文件变更」，而不是「历史是否有 P4 commit」：
  ```bash
  P4)
      # pre-commit 阶段：检查暂存区有代码文件（非纯文档/状态文件）
      git diff --cached --name-only | grep -qvE '\.(md|yaml)$|^\.state' && exit 0 || exit 1 ;;
  ```
- **B**：P4 gate 从 pre-commit hook 中移除，只在 CI backstop（post-commit）阶段检查——CI 阶段 commit 已存在，git log 查得到。
- **C**：P4 gate 返回 exit 2（需主 Agent 判断）而非 exit 1，避免硬拦截。但这弱化了 P4 的机器验证。

**建议 A**：因为它符合 pre-commit「验证即将提交的内容」的语义，且 P4 的本质就是「这次提交带来了实现代码」。

## 三、🟠 高优先级问题

### O-impl-1：CI backstop 的 timestamp 防补写验证被 `except: pass` 架空

**位置**：`scripts/ci-gate-backstop.py` timestamp 验证段
```python
try:
    ts = datetime.datetime.fromisoformat(recorded_ts.replace("Z", "+00:00"))
    commit_ts = datetime.datetime.fromisoformat(commit_ts_str)
    if ts > commit_ts:
        print(f"FAIL: ...timestamp...")
        return 1
except Exception:
    pass
```

**问题**：timestamp 比对是「防止 .gate-result.json 被事后补写」的核心机制。但任何解析异常（时区格式不符、git 命令失败、字段缺失）都被 `except: pass` 静默吞掉，验证直接跳过。攻击者只要让 timestamp 字段格式略微异常，就能绕过整个防补写检查——而 CI 不会报任何错。

**实测确认**：比较逻辑本身正确（两个都是 aware datetime，时区安全）。问题纯在异常处理——`pass` 让「防补写」在任何边界情况下静默失效。

**修法**：区分「字段缺失（合法跳过）」和「解析失败（应当报警）」：
```python
except Exception as e:
    print(f"WARN: timestamp 验证无法完成（{e}），跳过防补写检查")
    # 不 return 1（避免误杀），但至少在 CI 日志留痕，不静默
```

### O-impl-2：脚本可执行权限不一致

**位置**：`scripts/` 目录
```
-rw-r--r-- check-changelog.sh       ← 不可执行
-rw-r--r-- check-p6-evidence.sh     ← 不可执行
-rw-r--r-- check-state-transition.sh ← 不可执行
-rw-r--r-- gate-result.sh           ← 不可执行（但它是 source 的，OK）
-rw-r--r-- install-hook.sh          ← 不可执行
-rwxr-xr-x check-gate.sh            ← 可执行
-rwxr-xr-x pre-commit-gate.sh       ← 可执行
```

**问题**：6 个脚本里只有 2 个有可执行权限。当前能工作是因为它们都通过 `bash script.sh` 调用（不依赖 x 权限）。但：
1. install-hook.sh 文档说「运行 `bash scripts/install-hook.sh`」——能跑，但不一致
2. 如果未来有人改成直接 `./scripts/check-changelog.sh` 调用，会 permission denied
3. git 会记录这个权限差异，跨平台 clone 后行为可能不同

**修法**：`chmod +x scripts/*.sh`，统一权限。不影响当前功能，但消除潜在陷阱。

## 七、总评

**这是迄今为止 agate 最复杂的一次实现，质量经得起实测检验。** 上轮评审的 6 项修复全部真正落地——我把每一项都装上真实 hook 跑过，不是看代码声称修了。R1 防静默放行、O2 整数→列表、R2 扫 diff 不扫全文，这些都是容易「改了但没改对」的地方，实测证明改对了。

**但实测也证明了一件事：读代码评审会漏掉真实流程才暴露的 bug。** P4 鸡生蛋悖论（N1）在三轮评审（计划评审、维护者自评审、以及如果我只读代码的话）里都不会被发现——它只在「真实 git 仓库 + 真实 hook + 走到 P4 第一条 commit」时暴露。这印证了一个原则：**gate 系统这种东西，必须在真实提交流程里测，不能只靠静态审查。**

**最值得肯定的**：维护者的自评审主动发现了 `check-state-transition.sh` 的路径 bug 并修复了——虽然文档没同步更新，但「自己发现自己实现里最严重的 bug」这件事本身，说明这个项目的自我审查机制在起作用。

**最该立即做**：修 N1（P4 悖论），否则任何走完整 P0-P8 流程的真实任务都会卡在 P4。推荐用「检查暂存区有代码文件」替代「查 git log」，这同时修正了语义错位。
