## 🔴 阻断级（实现前必须修）

### R1：`gate-result.sh` 被 `source` 但 source 失败会静默放行

**位置**：`pre-commit-gate.sh`（Task 4）第 7-8 行
```bash
set -euo pipefail
source "$REPO_ROOT/scripts/gate-result.sh"
```

**问题**：`set -euo pipefail` 下，`source` 失败（路径错误、文件被删、权限问题）会直接触发 `set -e` 让脚本以 exit 0 退出——hook 不报错，commit 正常通过，gate 完全没跑。**这是最危险的失败模式：出了问题看起来像一切正常。**

`source` 本身不能被 `set -e` 的 pipefail 捕获为「命令失败」（它会让整个 shell 退出），所以必须显式检查。

**修法**：source 后立刻验证函数已加载：
```bash
source "$REPO_ROOT/scripts/gate-result.sh" \
    || { echo "GATE ERROR: 无法加载 gate-result.sh" >&2; exit 1; }
type write_gate_result >/dev/null 2>&1 \
    || { echo "GATE ERROR: gate-result.sh 加载不完整" >&2; exit 1; }
```

**验收**：注释掉 `gate-result.sh`，运行 hook，应该看到明确错误信息且 commit 被中止，而不是静默通过。

### R2：`PROD_TOUCHED` 检测扫文件全文而非暂存变更，必然产生误报

**位置**：`pre-commit-gate.sh`（Task 4）PROD_TOUCHED 检测段
```bash
if git diff --cached --name-only | xargs grep -l '\[PROD_TOUCHED\]' 2>/dev/null; then
```

**问题**：`xargs grep -l` 扫描的是**文件当前磁盘内容的全文**，不是这次 commit 暂存的变更。只要某个文件历史上曾经包含 `[PROD_TOUCHED]` 字样——比如 `LIMITATIONS.md`、`state-machine.md`、`dispatch-protocol.md`、任何 T026 复盘文档——本次只改了它的其他部分，hook 也会报「检测到 PROD_TOUCHED 标记」然后中止 commit。

验证：当前仓库里 `grep -r '\[PROD_TOUCHED\]' .` 有多少命中？`dispatch-protocol.md`、`state-machine.md`、`WORKFLOW.md` 都会命中——意味着每次修改这些核心协议文件就会被误拦。

**修法**：只扫描暂存的 diff 内容：
```bash
if git diff --cached | grep -q '\[PROD_TOUCHED\]'; then
    echo "GATE: 检测到 [PROD_TOUCHED] 标记，中止 commit" >&2
    exit 1
fi
```

**验收**：在 `dispatch-protocol.md`（本来就含有 PROD_TOUCHED 字样）里改一行不相关的文字，暂存后 hook 应该放行。再在暂存内容里实际加入 `[PROD_TOUCHED]` 字样，hook 应该拦截。

## 🟠 高优先级（实现时同步修）

### O1：`check-state-transition.sh` 的回退跳变检测依赖 commit message，违反协议明确规定

**位置**：`check-state-transition.sh`（Task 2）
```bash
if ! git log --oneline -5 --format='%s' | grep -qiE 'PAUSED|paused'; then
    echo "GATE STATE: 回退跳变 P${old_num}→P${new_num}，无 PAUSED 记录" >&2
    exit 1
fi
```

**问题**：这个检测的逻辑是「最近 5 条 commit message 里有没有 PAUSED 字样」。但 `state-machine.md` L371-373 明确写了：

> 检测基于 phase 编号差值，**不依赖 commit message 格式**。

协议的意图是通过状态机本身的历史判断（`.gate-history.jsonl` 或 `.state.yaml` 的 phase 序列），而不是通过 commit message 约定——因为 commit message 没有任何规范要求包含 PAUSED 字样。

另外，5 条 commit 的窗口是任意的：如果 PAUSED 发生在 10 条 commit 之前，检测就会误报。

**修法（分两档）**：

*短期（Task 2 实现时）*：降级为 WARNING，不中止 commit。`.gate-history.jsonl` 还没落地时，没有可靠数据源判断是否经过 PAUSED：
```bash
if [ "$diff" -ge 2 ]; then
    echo "GATE STATE: 警告 — 回退跳变 P${old_num}→P${new_num}（差 ${diff}），建议确认是否经过 PAUSED" >&2
    # 降级 WARNING，不 exit 1
fi
```

*长期（Task 4/9 之后）*：等 `.gate-history.jsonl` 积累了数据后，改为查历史记录：
```bash
# 查 .gate-history.jsonl 是否有 PAUSED 记录
if ! grep '"phase":"PAUSED"' .gate-history.jsonl 2>/dev/null | tail -10 | grep -q "$TASK_ID"; then
    echo "GATE STATE: 回退跳变，.gate-history.jsonl 无该任务的 PAUSED 记录" >&2
    exit 1
fi
```

### O2：`retries` 字段结构与 `.state.yaml` 实际模板不匹配

**位置**：`check-state-transition.sh`（Task 2）重试超限检查
```bash
retries_json=$(python3 -c "
...
retries = data.get('retries', {})
if isinstance(retries, dict):
    for phase, count in retries.items():
        if isinstance(count, (int, float)) and count >= ${MAX_RETRY}:
```

**问题**：脚本假设 `retries` 的值是整数（`count >= MAX_RETRY`），但 `state-machine.md` L418-425 的 `.state.yaml` 模板是：
```yaml
retries:
  P2:
    - round: 1
      failure_mode: quality
      prompt_changed: false
      adjustment: null
  P4: []
```

`retries[Pn]` 是**列表**（每次重试是一个对象），不是整数。`isinstance(count, (int, float))` 对列表永远是 False，重试超限检查永远不会触发。

**修法**：
```python
retries = data.get('retries', {})
if isinstance(retries, dict):
    for phase, attempts in retries.items():
        if isinstance(attempts, list) and len(attempts) >= MAX_RETRY:
            print(f'{phase}={len(attempts)}')
            break
```

### O3：`ci-gate-backstop.py` 的 `commit_sha` 字段语义错误，会造成审计链误解

**位置**：`gate-result.sh`（Task 1）
```bash
commit_sha=$(git rev-parse HEAD 2>/dev/null || echo "pre-commit")
```

**问题**：pre-commit hook 在 commit 发生之前运行，此时 `git rev-parse HEAD` 返回的是**上一个 commit 的 SHA**，不是本次 commit 的 SHA（本次 commit 还没有发生）。`.gate-result.json` 里的 `commit_sha` 字段记录的是「上一条 commit」，而 CI 里拿到的 HEAD 是「本次 commit」，两者不同。

如果 CI 用 `commit_sha` 字段做任何交叉核对，会一直失配。`ci-gate-backstop.py` 目前没有直接用这个字段做校验（timestamp 比对用的是 `git log -1 --format=%cI`），但：
1. 这个字段留在 `.gate-result.json` 里会让读者误以为它记录的是「本次 commit」的 SHA
2. `.gate-history.jsonl` 里的 `commit_sha` 作为审计链字段，指向错误的 commit，会污染历史记录

**修法**：重命名为 `prev_commit_sha` 并加注释，或者改为在 hook 结束时用 `COMMIT_EDITMSG` 里的内容（更复杂），最简单的是直接改名明确语义：
```bash
# 记录 hook 运行时的 HEAD（本次 commit 的上一个），不是本次 commit SHA
prev_commit_sha=$(git rev-parse HEAD 2>/dev/null || echo "pre-commit")
```
并在 JSON 里用 `prev_commit_sha` 作为 key 名。

## 🟡 中优先级（对齐协议再实现）

### M1：`check-p6-evidence.sh` 的格式假设与现行协议不一致

**位置**：`check-p6-evidence.sh`（Task 3）
```bash
BDD_COUNT=$(grep -cE '^##\s*BDD-[0-9]+' "$P6_FILE")
EVIDENCE_COUNT=$(grep -cE 'Evidence:\s*P6-evidence/' "$P6_FILE")
```

**问题**：这两个检查假设 P6-acceptance.md 里有 `## BDD-NN` 标题格式和 `Evidence:` 字段。但当前协议（`dispatch-protocol.md` L356-357）的约定是：

> 每条 BDD 验收结果必须用**行首 `- PASS` 或 `- FAIL`** 格式，便于 gate 命令 `grep -cE '^\s*- (PASS|FAIL)'` 可靠匹配。

协议里没有规定 `## BDD-NN` 标题和 `Evidence:` 字段格式。如果真实任务产出的 P6-acceptance.md 按协议约定用 `- PASS`/`- FAIL` 格式，但没有 `## BDD-NN` 标题，则 `BDD_COUNT` = 0，脚本走 exit 2（无 P6 文件）分支，检查完全失效。

**修法（两步走）**：

*第一步*：先在 `dispatch-protocol.md` 或 `assets/templates/dispatch-prompt.md` 里正式约定 P6-acceptance.md 必须包含 `## BDD-NN` 标题和 `Evidence:` 字段——这是协议扩展，需要单独的 PR。

*第二步*：协议约定落地后，`check-p6-evidence.sh` 才按新格式实现。

*临时替代方案（如果不想改协议）*：把证据检查退化为「P6-evidence/ 目录非空」（check-gate.sh P6 分支里已有），`check-p6-evidence.sh` 的 BDD 数量对照改为基于已有约定（`- PASS`/`- FAIL` 行数）：
```bash
BDD_COUNT=$(grep -cE '^\s*- (PASS|FAIL)' "$P6_FILE" || echo 0)
# 注意：此时无法检查每条 BDD 是否有独立 Evidence 引用，只能检查数量
```

## 设计层面确认正确的部分

以下几点刻意核实过，结论是正确，实现时不需要修改：

**触发条件逻辑正确。** `has_staged_phase_change`（`.state.yaml` 中 phase 字段有暂存变更）+ `has_staged_phase_output`（`P[0-9]+-*.md/.yaml` 文件有暂存变更），两个条件取 OR。覆盖了「只改 state 不改产出」和「只有产出变更而 state 未跟上」两种情形，且正常文档 commit 不触发，设计合理。

**exit code 语义贯穿一致。** exit 0 = 通过，exit 1 = 中止 commit，exit 2 = 允许 commit 但记录「需主 Agent 判断」——与 check-gate.sh 的现有语义完全对齐。`pre-commit-gate.sh` 的 gate 结果处理段（case 语句）正确实现了这个语义。

**`.gate-result.json` 不进 git，`.gate-history.jsonl` 进 git** 的区分正确。result 每次重新生成不应跟踪，history 是审计链应该保留。`.gitignore` 只忽略 `.gate-result.json` 的处理也正确。

**CHANGELOG 检查降级为 WARNING 不中止 commit** 的处理正确。CHANGELOG 是「应该做」不是「必须做才能 commit」的事，中止 commit 会产生太多摩擦。

**`check-state-transition.sh` 只在 `.state.yaml` 有暂存变更时运行**（`git diff --cached --name-only | grep -qF "$STATE_FILE" || exit 0`），避免在普通 commit 里误触发状态检查，设计正确。
