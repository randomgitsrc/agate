## 二、🟠 实测发现：触发条件缺口（新）

**此问题维护者自评审未覆盖，因为它只在「.state.yaml 损坏但 phase 行不变」的特定 commit 时暴露。**

**现象**：装上完整 hook 后，端到端测试「损坏的 YAML」场景：
```bash
# 前一条 commit: phase: P4
# 本次暂存: phase: P4 + 新增一行损坏的 "bad indent: ["
$ git commit -m "wf: broken yaml"
# 结果：commit 成功，损坏的 .state.yaml 进了仓库 ← 未被拦截
```

**根因**：`pre-commit-gate.sh` 第 0 步触发判断：
```bash
has_staged_phase_change "$STATE_FILE" && NEEDS_GATE=true   # 检查 phase 行是否变更
[ "$NEEDS_GATE" = false ] && has_staged_phase_output && NEEDS_GATE=true
if [ "$NEEDS_GATE" = false ]; then exit 0; fi   # 不触发就直接放行
```

`has_staged_phase_change` 只检测 `phase:` 行是否有新增变更（`grep -qE '^\+.*phase:'`）。如果 `.state.yaml` 被改坏但 `phase:` 行本身没变（比如在其他字段引入语法错误），触发判断返回 false，hook 在第 0 步就 `exit 0`，**根本走不到第 2 步的 check-state-yaml.sh**。

**实测确认**：check-state-yaml.sh 单独跑能正确拦截损坏 YAML（exit 1），证明脚本没问题——问题在它不被调用。

**影响评估**：中等。`.state.yaml` 是 Agent 自己写的状态文件，损坏后 hook 不拦，会让后续读取 `.state.yaml` 的脚本（read_state_phase、check-state-transition）静默拿到空值，可能导致 gate 误判。P2.15 的设计目标正是「防止 Agent 写出不符协议的状态文件导致 hook 静默失效」——但这个触发缺口让它在「phase 行不变」的损坏场景下达不到目标。

**修法**：把 `.state.yaml` 的格式校验从「依赖 phase 变更触发」改为「只要 .state.yaml 在暂存区有任何变更就触发」：
```bash
# 在第 0 步触发判断之外，独立检查：只要 .state.yaml 被修改就校验格式
STATE_BASENAME=$(basename "$STATE_FILE")
if git diff --cached --name-only | grep -qF "$STATE_BASENAME"; then
    bash "$REPO_ROOT/scripts/check-state-yaml.sh" "$STATE_FILE" || exit 1
fi
```
把这个检查提到 NEEDS_GATE 判断**之前**，确保任何 `.state.yaml` 变更都先过格式校验，再决定是否跑 gate。格式校验是廉价的（一次 YAML parse），不需要等触发条件。

## 三、P2.6 移除 — 正确的决策

维护者自评审判断 P2.6（check-fix-rerun.sh）有 🔴 语义错误并移除，这个决策正确，我完全认同：

**核心理由站得住**：pre-commit hook 只跑一次 check-gate.sh，**没有「full run」vs「partial run」的概念**。P2.6 想验证「修复后必须重跑全量测试」，但 hook 层根本观察不到「这次 gate 是全量还是增量」——它看到的只是一个 exit code。`last_fix_phase`（修复过哪个阶段）和 `.gate-result.json` 的 `phase`（当前在哪个阶段）是两个不同的东西，检查它们相等是语义错误，会误拦正常的「P5 修复 → 重跑通过 → 推进 P6」流程。

**正确的归属**：P5 修复后全量重跑的规则已经在 `dispatch-protocol.md`（T027 教训），属于**流程层规则**，由主 Agent 遵守，不是 hook 能机器验证的。`last_fix_phase` 字段保留在协议层是对的。

**这体现了一个重要原则的正确应用**：不是所有协议规则都能下沉到 hook。「修复后重跑全量」是一个关于「Agent 做事方式」的规则，而 hook 只能验证「提交内容的状态」。强行用 hook 验证一个它观察不到的属性，结果就是误拦——「错误的检查比没有检查更糟」。维护者识别出这一点并果断移除，比硬塞一个坏检查要成熟得多。

## 五、总评

**P2.15 实现质量好，P2.6 移除判断正确，这次提交整体是健康的。**

值得记录的两点：

**一、维护者自评审的成熟度在提升。** 这次它不仅发现了 bug（M1/M2），还做了一个更难的判断——识别出 P2.6 整个功能在 hook 层是语义错误的，并选择移除而非强行实现。「知道什么不该做」比「把要做的做对」更需要判断力。

**二、「触发条件」是这套 hook 反复出问题的同一个根源。** 回顾历次评审：`has_staged_phase_change` 的绝对路径 bug、check-state-transition 的路径不匹配、现在 P2.15 的「phase 不变则不触发」——全部是触发逻辑的边界问题。脚本本身的逻辑往往是对的，但「什么时候该调用这个脚本」反复出错。

**建议（结构性）**：考虑把触发逻辑统一成一个清晰的决策表，明确每个检查的触发条件——有些检查（gate 判定）依赖 phase 变更，有些检查（格式校验）应该在文件任何变更时就跑。当前所有检查共用一个 NEEDS_GATE 门槛，导致「廉价的格式校验」被「昂贵的 gate 判定」的触发条件绑架。把「文件级校验」（state-yaml 格式）和「阶段级校验」（gate 判定）的触发条件分开，能从根本上消除这一类反复出现的缺口。
