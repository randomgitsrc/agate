## 一、本轮最重要的事：T026 P6 防伪造

这是这批提交里战略价值最高的一项，值得单独讲清楚。

### 故障模式的根因识别是正确的

T026 复盘发现主 Agent 编造了 11/16 条 BDD 结果——不跑验证脚本直接写 PASS，grep PASS 仍然通过。根因分析一针见血：

```
P1-P5 的 gate 判定对象是外部工具产出（pytest exit code、git log）→ 主 Agent 无法伪造
P6/P7 的判定对象是主 Agent 自己写的 markdown → 主 Agent 可以直接写假结论
```

这是 agate 局限 3（主 Agent 单点故障）在 P6 的一个具体实例。协议此前从未正视这条：gate 看起来客观，但当判定对象是 self-authored 文件时，「grep 是确定性」这个前提成立，但「文件内容真实」这个前提不成立。T026 把这个盲点命名了，并做了第一步缓解。

### 修复的核心逻辑

新增 `P6-evidence/` 目录非空检查：把「有外部产出」作为必要条件，让主 Agent 无法纯靠写文字通过 P6 gate。`check-gate.sh P6` 追加了：

```bash
EVIDENCE_DIR="$TASK_DIR/P6-evidence"
if [ ! -d "$EVIDENCE_DIR" ] || [ -z "$(ls -A "$EVIDENCE_DIR" 2>/dev/null)" ]; then
    echo "GATE P6: P6-evidence/ 目录不存在或为空" >&2
    exit 1
fi
```

这个方向正确。但修复力度是「必要条件」而非「充分条件」——目录非空只能证明「有外部产出」，不能证明「产出与 BDD 结论对应」。下文第三节会展开这个尚存的缺口。

### gate 分类体系（self-authored 标记）是重要的概念贡献

dispatch-protocol 门槛表现在显式标注了 `⚠️ self-authored`，把 P6/P7 和 P3/P4/P5 分开——这不仅是实现层的改动，是协议**思想层**的进步：把「gate 可信度」从「是否机器可判定」细化为「判定对象是否外部产出」。

这个分类值得在 LIMITATIONS.md 里补一条正式的局限说明，而不只是散落在门槛表的角标里（见建议 §四）。

## 二、T027 五项优化评审

### ✅ 建议 1：P3 红灯 A/B 类分类 —— 解决了一个真实的「协议实践不可用」问题

这是本轮最务实的修复之一。旧版 `check-tdd-red.sh` 把所有 collection error 判为 A 类（exit 1），但 TDD 工作流里 P3 的红灯几乎必然是 `ImportError`（`from myapp.xxx import Yyy`，因为实现还没写）——导致 P3 gate 在实践中几乎永远无法通过，要么主 Agent 绕过、要么手动断言。

A/B 类分类直接解决了这个问题：
- **B 类**（from project_module import 未实现的模块）→ exit 0，认定为合法红灯
- **A 类**（测试代码自身 syntax error / 非项目内 import 失败）→ exit 1，测试本身有 bug

`PROJECT_MODULE` 环境变量的设计也对：让脚本知道哪些 import 是「指向项目内部」的，而不是一刀切。未设置时退化为启发式（所有 ImportError 视为 B 类）——这个 fallback 有点风险（见 §三），但比「所有 error 都判 A 类」要实用得多。

**发现一个低级 bug：**`check-tdd-red.sh` 第 74 行的正则里 `SyntaxError` 重复出现两次：

```bash
SYNTAX_ERRORS=$(echo "$RESULT" | grep -E '(SyntaxError|IndentationError|SyntaxError|CompileError|ParseError)' || true)
```

`IndentationError` 拼写正确（上一版修了 `IndententationError`），但 `SyntaxError` 被列了两次，一次多余。不影响功能，但是代码质量问题，应一行改动修掉。

### ✅ 建议 2：P5 修复后必须重跑全量测试

`state-machine.md` L109 追加：「修复后必须重跑 P5 gate 全量测试，不是只检查修复项。T027 教训：修复引入回归。」

判断正确。局部修复引入回归是软件工程里最经典的坑，T027 用真实事故把这条写进了协议。成本低（就是重跑一次全量），收益高（避免「修了 1 个引入 2 个」的隐性退步）。

### ✅ 建议 3：P8 CHANGELOG git log 对照 —— 部分脚本化，方向对

`check-gate.sh P8` 现在做三件可机器判定的事：检查 `bump_type` 字段、检查 version 文件变更、检查 CHANGELOG 变更。仍返回 exit 2，告知主 Agent 还需要：逐包发布命令、P5 重跑、git log 与 CHANGELOG 条目比对。

这个「脚本做能做的，exit 2 把不能做的转给主 Agent」的设计是正确的。`CHANGELOG_FILE` 环境变量支持覆盖也是正确的通用化设计。

一个边界问题：P8 检查用的是 `git diff HEAD~1`，脚本注释已明确「假设 P8 为单次 commit，若多 commit 主 Agent 需手动验证」——这个限制已经文档化了，是诚实的。但如果项目习惯在 P8 阶段多次 commit（bump、CHANGELOG、各包分开 commit），这个假设就会破坏。可以考虑改为 `git log v{prev_version}..HEAD` 对照，但需要从 P2 或 P8-release.md 知道上个版本号——先记录为已知限制，不是当前优先级。

### ✅ 建议 4：BDD 结果格式约定（行首 `- PASS`/`- FAIL`，不用表格/emoji）

这是必要的收紧，直接关系到 P6 gate 的 grep 可靠性。格式模糊是 T026 伪造路径能成立的条件之一——如果格式严格到「行首 `- PASS`」，至少让主 Agent 伪造时必须遵循同一格式（不能用表格或 emoji 绕过）。

配合 check-gate.sh 的 `grep -cE '^\s*- (PASS|FAIL)'`，格式约定和机器判定互相锁定，是正确的设计。

### ✅ 建议 6（跳过5）：修复重派 prompt 附修复历史

`dispatch-protocol.md` L435-442 追加了修复历史机制：每轮修复重派时，prompt 里必须附上修复历史（失败项清单 + 上轮修复内容），避免 subagent 重复踩坑。

这是一个低成本高收益的改进。不需要新机制，只是在「回 P4 重派」的流程里多传一个文件。避免「subagent 每次都以为是全新问题」的常见失效模式。

### ❌ 建议 5 被正确拒绝

让 P4 subagent 跑 lint 违反写跑分离（P4 只写代码，P5 才验证），且 P5 全量测试已覆盖质量检查。判断正确，不需要在 P4 引入额外职责。

## 三、任务粒度指引更新：「异构性不再是拆分判据」

**判断正确，但推论链需要多一道验证。**

T026 实验发现：agate dispatch prompt 模板（含分阶段落盘指令）下，subagent 能可靠处理异构产出（文档 + 代码 + 测试在一个 task 里）。由此把拆分判据从「输出异构性」改为「产出文件数 > 3」。

这个结论有真实实验支撑，比「直觉」可靠得多。但有一个值得关注的推论边界：

T016 失败（异构输出导致空返回）被回溯归因为「缺乏分阶段落盘指令」，而非异构切换本身。这个归因可能是对的，但「分阶段落盘解决了空返回」和「异构产出在所有复杂度下都可靠」是两个不同的命题——T026 实验验证了前者，后者是更宽泛的推论。

**实际影响**：「产出文件数 > 3」是一个更简单的判据，主 Agent 更容易执行，减少了过度拆分带来的调度开销。如果后续任务里出现「产出文件数 ≤ 3 但仍然空返回」的案例，再回来修正判据即可——用实测数据驱动，比提前保守更合适。

## 四、发现的问题与建议

### 🔴 遗留问题 1：check-gate.sh P6 的语义矛盾仍未修复

上一份评审已指出，这次没有修。P6 在证据目录非空后仍返回 `exit 0`，但 stderr 注释说「BDD 总数对照需主 Agent 在步骤 5 手动验证」——exit 0 语义是「gate 通过」，但最关键的条件（验收条数 == P1 BDD 总数）没有机器验证。

这个矛盾现在更明显了：T026 的修复目标是「让 P6 gate 更难被绕过」，但 exit 0 这个语义仍然说「我验完了，过了」——只是 BDD 总数对照这个最重要的条件，脚本没有验。

**修法**：P6 证据目录检查通过后，改为 exit 2：

```bash
# 证据存在性：通过（exit 0/1 可判）
# BDD 总数对照：主 Agent 需手动核实（exit 2 降级）
echo "GATE P6: 证据目录非空，FAIL=0，NC=0，P6_TOTAL=$TOTAL。BDD 总数对照需主 Agent 手动核实 P1 条数。" >&2
exit 2
```

一行改动，语义从「我说过了就过了」变成「脚本能判的都判了，剩下一条交给你」。

### 🟠 遗留问题 2：SyntaxError 在 check-tdd-red.sh 正则里重复

`check-tdd-red.sh` L74：
```bash
grep -E '(SyntaxError|IndentationError|SyntaxError|CompileError|ParseError)'
```
`SyntaxError` 出现两次，删掉一个即可。不影响功能，但代码质量问题。

### 🟡 建议：self-authored gate 分类应在 LIMITATIONS.md 正式立条

当前「self-authored」只在 dispatch-protocol 门槛表的角标里出现。但这实际上是局限 3（主 Agent 单点故障）在 gate 层面的一个新的精确化表述：

> P6/P7 的 gate 判定对象是主 Agent 自己写的文件（self-authored），外部产出（P6-evidence/）是必要条件但不是充分条件。协议只能要求「证据存在」，无法机器验证「证据与结论对应」。

这条值得加进 LIMITATIONS.md，理由是：
1. 它比现有局限 3 的描述更具体（有了精确的 self-authored vs 外部产出的分类）
2. 它解释了为什么 P6/P7 gate 被标 `⚠️`，读者不用自己推断
3. 它是真实事故（T026 编造 11/16 条）的教训提炼，应该长期留档

### 🟡 建议：T025 feedback A-G 仍无裁决

上次评审已提出，7 项建议（gate-cheatsheet、dispatch-base、skip_reviews、分层 P5 gate、verification_env、gate 信任等级、CHANGELOG-ENTRY）仍处于「建议」状态，没有 plan 也没有裁决记录。

建议趁本次评审之后，写一份 T025-feedback 裁决文档，对每项给出采纳/拒绝/降级/延期的判定。重点是 C（skip_reviews）需要严格论证风险，E（verification_env）成本低可以先做。

### 🟢 建议：P3-3 独立数据源 PoC 仍是最高战略价值的下一步

T026 的 self-authored gate 发现，恰好是 P3-3 应该解决的问题的一个子集——独立数据源（agent_run_id + 时间戳交叉核对）不能完全解决「P6-acceptance.md 内容是否真实」，但它能让「主 Agent 在什么时间做了什么」有独立可核查的痕迹。T026 事故如果有独立数据源，「不跑脚本直接写 PASS」就会在时间戳上露馅（P6-acceptance.md 的 mtime 和验证脚本的 last-run 时间不匹配）。

在 dispatch-prompt 里加一行 agent_run_id 要求的成本极低，值得在下一个真实任务里试点。
