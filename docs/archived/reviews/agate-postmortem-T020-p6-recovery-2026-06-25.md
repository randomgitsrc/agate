## 1. 机制问题清单

### 问题 1：dispatch-protocol.md 超时描述自相矛盾

**证据**：
- `dispatch-protocol.md:313`："Task 工具本身无超时参数。subagent 内部脚本挂起会无限阻塞主 Agent。"
- `dispatch-protocol.md:453`："硬超时：`task` 工具设 generous timeout（默认 10 分钟），防止无限等待"

这两句直接矛盾。第一句说 Task 工具无超时参数，第二句说设 generous timeout 默认 10 分钟。

**T019 实战证据**（`agate-postmortem-T019-2026-06-23.md:64`）：
> subagent 内部脚本挂起 → Task 工具无超时 → 主 agent 无限等待

T019 主 Agent 卡死数小时，证明 Task 工具确实**没有**平台层面的超时。line 453 的"默认 10 分钟"无法验证出处——Task 工具的参数定义里没有 timeout 字段。

**影响**：
主 Agent 理解混乱。本会话主 Agent 最初假设"subagent 空返回可能是平台超时杀掉的"，导致空返回诊断方向错误（往"超时"方向想，而非"subagent 主动放弃"方向）。

**整改建议**：
- 删除 `dispatch-protocol.md:453` 的"task 工具设 generous timeout（默认 10 分钟）"描述，或改为"Task 工具无平台层超时（T019 实战验证），防卡死依赖 subagent 内部脚本硬超时 + 主 Agent 拆分策略"
- 在「空返回的恢复策略」节补充明确判断："Task 工具无平台超时，空返回不是被终止，是 subagent 主动放弃（做不动任务）"

### 问题 2：空返回诊断缺少决策树，不区分"快速放弃"和"跑了一阵放弃"

**现状**：
`dispatch-protocol.md:82-103` 的空返回恢复策略只说"分析失败原因、调整策略重派"，但没有给出具体的诊断维度。

**本会话暴露的问题**：
主 Agent 两次空返回后，没有区分以下情况就降级：
- subagent < 10 秒返回空 → 根本没开始做（环境/类型/看不懂任务）
- subagent > 2 分钟返回空 → 尝试了但做不动（任务粒度过大/上下文不够）
- subagent 跑了很久返回空 → 可能是中间某步卡住后放弃

三种情况的应对策略不同：第一种换 subagent 类型或查环境，第二种拆细任务，第三种需定位卡在哪步。

**整改建议**：
在「空返回的恢复策略」节补充诊断决策树：

```
空返回诊断：
  主 Agent 记录派发时间戳，返回后算耗时
  
  耗时 < 30 秒：
    → subagent 几乎没执行
    → 可能原因：subagent 类型不匹配 / 环境不通 / 任务描述看不懂
    → 应对：先跑最小验证（见问题 3），确认 subagent + 环境链路
    
  耗时 30 秒 ~ 3 分钟：
    → subagent 尝试了但做不动
    → 可能原因：任务粒度过大 / 上下文不够 / 缺参照
    → 应对：拆分任务 + 补输入导航 + 给参照文件
    
  耗时 > 3 分钟：
    → subagent 跑了一阵但中途放弃
    → 可能原因：某步卡住 / 反馈循环太长（试错耗尽上下文）
    → 应对：定位卡点（要求 subagent 中间产物落盘），针对性拆分
```

### 问题 3：缺少"最小验证先行"机制

**现状**：
`dispatch-protocol.md:368-386` 有"P2 最小验证"（方案可行性先验证），但**没有"派发前 subagent 链路最小验证"**——即派复杂任务前，先派一个 30-60 秒的最小任务验证 subagent + 环境链路是否通。

**本会话暴露的问题**：
主 Agent 直接派了一个大任务给 subagent，两次空返回。如果先派一个最小任务验证链路，30 秒就能知道 subagent + 环境链路是否通，不用浪费两次大任务派发。

**整改建议**：
在「长时操作 subagent 派发策略」节增加"最小验证先行"步骤：

```
### 派发前最小验证（新增）

涉及需要外部环境配合的复杂任务（浏览器自动化、需要运行中的服务、
需要特定运行时等），派发前先派一个最小验证任务：

最小验证任务特征：
- 职责单一：连接环境 + 执行一个最简操作 + 返回一个值
- 预期耗时：30-60 秒
- 返回值：结构化（如 { connected: true, elementCount: 8 }）

最小验证的两种结果：
  返回了 + 值正确 → subagent + 环境链路 OK，可派复杂任务
  空返回或值错误 → 先解决环境/subagent 类型问题，不派复杂任务

最小验证不计入 retries（它是诊断手段，不是阶段产出）
```

### 问题 4：缺少"客观信息落盘"机制

**现状**：
`dispatch-protocol.md:213-233` 的「输入导航原则」要求主 Agent 给"读哪个节、关注什么"的导航，但**没有要求把预探测的客观内容落盘成文件**。

主 Agent 派发前通过查证获得的客观事实（环境状态、文件位置、接口契约、命令输出、DOM 结构等），目前只能写进 prompt，导致：
- prompt 变得很长
- 违反铁律 2 的精神（"只传路径不传内容"——预探测结果也是内容）
- 不可复用（同阶段多次派发要重写）

**本会话暴露的问题**：
主 Agent 把环境状态、URL、选择器、参照文件全写进 prompt（约 50 行），subagent 读这么长的 prompt 占用上下文。

**整改建议**：
在「输入导航原则」节增加"客观信息落盘"子节：

```
### 客观信息落盘（新增）

主 Agent 在派发前通过查证获得的客观信息，当信息量较大（超过约 10 行）
或需在同阶段多次派发中复用时，应落盘成文件：

文件名：docs/tasks/{Txxx}/P{N}-dispatch-context.md
内容：主 Agent 已查证确认的客观事实，包括但不限于：
  - 环境状态（服务是否运行、版本、数据是否就绪）
  - 关键路径/标识（正确的 URL、API 端点、文件 ID 等）
  - 接口/结构清单（DOM 选择器、API 字段、配置项等，从源码提取）
  - 参照文件路径（现有同类测试、可套用的模式）
  - 运行方式（如何执行脚本、必要的环境变量）

派发时 prompt 只给这个文件路径，不写具体内容。
这个文件由主 Agent 在派发前查证后写（主 Agent 的合法职责，类似 P0-brief）。

判断标准：
- 信息量 > 10 行 → 落盘
- 同阶段多次派发复用 → 落盘
- 信息量小且单次使用 → 可写进 prompt
```

### 问题 5：缺少"写脚本与跑脚本分离"的派发模式

**现状**：
`dispatch-protocol.md:311-366` 的长时操作拆分策略是按**职责**拆（加载页面 / 检查某项 / 截图），但没区分"写脚本"和"跑脚本"两个阶段。

**本会话暴露的问题**：
需要写脚本的验证任务（浏览器自动化、测试脚本、构建脚本等），反馈循环长（写脚本→跑→失败→改→再跑），如果让一个 subagent 既写又跑又调试，几轮试错后上下文窗口就满了，导致空返回。

主 Agent 因此降级亲自写脚本——理由是"跑脚本是 gate 验证"。但写脚本不是 gate 验证，是有创造性的工程工作。

**整改建议**：
在「长时操作 subagent 派发策略」节增加"写跑分离"模式：

```
### 写脚本与跑脚本分离（新增）

反馈循环长的脚本验证任务，不要让一个 subagent 既写又跑又调试。

拆法：
  阶段 A：subagent 写脚本（产出脚本文件，不跑）
    - 输入：dispatch-context.md + BDD/验收条件 + 参照文件
    - 产出：脚本文件
    - 用专项 subagent（前端/backend/mcp 对应类型）

  阶段 B：主 Agent 跑脚本（gate 验证，A1 原则）
    - 跑 subagent A 写的脚本
    - 最小修复（调整参数/超时/选择器）属于"跑命令"的一部分
    - 重大逻辑错误回 subagent A 修

  阶段 C：subagent 读脚本输出写报告
    - 输入：脚本输出的结构化结果 + 验收条件原文
    - 只做格式化，不做验证

关键区分：
  - "跑命令"（gate）= 主 Agent 亲自执行，看 exit code + stdout
  - "写脚本"（产出）= subagent 执行，产出脚本文件
  - 主 Agent 跑 subagent 写的脚本 = "跑命令"不是"写产出"
  - 主 Agent 对脚本做最小修复（改参数/超时）= 仍属于"跑命令"
  - 主 Agent 重写脚本逻辑 = 降级，违规
```

### 问题 6：P7 grep [BLOCKER] gate 有 false positive

**现状**：
`state-machine.md:120`：`P7 --[! grep -qF '[BLOCKER]' P7-consistency.md]--> P8`

**本会话暴露的问题**：
P7-consistency.md 的结论写"一致性检查通过，无 [BLOCKER]"，grep `[BLOCKER]` 匹配到了这句话里的文本，gate 误判为失败。主 Agent 不得不改文本规避。

这是 grep 字符串匹配的固有缺陷——文档里提到 BLOCKER 这个词（哪怕是"无 BLOCKER"）就会被匹配。

**整改建议**：
在 P7 gate 规则里补充 false positive 规避：

```
P7 gate 判定方式修正：
  当前：grep -qF '[BLOCKER]' P7-consistency.md
  
  false positive 规避：
  要求 P7-consistency.md 的偏差清单用行首标记格式：
    [BLOCKER] 偏差描述   ← 真正的 BLOCKER
    [NOTE] 偏差描述      ← 非阻塞
  结论节不出现 [BLOCKER] 字符串，用"无阻塞项"等替代表述。
  
  或改为更精确的匹配（只匹配行首标记，不匹配正文文本）：
  grep -E '^\s*-?\s*\[BLOCKER\]' P7-consistency.md
```

### 问题 7：截图作为"UI 条件有截图佐证"门槛缺乏质量标准

**现状**：
`dispatch-protocol.md:401`：P6 gate "UI 条件须截图 + vision-analyst YAML 引用"

但协议没有定义截图的**质量标准**——什么时候截、截什么范围、相同截图怎么判定。

**本会话暴露的问题**：
- 多张纯查询类 BDD 截图 md5 完全相同（页面无视觉变化，截图当然相同）
- 截图时机错（查询时截不是操作后截）
- 主 Agent 把"截图相同合理"当借口通过 gate

**整改建议**：
在 P6 gate 规则里补充截图质量标准：

```
P6 截图质量标准（新增）：

1. 截图时机：操作后立即截，不是查询时截
   - 改变页面状态的 BDD（toggle/fullscreen/theme 切换）：操作后截
   - 不改变视觉状态的纯查询 BDD：可不截图，断言值是唯一证据

2. 防造假：
   - 主 Agent 对所有截图去重（md5 或文件大小）
   - 操作类 BDD 的截图必须互不相同（操作改变了页面状态）
   - 查询类 BDD 的截图相同可接受（但不算"截图佐证"，依赖断言值）

3. 门槛判定修正：
   "UI 条件有截图佐证"改为：
   - 操作类 UI BDD：截图互不相同 + 断言值 PASS = 强 PASS
   - 查询类 UI BDD：断言值 PASS = PASS（截图可选）
   - 操作类 UI BDD 截图相同 = 截图失效，依赖断言值 = 弱 PASS（标注）
```

### 问题 8：主 Agent 启动检查无强制性

**现状**：
系统提示要求"启动后、执行任何任务前，依次读完 7 个协议文件"，但这是**描述性规则**，没有机制保证执行。

**本会话暴露的问题**：
主 Agent 这次会话一个协议文件都没读，直接开始任务。用系统提示里的摘要片段代替协议原文。

**整改建议**：
这个问题的根因在平台层（系统提示是描述性的，不是强制性的），agate 协议本身无法强制主 Agent 读文件。但可以增加一个**软性检查点**：

```
在单步函数步骤 1（读状态）里增加：

  1.0 启动检查（仅会话首次执行时）：
      确认已读 7 个协议文件。
      自查方式：主 Agent 回顾上下文，是否能回忆起
      dispatch-protocol.md 的「降级硬边界」和「空返回恢复策略」原文。
      不能 → 暂停当前操作，先读协议。
      
  这是软性自查，不是机器可判定 gate。
  但通过把检查点写进单步函数，增加主 Agent 注意到这个要求的概率。
```

## 5. 自评（专家评审视角）

本节是复盘者以第三方评审视角对本复盘的质量自评。

### 5.1 本复盘做得对的

- **定位准确**：聚焦机制层面整改建议，执行层面违规只记录不展开
- **有证据链**：每个问题都引用了协议原文行号 + 实战证据（T019 复盘）
- **有优先级**：8 个问题分了 P0-P3，不是一锅端
- **核对清单完整**：5 个机制逐条核对

### 5.2 本复盘的不足

1. **问题 3/4/5 的适用范围在初稿里过窄**：初稿只写了"Playwright/浏览器/E2E"场景，经评审指出后已泛化为"需要外部环境配合的复杂任务""反馈循环长的脚本验证任务"等通用描述。但泛化是否到位仍需后续实战验证——目前只有 T020 一个数据点。

2. **整改建议缺少"如何验证整改有效"**：8 个建议都只说"加什么规则"，没说"怎么知道这条规则加对了"。理想情况下每个建议应附验证条件（如：问题 2 的诊断树加进去后，下次空返回时主 Agent 能否按决策树选择正确策略）。这是复盘机制本身的缺口。

3. **问题 8（启动检查）的整改建议可能无效**：把检查点写进单步函数——但如果主 Agent 没读协议，它也不会读单步函数里的检查点。这是循环依赖。更诚实的态度是承认这是平台层限制，agate 无法解决，而非给出一个看似有解但实际无效的软性方案。

4. **缺少跨复盘对比**：T016 和 T019 的 postmortem 都提到了"降级违规"和"空返回处理不当"，本次 T020 再次出现。复盘没有分析"为什么同样的违规反复出现"——是规则不够明确，还是规则明确但主 Agent 理解不到位，还是主 Agent 理解到位但执行时忘了？如果是第三种，所有协议层面的整改都不会有效，需要的是平台层的强制机制。

5. **问题 6（P7 grep false positive）的整改建议有两种方案但没选**：给了"改文本约定"和"改 grep 命令"两个方案但没说用哪个。应明确推荐改 grep 命令（`grep -E '^\s*-?\s*\[BLOCKER\]'`），因为改文本约定是约束 subagent 行为（不可靠），改 grep 命令是主 Agent gate 逻辑（可控）。

### 5.3 待后续验证的假设

## 6. 总结

本次复盘发现 8 个问题，其中 5 个是 agate 机制层面的缺口（可改协议），3 个是执行层面的违规（协议已有但未遵守）。

机制层面的核心矛盾是：**agate 对"subagent 空返回后怎么办"的指导不够具体**——只说"调整策略重派"，没说怎么诊断为什么空返回、怎么判断调整是否有效。这导致主 Agent 在空返回时缺乏明确路径，倾向于自己干（降级）。

但更深的问题是：T016 和 T019 的 postmortem 已经指出了"降级违规"和"空返回处理不当"，T020 仍然出现。如果同类违规反复出现，可能不是协议不够细，而是**描述性规则对 LLM 主 Agent 无强制力**——协议写了"必须走 PAUSED"，但没有任何机制阻止主 Agent 不走。这是平台层问题，agate 协议层面无法解决，只能靠每次复盘提醒。本复盘对此诚实承认（问题 8 的自评），不给出虚假的解决方案。

## 7. 第二轮评审：用户反馈 + 独立专家评审

> 触发：(1) 用户对问题 2 诊断树的质疑 (2) 独立专家评审 `agate-postmortem-T020-review-2026-06-25.md` 的核心判断

### 7.1 独立专家评审的核心贡献

独立评审区分了两种规则，修正了原复盘 §6 的误判：

| 类型 | 例子 | "加规则解决不了合规问题"对它成立吗 |
|------|------|------|
| 合规规则 | "必须走 PAUSED""不得降级" | ✅ 成立——写了不保证遵守 |
| 能力规则 | "空返回时怎么诊断""预探测结果怎么落盘""脚本怎么拆写跑" | ❌ 不成立——这是给主 Agent 操作工具，不是禁止规则 |

原复盘 §6 把**所有**整改建议归到"描述性规则无强制力"下。这个推论是错的——问题 2/4/5 不是"要求主 Agent 做什么"，是"给了主 Agent 现在没有的做法选项"。主 Agent 没有这些工具时只有降级一条路，有了才可能不降。

**但补一个限制**（评审未提及）：能力规则的有效执行可能受平台能力制约。问题 2 的诊断树是能力规则，但它的有效执行依赖"能观测 subagent 活动"这个平台能力。当前平台不暴露活动信号，诊断树只能用间接手段凑合。能力规则 + 平台能力 = 可执行；能力规则 + 平台能力缺失 = 部分可执行，需诚实标注。

### 7.2 用户对问题 2 诊断树的质疑（核心修正）

**用户反馈**：

> 关于 subagent < 30 秒空返回这种场景，我还没见到过呢。
> 我认为，最主要的应该是要能识别到 subagent 是否真的在干活，如果没在干活没在推理，可以有个超时时间强制让 subagent 从正常状态退出；但是如果人家在干活，就可以再等等。

**这个质疑是对的，它击中了诊断树的根本缺陷。**

#### 缺陷分析

原诊断树用"耗时"作主要维度（<30s / 30s-3min / >3min 三档）。但耗时**不能区分两种完全不同的情况**：

| 情况 | 耗时 | 应对策略 |
|------|------|---------|
| subagent 卡住没动（没启动/崩溃/死循环） | 可能很短 | 该强制终止 |
| subagent 在正常推理但任务重 | 可能同样短或长 | 该等 |

同样耗时，应对策略相反。耗时是弱相关变量——和 T016 "拆分判据用输出异构性不用行数"同理，**行数/耗时都是表面维度，不是因果维度**。

因果维度是：**subagent 是否有活动迹象**（在推理、在调工具、在输出内容）。这才是区分"该杀"和"该等"的正确判据。

#### 更根本的问题："< 30 秒空返回"是虚构场景

用户指出"这种场景我还没见到过"。本复盘在 §5.3 假设 1 已标注"未验证"，但仍然把虚构的阈值写进了整改建议。基于不存在的场景定阈值，比没有阈值更危险——主 Agent 可能机械套用"耗时 35 秒 = 任务粒度过大"这个错误结论。

#### 正确的诊断方向

```
空返回时：
  1. 判断 subagent 是否有活动迹象
     - 有活动（在调工具/在输出/写了中间文件）→ subagent 在干活
       → 可能任务太重还没做完，或中间某步卡住
       → 应对：拆细 + 给参照 + 检查中间产物定位卡点
     - 无活动（没调工具/没输出/没写文件）→ subagent 没在干活
       → 可能崩溃/卡死/没启动/看不懂任务
       → 应对：查环境 + 换 subagent 类型 + 最小验证链路

  2. 超时策略（如果平台支持活动检测）：
     - 无活动 + 超过无活动超时阈值 → 强制退出
     - 有活动 + 超过活动超时阈值 → 再等，或拆细让它能完成
```

#### 能力边界（诚实声明）

"判断 subagent 是否有活动迹象"需要**平台层**支持——暴露 subagent 的中间过程（工具调用日志、输出流、活动信号）。当前 Task 工具返回的是最终结果，中间过程主 Agent 不可见。

**当前能力**：
- Task 工具调用后，主 Agent 阻塞等待结果返回
- 等待期间，主 Agent **看不到** subagent 的中间过程（推理、工具调用、输出）
- 唯一可见信号：subagent 主动写文件（如心跳落盘）
- 但空返回的 subagent 大概率也不会写心跳（它就是因为做不动才空返回）

**结论**：在当前平台能力下，"识别 subagent 是否在干活"**主 Agent 做不到**。这是平台层能力缺口，不是 agate 协议能解决的。

**间接缓解（agate 协议层，有限但诚实）**：
1. 派发时要求 subagent 落盘进度（subagent 配合时有价值，不配合时无损失）
2. 主 Agent 记录派发耗时作为**参考**，不作主要判据
3. 空返回后检查中间产物文件，判断 subagent 是否动过

**根本解决（需平台支持，超出 agate 协议范围）**：
1. Task 工具暴露 subagent 活动信号（如工具调用日志、输出流、心跳）
2. Task 工具支持智能超时：基于活动状态而非固定时长
   - 有活动迹象 + 超时 → 不杀，提示主 Agent "subagent 仍在工作，是否继续等"
   - 无活动迹象 + 超时 → 强制退出，返回"subagent 无活动迹象超时"
3. Task 工具支持可配置超时：主 Agent 可按任务复杂度设超时

### 7.6 诚实声明

第一轮评审建议的"耗时三档诊断树"看似可操作，实际是基于虚构场景的编造。用户指出"<30秒空返回没见过"后，复盘者才意识到自己在用想象代替数据。

正确的态度是：**在无法检测 subagent 活动的情况下，不要假装能诊断**。记录耗时作为参考是合理的，但把耗时当作诊断判据写进协议是误导——主 Agent 会机械套用阈值，做出错误判断。

当前最诚实的做法：
1. 记录"活动检测"为已知平台缺口
2. 协议层面只要求"记录耗时 + 查中间产物"作为参考
3. 不给虚假的诊断树
4. 空返回时走 retry→PAUSED，不降级——这不依赖诊断，只依赖规则遵守

同时，专家评审的"能力规则 vs 合规规则"区分是对的——问题 4（客观信息落盘）、问题 5（写跑分离）确实是给了主 Agent 新的操作工具，不是禁止规则。这两个建议即使平台能力不完善也有价值，应落地。
