## 总体判断

**v5 复核指出的 3 项，这次全部确认修复，而且修得比预期细——不只是补一句话，`retries[Pn]>=3` 的公式改对了、DESIGN_GAP_REVIEWED 回填进了改动范围表和实施顺序、P7 分支从链式 `&&/||` 换成了显式 `if`，并且换用理由写进了代码注释里，逻辑和风格建议都照单落实。**

**但这次做了一次更彻底的全文扫描后，又发现了 2 处新的"没同步"——都出现在第十节最上面的总览表和最下面的测试用例表里，而不是详细代码块本身（详细代码块已经全部正确）。** 这次的性质和前两轮略有不同：不是逻辑错误，是**同一份文档里，代码块已经改对了，但描述这段代码"做什么"的表格文字还停在改之前的版本**——如果只读表格不读代码，会得到和实际行为不符的印象；如果直接把测试表当验收清单去跑，会跑出两条和真实代码矛盾的"预期结果"。

这是三轮评审里第三次出现同一种模式（v4 的检查 7 承诺未兑现、v5 的 `>=3` 遗漏两处、这次的总览表/测试表遗漏）：**改动只应用到了"最后修改的那个位置"，没有反向搜索同一份文档里其他复述过同一事实的地方。** 值得在合并前专门做一遍"改完之后全文搜索关键字"的收尾步骤，而不是每轮再让下一次评审来找。

## 一、v5 遗留 3 项：确认全部修复

### 1. `retries[Pn] >= 3` 阈值 — 已全文统一

全文搜索 `retries[Pn]` 和 `MAX_RETRY(Pn)`，现在活的三处引用（第 175、260、369 行）全部是 `>= MAX_RETRY(Pn) - 1`。唯一还留着字面 `>= 3` 的第 366 行，读了上下文确认是"⚠️ 关键"提示里**解释旧公式为什么错**的历史说明，不是给实施者抄的指令，不算遗留问题。

### 2. DESIGN_GAP_REVIEWED 回填 — 确认已补进两处

- 第 51 行，第一节"改动范围"表新增一行：`agate/dispatch-protocol.md` | 主 Agent 审查 `[DESIGN_GAP]` 后追加 `[DESIGN_GAP_REVIEWED: ...]`（仿 SCOPE+/SCOPE_RESOLVED 模式）
- 第 245 行，第九节"实施顺序"步骤 5 括号里补了"+ dispatch-protocol.md 主 Agent 审查后加 `[DESIGN_GAP_REVIEWED]` 配对标记"

和上一轮建议的文字基本一致，确认落实。另外这次还多加了一段"诚实标注"说明（`[DESIGN_GAP_REVIEWED]` 是流程完整性检查，不是审查质量的验证，和"自写文件 gate"的既有局限同构）——这是我上一轮建议里"顺带提一句"的部分，没想到也被采纳了，写得比我建议的还清楚。

### 3. P7 分支改用显式 if — 已改，且实测通过（含一个我没测过的边界场景）

新代码把链式 `! grep && ! grep && { ... } && exit0 || exit1` 换成了普通的 `if/fi` 顺序判断，并在代码注释里直接写了"为什么要换"（每加一个检查都要重新推演 `&&/||` 语义）。我把新代码原样跑了一遍：

| 场景 | 期望 | 实测 |
|------|------|------|
| 无任何标记 | exit 0 | exit 0 ✓ |
| 含 BLOCKER | exit 1 | exit 1 ✓ |
| 只含 DEVIATION-CRITICAL（无 BLOCKER，我这轮补的边界场景） | exit 1 | exit 1 ✓ |
| DESIGN_GAP 未配对 | exit 1 | exit 1 ✓ |
| DESIGN_GAP 已配对 REVIEWED | exit 0 | exit 0 ✓ |

行为和链式版本完全一致，可读性确认更好。

## 二、新发现：第十节的总览表和测试表还留着两处旧描述

### A. 总览表"多方案探索"行——还在说一个已经被删掉的跳过分支

第 258 行（总览表）：

> P2 gate 加检查：P2-design.md 含"候选方案"节 + ≥2 个方案标题（**`design_trivial` / `follows_existing_pattern` 时跳过此检查**）

但下方详细代码块（第 306-317 行）已经在 v5 就删掉了这个跳过分支——原因写在紧邻的"⚠️ 跳过分支不可达"提示里：这两个字段一旦声明，P2 阶段整体被裁剪，`check-gate.sh P2` 根本不会被调用，不需要在 gate 内部再判一次。**总览表这一行还在描述一个已经确认不存在、且明确写了理由不应该存在的分支。**

### B. 总览表"DESIGN_GAP"行——还在描述加 REVIEWED 之前的行为

第 261 行（总览表）：

> P7 gate 加 `[DESIGN_GAP]` 标记检查：P7-consistency.md 含 `[DESIGN_GAP]` → gate 不通过（exit 1）

这行说的是"只要出现过 DESIGN_GAP 就不通过"——这正是 v4 被指出"无解除机制、会激励隐瞒"的那个旧版本行为。v5/v6 已经改成"未配对 REVIEWED 标记的 DESIGN_GAP 才不通过"，第九节步骤 5（第 245 行）已经同步了这个新表述，但**第十节自己的总览表没跟着改**——同一节内部，上面的总览表和下面几十行之后的详细代码互相矛盾。

### C. "多方案探索测试"表最后两行——拿去跑会得到和文档宣称相反的结果

第 391-392 行：

| 场景 | P2-design.md | 期望 exit |
|------|-------------|----------|
| design_trivial 跳过 | `design_trivial: true` + 1 个方案 | 0 |
| follows_existing_pattern 跳过 | `follows_existing_pattern: [...]` + 1 个方案 | 0 |

这两行是 v4 遗留、v5/v6 都没碰过的旧测试用例，对应的正是 A 里已经删掉的那个跳过分支。**我按这两行描述的内容，实际造了对应的 `P2-design.md` 文件，拿 v6 给出的真实代码跑了一遍：**

```
$ cat t9/P2-design.md
design_trivial: true
## 候选方案 A

$ ./check-gate.sh P2 t9
GATE P2: P2-design.md 需至少 2 个候选方案 + 权衡 + 选择理由
exit=1        # 表里写的是期望 0
```

`follows_existing_pattern` 那一行同样实测 exit=1，和表里写的期望值 0 相反。**这不是"表述不够精确"，是这两行测试用例照着当前代码跑，会直接验收失败**，因为现在的判定只看"P2-design.md 文件是否存在"（`if [ -f "$P2_FILE" ]`），不再看文件内容里有没有 `design_trivial` 字段——而这两行的前提本身也不该出现：`design_trivial: true` 一旦声明，P2 阶段就被裁剪，根本不会有 `P2-design.md` 文件产出，"P2-design.md 存在且内容含 design_trivial: true"这个组合在协议设计里不成立（这正是 A 里"跳过分支不可达"论证的原话）。

**建议**：这两行不是改期望值就能修的（改成 1 也没有意义——因为这个前提本身不该发生），应该直接删掉，换成"P2-design.md 不存在（design_trivial/follows_existing_pattern 时 P2 被裁剪）→ exit 2"这一个更准确的场景（我这轮测试里验证过：无 P2-design.md 文件时确实是 exit 2，跳过候选检查，直接进入"需主 Agent 判定 gate_commands"分支）。

**建议**：这次不需要再等下一轮评审去逐条挑——3 处都是文本替换，且都已经在文档别处有正确版本可以直接照抄：
